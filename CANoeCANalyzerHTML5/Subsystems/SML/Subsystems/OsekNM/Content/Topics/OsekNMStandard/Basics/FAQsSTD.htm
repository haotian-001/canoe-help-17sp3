Frequently Asked Questions

 

Problems using two or more busses

Symptom
The network management does not work for more than one bus.

Possible cause
CANoe cannot find the correct correlation between nodes, busses and databases.

Suggestions
Depending on the Simulation Setup, two approaches are recommended.

  • The busses do not possess common nodes, i.e. no node name appears in databases assigned to different busses: In this case you may use the node attribute NmCAN to assign the nodes to CAN channels. Note that you have to use the compatible mode of newer CANoe versions (same behavior than versions before 3.2 SP3) too.
  • There are nodes that should participate in the NM of more than one bus: Here, you have to use a newer CANoe version (3.2 SP3 or later) in standard mode. Only then it is possible to correlate each node instance with the bus it is attached to, and also with the correct database! (Cf. Example — Hints for Gateway Programming).

Some nodes/ECUs are ignored in the NM communication

Symptom
Some ECUs send their NM messages on the bus, but the simulated nodes do not react on them, i.e. skipping the ECUs.

Possible cause
The ECU number is too high, i.e. the value of the network attribute NmMessageCount is too low. For example, if the base address of the NM is 0x400 and this attribute is not defined, then the simulated nodes will use the default of 32. A node with ID 32 sends a NM message with ID 0x420 that is not recognized, hence the node is ignored.

Suggestions
add the network attribute NmMessageCount to the database and set it to the appropriate value, e.g. 64.

CANoe reports callback functions not found

Symptom
After starting the measurement, CANoe reports Addon OSEK NM for node X: callback function … not found, i.e. the NM does not find certain callback functions in the CAPL program assigned to the node.

Possible cause
The NM INI file may be missing or not found. Or the functions are not defined in the CAPL program.

Suggestions
Check if the NM reports the correct INI file location, otherwise copy the INI file to the location of the DLL/SO file or the configuration file. If the file is found, then this warning may be ignored if it is certain that the events reported by the NM are not necessary for the CAPL application.

Node parameters are not recognized

Symptom
Although a value for node ID or base address is given in database attributes, the values are not used.

Possible cause
The attribute names are not written correctly – the names are casesensitive, e.g. NMStationAddress is not recognized as NmStationAddress. Also note that some older attribute name variants are no longer supported.

Suggestions
Confirm that the NM attributes are spelled exactly as documented in CANdb Attributes.
