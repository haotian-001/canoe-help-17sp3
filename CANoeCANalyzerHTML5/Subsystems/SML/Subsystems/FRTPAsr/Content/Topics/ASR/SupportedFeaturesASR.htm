Supported Features

 

Introduction

The first version of the DLL supports a subset of the features defined in the standard. Please contact Vector Informatik GmbH if a feature you need is not yet supported.

Feature

Short Description

Protocol version

This DLL supports three different protocol variants according to a given AUTOSAR version. Cf. FrTP_SetProtocolVersion

Channel Options

The DLL supports transport connections over channels occupying one or more FlexRay slot per direction.

Connection Options

It is possible to transfer data using the following types of connections:

  • Unicast: One-to-one communication
  • Acknowledged unicast: One-to-one communication where the receiver acknowledges the reception
  • Multicast: One-to-many communication

All unicast connections can principally be used in full-duplex mode (transmission and reception simultaneously).

Slot Usage

The DLL allows using static and dynamic slots Frames or PDUsfor the reception and transmission. It is possible to reserve multiple slots per direction, i.e. one FlexRay cycle may carry several TP C-PDUs in different slots.

Several connections can share their Tx and Rx slots.

PDU Definitions

It is possible to define PDUs in the FIBEX database. This DLL will detect those PDUs of type TPL (5) and DIAGNOSTICS (6, 7, 8), and register for their reception or transmission. Please refer to the Creation of Connections on PDUs or Frames function group.

ISO TP Compatibility

All ISO TP compatibility modes are supported:

  • FRTP_ISO: Only ISO TP frame types are used.
  • FRTP_ISO6: Only ISO TP frame types with a maximum of 6 byte payload are used.

Extended TP Frame Types

The extended TP frame types are supported:

  • FRTP_L16M: All defined TP frame types except FF-E4 can be used.
  • FRTP_L4G: All defined TP frame types can be used.

Address Types

One (FRTP_OB) and two byte (FRTP_TB) addresses are supported.

Maximum PDU Size

It is possible to specify any PDU length up to a FlexRay maximum of 254 byte. The padding byte can be specified.

Dynamic Bandwidth Control

The bandwidth control algorithm is supported. CANoe simulation nodes can request a specific behavior from a sender, and follow the behavior requested by a receiver.

The receiver may indicate the number of slots the sender can use in one FlexRay cycle, or specify the number of separation cycles between the sending of consecutive frames.

TP Parameters

The ISO TP parameters separation time, block size and timer values can be set.

Configuration via CAPL

FlexRay TP connections and parameters can be configured by calling CAPL functions during initialization of the measurement. Some settings like the length of PDUs may be changed at any time, with effect on future data transfers only in some cases.

Small Buffer Handling

Instead of providing all data immediately, it is possible to start sending without having the complete data. During reception, it is possible to access the already received data at any time.

This helps to implement efficient gateway code that does not need to receive completely the data request before forwarding the request to another bus.

Fault Injection

Some miss-behaviors of the protocol engine can be specified in order to test the peer.

Connection Priorities

A connection can be given a Tx priority. Thus, their transmitted C-PDUs may be prioritized over other connections that share the same Tx slots.

Exclusive Slot Usage

The Tx slots of the connection can exclusively be used for one connection (although they are shared with other connections).

PDU Pools

PDU pools are supported implicitly. There is no need to define explicitly a PDU pool for the use of this DLL, because the Tx and Rx PDUs (of an appropriate pool) must be assigned explicitly during connection instantiation (cf. function groups Creation of Connections on PDUs or Frames and Slot Assignment). Refer also the CANoe TP sample configuration coming along with this add-on and the Example.

Maximum Number of Tx Priority PDUs per Connection

The number of prioritized Tx PDUs per connection can be defined. (Only applicable to protocol version 3.2.2; cf. function FrTP_SetProtocolVersion.
