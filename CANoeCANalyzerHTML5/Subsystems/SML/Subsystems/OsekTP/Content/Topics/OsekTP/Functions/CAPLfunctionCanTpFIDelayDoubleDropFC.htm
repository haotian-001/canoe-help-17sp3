CanTpFI_Delay/_Double/_DropFC

 

Note

The functions in this section influence the sending of Flow Control Frames when receiving data. Only the next data reception is affected!

The fault injection status is determined after the call to the CAPL callback CanTp_FirstFrameInd has returned. Therefore it is possible to setup fault injection in this function.

Function Syntax

long CanTpFI_DelayFC(long handle, long FCno, dword delay_ms)

long CanTpFI_DoubleFC(long handle, long FCno)

long CanTpFI_DropFC(long handle, dword FCno)

Description

These functions allow manipulating the sending of Flow Control Frames, where 0 indicates the first Flow Control Frame sent in response to the First Frame.

Parameters

—

Return Values

—

Example

The first Flow Control Frame is delayed for 100 ms, the second one is doubled, and the third one is dropped, causing an transmission error in the sender.

CanTpFI_Enable( handle);
CanTpFI_DelayFC( handle, 0, 100);
CanTpFI_DoubleFC( handle, 1);
CanTpFI_DropFC( handle, 2);

Trace (normal addressing, protocol information in brackets).

1.000 [10 40] 01 02 03 04 05 06   // First Frame announcing
                                  // 64 byte (received)
1.100 [30 02 14] 00 00 00 00 00   // Flow Control 0 (delayed)
                                  // block size 2, STmin 20 ms
1.120 [21] 07 08 09 0a 0b 0c 0d   // Consecutive Frame (received)
1.140 [22] 0e 0f 10 11 12 13 14   // Consecutive Frame (received)
1.149 [30 02 14] 00 00 00 00 00   // Flow Control 1
1.150 [30 02 14] 00 00 00 00 00   // Flow Control 1 (doubled)
1.170 [23] 15 16 17 18 19 1a 1b   // Consecutive Frame (received)
1.190 [24] 1c 1d 1e 1f 20 21 22   // Consecutive Frame (received)
// Transmission aborts here since the third Flow Control frame
// is dropped. Sender and Receiver run into timeouts.
