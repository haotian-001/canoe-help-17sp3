Callback and Small Buffer

Callback [▲ back]

The CAPL program has to define these optional functions (all marked with the prefix CanTp_) to receive indications about TP events like the reception of data or a protocol error. It is highly recommended to implement at least the error indication function.

Functions

Short Description

CanTp_ErrorInd

An error occurred for the connection indicated.

CanTp_FirstFrameInd

These function allows the CAPL program to determine the start of data transmissions.

CanTp_PreSend

This callback function is called every time a CAN message is scheduled to be sent.

CanTp_ReceptionInd

This function is called when data has been received on the connection with the given handle.

CanTp_SendCon

These function allows the CAPL program to determine the end of data transmissions.

CanTp_TxTimeoutInd

Called if a CAN message cannot be sent within the timeout Ar or As.

Small Buffer Handling [▲ back]

Introduction: Small Buffer Handling

Functions

Short Description

CanTp_ProvideTxDataInd

This callback is called when the DLL is not able to fill the data part of the next Consecutive Frame it has to send.

CanTp_RxBufferFullInd

This CAPL callback function is called by the DLL when the receiver buffer is full.

CanTpProvideTxData

Second Syntax: The data provided with this function will be sent as soon as the respective Consecutive Frame has to be filled.

CanTpSetRxBufferSize

Second Syntax: This function limits the receive buffer for the connection to the given number of bytes.

CanTpSmallBufferSend

This function is provided as a replacement for CanTpSendData in case small buffer handling is to be used.

CAPL Callback Information [▲ back]

The first three functions in this section may only be called from the following CAPL callback functions to retrieve additional information:

  • CanTp_ErrorInd
  • CanTp_FirstFrameInd
  • CanTp_PreSend
  • CanTp_ReceptionInd
  • CanTp_SendCon
  • CanTp_TxTimeoutInd

If called at any other time, an error code is returned.

Functions

Short Description

CanTpGetRecentAddressExtension

CanTpGetSenderAddress

When data is received, this functions returns the actual source address used.

CanTpProvideTxData

First Syntax: After a transmission has been started with incomplete data, it is necessary to provide this data once it is available.

CanTpSetRxBufferSize

First Syntax: During callback indications for the reception of data, it is possible to limit the size of the receiver buffer to cause, stop or modify calls to the callback CanTp_RxBufferFullInd.

Configuration of Addressing

Every TP connection is configured to exactly one addressing mode that defines the usage and interpretation of the CAN IDs and the message’s data bytes.

Her you get further information about the available Addressing Modes.

The following parameters can be configured, though not all of them are used in a concrete addressing mode.

Parameter

Description

Modes

txId, rxId

CAN message IDs (11 or 29 bit) of the messages sent (txId) or received (rxId).

normal, extended (free), mixed (11 bit)

ecuAddress, targetAddress

Byte values indicating a number assigned to the local (ecu) or remote (target) TP node

extended, normal fixed, mixed (29 bit)

addressExtension

Byte value used to distinguish between otherwise identical connections

mixed (11 and 29 bit)

txPriority

3 bit J1939 CAN ID field indicating the CAN message priority

normal fixed, mixed (29 bit)

Faults Introduced on Sender Side [▲ back]

The following functions affect the sender side of a data transfer, i.e. they have an effect only on the next data sending on the specified connection. The possibility of a fault injection is determined during the CanTpSendData function call, i.e. it is not possible to change the transmission behavior of connections already started, except from the CanTp_PreSend callback function.

Functions

Short Description

CanTpFI_AllCTS

Treats all Flow Control frames as Clear to send.

CanTpFI_DelayCF

Delay Consecutive Frames sent.

CanTpFI_DoubleCF

Double Consecutive Frames sent.

CanTpFI_DoubleTx

Double any frame.

CanTpFI_DropCF

Drop Consecutive Frames sent.

CanTpFI_SendXByte

Aborts transmission after the specified number of bytes has been sent.

General

These functions are implemented in the modeling library (DLL) and can be called in CAPL programs.

All functions returning long values indicate the success by returning 0 (no error), or a positive value indicating the requested information, e.g. the connection handle.

A negative value indicates an error; please refer to the page Error Codes for details. Exceptions are indicated at the functions.
