Supported Features

 

Feature

Short Description

Background

Since CANoe version 3.0 the node layer OSEK_TP.DLL is provided. Originally implementing the OSEK specification, it has been adapted to the changes defined by the ISO 15765-2 standards. Many CANoe configurations use this DLL to transfer data longer than 8 bytes between nodes.

Backward Compatibility

The OSEK_TP.DLL now provides two separate interfaces, the old OSEK Transport Layer interface, where all functions are identified by an OSEKTL_ prefix, and the new CanTp interface, where all function names start with CanTp.

The DLL is released for usage with CANoe 6.0 or later.

This means that all older simulations will continue to work with this version of the DLL. For this functions ask the Vector Support to get the OsekTPobsolete.chm. Newly developed simulations, which are to be used in CANoe versions 6.0 and up, should use the new interface (see this help pages), and it is even possible (though not recommended) to use both interfaces in parallel.

Multi Connection Support

The CanTp interface allows CAPL programs to operate on multiple ISO TP connections in parallel. Each connection is identified by a handle that has to be given to the interface functions. The handle is also provided when CANoe calls a CAPL callback function, allowing the CAPL program to identify the event’s connection immediately.

Transfer Acceleration

For applications like flashing an ECU's memory, the TP DLL supports data transfer at the highest possible speed and hence decreases flashing times significantly. Bursts of up to 64 Consecutive Frames can be sent if STmin = 0.

Small Buffer Handling

For applications like simulated gateways, the speed of data routing between networks is increased when using this feature. TP frame data from the source network can be relayed to the target network even before the TP frame has been completely received.

API Redesign

The CanTp CAPL API has been redesigned completely to make usage easier than before. The OSEKTL function name length restriction (due to compatibility with old programs) has been relieved, making function names much more expressive. New functions have been added that handle interrelated configuration tasks in one step, reducing the possibility of misconfiguration.

Configuration Via Database

CANdb database may configure the TP settings of simulation nodes with the standard attributes. The database connection can be accessed in a simple way and will be used by the CanTp interface only from that point on.

Fault Injection

It is possible to violate the TP specification in a defined way to observe and evaluate the reaction of the connection peer. This functionality requires detailed understanding of the protocol.

Supported Addressing Modes
  • Normal
    Connections use arbitrary CAN IDs for Tx and Rx messages.
  • Extended (based)
    One CAN ID is used as base address, and each ECU gets an address that is added to the ID (to identify the sender), and put into a target byte (to identify the receiver).
  • Extended (free)
    Connections use arbitrary CAN IDs for Tx and Rx messages, but ECU addresses are also used to identify the receiver.
  • NormalFixed
    J1939 messages are used with ECU addresses that are coded into the CAN IDs.
  • Mixed
    J1939 mode like normal fixed, but an additional address extension byte is used to distinguish otherwise identical receivers.
  • Mixed (11 bit)
    Like Normal mode, but an additional address extension byte is used to distinguish otherwise identical receivers.
