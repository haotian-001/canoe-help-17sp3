CanTpGet/SetTxTimeoutThreshold

 

Note

In order to handle very short Tx timeout values (i.e. Ar and As, both default to 1000 ms), the TP DLL uses another time called Tx timeout threshold.

The threshold value defaults to 125 ms, but it can be modified with the following functions.

Function Syntax

long CanTpGetTxTimeoutThreshold(long connHandle);

long CanTpSetTxTimeoutThreshold(long connHandle, dword TxTO_ms);

Description

Depending on the threshold value and the values for Ar or As, the DLL will behave differently:

  • If a Tx timeout occurs and Ar or As are smaller than the threshold value, the first timeout does not lead to the abort of the data transfer, but the timeout is restarted until the threshold time has passed completely.
  • If a Tx timeout occurs and the threshold value is 0, the reaction of the DLL depends on the result of a call to the CAPL callback CanTp_TxTimeoutInd (cf. 0):
    • If the callback function is not defined or returns 0, the data transfer is aborted immediately. This means that the Ar or As timeout is reported without any delay.
    • If the callback function marks the frame for dropping (i.e. changes its ID to -1) and returns 1, the DLL assumes that sending was successful, and continues the protocol. This might cause an error later in case the frame is not actually sent.
    • If the callback function does not drop the frame and returns 1, the DLL will send the frame again and restart the Ar or As timeout. This can cause the frame to be sent several times!
  • If a Tx timeout occurs, but Ar or As are larger than the threshold value, or it is not the first Tx timeout, the handling (1) is also performed.
