Attributes

 

vCDL allows to specify attributes for core language declarations, which provide extended configuration information and are by nature optional. Syntactically attributes are a comma separated list of key-value pairs enclosed in square brackets [...]. Valid attribute keys are predefined identifiers, optionally qualified by one or more namespaces, where the list of valid identifiers is individually defined for each language element. Most predefined attributes are not defined within any namespace. However, binding specific attributes, either for communication- or distributed objects, have the binding type as namespace. For some attributes, mostly for those with boolean type, the value is optional. Besides predefined attributes you can define your own attributes (see below).

Attribute values are expressions over primitive type operands (bool, string, integer, float), references to named elements, primitive type names, encoding formulas or homogeneous collections of these. Each attribute has a specific data type associated, which constrains the possible values. The expected data types are documented in the reference to the corresponding language element.

Attributes

Description

Boolean attribute value

Expression evaluating to true or false. The language supports a shortcut notation, where only the attribute key (without the equal sign) is given, implying the value true. The boolean type supports literals true and false.

Integer attribute value

Expression evaluating to an integer value. In the simplest case this is a decimal or hexadecimal integer literal.

Floating-point attribute value

Expression evaluating to a floating-point value. In the simplest case this is a floating-point or integer literal.

String attribute value

Expression evaluating to a string value, where the parser does not make any assumptions about the string content. Also enumerated attribute values can be specified as string literals, where the content is restricted to predefined strings. In particular, this holds for attributes in vCDL language versions prio to v1.2.

Enumeration attribute value

Since vCDL v1.2 newly added attributes may support single identifiers as values that are reinterpreted as enumeration literals if the attribute expects enumeration values. Otherwise, the identifier is treated as a reference. Every new attribute since v1.2 that expects an enumeration value still accepts the same values as string literals.

Reference attribute value

Reference to a named language element.

Primitive type reference attribute value

Name of a primitive data type. This is for instance used to specify wire types associated with application data types.

Formula

Expression applied for shortcut value encoding notation.

Collection of attribute values

Shall be put in parentheses (...), separating collection elements by comma, with an optional trailing comma. Each collection element in turn must have the same type, which shall be one of the value types in this list, except for encoding formula. A single element list must either have a trailing comma or omit the parentheses, since otherwise it would not be distinguishable from an expression. It depends on the attribute semantics whether duplicate collection elements are allowed or whether the sequence of elements is relevant.

Attributes are associated with declarations by placement, which is individual per type of declaration. Mostly it is allowed to place the attributes before or after the declaration (except if the declaration has a body). When placed after the declaration it must still be before the trailing statement delimiter or body, otherwise it would be associated with the subsequent declaration.

Example: List of attributes using different value types.

[boolAttr,            // implicitly sets the attribute to true
  intAttr=1 << 10,    // expression evaluates to integer
  floatAttr=2.0e-10,  // floating-point constant
  stringAttr="enumA", // string attribute encodes a string or an enumerated value
  stringAttr=enumA,   // enumeration attribute using an identifier
  refAttr=ns::if,     // reference to interface in namespace ns
  listAttr={2, 3, 5, 7, 11, 13, 17}]

Custom Attributes

As stated above you can define your own attributes, either on root level or in any namespace. This way you can define if and which namespace must be written prior to the attribute name. The attribute definition consists of the keyword attribute followed by the attribute’s primitive data type and name. Afterwards it may …

  • … be terminated with a semicolon.
  • … have a default value assigned using the assignment operator and a semicolon afterwards.
  • … contain a body with brackets in which several properties of the custom attribute can be defined. These are:

Name

Default

Description
default none Value that is used when the attribute is not specified at all
flag none Value that is used when the attribute is used without a right-hand-side, thus without an explicitly assigned value

mutable

false

Indicates whether the attribute can be modified during measurement

By offering three possibilities, you can keep a uniform coding style depending on your requirements regarding custom attributes. If, for example, flag values are not necessary at all and values shall not change, you are not forced to write a body to the attribute definition. Following shows all three ways applied to a boolean attribute.

attribute bool MyAttribute1;
attribute bool MyAttribute2 = false;
attribute bool MyAttribute3
{
  default false;
  flag    true;
  mutable;
}

Note that if you want the attribute being used without a value, e.g. [oneWay], you need to define a flag value.
