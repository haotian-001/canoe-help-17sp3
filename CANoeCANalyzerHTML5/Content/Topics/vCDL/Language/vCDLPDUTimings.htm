PDU Timings

 

An interaction layer definition allows to specify how and when PDUs or Service PDUs are transmitted on the network. The keyword pdutiming allows to define named timings, which can be referenced from PDUs and Service PDUs through attribute timings. In case multiple timings are referenced from a communication object, the first is considered the active timing.

The timing definition allows to specify event controlled and cyclic transmission mode timings in its body. These are defined by the keywords onupdate and cyclic and both have a body definition themselves. They support the attributes onTrue and onFalse, which identifies them as a timing to be used when the PDUs timing conditions evaluate to true or false respectively. Within a timing definition there must be at most one cyclic timing for the true condition and at most one for the false condition. The same restriction applies to event-controlled timings.

The body of a cyclic timing requires the specification of an offset and a period, both time values which together define the cyclic transmission. Each of these values can be attributed with a tolerance specification, either an absolute time value or a relative value, i.e. a fractional number between 0 and 1. The offset specification can be omitted, in which case it defaults to 0s with 0 tolerance.

The body of an event-controlled timing requires the specification of a repeat count and a period, which define how to transmit the PDU when a signal is updated. Again, the period allows to specify an absolute or relative tolerance. The repeat specification can be omitted, in which case it defaults to 1.

Example: PDU timing with event controlled and cyclic timings.

pdutiming Timing
{
  [onTrue,onFalse]
  cyclic
  {
    offset 10ms;
    period 50ms [tolerance=0.01]; // 1% tolerance
  }

  [onTrue]
  onupdate
  {
    repeat 3;
    period 100ms [tolerance=2ms];
  }

  [onFalse]
  onupdate
  {
    repeat 5;
    period 10ms [tolerance=1ms];
  }
}

The timing definition is associated with a specific PDU through attribute timings. In addition, for selecting onTrue or onFalse timings each integer signal within the PDU can be associated with timing conditions through an attribute referencing a named timing condition. The condition is a simple statement with a name and all parameters defined through attributes. If conditions are specified and no timings are referenced from the PDU, the conditions are ignored.

Example: Timing condition definitions.

[filterType="NewIsWithin", min=-500, max=100]
condition Condition1;

[filterType="MaskedValueChanged", mask=0x1F]
condition Condition2;

[timings=Timing]
pdu PDU
{
  [conditions=(Condition1,Condition2)]
  uint8 Signal1;

  [conditions=Condition2]
  int32 Signal2;
}
