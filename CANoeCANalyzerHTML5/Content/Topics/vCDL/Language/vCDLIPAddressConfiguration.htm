IP Address Configuration

 

For IP-based communication, as for example SOME/IP, an IP configuration must be provided. It consists of various types of elements that are described in detail in the upcoming sections.

VLANs

VLANs are declared on namespace level using the vlan keyword followed by its name, its unique ID (in laces) and a delimiter.

Example: VLAN definition

vlan VLAN1<1>;
vlan VLAN2<2>;

Supported Attributes (VLAN)

There are no specific attributes for measurement point declarations.

Stacks

TCP/IP stacks are used to easily handle the IP-based communication. An Ethernet adapter – also called network interface – connects the TCP/IP stack to the actual Ethernet network. Each Ethernet adapter is expected to have a unique MAC address. An Ethernet adapter may be assigned to one or multiple VLANs. In vCDL all three entity types (IP stacks, Ethernet adapters and VLAN configurations) are declared in a nested way. Note that the VLAN configurations within an Ethernet adapter reference VLAN instances that have to be explicitly defined somewhere else within the imported vCDL files.

Example: IP-Stack definition

[gateway ="192.168.2.1",
 defaultTTL=128,
 ipForwarding=true,
 tcpDelayedAcknowledge=false,
 tcpTimestampOption=false]
ipstack IPStack
{
  [mac="B1:8D:F7:75:99:96",
   mtu=1000]
  ethernetadapter Adapter1
  {
    [priority=2] VLAN1,
    [priority=4] VLAN2
  }
}

Network Endpoints (NEPs)

Network Endpoints have an IP address (unicast or multicast) and reference either directly an Ethernet Adapter (without VLAN) or an Ethernet Adapter’s VLAN configuration. In vCDL they are declared on namespace level outside any IP stacks using the ipnep keyword, followed by its name and a delimiter. As commonly used by the Classless Inter-Domain Routing (CIDR) notation, a prefix specifies the network-part of the IP address.

Example: Network endpoint definition

[address="192.168.2.10",
 vlan=IPStack.Adapter1.VLAN2,
 prefix=24]
ipnep NEP1;

[address="192.168.2.20",
 ethernetAdapter=IPStack.Adapter2,
 prefix=24]
ipnep NEP2;

Application Endpoints (AEPs)

Application Endpoints are used for reliable (TCP) or unreliable (UDP) communication at a specific port. An AEP is bound to a specific IP address by referencing a NEP. The transport protocol (UDP or TCP) is determined by higher level references to the AEP, e.g. by a SOME/IP endpoint. In vCDL they are declared on namespace level using the ipaep keyword, followed by its name and a delimiter.

Example: Application Endpoint definition

[port=2048,
 nep=NEP]
ipaep AEP;

IP Address Format

Several attributes require IP addresses as right-hand-side. vCDL offers several ways in defining them:

  • Unsigned integer (only IPv4): The IP address is written as unsigned integer, preferably in hexadecimal form. The address has to be provided in Network Byte Order (Big Endian), e.g. 0xC3A67D03 (= "195.166.125.3").
  • Only the address: Any valid address (IPv4 or IPv6) as string, e.g. "192.168.2.10".
  • Address and prefix: Any valid address (IPv4 or IPv6) as string followed by the prefix, e.g. "::1/24".
  • Address and port: Any valid address (IPv4 or IPv6) as string followed by the port, e.g. "[2001:db8:0:8d3::7347]:40000". Note that only for IPv6 square brackets enclosing the address are required.
  • Address, prefix and port: Any valid address (IPv4 or IPv6) as string followed by prefix and port, e.g. "[2001:db8:0:8d3::7347/64]:40000". Note that only for IPv6 square brackets enclosing address and prefix are required.

MAC Address Format

Ethernet Adapters require MAC addresses. These can be specified the following ways:

  • Unsigned Integer: The MAC address is written as unsigned integer, preferably in hexadecimal form to see each byte. The address has to be provided in network byte order (Big Endian), e.g. 0x331122AABBFF.
  • Colon separated string, e.g. "33:11:22:AA:BB:FF"
  • Dash separated string, e.g. "33-11-22-AA-BB-FF"
  • String without any separators, e.g. "331122AABBFF"

Implicit Element Generation

vCDL offers great flexibility when it comes to setting up your IP configuration. However, this degree of freedom can become annoying when the configuration is very small, e.g. you only have two AEPs (TCP and UDP) using the same NEP, thus also having the same IP address. Fortunately, vCDL allows IP stack related attributes to be written at NEPs, all attributes allowed at NEPs are also valid at AEPs and all IP configuration attributes are allowed at SOME/IP endpoints. If, for example, attributes that belong to a NEP are found at an AEP, but the AEP does not refer any explicit NEP, those attributes are used to create a NEP implicitly. Same applies to all other IP attributes.

Note

Distinct names need to be generated for implicit elements, that can become lengthy, depending on the element type. If you plan to modify the model within CANoe or use it in any other form that requires referring the elements by their name, you might find it useful to define them explicitly in vCDL, even though your IP configuration is very simplistic.

Example: Implicit IP configuration definition

[address="192.168.2.10/24:2048", // also contains prefix and port
 gateway="192.168.2.1",
 mac="B1:8D:F7:75:99:96",
 mtu=1000,
 priority=2,
 vlan=VLAN1]
ipaep AEP;
