E2E Protection

 

The E2E protection settings augment event and PDU based communication with counter measures against communication failure by software errors, hardware failure or environmental influences. They are designed along the AUTOSAR E2E library specifications. Since the E2E protection requires quite some parameters, not all of these are directly realized as attributes on the corresponding language elements but an E2E Profile definition allows to reuse the basic settings. The profile is declared by the keyword e2e followed by an identifier and the name of the profile to use. The actual profile settings are realized as attributes on that statement.

Example: E2E profile definition.

[ counterOffset=4,
  crcOffset=8,
  dataIdMode="E2E_P11_DATAID_NIBBLE",
  dataIdNibbleOffset=0,
  maxDeltaCounter=3]
e2e E2EProfile11 "E2E_P11";

The parameters of the profile are the usually more stable and thus reused settings. More specific parameters are attached to individual Signal Groups, Service PDUs, PDUs or SOME/IP events and fields besides a reference to the profile. All these attributes can already be attached to the corresponding type definitions (e.g. a data group definition within a PDU type definition) and can be refined at the instance level.

Example: Event refinement with E2E protection settings.

refine EngineControl.OilPressure
[ e2eDataLength=127*8,
  e2eDataIdList=(180,134,53,233,243,121,67,176,9,94,168,132,198,142,201,138),
  e2eProfile=E2EProfile02];

Depending on the selected profile, there are different restrictions on the E2E attributes to use and on their values. These restrictions are corresponding to the AUTOSAR specifications. The Importer will check the constraints per profile and warn about the use of ignored attributes or out of range attribute values.

Supported dataIdMode attribute values

The dataIdMode attribute is enumerated, allowing the following values (for AUTOSAR E2E Profile 1 and 11 only).

Name

Semantics

"BOTH"

Two bytes are included in the CRC.

"ALT"

Depending on parity of counter alternatingly low or high byte is included in the CRC.

"LOW"

Only the low byte is included in the CRC.

"NIBBLE"

The low byte is included in the CRC and the low nibble of the high byte is transmitted along with the data.
