SOME/IP Binding

 

Binding communication objects to the SOME/IP protocol stack requires to configure numerical identifiers, addresses, event groups, service discovery and serialization parameters. Most of these are specified as attributes on language elements, though the address and event group configuration add some binding specific language elements. All attributes are defined in the binding namespace SOMEIP.

Service Configuration [▲ back]

The service configuration is concerned with the specification of SOME/IP identifiers to services, endpoints, events, methods, and PDUs as well as the definition of and subscription to event groups. All identifiers are simply attached to the corresponding language elements by binding specific attributes. The qualification with the binding namespace can be omitted when the service has SOME/IP binding.

Example: Service definition with inlined identifier attributes.

[SOMEIP::serviceId=12]
service EngineControl : IEngineControl
{
  binding SOMEIP;

  provider Engine     [instanceId=17];
  consumer Dashboard  [clientId=23],
           Navigation [clientId=24];

  void RequestFuelSaveMode()  [methodId=137];
  event bool OilPressureLow   [eventId=287];
  field uint16 EngineSpeed     [getterId=28, eventId=91];
  pdu EngInfoType EngineInfo  [pduId=67];
}

The definition of event groups requires a binding specific keyword eventgroup which has a name and is assigned a list of references to events, fields and service PDUs. The event group definition is associated with a specific service and may only reference members of that service. This can either be done directly within a service definition or in a service refinement block.

Example: Refined service with event group definitions.

refine EngineControl
{
  binding SOMEIP
  {
    [groupId=17, multicastThreshold=5]
    eventgroup Alarms = OverTemperature, OilPressureLow;

    [groupId=24, multicastThreshold=2]
    eventgroup State = EngineSpeed, Temperature, Mode;
  }
}

For defined event groups, a static subscription can be configured with additional attributes for service consumer and provider endpoints. A consumer may list the event groups for which it should be subscribed at the start of measurement and a provider may list the consumers which shall be statically subscribed.

Example: Event group subscription.

refine EngineControl
{
  refine Dashboard  [subscribedEventGroups={State,Alarms}];
  refine Navigation [subscribedEventGroups=State];

  refine Engine [subscribedConsumers=Dashboard];
}

The referenced event groups and consumers must be part of the same service in which the reference is declared.

Address Configuration [▲ back]

For being able to transmit data via SOME/IP, each endpoint of the communication needs to be associated with an IP address and a UDP or TCP port. Since the overall configuration may require multiple addresses for reliable and unreliable as well as multicast and service discovery communication, an endpoint can be defined, which identifies the AEPs for reliable, unreliable and multicast communication and can be referenced from service consumers or providers.

A SOME/IP endpoint can be defined in a shortcut way, by directly specifying IP addresses and ports in the attributes. In that case no additional ipaep or ipnep definition is required, but the configuration is not as flexible, since no individual subnet addresses and masks, MAC addresses or VLAN IDs can be specified per IP address.

Syntactically the endpoint keyword starts a statement, followed by the endpoint name. Address, transport protocol and port configuration are specified by attributes. All SOME/IP endpoints must be declared within a global SOME/IP binding block.

Example: Inlined endpoint definition in IPv4 format.

binding SOMEIP
{
  [udpEndpoint="192.168.1.10:40000",
   tcpEndpoint="192.168.1.10:40000",
   multicastEndpoint="239.0.0.1:40000",
   sdMulticastEndpoint="239.0.0.1:30490",
   mask="255.255.255.0",
   vlan=0]
  endpoint MyEndpoint;
}

For endpoint string syntax and restrictions between inlined attributes, see also IP Address Configuration.

The same address configuration can be achieved through explicit AEP and/or NEP declaration.

Example: Explicit endpoint definition in IPv4 format.

ipnep MyNEP          [address="192.168.1.10"];
ipnep MyMulticastNEP [address="239.0.0.1"];

ipaep MyAEP            [port=40000, nep=MyNEP];
ipaep MyMulticastAEP   [port=40000, nep=MyMulticastNEP];
ipaep MySDMulticastAEP [port=30490, nep=MyMulticastNEP];
binding SOMEIP
{
  [udpEndpoint=MyAEP,
   tcpEndpoint=MyAEP,
   multicastEndpoint=MyMulticastAEP,
   sdMulticastEndpoint=MySDMulticastAEP,
   mask="255.255.255.0",
   vlan=0]
  endpoint MyEndpoint;
}

MAC addresses are expected as a string of six groups of two hexadecimal characters, separated by a dash ("00-11-22-aa-BB-FF"), a colon ("00:11:22:aa:BB:FF") or not separated at all ("001122aaBBFF") in network byte order. Alternatively, the MAC address may as well be specified as 48 bit integer number.

The defined SOME/IP endpoints can be referenced from service consumers and service providers, associating these with the address information.

Example: Service provider bound to a SOME/IP endpoint.

service MyService
{
  binding SOMEIP;
  [endpoint=MyEndpoint]
  provider MyProvider;
}

All IP address information can be inlined directly with service consumers and providers. Therefore, these do also support the same attributes as a SOME/IP endpoint definition, implicitly defining an anonymous SOME/IP endpoint.

Example: Service provider bound to a SOME/IP endpoint with inlined address settings.

service MyService
{
  binding SOMEIP;

  [SOMEIP::udpEndpoint="192.168.1.10:40000",
   SOMEIP::multicastEndpoint="239.0.0.1:40000",
   SOMEIP::sdMulticastEndpoint="239.0.0.1:30490",
   mask=0xFFFFFF00,
   vlan=1]
  provider MyProvider;
}
