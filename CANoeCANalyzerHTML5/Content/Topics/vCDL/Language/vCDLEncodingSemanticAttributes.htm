Encodings and Semantic Attributes

 

Encodings can be defined to support the use of distinct raw and implementation data types and formulas for calculating physical and textual representations from one or the other. In general, the model distinguishes an implementation data type, a raw (wire) data type and a physical data type. The actual declaration of a signal, event, field etc. always contains its implementation data type, e.g. declaring an int32 struct member implies that this is its implementation data type. Thus, an implementation data type is always given and cannot be omitted.

A specific raw data type can be associated with any declaration using a numeric data type by adding the rawType attribute. Without further specification the raw value will then be equivalent to the implementation value simply converted to the raw data type. In a network transmission this may for instance mean that you are losing precision or resolution.

A physical data type is again always defined and can be any of uint64, int64 or double, whichever can hold all values from the implementation data type. You never need to explicitly specify the physical data type. Physical values are by default the same as implementation values, i.e. there is no scaling or offset and because of the automatic choosing of their data type you cannot lose any precision or resolution by conversion.

If there should actually be a conversion between implementation and physical values, this can be specified in the attribute impl2phys. The attribute value can either be a linear formula over the parameter x, or a reference to a more complex encoding definition. A second attribute raw2phys allows to specify another such encoding for calculating between raw and physical values. If both formulas are specified, the raw values are calculated from implementation values by first converting the implementation values to physical values and then applying the reverse formula for calculating raw values from physical values.

Example: Encoded data type as type alias.

[rawType=int16, impl2phys=x*0.02, raw2phys=x*0.1, min=-274, max=1000, unit="°C"]
typedef Temperature = int32;

Minimum and maximum values as well as the unit are specified in terms of physical values. They have only informational character and will not influence the simulation behavior.

In case the encoding formulas need to be specified per interval of implementation or raw values or if specific values shall be mapped to textual representation, the shortcut notation above is not useful. You need to define a named encoding in these cases, which allows to define intervals of values mapped to specific formulas or textual values.

Each interval definition requires a differentiation of open/closed boundaries for defining a range of encodings without gaps, considering floating point data types. This is achieved by the keyword where followed by an equation over the variable x. The equation takes the form [c1 op1] x [op2 c2] where c1 and c2 must be constant expressions and op1 and op2 are comparison operators. It is not allowed to skip both operators and the overall expression must not define an empty interval (e.g. 10 < x < 5 is not allowed). If the equality comparison operator == is used, then only one operator and operand is allowed. If two inequality operators are used, then both must have the same direction (e.g. 10 < x >= 5 is not allowed). If the where clause is omitted for an encoding item, then the formula or textual encoding applies to the full value range except other intervals with specific mappings. It is not allowed to specify multiple statements without an interval equation. The linear encodings applied to a value must not have overlapping intervals, i.e. no two different formulas must be mapped to the same value. The same applies for textual encodings but it is allowed to define textual and linear encodings for the same range of values.

The physical values’ unit can be specified globally (e.g. as part of the typedef as in above example) or as part of the named encoding, or it can be specified per formula thus allowing different units for different intervals. If both a global unit and an interval unit are specified, the interval unit overrides the global unit (this does not cause any validation errors or warnings).

It is possible to split the definition of encodings into multiple separately named objects. The corresponding reference attribute can then be written in the form of a collection, merging the intervals from all referenced encodings.

Example: Encoded data type as alias with multiple named interval encodings.

[unit="°C"]
encoding LinearEncoding
{
  2 * x wherex < 0,
  -x    where 0 <= x <= 10 [unit="°F"],
  2 * x - 30, // default
}

encoding TextualEncoding
{
  "null"    where x == 0    [display="freezing point"],
  "minimum" where x == -273 [display="absolute zero"],
}

[impl2phys=(LinearEncoding, TextualEncoding)]
typedef Temperature = int32;

Explicit encodings must be declared at namespace level, i.e. they cannot be nested into complex types, interfaces or services.

Supported Attributes (Encoding)

The following attributes are supported by encodings:

Name

Type

Default

Semantics

unit

String

""

The physical unit to use for displaying an encoded value.

Attributes for the encoding declaration may be placed before the encoding keyword only.

Supported Attributes (Encoding Item)

The following attributes are supported by encoding items:

Name

Type

Default

Semantics

unit

String

""

The physical unit to use for displaying an encoded value.

display

String

""

A verbose description of a textual encoding item.

Attributes for an encoding item may be placed before the textual value or formula of the item or before the item separator or after the last item.
