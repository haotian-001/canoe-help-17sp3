Interfaces

 

A detailed explanation for each member along with the concept of member directions can be found on the Distributed Objects page.

If features are mixed, e.g., if a distributed object interface contains PDUs or a service interface inherits from other interfaces, there will be errors during import. See the common errors section below for what may be wrong. As rule of thumb respect following guidelines:

  • Have a direction for each member (except for embedded members) when using the interface for distributed objects.
  • Have no directions at all in the interface when using it for services.
  • Only inherit from other interfaces when using the interface for distributed objects.

Example: Two interfaces, one for distributed objects, the other for services

interface DOInterface : DOBaseInterface
{
  internal data   Struct      DataMember;
  provided method Prototype   MethodMember;
  provided        uint48      InlinedMethod(float Param);
  consumed event  void        EventMember;
  consumed field  list<int32> FieldMember { get; notify; }

  embedded OtherDOInterface            EmbeddedInstance;
  embedded reference<OtherDOInterface> EmbeddedReference;

  embedded InlinedEmbeddedMember
  {
    internal data int32 EmbeddedDataMember;
  }
  embedded array<OtherDOInterface, 6> EmbeddedMemberArray;
  embedded list<OtherDOInterface> EmbeddedMemberListUnbounded;
  [minSize=5, initialSize=7];
  embedded list<OtherDOInterface, 10> EmbeddedMemberListBounded;
}

interface ServiceInterface
{
  method Prototype   MethodMember;
  event  void        EventMember;
  field  list<int32> FieldMember { set; }
  pdu    PDUType     ServicePDU;
}

The syntax is similar to other object types in vCDL. The keyword interface followed by its name and a bracket enclosed body containing the interface members. Optionally, one or more base interfaces may be declared by a colon followed by a comma-separated list of interfaces. Named interfaces are always declared on namespace level and cannot be nested . Note the different member types and the missing directions for service interface members.

Reversed Interfaces

The distributed object concept often requires complementary interfaces, such that both interfaces contain the same members, but with complementary directions. Provided members in one interface become consumed members in the other interface and vice versa. vCDL provides a convenient way to declare the interface and its members only once. To refer to the reversed distributed object interface, simply enclose the reference as follows: reverse<DOInterface>.

Inheritance

Distributed object interfaces support multiple inheritance. All direct base interfaces are listed comma separated between the interface name and its body. Distributed object interfaces may inherit from reversed interface as well. Due to multiple inheritance a derived interface can inherit the same base interface multiple times via different intermediate interfaces. Normally, this is not a problem, but with reversed interfaces a base interface may be inherited from in both its original and its reversed form at the same time. Of course, this is not allowed as the same base interface member would be both consumed and provided. Following example shows inheritance from a base interface and two intermediate interfaces using different combinations of reversed and non-reversed interfaces.

// Base interface
interface InterfaceA
{
  provided dataint32 Member;
}

// Intermediate interfaces
interface InterfaceB1 : InterfaceA;
interface InterfaceB2 : reverse<InterfaceA>;

interface InterfaceC1 : InterfaceB1;                      // Ok
interface InterfaceC2 : InterfaceB1, InterfaceA;          // Ok
interface InterfaceC3 : InterfaceB1, reverse<InterfaceA>; // Error
interface InterfaceC4 : InterfaceB2, reverse<InterfaceA>; // Ok
interface InterfaceC5 : InterfaceB2, InterfaceA;          // Error
interface InterfaceC6 : InterfaceB1, InterfaceB2;         // Error
interface InterfaceC6 : reverse<InterfaceB2>;             // Ok (InterfaceA reversed twice)
interface InterfaceC7 : reverse<InterfaceB2>, InterfaceA; // Error
interface InterfaceC8 : reverse<InterfaceB1>, InterfaceB2, reverse<InterfaceA>; // Ok

Inheriting from a reversed base interface is propagated to its base interfaces as well. It is therefore usual that, when traversing the inheritance hierarchy up to the base interface, an interface is reversed more than once. This is not a problem as reversing an interface twice yields the original interface. You can see this behavior in the example above when the reversed intermediate interface is used which itself reverses the base interface. In the end the base interface is not reversed at all.

Other programming languages supporting multiple inheritance deal with the problem that intermediate interfaces introduce the same members or at least members with the same name. In the derived interface a mechanism to distinguish both members is necessary. In vCDL we avoid such problems by forbidding conflicting members between intermediate and/or base interfaces in first place. Doing otherwise yields an error during import. Therefore, shadowing and overriding of inherited members cannot occur in vCDL.

Common Errors

However, there are plenty of possible errors that can cause the interface to be ambiguous. The following examples show some common errors:

// Error: Some members have a direction, some do not
interface InterfaceA
{
  internal data Struct    DataMember;
  method        Prototype MethodMember;
}

// Error: Interface contains DO-only members as well as service-only members
interface InterfaceB
{
  internal data Struct  DataMember;
  provided pdu  PDUType ServicePDU;
}

// Error: Interface looks like a service interface, but inherits from another interface
interface InterfaceC : InterfaceA
{
  event int32 EventMember;
  field int32 FieldMember { get; notify; }
}

Attributes

Which attributes are allowed at the interface depends on whether the interface is used for services or distributed objects.
