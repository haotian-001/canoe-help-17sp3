Motivation and Delimitation

 

The cross-network communication concept of CANoe offers the possibility to import communication objects, distributed objects, or just data types and interfaces from different data sources. These are generated in different steps of the process and can be incorrect and incomplete:

  • When commissioning an ECU, no database may be present or data may be missing that are only required to simulate the network.
  • During the development of software components or individual algorithms, the communication structure is still undefined.
  • The network design is incorrect or incomplete and the databases generated therefrom are patchy.
  • Different versions of databases with incompatible definitions do not allow a network to be simulated or partly put into operation.
  • An automated test requires additional communication objects that are not part of the system design.

These gaps or errors in the definition either mean that information must be added via the respective data source (which can lead to a lengthy process) or that changes must be made in the test / simulation tool. In the latter case, the CANoe data sources can be extended and corrected using the Model Editor or by importing vCDL. The Communication Object Editor is particularly suitable for changing parameters and bindingsNetwork connection of communication objects, defines the communication stack and its parameters. of existing communication objects, while the import of vCDL allows you to define additional communication objects more easily.
