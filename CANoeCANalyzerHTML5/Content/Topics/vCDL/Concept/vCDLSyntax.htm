Structure and Syntax

 

The structure of vCDL is similar to the most common programming languages, except that vCDL is a pure declarative language and therefore does not include any program flow.

Communication Objects

The basic objective of this language is to define communication objects that can be identified by a unique name. In the case of service-oriented architecture, these are services that can include events, fields, methods, and PDUs. In signal-based communication, signals and PDUs are available as independent communication objects.

signal uint16 Speed
{
  sender WheelFL, WheelFR, WheelBL, WheelBR;
  receiver TractionControl;
}

Unions

Type alias definitions allow application-specific names to be assigned to the defined data types; furthermore, properties for serialization and simulation can be added by means of data-specific attributes.

const int32 AbsZero = -273;

[min=AbsZero, max=1000, unit="C"]
typedef Temperature = int32;
typedef History     = array<Temperature, 100>;

Namespaces

As for conventional programming languages, defined communication objects are organized in namespaces that are used for an unambiguous classification in case of mutual references. Any type definitions and other independent constructs such as linear coding rules, constant declarations, function prototypes, interfaces, PDU types and binding-specific definitions are also identified by the respective namespace.

namespace Multimedia
{
  namespace Audio
  {
    interface IAudioSource
    {
      field uint16 CurrentTrack { get; set; notify; }
    }
  }

  service CDPlayer : Audio::IAudioSource
  {
    consumer MediaController;
    provider Console;
  }
}

Bindings and Refinements

To define the network binding of communication objects, vCDL provides special language constructs through which the binding is identified by a reserved identifier. For example, the abstract binding for the pure simulation of communication is identified by the identifier Abstract. Depending on the complexity of the communication stacks, parameters and serialization rules are set by means of binding-specific attributes or additional language constructs. The use of a specific binding can be completely separated from the actual definition of communication objects by defining additional attributes elsewhere in the source file or even in a separate source file by means of the so-called refinements. The same applies to distributed objects, where all bindings are configured exclusively via attributes.

[delay=100ns]
refine Multimedia::CDPlayer
{
  binding Abstract;

  [subscribedConsumers=MediaController]
  refine.Console.CurrentTrack;
}

Imports

vCDL allows to divide a model into several source files as desired. Source files can reference each other via an import; furthermore, references between communication objects, data types and other identifiable vCDL objects can work across file limits. It is recommended to specify paths in imports in a relative way so that all directories can be simply moved and copied.

import"../common/types.vcdl";

Syntax

From a syntactic point of view, a vCDL source file includes declarations which either can end with a semicolon or which contain a body definition between curly brackets. Declarations of objects are always located in a namespace. Only the declaration of the language version and the import statements are located at the beginning of each source file outside all namespaces.

version1.4;

Attributes are assigned to declarations in the form of a list of key-value pairs between square brackets. The assignment is made using the placement rules for attributes that are documented in the technical reference for all types of declarations. In general, attributes can always be written directly before and after the declaration, but before the final semicolon.
