MOST Light & Lock Observation

 

Name

MOST Light & Lock Observation

Description

This check is used to observe the Light & Lock condition of the MOST network interface. There is a corresponding check function for each of the event types delivered by the hardware.

Variant 1

The diagram shows the sequence of Light & Lock states on light failure for 20 ms (tLightOff < tUnlock, see MOST specification):

The Light & Lock checks show the following behavior, depending on their type and parameterization ("Light, No Lock" and "No Light, No Lock" appear directly after each other and have the same time stamp):

1. LightOff (ChkStart_MostLightOff(), <mostlightlock eventtype="lightoff">)

2. ShortUnlock (ChkStart_MostShortUnlock(), <mostlightlock eventtype = "shortunlock">)

3. CriticalUnlock (ChkStart_MostCriticalUnlock(), <mostlightlock eventtype = "criticalunlock">)

In this scenario the Light & Lock check doesn't fire since no CriticalUnlock state is reached.

4. StableLock (ChkStart_MostStableLock(), <mostlightlock eventtype = "stablelock">)

Variant 2

The diagram shows the sequence of Light & Lock states when turning off light modulation for 80 ms (tLightOff > tUnlock, see MOST specification):

The Light & Lock checks show the following behavior, depending on their type and parameterization:

1. LightOff (ChkStart_MostLightOff(), <mostlightlock eventtype="lightoff">)

In this scenario the Light & Lock check doesn't fire since the LightOff state isn't reached.

2. ShortUnlock (ChkStart_MostShortUnlock(), <mostlightlock eventtype = "shortunlock">)

3. CriticalUnlock (ChkStart_MostCriticalUnlock(), <mostlightlock eventtype = "criticalunlock">)

4. StableLock (ChkStart_MostStableLock(), <mostlightlock eventtype = "stablelock">)

Remark

The check works on the LightLock events that are generated by CANoe, as soon as a change of the LightLock is recognized by the hardware. It will not automatically recognize the current LightLock state at the time of its activation. Therefore it's possible, that a check waits for a LightLock state, which is currently active but has been established before the check was started.

CAPL Function

ChkCreate_MostCriticalUnlock, ChkStart_MostCriticalUnlock, ChkCreate_MostLightOff, ChkStart_MostLightOff, ChkCreate_MostShortUnlock, ChkStart_MostShortUnlock, ChkCreate_MostStableLock, ChkStart_MostStableLock
