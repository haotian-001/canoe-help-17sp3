Selection in Execution Units [▲ back]

In C#, you often do not need to explicitly select the endpoints of a communication object, for example, because there is only one sender and one receiver. In the general case, a communication object can, however, have several endpoints on each side. In CAPL, you must always specify endpoints even if they are unambiguous.

With point-to-point communication, you must specify in the API from which endpoint to which endpoint communication takes place.

With broadcast communication, you must only specify the endpoint used by the code. For some values, however, you must additionally specify on which side the value should be read. For example, in case of an event, for each pair, consisting of a consumer and provider there is a value both on the consumer side (the last received value) and on the provider side (the value last sent to the consumer or the value to be sent next to the consumer). If you only want to measure transferred values between the endpoints, in particular if both endpoints are not simulated, there is also a measured value in the center.

In C#, selection per communication object and per execution unit is saved, thus avoiding the need to select endpoints every time values are accessed or functions are called. An execution unit is a group, consisting of one or several programs. This group is configured together and created for a particular purpose, for example an application model or a test unit. If an endpoint is selected for an execution unit, this has no effect on other execution units. The selection of the endpoints is reset when the measurement is started.

Note

You can assign the mentioned endpoints only if they have been configured in CANoe and switched to simulated or measurement. This is not necessary for the opposite side of the communication.

Default Selection (C# only)[▲ back]

Participants can be assigned to each execution unit during its configuration. By default (when starting the measurement), the following endpoint is selected for a communication object:

  • If at least one of the assigned participants has an endpoint on the object, this object is selected.

Example

The MirrorProvider service has the provider endpoints LeftMirror and RightMirror. RightMirror belongs to the RightMirror participant which can be assigned by means of configuration to a C# model in the Communication Setup (Application Models). The RightMirror endpoint is now preselected for this C# model.
  • If several participants have endpoints on the object, the first endpoint is selected.

Example

If the LeftMirror participant has been additionally assigned to the C# model as the first participant in the list, then the LeftMirror endpoint is selected.
  • Otherwise, the first simulated endpoint is selected.

Example

If no participant at all has been assigned, but LeftMirror is set to real and RightMirror to simulated, the RightMirror endpoint is selected.
  • If there is no simulated endpoint (on the side), the first endpoint is selected.

Example

However, if RightMirror is also set to real, then LeftMirror is selected again, since this is the first endpoint.
  • If an endpoint has been selected via an assigned participant, then its side is selected.

Example

If the RightMirror participant has been assigned to the model and its endpoint has been selected because of this, then the provider side is selected, since this endpoint is a provider endpoint.
  • Alternatively, the side with a simulated endpoint is selected if only one side has a simulated endpoint.

Example

If no participant has been assigned, all consumers are set to real and the RightMirror provider is simulated, then the provider side is selected.
  • Alternatively, Measure is selected if measured values between endpoints are available.
  • Otherwise, the consumer side or the sender side is selected.

Example

However, if a consumer (Control) is also simulated, then its side is selected as long as no participant has been assigned to the model.

API for Selection (C# only)[▲ back]

The endpoints are selected by calling methods which with service-oriented communication are called SelectProvider and SelectConsumer, and with signal based communication are called SelectSender and SelectReceiver. You can pass the following as a parameter:

  • a participant
  • an endpoint directly
  • the complete path to an endpoint (as a string, as in CAPL, e.g. Cameras::Camera[Control])
  • the index of an endpoint

To select the side, use SelectSide; you can pass ProviderSide, ConsumerSide, SenderSide, ReceiverSide or Measure as a side (depending on the communication type of the object). You can query each communication object to find out whether currently valid endpoints are selected (IsConsistent) and, if so, whether the currently selected endpoint is accessible for the currently selected side (IsValid).

You can use the properties SelectedConsumer / SelectedProvider / SelectedSender / SelectedReceiver to query the participant which belongs to the currently selected endpoint (if there is one). You can use Selected…Index to also query the index of the selected endpoint.

C#

// Call "SetActive" method on all camera providers
Camera.SelectConsumer(Camera.Control.Instance);
for (uint i = 0; i < Camera.ProviderCount; ++i)
{
  Camera.SelectProvider(i);
  var cco = Camera.SetActive.CallAsync(true);
  // ...
}

Note that the index of an endpoint can change if endpoints are added or removed (see Service Discovery).

Direct Specification of Endpoints[▲ back]

C#

Instead of using selected endpoints, you can also specifically define endpoints every time values are accessed or functions are called. This is particularly necessary for handlers, since they are linked to a value even before the measurement starts. In C#, you can also specify the endpoints directly to ignore the current selection.

In C#, you specify the endpoints in the order Side, Consumer, Provider or Side, Sender, Receiver. With broadcast communication, you only need to specify one of the two endpoints; this endpoint then will be used to determine the side. The individual parts are separated by points.

Example

MirrorAdjustment.consumerSide.Control.LeftMirror; StatusPdu.RightMirror

CAPL

In CAPL, you specify the endpoints with point-to-point communication using Side[Consumer,Provider] or Side[Sender,Receiver]. If the side is already clearly identified by the type of use, for example with on fct_Called handlers or with CallAsync call, then you can omit it. With broadcast communication, specify the endpoint using [Endpoint].

Example

MirrorAdjustment.consumerSide[Control,LeftMirror]; StatusPdu[RightMirror]

You can specify the endpoints using their name or an index; the index can also be determined via a variable or more generally via an expression.

With service-oriented communication, you may use consumerSide, providerSide and measure for the side, otherwise use senderSide, receiverSide and measure.

CAPL

// Call "SetActive" method on all camera providers
callContext Camera.SetActive cco;
dword index;
for (index = 0; index < getNrOfCOProviders(Camera); ++index)
{
  cco = Camera[Control,index].SetActive.CallAsync(1);
  // ...
}

Collective Endpoints [▲ back]

Collective endpoints are virtual endpoints which are used to collect incoming data for a whole group of endpoints or to distribute data to be sent to a whole group of endpoints.

There are collective endpoints for methods on provider side. You can use them to program the simulation or implementation of the method independently of the consumer. Use all instead of a concrete consumer to specify the collective endpoint.

C#

[OnCall(typeof(Camera.providerSide.all.Camera1.SetActive))]

CAPL

on fct_Called Camera[all,Camera1].SetActive
