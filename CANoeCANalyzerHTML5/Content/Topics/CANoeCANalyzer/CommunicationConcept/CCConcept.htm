Basics of the Communication Concept

 

The key of the CANoe communication concept is the strict separation of the communicating parties (the application parts) and the medium which is used for transmission. For this, CANoe provides a layer between the application and the transmission medium, the so-called middleware.

In the middleware, the required parameters are configured, for example endpoints for the communication partners or interfaces for the distributed objects. Based on the configured parameters, the objects communicate with their configured counterparts. This is done according to a configurable communication pattern. Typical patterns for communication objects are, for example point-to-point, broadcast or multicast connections. Typical patterns for distributed objects are Publish/Subscribe or Send/Receive. For communication objects (CO) or distributed objects (DO), all usual functions for analysis, test and simulation are available in CANoe:

  • Interactive graphic interactions, such as Trace, Logging, Graphics Window, State Tracker, etc.
  • Test modules and test scripts
  • CAPL and C# interface
  • MATLAB/Simulink
  • …

The focus here is on the support of the service-oriented communication in the way it is required in modern vehicle architectures for application servers, for example of ADAS or modern infotainment systems. However, classic signal-based and PDU-oriented communication via broadcast is also supported.

A further goal is to support modeling of dynamic scenarios in CANoe in a more comfortable way. For such scenarios, the communication matrix or the involved participants are typically specified not statically by a system description. Instead, connections are established during runtime.

In this case, communication can be mapped on many levels. The beginning of the application layer is not specified by CANoe, but can be configured individually. For example, it is possible to model communication between software components, vehicle functions, to an Internet of Things Backend or, of course, in the classic way between ECUs.

The objects are connected to the transmission layer via the so-called bindings. Here, the specific network technology (Ethernet, CAN, FlexRay, …) and – similar to the interaction layer configuration – the transmission behavior of the information units are configured. If both communication partners are represented by CANoe, for example in an early phase of development with a remaining bus simulation by CANoe, transmission can also take place on a very high layer in the communication stack (Abstract binding). In this case, no in-depth configuration of the transmission layer is required.

A design goal of the communication concept is a CANoe centric configuration; generally, only items which are actually required by CANoe must be configured. In particular, it is not necessary to first create an all-encompassing "world model" from which only few required data is filtered. Of course, as usual with CANoe, it is still possible to import communication databases or system descriptions from different formats, including DBC files, AUTOSAR-ARXML system extracts and FIBEX descriptions.
