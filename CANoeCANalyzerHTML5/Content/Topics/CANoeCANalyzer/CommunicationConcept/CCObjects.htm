Communication Objects

 

General

Besides distributed objects, communication objects (CO) are a core feature of the communication concept. They are the central configuration point of communication. Here, communication endpoints, communication type and pattern as well as transmission properties are set:

  • The endpoints correspond to the participating communication partners.
  • The communication type determines whether transmission must be signal based or service oriented.
  • The communication pattern specifies whether it is a point-to-point or broadcast communication.

According to their role, the endpoints are divided into the sides sender/receiver for signal-based communication or provider/consumer for service-oriented communication. In addition, the service interface is set for service-oriented communication and the data type of the exchanged information units is set for signal-based communication. The configuration also specifies which endpoints must be operated by CANoe.

Service Interfaces

Service interfaces describe service functions, service signals, service PDUs, etc. available in the service. For signal-based communication, the usual fundamental data types, such as glide ratio (float, double, etc.), integer or strings may be used. It is also possible to define and use complex structured data types.

Theoretically, for a communication object, connections between the endpoints are established according to the set communication pattern. CANoe provides access opportunities for "its" connection ends. The actual data storages for interaction with the system are located there. Their structure is derived from the configured data type.

Data Storage and Call

Similar to the existing signal concept in CANoe, a write access to these data storages triggers sending data according to the configured transmission behavior. On the reception ends, the most recent received values are buffered and as usual handlers are triggered when new data is received. Service functions play a special role in this process. Here, data storages also keep the current values of the input and output parameters of the function. However, in particular in CAPL and C#-APIs, the service function is called via the usual comfortable call syntax.

The name "communication object" and particularly its part "object" is derived in terms of object orientation: CANoe provides objects for communication. This is reflected in the syntax which is mainly based on the way the objects are described by C++ / C# / Java.

Binding

Similar to distributed objects, the binding that ensures the transmission of data can be selected for communication objects.

Abstract Binding

Abstract binding enables transmission between simulated endpoints within CANoe. Thus, abstract binding enables application-related communication between simulated components in CANoe without describing and simulating lower protocol levels.

Abstract binding is well suited, for example of virtual prototyping, but is also suitable for "test the test" scenarios.

You can find an example of using the abstract binding in the SOASystem sample configuration.

SOME/IP Binding

SOME/IP binding enables transmission between endpoints on the bases of the SOME/IP protocol via TCP/UDP/IP and Ethernet. In this case, one endpoint is typically simulated and one endpoint is real. SOME/IP binding supports the transmission of events, the call of methods as well as finding and acquainting services by means of Service Discovery. In addition, the typical AUTOSAR extensions of the SOME/IP protocol and the transmission of Signal-I-PDUs with end-to-end protection and Secure Onboard Communication are supported. Currently the SOME/IP binding only supports communication objects, but no distributed objects.

Communication objects of the Service Interface type can be directly linked with a SOME/IP binding. To link a communication object with a SOME/IP binding, the following two steps are required:

SOME/IP Interface Mapping

The SOME/IP interface mapping links the Service Interface elements with the required SOME/IP protocol parameters. Essentially, the following SOME/IP protocol parameters must be configured:

  • Service ID: …
  • Method ID: …
  • Event ID: …
  • Event group: …

SOME/IP Endpoint Mapping

The SOME/IP endpoint mapping links the endpoints of the communication object with the network addresses of the TCP/UDP protocol, i.e. with an IP address and with a TCP and UDP port number. Furthermore, the SOME/IP-specific address parts, such as the instance ID are configured here.

If Service Discovery is used, the network addresses for real participants can also be configured automatically.

In case of static scenarios, where communication participants are known in advance, this step can be carried out manually via a GUI. In case of dynamic scenarios, this step can be carried out via a corresponding API during simulation.
