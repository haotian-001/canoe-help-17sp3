Placeholders in Attributes

 

String attributes for distributed objects can contain placeholders that are replaced by the value of another attribute when the attribute value is used and queried.

This is especially useful if the value of the attribute should be similar for many objects or members. Then the value can be set with one or more placeholders, e.g. at the interface or at the object, while at the member only the value for the placeholder is set.

Example

For the MQTT binding, the topic is defined by attribute. Here you could have several distributed objects of type ClimateControl, one per room; and each of these objects receives temperature data from several sensors:

namespace SmartHome
{
  interface ClimateControl
  {
    consumed data int64 Temperature1;
    consumed data int64 Temperature2;
    // ...
  }
  object LivingRoomClimateControl : ClimateControl;
  object OfficeClimateControl : ClimateControl;
  // ...
}

Often the MQTT topic is similar for all sensors, only room and sensor ID change. So you can define own attributes for these values:

attribute string Room;
attribute int32 SensorId;

Then you define the topic at the interface with placeholders for all members:

[MQTTBinding::Topic="rooms/{Room}/devID{SensorId:d}/Temperature"]
interface ClimateControl

The value of the new attributes can now be set for each object or member:

[SensorId=1]
consumed data int64 Temperature1;

[Room="Office"]
object OfficeClimateControl : ClimateControl;

These examples model the distributed objects using the vCDL description language. You can also use the Model Editor.

Predefined Placeholders

Some placeholders are predefined and are replaced with specific values:

Placeholder

Description

ObjectName

Name of the distributed object, e.g. OfficeClimateControl

MemberName

Name of the member, e.g. Temperature1

ObjectPath

Complete path of the member, contains namespaces and object name, e.g. SmartHome::OfficeClimateControl.Temperature1

MemberPath

Relative path of the member to the object, contains in case of embedded members the parent members, but not namespaces and object name.

NamespacePath

Only namespaces, e.g. SmartHome.

Container::IndexNr

Index of the object in its container, in the dimension Nr. The counting of the dimensions starts at the very outside with 0.

Container::IndexNrValue

The attribute Container::IndexNrValues must be given a value at the container, which is a string with comma separated values for each element of the container. The placeholder then takes the value from this list, which corresponds to the index of the element. Nr is again the dimension of the container.

If the attribute IndexNrValues does not have a matching value, the placeholder corresponds to IndexNr, thus returning the value of the index.

Example

The above example can be further simplified. If you define one sensor per object, but put them into an array which is called like the room, the following definition is sufficient:

[MQTTBinding::Topic="rooms/{ObjectName}/devID{Container::Index0}/{MemberName}"]
interface Sensor
{
  consumed data int64 Temperature;
}
array<Sensor, 3> Office;
array<Sensor, 3> LivingRoom;

If the topic becomes more complex in parts, e.g. the devID part does not simply correspond to the index, you can use IndexNrValue, e.g. like this:

[MQTTBinding::Topic="rooms/{ObjectName}/{Container::Index0Value}/{MemberName}"]
interface Sensor
{
  consumed data int64 Temperature;
}
[Container::Index0Values="devIDOne,devIDTwo,devIDThree"]
array<Sensor, 3> Office;
[Container::Index0Values="devIDEins,devIDZwei,devIDDrei"]
array<Sensor, 3> LivingRoom;

The second temperature sensor in the office would thus have the topic rooms/Office/devIDTwo/Temperature.

Placeholder Syntax

The exact syntax for a placeholder is '{'AttributePath [':' FormatSpec]'}'

  • There may be spaces between the individual elements. They will be ignored in the replacement. Other whitespace is not allowed.
  • AttributePath is the path of the attribute to which the reference is made. The namespaces of the attribute can also be partially omitted, in which case the attribute is searched for relative to the following namespaces:
  • Namespace of the attribute that contains the placeholder
  • Superior namespaces of this namespace
  • Namespace of the distributed object for which the attribute value is determined
  • Superior namespaces of this namespace
  • To specify a set parenthesis in a string attribute value, it must be masked by doubling so that it is not interpreted as a wildcard character. This applies to both the opening and the closing set parentheses.
  • If the replaced value itself contains placeholders again, they will continue to be replaced. However, placeholders that are only created by replacing a combination of placeholders are not replaced.
  • The FormatSpec is optional and only allowed if the attribute to be referenced has a numeric type. Its syntax is ['.'precision]type.
  • Precision (if specified) is a number that specifies the number of digits in the formatted string for floating-point numbers
  • Type must match the type of the attribute that is referenced. It can be:
  • d: Integer, is formatted in decimal system. Precision is not allowed. A minus sign is placed on the left of negative numbers, a plus sign is not placed on the left of positive numbers.
  • x: Integer, is formatted in hexadecimal system with lower case letters. Precision is not allowed. No 0x is added.
  • X: Like x, but capital letters are used.
  • f: Fixed point representation, only for double. Precision specifies the number of decimal places. A decimal point is used as separator.
  • e: Exponential representation with small e. Precision specifies the number of decimal places.
  • E: Like e, but with large E.
  • g: Either fixed point or exponential representation, only for double. Precision specifies the total number of digits before and after the decimal point.
  • s: Symbolic representation. For bool types true or false is output. For enumerations, the name of the enumerator is output, if available; otherwise it acts like d.
