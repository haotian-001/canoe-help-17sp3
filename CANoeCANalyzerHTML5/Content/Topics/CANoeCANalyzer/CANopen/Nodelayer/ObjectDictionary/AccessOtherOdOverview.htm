Access to the Object Dictionary of other Nodes (deprecated)

 

Frequently an application is permitted to access the object dictionary of other nodes. This is especially necessary if data should be accessed that is not mapped in a PDO (or cannot be mapped).  The node layer makes a SDO client functionality available for this purpose. The user does not have to worry about the details of the SDO communication. Here the accesses always follow the same scheme:

  • Read access via SDO
    • Call of coUpload
    • Reading out of the data with coThisGet...() in the event function coOnUploadResponse
  • Write access via SDO
    • Call of coDownload or coDownloadExpedited
    • Successful completion of the transfer is signaled via coOnDownloadResponse

If during the transfer an error occurs, then the event handling is called in coOnError.

 

A SDO access to a node may only be executed if the previous SDO access is ended on the same node. Therefore it is not possible, e.g. to read out the mapping for a PDO in a loop. Several SDO accesses to various nodes can be initiated in parallel without limitations.

Example

int i;
dword errCode[1];
char errBuffer[256];
dword index = 0x1600;

for (i = 0; i < 9; i++) {
coUpload(nodeID, index, i, 0, errCode);
  if (errCode[0] != 0) {
    coGetLastError(errBuffer, elCount(errBuffer));
    write("Could not trigger SDO with coUpload...(%d): %#x - %s", nodeID, errCode[0], errBuffer);
  }
  else {
    write("SDO triggered with coUpload...(%d): (%#x/%#x)", nodeID, index, i);
  }
}

The example above will not work. Instead, the SDO access would have to be triggered thus:

Example

dword errCode[1];
char errBuffer[256];
byte  nodeID = 1;
dword Index = 0x1600;
dword SubIndex = 0;

on timer SdoRead {
coUpload(nodeID, Index, SubIndex, 0, errCode)
  if (errCode[0] != 0) {
    coGetLastError(errBuffer, elCount(errBuffer));
    write("Could not trigger SDO with coUpload...(%d): %#x - %s", nodeID, errCode[0], errBuffer);
  }
  else {
    SdoRun = 0;
  }
}

This call sequence can be placed in an onTimer event function. The timer itself is installed at the press of a key.

Example

on key 'u' {
  if (!SdoRun) {
    SdoRun = 1;
    setTimer(SdoRead, 0);
  }
}

In the event handling coOnUploadResponse the data is then available and the next SDO access can be started. The access to the read data occurs exclusively via the functions coThisGet...().

Example

void coOnUploadResponse(...) {
  dword errCode[1];
  switch (index) {
  case 0x1600:
    if (subindex < 10) {
      write("Read data from %d (%#x/%#x) : %#x", nodeId, index, subindex, coThisGetUnsigned(errCode));
      SubIndex++; // this is the global variable !!!
      SetTimer(SdoRead, 0);
    }
    break;
  }
}

In order to be able to access the read SDO data, there are special functions available. These functions are conceived exclusively for use in the event functions coOnUploadResponse and coOnDownloadIndication.

The user must take care that the correct function is used for the corresponding read value. For data of the type INTEGER/UNSIGNED and REAL, the functions coThisGetSigned, coThisGetUnsigned, and coThisGetFloat are available. For objects larger than 32 bits and string and domain objects, the function coThisGetData is used.

Write accesses via SDO are also subject to the same limitations as read accesses. In principle, a write access can also be programmed the same way (trigger in the timer). It must only be noted that the respective data to be written and its length must also be available. To trigger a download, two functions are available.

With the function coDownloadExpedited, an expedited download is executed. That is, the data to be written is already transmitted during the initiation of the transfer. The maximum number of data is limited to 4 bytes. For data quantities > 4 Bytes, the function coDownload must be used. Here it is possible to specify whether the transfer is executed as a "segmented transfer" or as a "block transfer."

If write access via SDO is successful, then the event function coOnDownloadResponse is triggered. If the write access was not successful, then in any case an error will be signaled via the event function coOnError.

Note

The CANopen node layer is still supported for capability with versions up to and including 9.0 SP6.

It was replaced by the CANopen Simulation Concept that is supported starting with CANoe 10.0.
