Access to the Local Object Dictionary (deprecated)

 

Generally there are two possibilities for accessing entries in the local object dictionary:

  • with the functions coODGet...()/coODGet...()
  • via environment variables (putValue()/getValue()), after a connection with coODConnectEnvVar was established.

Note

The CANopen node layer is still supported for capability with versions up to and including 9.0 SP6.

It was replaced by the CANopen Simulation Concept that is supported starting with CANoe 10.0.

Note

In order to simplify handling for the user, the functions mentioned above work synchronously. This means, they only return if the data could be written or read.

Based on the event character of CANoe, the simulation time only continues if an event occurs.

Therefore, it is strongly advised that you not call these functions multiple times (e.g. in a loop) within an event, e.g. onTimer. Otherwise there can be time jumps in the simulation procedure.

In order to simplify access to entries in the object dictionary, these can be connected with an environment variable via the function coODConnectEnvVar. Now access can occur via the functions putValue() and getValue().

Note

Environment variables must be created in a database.

This database is then assigned to the CANoe configuration.

For detailed information about environment variables see the CANoe help /1/.

A "mixing" of the access functions should be avoided. Thus after setting a value in the object dictionary with coODSet...() you can read the associated environment variable immediately with getValue(). However, it is not possible to read back a value of an environment value written with putValue() immediately with coODGet...(). CANoe requires a particular time span in order to update the data.

In principle, objects in the local object dictionary can also always be read and written by another node on the network. By connecting an object with an environment variable, it is possible to be informed about the write procedure (on envVar event), however, it is not possible to intervene in the transfer. This means that if the event of the environment variable occurs, the transfer is already complete. Thanks to this procedure it is not possible, for example, to make a value range check of the written data and to confirm or abort depending on the result of the transfer. In order to intervene in the transfer, various CAPL and event functions are available.

Note

Objects that should be handled this way must be created with the special access type 7.

This is only possible with the coODCreate...() functions. It is not possible to specify this type in an EDS or DCF file.
