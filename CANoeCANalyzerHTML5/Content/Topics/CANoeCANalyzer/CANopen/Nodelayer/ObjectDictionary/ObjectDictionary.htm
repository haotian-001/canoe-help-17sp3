Node Layer Object Dictionary (deprecated)

 

The functionality of the node layer can be configured largely via existing object dictionary entries both for the master and for slaves. If entries do not exist, then functionalities can also be completely switched off. Nodes that are simulated via the node layer should naturally be as close to reality as possible. Therefore, this section describes the behavior of the node layer with respect to CANopen.

Note

The CANopen node layer is still supported for capability with versions up to and including 9.0 SP6.

It was replaced by the CANopen Simulation Concept that is supported starting with CANoe 10.0.

General

The node layer can also be used if no objects are created in the object dictionary. All access to the object dictionary then returns the error "object does not exist." In most cases, at least one basic initialization of the object dictionary is necessary.

Required Objects

Some objects are required according to /2/ in order to fulfill the CANopen conformity.

Device Type

Device type

Index

0x1000

Type

Unsigned32

Description

This object contains the device profile number that is supported by this device. This object is required in order to support the defined boot-up in accordance with /3/.

Error Register

General error register

Index

0x1001

Type

Unsigned8

Description

This register depicts the current error state of a node. If the emergency message is supported, the content of this register is transmitted via the bus.

Identity Object

Identity object

Index

0x1018

Type

Array of Unsigned32

Description

The current node identification is located in this array:

  • subindex 1: Vendor ID (only this entry is required)
  • subindex 2: Product code
  • subindex 3: Revision number
  • subindex 4: Serial number

Furthermore, the corresponding entries must also exist in the object dictionary for the PDOs to be set up. The node layer supports the mapping of non-structured data types with a length of 1 - 64 bits (BOOLEAN, user-specific data types). Therefore, the mapping tables of the PDOs can include up to 64 entries. With a RPDO, the following communication parameters are supported:

  • COB-ID (CAN identifier)
  • Transmission type

In addition for these entries there is an

  • inhibit time for a TPDO
  • Event Timer

Handling of Synchronous PDOs

A PDO is handled as a synchronous PDO if the value for the transmission type lies in the range from 0 to 240 (inclusive). So that the SYNC message can be received, the object 0x1005 must exist in the object dictionary.

COB-ID Sync Message

COB-ID of the synchronization message

Index

0x1005

Type

Unsigned32

Description

This object contains the COB-ID of the synchronization message.

Bit 0..28

COB-ID

Bit 29

0 = 11-Bit ID

1 = 29-Bit ID

Bit 30

0 = the node does not generate the sync message

1 = the node generates the sync message

If the SYNC message should be sent, the object 0x1006 is also necessary.

Communication Cycle Period

Cycle time of the sync message

Index

0x1006

Type

Unsigned32

Description

This object contains the cycle time in microseconds with which the sync message is generated. The object is only required if the node should generate the sync.
