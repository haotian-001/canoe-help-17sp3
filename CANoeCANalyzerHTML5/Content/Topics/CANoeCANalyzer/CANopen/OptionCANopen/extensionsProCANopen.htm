Simulation of CANopen Nodes

 

Within the scope of the .CANopen option, CANoe offers the opportunity to use simulations of CANopen devices. The CANopen network to be simulated is set up and configured using the CANopen Configuration window. Then simulation nodes can then be generated automatically for each device in the Simulation Setup.

CANopen Simulation Concept

With CANoe you can simulate both individual CANopen devices and complete networks. A device configuration file (*.dcf, *.xdc) containing the current configuration is used for each simulated device. If a simulation node is assigned to a CANopen device, the simulation node behaves like a real CANopen device and sends a boot-up message at the start of a measurement, for example.

System Variables for the Object Directory

Each CANopen device has an object directory whose content is described in the device configuration file. Each object in the object directory is represented by a system variable in CANoe. These variables are created in the reserved CANopen namespace.

The system variables are structured by namespaces as follows:

  • CANopen – network name – node name.
    All objects in the object directory are created with a flat structure in the namespace of the node. Complex objects are created in a separate namespace, which is derived from the name of the main object. The variable name corresponds to the name of the respective object. The comment for the system variable contains the index, subindex, and name of the object, so that you can use the Symbol Explorer search, for example, to find a variable based on the object index. Note: The name generation is freely configurable in the dialog CANoe Options (CANopen Settings). If objects are required but missing in the configuration file, they are create with the suffix GENERATED. When SDO access occurs, the CANoe simulation automatically uses the value of the system variable. That means that in the event of read access via the bus (SDO Upload), CANoe responds with the value from the corresponding system variable. If write access occurs (SDO Download), the variable changes to the new value.
    If an object is mapped to a PDO via the TxPDO mapping, the corresponding PDO is sent, taking into account the send settings. Conversely, system variables are assigned new values that are read from a PDO on the bus via the RxPDO mapping. The mapping is part of the device configuration file and can be configured conveniently in the CANopen Configuration window. To do this, select a network and then open the PDO/SRDO Connections page.

  • Within the node namespace there exists also the namespace Control. It contains variables for controlling and monitoring the node state.
