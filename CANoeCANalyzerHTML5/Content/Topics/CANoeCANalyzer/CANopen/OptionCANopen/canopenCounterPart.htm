CANopen Counterpart

 

Introduction

If individual CANopen devices are to exchange data with CANoe or if only a part of a CANopen network is to be put into operation, these devices may lack Rx PDOs/SRDOs for correct functioning. On the page PDO/SRDO Connections you can use the button in the toolbar to create a counterpart. Hereby a new CANopen simulation node is created, whose object dictionary is built up so that the node sends required objects and receives sent objects.

Procedure

  1. In the CANopen Configuration window, first add the EDS files of all existing CANopen devices.
  2. Start the measurement and read the PDO/SRDO settings .
  3. Check in the PDO/SRDO Connections view whether all desired PDOs and SRDOs are valid. If necessary, use the button or to set or remove the valid flag. Note: The counterpart only considers valid PDOs or SRDOs that are not connected to other devices in the network.
  4. If you made changes in step 3, write them back to the respective devices .
  5. Stop the measurement.
  6. Now create the counterpart via the button . In the Simulation Setup and in the CANopen configuration a new node CANoe_Counterpart is created with a node ID that has not been used so far.
  7. Customize the transmission behavior of the PDOs/SRDOs as well as SYNC, Heartbeat and Master configuration. Especially for TPDOs the Transmission Type must be set to the required value.
  8. Now save the configuration and generate the DBC file via button for a complete display of the bus communication in the Trace Window.

Object Directory

Like any other CANopen node, the CANoe_Counterpart node has a DCF file with an object dictionary. The object dictionary contains all entries required for operation in a CANopen network such as heartbeat and for sending and receiving PDOs/SRDOs. Also for master functionality like NMT or CMT all objects are available.

For each PDO and SRDO of the other devices an entry starting with index 0x2000 is created in the object dictionary. For each mapped object in the PDO/SRDO a subentry is created. Here the definition of the original object is used, but the Access Type is inverted and the postfix _rev is appended to the name.

Update

If a counterpart has already been created, it can be updated after changes to the PDO/SRDO configuration, for example to send additional data. The object dictionary of the counterpart is completely rebuilt and all manual changes, e.g. to the heartbeat configuration, must be made again. It can be helpful to create a copy of the DCF file beforehand and merge the changes into the new file using a merge tool.
