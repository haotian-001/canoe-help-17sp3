VH5110 CCS Listener

 

Listening Strategy

The CCS Listener offers two ways to access a PLC (Power Line Communication) connection between a Charge Point and a Vehicle. The first way requires a direct connection to the CP (Control Pilot) as well as the PE (Protective Earth) lines of the Charging Plug. By using this way, the PWM signal can be measured.

The second way is more passive because it uses inductive coupling, usually a negative side effect of the PLC. By using this way, the PWM signal cannot be measured.

For being able to see any IPv6/UDP/TCP based communication, like SECC Discovery or V2G, the CCS Listener must monitor the SLAC protocol run already from the beginning. The SLAC protocol is responsible for establishing a network connection between the Vehicle and the Charge Point.

The CCS Listener is just forwarding all the traffic between the Vehicle and the Charge Point to the Host computer. It is neither capable of decrypting TLS secured communication nor can it be used to send any traffic on the network listened in on.

Active

To get the VsGetPWMStats.Ind the Mediatek chip must be configured with the Vector GreenPHY Configurator. To do this, set the Report Mode to Push and adjust the threshold values accordingly. Alternatively, these can be actively polled as follows.

The passive messages are received while the communication is in progress and do not require any special configuration. They contain general information about the communication process.

The active messages on the other hand need to be actively polled or at least configured to be requested once certain criteria are met, otherwise they will not be received. See also a code example how to get the active message VsGetPWMStats.

Passive Messages

Description

UnassociatedSta

Contains data about the networkID and CCO capabilities.

CcAssoc

The request contains data about the request type, the networkID, CCO capabilities and the proxy networking capabilities. The confirmation (confirm) on the other hand contains information about the result of the request, the networkID, the short network identifier and the lease time in minutes.

CcSetTeiMap

Contains data about the mapping mode used, the number of nodes, as well as their respective MAC addresses.

CmEncryptedPayload

Contains data about the selected encryption key for the payload, the status of the source AVLN, as well as the ID, run number and message number of the protocol.

CmStaIdentify

Contains data about the GreenPHY-, power save-, GP preferred allocation-, and repeating & routing capabilities, as well as the HomePlug AV version and extended fields length.

CmBridgeInfo

Contains data about the bridge STEI, bridging status flag, and the number of bridged destinations including their respective MAC addresses.

CcLinkInfo

The indication contains data about the current connection, like the connectionID, STEI, DTEI, forwardLinkID and ReverseLinkID.

CmHFID

The request contains data about the kind of request and some additional request data. The confirmation on the other hand contains data about the HFID result type and the HFID.

Active Messages

Description

VsGetPWMStats

The indication contains data about the frequency, voltage and duty cycle measured between the CP and the PE line. This feature requires a conductive connection to those lines.

System variables provided by Monitor.module

As of now only the values of the VsGetPWMStats message (frequency, dutyCycle and voltage) are mapped to CANoe system variables within the SCC Monitor module. These variables are in the CCS_Listener namespace.

Cross Reference

The monitor is needed to use the system variables of the CCS Listener.

Code - Polling of the PWMStats

Example

/*@!Encoding:1252*/
includes
{

}
variables
{
  const long cLengthOUI = 3;
  const long cLengthMAC = 6;
  const long cLengthMACString = cLengthMAC*2+(cLengthMAC-1)+1;
  const word cEtherTypeHomePlugAV = 0x88e1;
  byte cMediatekOUI[cLengthOUI] = {0x00, 0x13, 0xD7};

  char gMediatekReqDestMAC[cLengthMACString] = "00:B0:52:00:00:01";
  char gMediatekReqSrcMAC[cLengthMACString] = "20:00:00:01:01:01";

  _align(1) struct stMediatekGetPWMStatsReq
{
  byte MMV;
  word MMType;
  word FMI;
  byte OUI[cLengthOUI];
};

} // end of variables section

on key '1'
{
  Mediatek_SendGetPwmStatsReq();
}

/// <TX>
void Mediatek_SendGetPwmStatsReq()
{
  ethernetPacket pkt;
  struct stMediatekGetPWMStatsReq mme;

  pkt.source = ethGetMacAddressAsNumber(gMediatekReqSrcMAC);
  pkt.destination = ethGetMacAddressAsNumber(gMediatekReqDestMAC);
  pkt.type = cEtherTypeHomePlugAV;
  pkt.Length = __size_of(struct stMediatekGetPWMStatsReq);
  pkt.msgChannel = %CHANNEL%;

  mme.MMV = 0x01;
  mme.MMType = 0xA10C;
  mme.FMI = 0x0000;
  memcpy(mme.OUI, cMediatekOUI, cLengthOUI);

  pkt.SetData( 0, mme );

 output( pkt );
}
