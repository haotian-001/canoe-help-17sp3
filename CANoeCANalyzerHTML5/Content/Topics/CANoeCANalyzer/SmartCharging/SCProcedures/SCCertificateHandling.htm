Certificate Handling

 

A simulation of SmartCharging communication according to ISO 15118‑2 with PnC mode needs access to public key infrastructure. Depending on the simulated node (vehicle or charge point), and the features, one or more certificates and their keys must be available. These are configured using the Vector Security Manager and its PKI Security Source.

PKI Structure

A complete sample PKI according to ISO 15118‑2 is provided with the CANoe installation, which is titled SmartCharging Default PKI v2.0 (replacing the former SmartCharging Default PKI). The naming and structure is chosen according to ISO 15118‑2:2014 Annex E. The SmartCharging sample configurations are preconfigured to use this PKI for TLS and application layer security (XML signatures and encrypted key transfer). Both participants (vehicle and charging station) share the PKI in these sample configurations. It contains the required certificates for both sides.

Installation in a Counterpart

To enable security features with a real controller as counterpart, the controller must be configured to be part of the PKI. For that purpose, the certificates and keys of the SmartCharging sample PKI are also available as regular files. You can find them in the archive:

%ProgramData% \Vector\Security Manager\PKI\SmartCharging Default PKI.zip

Depending on the participant and the test scenario, different certificates and keys may be required. For a vehicle controller, the minimum set of files is:

  • OEMProvCertA_X_cert.c
  • OEMProvCertA_X_key.c
  • V2GRoot_X_cert.c

After installation of these, a Certificate Installation step can be executed to install a valid Contract Certificate and its associated key.

Configuration of Different Certificates

In order to configure different certificates than those provided, it is recommended to first create a new PKI security profile in the Vector Security Manager (please refer to the tool documentation for more information). Please make sure that private keys are available for all owned certificates of the simulated participant.

The new profile must then be assigned for the TCP/IP stack of the respective node using the Security Configuration dialog.

Moreover, the SmartCharging simulation module must be aware of the updated certificates. The certificate names, as given by the Name field in the Security Manager, must be supplied to enable the application layer to map the certificates to the required roles (e.g. the Contract Certificate, OEM Provisioning Certificate, â¦).

  • When using the Simulation Setup, the certificate names are part of the XML configuration of the simulation DLL.
  • When using the Communication Setup, the certificate names are specified in the SmartCharging Configuration dialog.
