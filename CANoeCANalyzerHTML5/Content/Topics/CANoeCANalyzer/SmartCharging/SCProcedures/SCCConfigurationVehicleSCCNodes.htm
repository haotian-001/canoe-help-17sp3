Configuration of Vehicle SCC Nodes

 

General information

An SCC entity is simulated by assigning one of the modeling libraries to a network node of an Ethernet network. The node then reacts automatically to incoming SCC messages according to its internal state machine. Its addresses, protocol configuration and the data to be sent and the must be configured by the user using one or more XML configuration files.

The parameters available are described below for the charge point, respectively for the vehicle.

Configuration of a vehicle

The vehicle parameters are configured using an XML file. The meaning of each parameter is listed below.

Example

XML example files accompany the supplied sample configuration. You can use them as the basis for your own configuration. They contain the most relevant parameters out of the list below.

The configuration file must be named according to the schema [Network name]_SCC_Vehicle.xml and be located in the directory of the loaded CANoe configuration.

Thus, different parameters can be defined for various networks of a CANoe configuration. In addition, multiple configurations can also be stored in the same file. You achieve this by integrating several PEVConfiguration elements, which must each have their own unique ID. Within the simulation, you can switch between the configurations using CAPL functions.

General Parameters

Parameter

Description

V2GProtocolVersion

Transport Protocol version to be used (decimal), default = 1

V2GTimeout

Timeout for a connection to be considered inactive, in milliseconds.

ErrorTimeout

Timeout after protocol errors in milliseconds before the vehicle restarts the protocol.

V2GTimeoutSettings

Behavior in case of Timeouts according to norm specification (EVCCMsgTimeout, EVCCCableCheckTimeout, EVCCPreChargeTimeout). In case of timeout: NoSpecTimeOut: Timeouts are igored NoBreakOnSpecTimeout: callback and Write window warning BreakOnSpecTimeout: additionally closes the TCP connection

V2GRetries

Number of retries for the communication after connection has been closed.

ChargePointDiscoveryRetries

Number of retries when sending a SECC Discovery Request (default: 4, which equals to 5 request messages in total)

ChargePointDiscoveryTimeout

Time between SDP retries, in milliseconds (default: 1000)

SessionStopOnError

If enabled, the vehicle will send a SessionStopReq message before terminating the protocol due to errors. This is a non- standard form of error handling. (default: 0)

StopOnVerificationError

If enabled, the charge point will stop the communication in active simulation mode when an invalid signature is received.

ServiceScope

Service Scope string to use for the Service Discovery Request (ISO 15118‑2)

PreferredPaymentOption

Payment option to select if offered by the charge point (Contract or ExternalPayment). Contract selects PnC mode.

ScheduleEntrySelectionMode

Defines which schedule entry (SAScheduleTuple) is selected (first for the first tuple, latest to select the tuple with the highest ID)

SchemaNamespace

SchemaVersionMajor

SchemaVersionMinor

Default schema to be requested in the SupportedAppProtocolReq message, which determines the target protocol version. Also defines the schema to use if no protocol handshake takes place (free sending).

PublishAllSchemas

If enabled, the vehicle will send all other supported schemas in the SupportedAppProocolReq, in addition to the chosen schema.

InvalidValueSigned

32 bit value to be returned when a missing optional message parameter is queried (default = -1, applies to types long and float)

InvalidValueUnsigned

32 bit value value >= 0 to be returned when a missing optional message parameter is queried (default = 0xFFFFFFFF/4294967295, applies to type dword)

NoncriticalShutdownBehavior

Set default value for ChargingSession in SessionStopReq when responding to EVSENotification = StopCharging. Possible values: pause, terminate. Default = pause. (ISO 15118‑2)

Connection Parameters

Parameter

Description

PEVUDPPort

The port used for UDP. Set to auto for random port. (default: auto)

PEVTCPPort

The port used for TCP. Set to auto for random port. (default: auto)

UseTLS

If set to 1, the SECC Discovery Request will be sent with security option TLS and the vehicle will try to establish a TLS connection.

Number of WeldingDetectionReq messages to be sent (default: 1)

Number of PreChargeReq messages to be sent. If set to AUTO (default), the vehicle will continue sending PreCharge messages until voltage and current provided by the EVSE match the requested values.

MaxEntriesSAScheduleTuple

Maximum number of SAScheduleTuples (remove to disable sending)

DepartureTime

Planned time of departure in seconds from now (remove to disable sending)

Note

The XML structure and data types of the configuration parameters match their definition within the SCC specification in the context of the associated SCC messages. In particular, for float types, the float representation according to the specification must be used.

SLAC Parameters

Parameter

Description

UseSLAC

Configures default SLAC handling for both active and passive mode. 0 = SLAC disabled 1 = SLAC enabled 2 / auto = use SLAC depending on link status (not recommended due to HomePlug Green PHY chip instabilities).

PEVMACAddress

MAC address to be used as source address for SLAC frames. This should equal the address configured for the adapter / the CANoe node.

SLAC_MACFilter

Defines a MAC address whose messages will be ignored. This parameter can be contained multiple times in a configuration.

SLAC_AttnThresholdMin

Maximum averaged attenuation to consider a Charge Point as Found.

SLAC_AttnThresholdMax

Maximum averaged attenuation (>SLAC_AttnThresholdMin) to consider a Charge Point as Potentially found. Only relevant for the validation process.

SLAC_AttenuationTx

Attenuation of the Tx path to be subtracted from the AAG values. This value may be negative and have fractional digits, e.g. -0.5. Recommended value: 7.0.

SLAC_RunID

User configured ID (8 byte) for SLAC sessions. If not defined, IDs will be random.

SLAC_UseValidation

Enables or disable the possibility to validate Charge Points considered Potentially found. If 0 (default: 1)

Note

  • The following validation parameters apply only in combination with SLAC_UseValidation=1.

SLAC_ForceValidation

Validate even if EVSE answers with not required. (default: 0)

SLAC_ConcurrentValidationBehavior

Configures EV to either retry the validation on current or iterate to next EVSE if EVSE answers with Not Ready. (default: retry).

SLAC_FallbackValidationFailedBehaviour

Configures EV whether to skip the validation or to continue with the next EVSE if EVSE answers with Failed (default: continue).

SLAC_ValidationTimer

Timer value for the validation, if enabled, in multiple of 100 milliseconds.

SLAC_AmpMapDelayTime

Defines the delay for sending a CM_Amp_Map.Req after a link has been detected. Default = 50 ms.

Note: Only applies if a PSD is configured with SCC_SLAC_SetPSD()

Note

  • The following link status parameters are marked as deprecated. It is recommended to use VS_Host_Action.Ind as described in SCC_Generic_VS_HostMessage_Ind instead.

SLAC_LinkStatusPollingType

Determines the message(s) for querying the link status. Possible values are:

  • All (0) : use both available messages
  • PlLnkStatus (1): use only VS_PL_Lnk_Status
  • NwInfo (2): use only VS_Nw_Info

  • NwInfoWithPing (3): use two-step polling of the remote station using VS_Nw_Info (recommended)

SLAC_LinkStatusPollingInterval

Interval in ms for the sending of VS_PL_Lnk_Status messages to query the link status. If set to 0, polling is disabled. (default: 100)

SLAC_LinkStatusDebounceTime

Debounce time in ms for switching the link status (default: 0).

SLAC_UseSTPLinkStatus

Enables usage of STP_Link_Status (vendor specific) messages instead of VS_PL_Lnk_Status.

SLAC_DelayTime

Duration in milliseconds that the vehicle waits after the SLAC process, before sending the SECC Discovery Request (default: 0). Only applied if link status polling is disabled.

SLAC_ChipMACAddress

MAC address of the local HomePlug Green PHY chip. If provided, it is used to filter link status messages, to ensure that no messages for other senders are interpreted.

SLAC_ChipMagicAddress

Generic address used to address the HomePlug Green PHY chip. (default: 00:B0:52:00:00:01)

Note

  • The HomePlug Green PHY chip may need to reset after setting a new NMK. During this interval, no communication is possible. You can either use link status detection to automatically continue the communication when the link is established, or simply bridge this gap by using SLAC_DelayTime.
  • Although the validation messages can by sent by the EV simulation, no actual validation takes place at the moment. The simulation will always accept the result sent by the Charge Point.

Security Parameters

The following elements are only required for the ISO 15118‑2 PnC profile. Certificates are referred by their Name property as displayed in the Vector Security Manager after importing the certificate.

Note

Only leaf certificates are referred in the XML config. Please make sure that the base certificates are also part of the Security Profile where full chains are required, as they will be resolved automatically using the Signer ID.

Parameter

Description

ContractID / eMAID

Contract ID which is sent along with the certificates (may be overwritten by a CertificateInstallation response).

TLSClientCert

(Certificate for initiating a TLS connection only if Client Authentication is required).

TLSHostCert

Certificate of the TLS host, if known.

ContractCert

Contract certificate of the vehicle.

MOSub2Cert

Issuer certificate (the first SubCertificate in the ContractSignatureCertChain) for verification of the SalesTariff.

OEMProvisioningCert

OEM provisioning certificate to be used for the certificate installation process.

ListOfRootCertificateIDs

List of root certificate IDs to be sent during certificate installation. If not defined, these are read from the security profile.

UseCertChainVerification

Deactivates the certificate chain check. Deactivating this feature can help resolve performance issues.

AllowAsyncCertVerification

Certificate (chain) verification by default is done synchronously in Simulated Bus mode and asynchronously in Real Bus mode. You can force the method with this switch.

Note: Setting this to 1 in Simulated Bus mode may cause timing issues if run As fast as possible or with an animation factor faster (smaller) than 1, because the simulation may then run into a message timeout (fast simulation time) before receiving the verification result (real time).
