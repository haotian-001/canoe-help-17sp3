FAQ â General hints

Using CANoeâs XCP option customers have regularly problems with the A2L file.
In most of the cases the used A2L file does not fit to the XCP_SLAVE (ECU), because the file is not up to date/does not contain all required information.
CANoe requires a 100% appropriate A2L file; CANape does not.
CANape trusts the parameters and commands of the ECU. This means, (almost) any A2L file can be used in CANape and missing parameters and commands will be added automatically by CANape. So CANape can work with incomplete A2L files.

Please consider to use an A2L file which fits to the ECU 100%.

  • ADDRESS_EXTENSION
  • ACTION_SERVICE

FAQ â The Seed & Key DLL could not be loaded

  • Cause: The DLL was linked with another DLL or a manifest, which could not be found on the target system.
    Solution: Link the DLL statically, do not create a manifest (Visual Studio settings under Linker and General).
  • Cause: The DLL does not correspond to the checksum calculation API according to the ASAM Standard.
    Solution: Adapt the DLL according to the ASAM Standard.

  • Cause: The DLL is an old checksum-DLL or an incompatible DLL.
    Solution: Old checksum-DLLs or incompatible DLLs are not supported.

  • Cause: Inside the DLL a crash occurs. Due to this CANoe cannot load the DLL.
    Solution: A measurement is not possible with this DLLs.

FAQ â Upload command is locked (Seed & Key on demand)

The Upload command does not work.

Cause: CANoe assumes that the calibration resource is unlocked before the Upload command.
Solution:
Workaround until CANoe 7.6 SP3: Trigger the calibration connection using a Download command after CONNECT.
In CANoe 7.6 SP3 there is an option Seed & Key on Demand in the XCP/CCP Configuration dialog under Device Configuration|Configureâ¦|Extended Settings. With this all resources are unlocked with the CONNECT.

FAQ â CONNECT does not work. When changing the BYTE-ORDER (e.g. Intel > Motorola) then it works.

Cause: In CANoe 7.5 SP2 the addressing bytes were interchanged by mistake.
Solution: This is resolved in recent versions.

FAQ â Upload of GetID does not work

Following error message/ warning appears in the Write Window:

  • XCP: Error in device TGW: Upload generic BSW-service failed. Request "GetId" has undefined state
  • XCP: Communication to device TGW stopped, because no upload of generic measurement information was possible

Cause: There is a malfunction in CANoe 7.6 SP3.
Solution: Update to 7.6 SP4.

FAQ â DAQ measurement does not work

Different error messages/ warnings appear in the Write Window:

  • Protocol initialization warning: Optional commands to allocate dynamic daq lists are not provided. Daq mode will not be available
  • Protocol initialization info: Required optional commands to allocate dynamic daq lists are: FREE_DAQ, ALLOC_DAQ, ALLOC_ODT and ALLOC_ODT_ENTRY
  • Error in device ECU_2: Couldn't get Start Daq-service from protocol -> Request rejected
  • Cause: CANoe uses the DAQ type as defined in the A2L file. Please ensure that the corresponding DAQ type is supported by the ECU. If the DAQ measurement does not work the wrong DAQ type is defined in the A2L file.
    Solution: Please change the DAQ type in the A2L file. The DAQ type could be either DYNAMIC or STATIC.

/begin DAQ
DYNAMIC 0x0 0x3 0x1 OPTIMISATION_TYPE_DEFAULT
ADDRESS_EXTENSION_FREE IDENTIFICATION_FIELD_TYPE_ABSOLUTE
GRANULARITY_ODT_ENTRY_SIZE_DAQ_BYTE 0x7 OVERLOAD_INDICATION_PID
PRESCALER_SUPPORTED
/begin DAQ_LIST
...
/end DAQ_LIST
...
/end DAQ

SET_MTA does not work, or rather after changing the BYTE_ORDER in the A2L file SET_MTA does not work correctly or does not work anymore.

  • Cause/Solution: The BYTE_ORDER on the bus (SET_MTAâ¦) has to be defined in section TP_BLOB.

Note

Often the attribute MOD_COMMON was changed by mistake. With this only the interpretation of the measured data can be defined.
  • Cause: In CANoe 7.5 SP2 there was a malfunction in combination with CCP and MOTOROLA.
    Solution: Update to 7.5 SP3.

FAQ â Intel/Motorola

During start following error messages/ warnings are displayed in the Write Window:

  • XCP: Couldn't initialize protocol layer. Reason: Illegal byte ordering configuration

  • XCP: Protocol initialization Warning: Parameter 'ADDRESS_EXTENSION' is not yet supported. Daq generation will NOT consider this option.

  • Couldn't initialize protocol illegal byte ordering configuration

Cause: The A2L file contains an invalid value for the BYTE_ORDER, e.g. 3.
Solution: Possible are 1 and 2 (Motorola and Intel).

FAQ â ODT Settings

In CANape the option ODT_ENTRY_ADRESS_OPT_DISABLED can be set in the canape.INI file. The same option is required for CANoe.

Cause: ODT_ENTRY_ADRESS_OPT_DISABLED cannot be disabled in CANoe.
Solution: One option to change the address transmission is to set the Consistency parameter = ODT in the XCP/CCP-Configuration dialog under Device Configuration|Configureâ¦|Extended Settings|Protocol Settings. Then addresses are not disposed via multiple messages.
