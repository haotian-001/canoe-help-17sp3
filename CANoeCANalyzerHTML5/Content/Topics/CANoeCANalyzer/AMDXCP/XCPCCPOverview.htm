XCP and CCP Functionality in CANoe

 

There are two options for linking XCP/CCP in CANoe:

CANoe Option Option AMD/XCP

  • The XCP protocol (XCP on CAN, XCP on CAN FD, TCP, UDP, FlexRay and LIN) and the CCP protocol are supported. Here:
  • Values can be written.
  • Values can be read out via request/response. The cyclical polling time can be configured separately for each parameter.
  • Values can be read out via the DAQ measurement (Data Acquisition Mode, free-running measurement data acquisition). The measurement mode can be configured separately for each parameter.
    Not available for XCP on LIN.
  • Time stamps are used for the parameters measured via DAQ
  • Realtime-capable coupling â The reading out and writing of values occur directly in CANoe. CANoe represents the XCP master here.
  • The distributed mode and VN8900 are supported.

An A2L file is configured in CANoe for the .XCP option. The XCP or CCP parameters required during measurement in CANoe must be configured beforehand in the parameter list. A system variable for read and write operations are automatically available for every configured parameter. It can be used in test modules as usual. No CAPL code whatsoever is required.

The focus of the .XCP option is the testing of control units. Scalar value types (e.g. integer or float ), two-dimensional value blocks (arrays) and maps and curves are supported here.

Note

All value blocks are represented as one-dimensional array system variable regardless of their orientation defined in the A2L file. I.e. the order of the elements represents the order in the ECU's memory.

New Technology available!

Merely due to compatibility reasons ASAM MCD3 via CANape is currently still available.
Please use CANoe option .AMD/XCP instead.
  • The CCP and XCP protocols are supported. Here:
  • Values can be written.
  • Values can be read out via the DAQ measurement (Data Acquisition Mode, free-running measurement data acquisition).

You can map the values to system variables automatically by calling up the MCDMapECUParamToSysVariableRead and MCDMapECUParamToSysVariableWrite functions once for each MCD parameter in CAPL. The system variables are to be created separately for the read and write directions.
The two CAPL functions cannot be called up for values from DAQ measurements until after the DAQ measurement is started.

If the MCD parameters were mapped to system variables as described, they are available in test modules for read and write access or monitoring as usual.

  • Values can be read out via request/response.
    CAPL commands are required for this purpose. They can be called for cyclical procedures via a CAPL timer as well
  • "Not" realtime coupling, as the calls to the MCD3 server are swapped out asynchronously
  • The distributed mode is not supported.

For more information on the MCD3 server link, see ASAM-MCD3: Overview.
