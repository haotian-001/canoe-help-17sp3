Device Configuration

 

If you have selected a device in the tree view on the left side, all XCP/CCP-relevant setting options are displayed, grouped according to subject, on the right side. You can change the settings or reset them to the default value with in the toolbar.

General

  • ECU Qualifier: Here you have to enter a unique ECU name. With this name the ECU is identified within CANoe.
  • Active: Here you can activate/deactivate the device for the following measurement. If the device is completely deactivated, here, you cannot re-activate it during the measurement; also XCP CAPL commands won't re-activate it.
    You can also activate/deactivate the device via the context menu.
  • Observer: Here you can activate/deactivate the protocol observer.
  • Network: Select the network to which the ECU is connected.
  • CAN FD: If the channel is operated in CAN FD mode, you can set here whether CAN or CAN FD messages are used for the XCP communication.
  • Use Bitrate Switch: Here you can set whether the higher data rate should be used for CAN FD messages.
  • CCP Version: Here you can switch between the CCP versions 2.0 and 2.1.
  • TCP/IP Stack: Here you can select from a list of all TCP/IP stacks of the Ethernet simulation nodes via which the XCP connection should be established.
  • Protocol: Here the protocol is displayed.
    • In case of XCP on CAN, you can activate the CAN FD function and after that also the Use BRS check box for the CAN FD Bitrate Switch, to use a higher bitrate during the data phase of an CAN FD message.
    • In case of CCP, you can switch between the CCP versions 2.0 and 2.1.
  • Database: The selected A2L file.
  • Instance Name (XCPplus): Displays the instance name of the transport protocol (only for A2L files that use XCPplus).

Protocol Settings

Note

The additional protocol settings are read from the A2L file and may be reset to these settings by clicking the [Reset] button, if you manually changed them.
  • XCP on CAN:
  • Response Timeout: Amount of time that the XCP master waits for a response from the XCP slave. Set this value according to the response time of your control unit.
  • Request ID: CAN ID that the XCP master uses for sending.
  • Response ID: CAN ID that the XCP slave uses for responding.
  • Use max. DLC: If this setting is True the maximum message size is used.
  • XCP on Ethernet:
  • Response Timeout:Amount of time that the XCP master waits for a response from the XCP slave. Set this value according to the response time of your control unit.
  • Host (IPv4, IPv6 or Comuter Name): IPv4, IPv6 address or computer name of the XCP slave. The appropriate type is detected automatically.
  • Port: Here you can set the ports for master and slave (TCP/UDP Ports used by CANoe).
  • Slave
  • Port (Rx): Port on which the XCP slave listens.
  • Port (Tx): Port on which the XCP slave transmits.
  • Master
  • Port: Port that is used by CANoe for the communication.
  • XCP on FlexRay:
  • Response Timeout: Amount of time that the XCP master waits for a response from the XCP slave. Set this value according to the response time of your control unit.
  • Node Address: Node address of the XCP slave.
  • ECU Tx Queue: Select this option if the XCP device contains a transport layer buffer for optimized use of the bandwidth.
  • CCP
  • Use SET_S_STATUS: Here you can select whether SET_SESSION_STATUS command should be sent or not.
  • off: The SET_SESSION_STATUS command will not be sent.
  • Cal: The SET_SESSION_STATUS command will be sent, if the RAM Page Switching is also active.
  • Daq: The SET_SESSION_STATUS command will be sent, if DAQ lists are used.
  • CAL + DAQ: The SET_SESSION_STATUS command will be sent with active RAM Page Switching or when DAQ lists are used.
  • RAM Page Switching: If this option is activated, CANoe sends the SET_CAL_PAGE command to the ECU during connection establishment. That way the ECU is switched to the RAM page. CANoe.XCP only uses parameters that are located in the RAM page.
  • Connect Mode: The value of parameters Mode of the CONNECT command.
  • Check EPROM Identifier from Database
    If this setting is True the EPK identifier that is defined in the A2L file is compared with the values of the ECU at the specified memory addresses. This is done to ensure that the A2L file and the ECU software match. For this, an UPLOAD of the specified addresses is executed after CONNECT. The returned values are then compared with the EPK Identifier from the A2L file. If the check was activated and one of the values does not match the EPK Identifier, a DISCONNECT is executed.
  • Optional Commands: Activates/deactivates the usage of optional protocol commands.

DAQ Settings

  • Use DAQ time stamp of the ECU
    If you activate this option, system variables receive the time stamp supplied from the ECU when an update is made via XCP in DAQ mode. If the option is deactivated, system variables receive the bus time stamp.
    This option must be activated if a transport frame contains several DAQ packages with the same measurement value.
  • Time Stamp Calculation: In earlier versions of the XCP specification, it was unclear whether the calculation of the timestamps was to be multiplied by the number of ticks or divided by the number of ticks. Therefore you can choose between multiplication and division. Currently, only multiplication is used.
    The resolution of the counter (ctr) and the number of ticks are taken from the A2L file.
  • ECU delay: CANoe tries to calculate the response time of the ECU to the request for the current value of the DAQ time stamp. Here you can set in percent which part of the duration between request and response should be interpreted as processing time in the ECU. 0% means that the time stamp is taken immediately after the request. 100 % means that the time stamp is taken when the response is sent.
  • Enable drift compensation: To avoid a deviation of CANoe and ECU time the time is redetermined once per second.
  • DAQ Timeout: Timeout detection for DAQ lists. Amount of time that the XCP master waits for a DAQ list from the XCP slave. In case of a timeout the connection will be detected as interrupted.
  • Consistency Mode: Consistency setting for DAQ lists. Defines how DAQ lists are organized if the number of data bytes exceed the available bytes in the DAQ frames. Possible values:
  • Event
  • Daq
  • Odt
  • DAQ Configuration
    In this dialog you can
  • Display DAQ events from A2L files or add , edit and delete user defined DAQ events.
  • The following functions are available for DAQ lists:
  • XCP:
    If you select the dynamic usage of the DAQ lists, CANoe uses the available DAQ lists of the ECU automatically.
    If you select the static usage of the DAQ lists, you can display DAQ lists from A2L files or add , edit and delete user defined DAQ lists.
    If you select the predefined usage of the DAQ lists, CANoe uses the DAQ lists from the A2L file.
  • CCP:
    If the Automatic Detection option is active, CANoe uses the DAQ lists 0-254 automatically.
    With deactivated Automatic Detection option you can display DAQ lists from A2L files or add , edit and delete user defined DAQ lists.

With you set the user defined settings to the settings of the A2L file.

  • Check DAQ Configuration
    Checks whether the DAQ lists/DAQ events described in the A2L file are present in the ECU. If a non-existent DAQ list or DAQ event is used in the parameter list, the connection setup is aborted and corresponding messages are displayed in the Write Window.

Technical Details

The response of the following commands is evaluated for the check:
GET_DAQ_PROCESSOR_INFO, GET_DAQ_EVENT_INFO and for non-dynamic configurations also GET_DAQ_LIST_INFO.

For GET_DAQ_PROCESSOR_INFO the parameters MAX_EVENT_CHANNEL (number of event channels) are checked and for non-dynamic configurations also MAX_DAQ (number of DAQ lists).

For GET_DAQ_EVENT_INFO the parameters MAX_DAQ_LIST, EVENT_CHANNEL_TIME_CYCLE, EVENT_CHANNEL_TIME_UNIT, EVENT_CHANNEL_PRIORITY and DAQ/STIM-Support are checked.

For GET_DAQ_LIST_INFO the parameters MAX_ODT, MAX_ODT_ENTRIES, FIXED_EVENT and DAQ_LIST_TYPE DAQ/STIM are checked.

  • XCP on FlexRay
  • Minimum Buffer Count: Minimum number of requested FlexRay buffers
  • Additional Buffer Count: Additional number of requested FlexRay buffers to compensate transmit delays inside the ECU.
  • ODT Optimization: If possible, several parameters that are stored in series in the memory are configured in series in a common, large ODT.
    This optimization is not possible for devices whose XCP address is not related to the internal memory structure (for example if the XVP address is used as handler to identify an object).

Security Settings

  • Seed & Key Path: Path to Seed & Key DLL.
  • Seed & Key Active: To activate the Seed & Key mechanism, you must activate this option.
  • Seed & Key on Demand: If this check box is activated, in case of CCP the Seed & Key function is only executed when it is needed for a command. This function does not affect to a XCP connection.

Connection Settings

  • Connect on Measurement Start: In the Measurement Start dialog you can set if the CCP/XCP connection is established immediately after the start of measurement or with the set delay. Alternatively you can initiate the connection setup via the xcpConnect CAPL command or manually in the XCP configuration dialog.
  • Disconnect on Measurement Stop: If you deactivate this option, CANoe doesn't stop the XCP/CCP connection automatically on measurement stop. You have to stop the measurement manually or via CAPL.
  • Connection Attempts: On measurement start and on automatic connection after connection errors.
  • Allow Reconnect After Connection Error: This function activates automatic reconnection after a connection error. A connection error occurs when an upload or download could not be performed correctly, for example. If the connection has still not been established after this number of connection attempts, communication is disabled for the remainder of the measurement.

System Variables Settings

  • Reset Variables After Disconnect: If activated, system variables are reset to 0 after disconnect (this setting has no effect on axes of type FIX_AXIS).
  • Disable Updates for Higher Throughput: If activated, an event is only generated if the value of a system variable is changed.
  • Hierarchical Partial Identifiers: System variable namespaces are used for partial identifiers of the parameter's name.
  • Extended Data Types: Enables support of 64 bit as well as unsigned system variable types.
