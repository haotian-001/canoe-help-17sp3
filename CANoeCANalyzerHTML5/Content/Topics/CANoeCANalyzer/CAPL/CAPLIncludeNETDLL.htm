.NET DLL

 

From CAPL, it is possible to call functions which have been implemented in a separate .NET DLL. For this, the function must fulfill the following requirements:

  • It must be defined as public in a public class
  • It must be static
  • Its name must be a valid CAPL identifier
  • Neither the function nor its class must have generic parameters
  • Its return type must be void, an integer type, bool or double
  • All parameters must be integer types, bool, double, string or one-dimensional arrays from integer types or double.

The data type bool is mapped to DWORD in CAPL. The function is called in the same way as a CAPL function. The name, however, must be qualified with a class name and, if necessary, with a namespace.

Example

.NET Function (in .NET DLL)

namespace MyNetHelpers {
  public class MyClass {
    public static Int32 SomeFunc(String s, out double d) { /* ... */ }
  }
}

Function Call in CAPL

double d; long ret;
ret = MyNetHelpers::MyClass::SomeFunc("Vector", d);

It is not possible to look into the NET function from the debugger because CANoe does not compile the .NET DLL and the source files are not known to the debugger. Test and debug the .NET DLL with Visual Studio, ideally by means of a unit test outside of CANoe.

Note

The execution of .NET code always influences the realtime capability of the program. Especially during the first execution (by loading the IL code and just-in-time compilation) and during garbage collection, delays may arise. Additional delays may occur during the measurement setup since a special AppDomain will be used for the .NET DLLs.

Configuration of .NET DLL

To enable the CAPL compiler and the CAPL Browser to detect the DLL, they must be included in the CAPL program.

For this, the following procedures are available:

  • Enter the DLL on Programming/CAPL DLL page (Options dialog) of CANoe.
    In this case, the DLL will be available to all CAPL programs.
  • You can enter the DLL in the includes section of a CAPL program using the #pragma netlibrary command.
    In this case, it will only be available to this program.
  • You can indicate the DLL in the configuration dialog of a simulation node or a CAPL test module in CANoe on the page Components.

The platform target set for the .NET DLL must be Any CPU or the architecture adapted to the node. In the measurement setup, this is normally x64. In the simulation or test setup, it is x86.

The target framework of the .NET DLL should be .NET Framework 4 or higher.

Using Additional Assemblies (References)

You can use other .NET assemblies from the .NET DLL by adding them to the .NET project as references. In addition, you should configure these .NET assemblies also in CANoe or per #pragma netlibrary so that they are taken into account, e.g.

  • for the packed transmission of the configuration
  • when using the distributed mode or the standalone mode.

Exceptions are .NET framework assemblies or CANoe assemblies (see below).

Moreover, Copy Local in the properties of the reference should be set to true so that they are copied to the directory of the .NET DLL.

Using CANoe .NET API

You can also use the CANoe .NET API in .NET DLL. For this, the assemblies Vector.CANoe.Runtime, Vector.CANoe.Runtime.Internal, Vector.Tools and Vector.Tools.Internal must be added manually as references to your .NET project. It is not necessary to configure these assemblies additionally in CANoe or per #pragma netlibrary. They also have not to be copied into the directory of .NET DLL. If you want to use also the types generated for databases, it is possible for you to generate the type libraries on the page Programming / .NET (Options dialog) of CANoe and add them as further references.

The attributes On… as a reaction to simulation events do not work in such DLLs. Instead of this, use the respective CAPL handler to call .NET functions, if required.

Note

When linking your own .NET DLLs in CAPL programs, all methods are available to you of accessing system resources (Memory management, hardware, etc).

However, only experienced programmers should make use of this option due to the multitude of associated problems.

We ask for your understanding that Vector is not able to provide any support for creating .NET DLLs.

Note

.NET-DLLs can be used by CAPL from CANoe version 12.0 SP3.
