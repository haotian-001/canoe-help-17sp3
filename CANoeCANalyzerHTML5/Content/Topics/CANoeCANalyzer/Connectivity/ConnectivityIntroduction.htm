Use Cases and Certificates

 

Getting Started

To use the cloud functionality of the Connectivity Features Set (CFS), a so-called owner certificate is required. This can be acquired through Support. No certificates are required for the local use case.

For this purpose, please contact Vector Support. Following this, you’ll receive an owner certificate.

For information on how you must configure CANoe, refer to Configuration: Connectivity.

Use Cases

What is the CFS for?

The CFS extends the connection functions of CANoe. With the CFS you currently have the possibility to connect Connected Devices to CANoe via the MQTT protocol, for example to simulate and test Internet-of-Things (IoT) use cases. This connection can be made either via a local broker (see use cases 2,3) or via the Connectivity Hub, which is operated by Vector (see use case 1). The Connectivity Hub provides each Vector customer with a dedicated MQTT broker, which can be used free of charge to a certain extent. (For details, see section Getting Started above).

Use Case 1 – Connected Device Cloud Communication

A connected device connects to an MQTT broker run in the Connectivity Hub. If some sensors are not available, they can be simulated with CANoe in order to stimulate the connected device and check whether the response occurs in the form of a PUBLISH message, for example. This message can be received in CANoe and further processed as an application message.

Use Case 2 – “Local” Broker Device

In this use case, a device connects to a local broker running on a laptop or connected device, for example.

Use Case 3 – CAN -> MQTT Gateway

Here CANoe acts as a gateway to send data from various sensors, implemented e.g. by a LIN/CAN ECUs, to the broker in the Connectivity Hub or to the local broker. This data can then be processed by a connected device.

Certificates (General)

Certificates are used to enable communication partners identify itself and thus for its authenticity to be checked. Secure TLS connections can be established using certificates. This prevents the data from being easily read and manipulated by third parties.

Various different formats for certificates exist, such as PEM and PFX.

PEM

The certificate is represented by a text file that is Base64 coded in a text file. The extension is *.pem. You can also open this file in a text editor. With this format, the certificate and key are saved in two separate files.

PFX

This represents a binary format. It contains all certificates and the private keys. The data is protected with a password.

Certificates (CANoe)

CANoe follows the approach of mutual authentication. This means that CANoe must identify itself to the Connectivity Hub with a certificate and the Connectivity Hub functioning as a server must identify itself to CANoe. This also applies for the System Under Test (SUT).

For this reason, three certificates are required for using the MQTT broker in the Connectivity Hub:

Owner Certificate:

The owner certificate is required for creating the following two certificates.

CANoe Certificate:

This certificate establishes the connection to a created instance (i.e. a CANoe configuration) and is used to establish a data channel to the Connectivity Hub. This certificate is required for CANoe.

SUT Certificate:

To connect an SUT to the Connectivity Hub, this certificate is required. It must be installed on the SUT to establish a connection to the Connectivity Hub.

Note

A PEM format as well as a PFX format for the CANoe Certificate and SUT Certificate are always provided when creating an instance. The PFX certificates are imported directly into the Windows Certificate Store for the current user. The PEM formats are defined by a path which can be set under Configuration: Connectivity.
