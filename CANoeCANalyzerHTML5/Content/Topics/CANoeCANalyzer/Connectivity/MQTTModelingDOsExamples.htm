MQTT Modeling with DOs - Example

Data Models and Application Models

With the following example a simple client with a PUBLISH and a SUBSCRIBE can be modeled. For this purpose the model is represented in vCDL and an application model in C#, in CAPL and in Python.

vCDL

version 1.4;

import module "MQTT";

namespace ClimateSimulation
{
  //----------------------------------------------------------------------------------------
  //Modulation of the Fan which is consuming the temperature and publish the rotation speed
  //----------------------------------------------------------------------------------------
  namespace FanSimulation
  {
    interface IFan
    {
      [Binding="MQTT"]
      [MQTTBinding::ClientId="CANoe"]
      [MQTTBinding::Topic="rooms/living/devLowerLeft/temperature"]
      consumed data int64 ConsumedTemperature;

      [Binding="MQTT"]
      [MQTTBinding::ClientId="CANoe"]
      [MQTTBinding::Topic="rooms/living/devFan/Speed"]
      provided data int64 ProvidedSpeed;
    
  }

    object Fan1 : IFan;
  }
  //------------------------------------------------------------------------------------
  //Modulation of the temperature sensor
  //--------------------------------------------------------------------------------------
  namespace TemperatureSimulation
  {
    interface ITemperatureSensor
    {
      [Binding="MQTT"]
      [MQTTBinding::ClientId="CANoe"]
      [MQTTBinding::Topic="rooms/living/devLowerLeft/temperature"]
      provided data int64 ProvidedTemperature;
    }

    object TemperatureSensor : ITemperatureSensor;
  }
}

C#

using System;
using Vector.Tools;
using Vector.CANoe.Runtime;
using Vector.CANoe.Sockets;
using Vector.CANoe.Threading;
using Vector.Diagnostics;
using NetworkDB;
using NetworkDB._SystemDataTypes;

using IServiceProvider = Vector.CANoe.Runtime.IServiceProvider;

public class MQTTTester : MeasurementScript
{
  private Timer mTimer = null;

  public override void Start()
  {
    mTimer = new Timer(TimeSpan.FromMilliseconds(1000), new EventHandler<ElapsedEventArgs>((o, a) =>
    {
      SendValues();
    }));
    mTimer.AutoReset = false;
    mTimer.Start();
  }

  // Sending sensor data
  private void SendValues()
  {
    ClimateSimulation.TemperatureSimulation.TemperatureSensor.ProvidedTemperature.Value = 20;
  }

  // Receiving sensor data
  [OnUpdate(ClimateSimulation.FanSimulation.Fan1.MemberIDs.ConsumedTemperature)]
  public void OnTemperatureReceivedOnFan ()
  {
    Output.WriteLine ("Temperature received " + ClimateSimulation.FanSimulation.Fan1.ConsumedTemperature.Value + " °C");
    ClimateSimulation.FanSimulation.Fan1.ProvidedSpeed.Value = ClimateSimulation.FanSimulation.Fan1.ConsumedTemperature.Value * 2;
  }
}

Python

import vector.canoe
import application_layer

@vector.canoe.measurement_script
class MQTTTester:

  def start(self):
    self.timer = vector.canoe.MsTimer(1000, self.send_values)
    self.timer.start()

  def send_values(self):
    application_layer.ClimateSimulation.TemperatureSimulation.TemperatureSensor.ProvidedTemperature = 20

  @vector.canoe.on_update(application_layer.ClimateSimulation.FanSimulation.Fan1.ConsumedTemperature)
  def on_value_update(self):
    value = application_layer.ClimateSimulation.FanSimulation.Fan1.ConsumedTemperature.copy()
    vector.canoe.write(f"Temperature received {value}°C")
    application_layer.ClimateSimulation.FanSimulation.Fan1.ProvidedSpeed = value * 2

Serialization

JSON

Struct

The following vCDL description models the DO member demo_structure, which should send JSON-serialized data of the data type structure:

version 1.4;

import module "MQTT";

namespace Demo
{

struct innerstruct{
    int64 innerint;
    }

struct structure{
        int64 some_integer;
        double some_double;
        uint64 some_unsignedinteger;
        string some_string;
        innerstruct some_innerstruct;
}

    namespace MQTTClients
    {
        [MQTTBinding::ClientId="CANoe"]
        [Binding="MQTT"]
        interface ISending
        {
            [MQTTBinding::Topic="test"]
            provided data structure demo_structure;

        }

        object Sending : ISending;


    }
}

The JSON (which is default if no explicit serialization attribute is set) serialized MQTT payload in this case has the following form (the numerical values are arbitrary):

{ "some_integer": -233,
  "some_double" : 233.0,
  "some_unsignedinteger": 233,
  "some_string" : "teststring",
  "some_innerstruct":{
                      "innerint":250}
}

Union

The vCDL description models the DO member demo_union, which should receive JSON-serialized data and write it to the data type uniontype.

version 1.4;

import module "MQTT";

namespace Demo
{
struct innerstruct {
    int64 innerint;
    }

union uniontype{
            int64 some_integer;
            double some_double;
            string some_string;
            uint64 some_unsignedinteger;
            innerstruct some_innerstruct;
}

    namespace MQTTClients
    {
        [MQTTBinding::ClientId="CANoe"]
        [Binding="MQTT"]
        interface IReceiving
        {
           [MQTTBinding::Topic="test"]
           consumed data uniontype demo_union;

        }

        object Receiving : IReceiving;


    }
}

Below is shown which received JSON strings are written to which union member:

Note the order of the union members regarding the int64 and uint64 data types. If a positive integer is to be written to a uint64 member, the uint64 member must come before an int64 member in the order in which it is defined.

Google Protocol Buffers

This chapter assumes that the reader has basic knowledge about Google Protocol Buffers, since Protobuf is not as self-describing as JSON. If this is not the case, the necessary information about Protobuf can be found on the internet.

Struct and Array

Consider the following vCDL description:

version 1.4;

import module "MQTT";

namespace Demo
{
  namespace InnerNamespace{
    struct innerstruct{
    float some_float;
      uint32 some_unsignedinteger;
      list<bool> some_bool_array;
    }
}

  struct structure{
    int64 some_integer;
    double some_double;
    list<list<bool>> some_nested_bool_array;
    InnerNamespace::innerstruct some_innerstruct;
  }

namespace MQTTClients
  {
    [MQTTBinding::ClientId="CANoe"]
    [Binding="MQTT"]
    interface ISending
    {
      [MQTTBinding::Topic="test"]
      [Serialization="GoogleProtobuf"]
      provided data structure demo_structure;
    }
    object Sending : ISending;
}
}

Importing this vCDL description results in three (actually six because of the second set of proto files for syntax proto3) generated proto files:

1. non_complex_array_data_types_proto2.proto

This file is generated because of the presence of (nested) arrays with primitive/simple base types in the data model. In this specific case, the arrays are some_bool_array and some_nested_bool_array. The Protobuf message types defining their data types are in this proto file.

File content:

syntax = "proto2";
message bool_array_1{
repeated bool array = 1;
}
message bool_array_2{
repeated bool_array_1 array = 1;
}

Here, the first message type defines the list<bool> data type and the second one the nested list<list<bool>> data type.

2. Demo_InnerNamespace_proto2.proto

This file is generated for the data types defined in the (inner) namespace Demo::InnerNamespace, i.e. the data type innerstruct:

  syntax = "proto2";

  import "non_complex_array_data_types_proto2.proto";

  package Demo.InnerNamespace;

  message innerstruct{
  required float some_float = 1;
  required uint32 some_unsignedinteger = 2;
  optional bool_array_1 some_bool_array = 3;
}

The package specifier corresponds to the namespace, with the namespace-level separators as dots.

3. Demo_proto2.proto

This file includes the data types defined (directly) in the namespace Demo, i.e. the data type structure:

syntax = "proto2";

  import "non_complex_array_data_types_proto2.proto";
  import "Demo_InnerNamespace_proto2.proto";

  package Demo;

  message structure{
  required int64 some_integer = 1;
  required double some_double = 2;
  optional bool_array_2 some_nested_bool_array = 3;
  required Demo.InnerNamespace.innerstruct some_innerstruct = 4;
}

The file imports the file Demo_InnerNamespace_proto2.proto because structure contains a field with the data type innerstruct defined in package Demo.InnerNamespace.

Union

Redefine the data type innerstruct to an union data type, here named innerunion. This results in the following vCDL file:

version 1.4;

import module "MQTT";

namespace Demo
{
  namespace InnerNamespace{
    union innerunion{
      float some_float;
      uint32 some_unsignedinteger;
      list<bool> some_bool_array;
    }
  }

  struct structure{
    int64 some_integer;
    double some_double;
    list<list<bool>> some_nested_bool_array;
    InnerNamespace::innerunion some_innerstruct;
  }

namespace MQTTClients
  {
    [MQTTBinding::ClientId="CANoe"]
    [Binding="MQTT"]
    interface ISending
    {
      [MQTTBinding::Topic="test"]
      [Serialization="GoogleProtobuf"]
      provided data structure demo_structure;

    }

    object Sending : ISending;
}
}

This will result in a redefinition of the message type innerstruct to innerunion with a oneof in its body. The name of the oneof inside a message type describing a union is always chosen as oneof_body. The content of the file Demo_InnerNamespace_proto2.proto after reimport is now:

syntax = "proto2";

import "non_complex_array_data_types_proto2.proto";


package Demo.InnerNamespace;

message innerunion{
oneof oneof_body {
float some_float = 1;
uint32 some_unsignedinteger = 2;
bool_array_1 some_bool_array = 3;
}
}

Simple/Primitive Data Types

Consider the following vCDL-setup (note the serialization attribute!):

version 1.4;

import module "MQTT";

namespace Demo
{

namespace MQTTClients
  {
    [MQTTBinding::ClientId="CANoe"]
    [Binding="MQTT"]
    interface ISending
    {
    [MQTTBinding::Topic="boolTopic"]
    [Serialization="GoogleProtobuf"]
    provided data bool Demo_bool;

    [MQTTBinding::Topic="stringTopic"]
    [Serialization="GoogleProtobuf"]
    provided data string Demo_string;

    [MQTTBinding::Topic="stringTopic"]
    [Serialization="JSON"]
    provided data int64 Demo_int64;
    }

      object Sending : ISending;
}
}

Importing this model will result in a single generated proto file (again, actually 2 because of the file for syntax proto3) simple_data_types_proto2.proto, which contains the Protobuf message type definitions for all the simple/primitive DO/interface member data types to be serialized with Protobuf.

File content:

syntax = "proto2";

message boolType{
required bool boolField = 1;
}

message stringType{
required string stringField = 1;
}

Note, that there is no message type definition for the int64 data type, because the serialization attribute for the member Demo_int64 is set to JSON.

Frequent Use Cases

Send Retained Messages

For each topic, you can store a message in MQTT that is sent first to each receiver that subscribes to the topic. To do this, send a Retained message with the Retain flag set to True. In the DO modeling, the [MQTTBinding::Retain] attribute is available therefor. The stored message is overwritten by the last Retained message sent. To delete the stored message, a Retained message can be sent with a zero byte payload. Plain serialization can be used to send a zero byte payload, which transmits the empty payload unchanged. The following example demonstrates how to set and delete retained messages. The model is represented in vCDL and an application model in C#, in CAPL and in Python.

vCDL

version 1.4;

import module "MQTT";

namespace ClimateSimulation
{
  //------------------------------------------------------------------------------------
  // Info about the temperature sensor via retained messages
  //--------------------------------------------------------------------------------------
  namespace TemperatureSimulation
  {
    interface ITemperatureSensorInfo
    {
      [Binding="MQTT"]
      [MQTTBinding::ClientId="CANoe"]
      [Serialization=Plain]
      [MQTTBinding::Topic="rooms/living/devLowerLeft/temperature"]
      [MQTTBinding::Retain=true]
      provided data string ProvidedTemperatureInfo;
    }

    object TemperatureSensorInfo : ITemperatureSensorInfo;
  }
}

C#

using System;
using Vector.Tools;
using Vector.CANoe.Runtime;
using Vector.CANoe.Sockets;
using Vector.CANoe.Threading;
using Vector.Diagnostics;
using NetworkDB;
using NetworkDB._SystemDataTypes;

using IServiceProvider = Vector.CANoe.Runtime.IServiceProvider;

public class MQTTTester : MeasurementScript
{
  public override void Start()
  {
    SendRetainedMessage();
  }

  public override void Stop()
  {
    DeleteRetainedMessage();
  }

  private void SendRetainedMessage()
  {
    ClimateSimulation.TemperatureSimulation.TemperatureSensorInfo.ProvidedTemperatureInfo.Value = "Temperature / °C";
  }

  private void DeleteRetainedMessage()
  {
    ClimateSimulation.TemperatureSimulation.TemperatureSensorInfo.ProvidedTemperatureInfo.Value = "";
  }
}
