MQTT Protocol

 

Introduction

MQTT is a lightweight transport protocol which sits atop TCP/IP. It is primarily used in the machine-to-machine (M2M) and Internet of Things (IoT) environments.

The publish/subscribe pattern, which enables 1-n distribution of the messages, is used for communication. Here, distribution occurs solely over a server commonly also called an MQTT server.

Topic

A topic serves as an ID for PUBLISH messages and as a filter for SUBSCRIBE messages to enable interest-based distribution of application messages between the clients.

Client

Following establishment of the connection with MQTT broker, the client can send application messages via PUBLISH-messages and/or declare interest in specific application messages via SUBSCRIBE-messages.

Broker

The main task of the broker is to distribute PUBLISH messages between clients. It also functions as a filter for the clients here, as only those PUBLISH messages for which the client has subscribed reach them.

Note

A detailed description can be found in the specifications of the Organization for the Advancement of Structured Information Standards (OASIS):

  • Version 3.1.1 (from 2/25/20)
    http://docs.oasis-open.org/mqtt/mqtt/v3.1.1/mqtt-v3.1.1.html
  • Version 5 (from 2/25/20)
    http://docs.oasis-open.org/mqtt/mqtt/v5.0/mqtt-v5.0.html
  • By default, many brokers refuse to connect a client with the KeepAlive mechanism disabled (value 0). In case of connection problems, please make sure that this does not apply to your broker or set a value greater than 0 for your clients.

Example

In the following scenario, a sensor for temperature and a sensor for humidity connect as MQTT clients to an MQTT broker. The sensors act as PUBLISHERs for the TEMP or HUMID topics. An ECU and/or a mobile device also connect to the MQTT broker as MQTT client and subscribe to both topics.

To send PUBLISH and SUBSCRIBE messages, the clients must send a CONNECT message to the MQTT broker. If connection establishment is successful, the MQTT broker sends a CONNECT message to the respective client.

Following the CONNECT message, the clients can send PUBLISH and SUBSCRIBE messages. They don’t absolutely have to wait for the CONNECT message here, however.

If one of the sensors sends a PUBLISH message to the MQTT broker, it distributes this message to all subscribed clients. The MQTT broker also sends PUBLISH messages for this purpose.

The application message is processed by the client following this.

Supported Scope of the MQTT Protocol

CANoe currently supports a subset of the MQTT transport protocol in version 3.1.1. Basic use of MQTT is possible with it.

The following limitations currently exist:

  • The flags for the connection establishment can be set via predefined attributes for the MQTT binding. However, it is not possible to configure a LastWill message.
  • Clients can subscribe to topic filters.
  • Clients can send PUBLISH messages.
  • SUBSCRIBE and PUBLISH messages are sent with a
    Quality of Service
    QoS) = 0 or
    QoS = 1.
    QoS = 2 is not available.
