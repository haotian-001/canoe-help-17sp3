Protocol Information

To protect the data of a GeoNetworking packet, they can be transmitted in a secured packet (ETSI TS 103 097).

The secured packet follows the GeoNetworking Basic Header. It consists of the Security header, the data to be protected and a trailer that contains the signature. The payload to be protected always begins with the GeoNetworking Common Header, and optionally followed by the GeoNetworking Extended Header, Basic Transport Protocol and application message. The signed data begin with the security header and extend up to the end of the payload.

Note

Various document versions of the standard are supported.

You can set which document version is used by CANoe in the area Car2x Security Layer (Options dialog). The document version used is displayed for the SH layer in the Detail view of the Trace Window.

The functions of the packet APIs support only the version set in each case. Different protocol designators are available for the different versions.

Trace Window [â² top]

Since the data structure of the security header is unclear, a summary of the most important transmitted values is shown under Summary in the Detail view of the Trace Window. Some values are shown that do not occur in the message but are known in CANoe, such as the signature status or certificate name. The complete data structure that is transmitted can be found under Message Tree. The payload of the secured packets is shown in interpreted form after the trailer of the secured packet.

Invalid signatures are shown in the Trace Window with an error icon. If the signature of a message cannot be checked, a comment appears in the Protocol Info column. This comment remains there, even if a certificate was received at a later time that could be used to check the validity of the older message.

Protocol Dependencies [â² top]

The Security Header is available if the Next Header field of the GeoNetworking Basic Header has the value 2 (Next Header=Secured Packet). The Security Header is always followed by the GeoNetworking Common Header.

Signature Status

signatureStatus

â

â

â

The status of the signature is shown in the Detail view of the Trace Window.

  • 0x1000..0x1FFF: unknown
    The signature cannot be checked, because only the digest, but not the certificate is known, or because the validity of the certificate cannot be checked.
  • 0x2000..0x2FFF: invalid
    The signature is invalid, because either the signature does not match the certificate, or because the certificate is invalid.
  • 0x3000..0x3FFF: valid
    The validity of the signature is based on one of the certificates from the certificate chain used (certificate, certificate of Pseudonym CA or root certificate) being marked as trustworthy in the Car2x Certificate Manager.

The status of the signature (value of the token) can be read out with the C2xGetTokenInt function.

Optional Tokens

For received messages it has to be checked that the token was transmitted. When sending a message, the token is only transmitted if it was set.
