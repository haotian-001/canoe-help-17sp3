Protocol Information

In order to protect the data of a WAVE packet, these can be secured by the WAVE Security Services (IEEE 1609.2) protocol layer.

The WSS layer consists of information on the signer, information on the user data, the user data itself, and finally the signature (signature over the information of the user data and the user data itself).

Notes

Different document versions of the standard are supported.

You can set which document version is used by CANoe in the area Car2x Security Layer (Options dialog). The document version used is displayed for the WSS layer in the Detail view of the Trace Window.

The functions of the Packet API support only the version set in each case. Different protocol designators are available for the different versions.

Protocol Dependencies of 1609.3-2010 [▲ top]

It is possible to secure the payload data of a WAVE Short Message Protocol (WSMP) layer as well as the content of the WAVE Service Advertisement (WSA) layer. In the first case, the WSS layer is between the WSMP layer and the payload data, which consists mostly of a SAE J2735 message (BSM, TIM, etc.). In the second case, the WSS layer directly follows the WLAN header, and the WSA layer follows that.

Case 1: WSMP

In the WLAN Data Frame with WSMP layer, the WSS layer is interpreted if the first byte of the WSMP payload data has the value 2 or 3 (WSS version 2 or 3). If the WSS layer with version 2011/D9 and 2013 contains the value 1 (signed data) as Content Type or if the WSS layer with version 2016 does not contain the value 135 as PSID, an application message is interpreted in the following (if present).

Case 2: Action/WSA

In the WLAN management frame with subtype Action, a WSS layer is interpreted under certain conditions. If the WSS layer with version 2011/D9 and 2013 contains the value 11 (signed WSA) as Content Type or if the WSS layer with version 2016 contains the value 135 (WAVE_WSA) as PSID, a WSA layer is interpreted in the following (if present).

Trace Window [▲ top]

Since the data structure of the WSS layer is unclear, a summary of the most important transmitted values is shown under Summary in the Detail view of the Trace Window. Some values are shown that do not occur in the message but are known in CANoe, such as the signature status or certificate name. The complete data structure that is transmitted can be found under Message Tree. The payload is shown in interpreted form after the signature.

Invalid signatures are shown in the Trace Window with an error icon. If the signature of a message cannot be checked, a comment appears in the Protocol Info column. This comment remains there, even if a certificate was received at a later time that could be used to check the validity of the older message.

Signature Status

signatureStatus

–

–

–

The status of the signature is shown in the Detail view of the Trace Window.

  • 0x1000..0x1FFF: unknown
    The signature cannot be checked, because only the digest, but not the certificate is known, or because the validity of the certificate cannot be checked.
  • 0x2000..0x2FFF: invalid
    The signature is invalid, because either the signature does not match the certificate, or because the certificate is invalid.
  • 0x3000..0x3FFF: valid
    The validity of the signature is based on one of the certificates from the certificate chain used (certificate or root certificate) being marked as trustworthy in the Car2x Certificate Manager.

The status of the signature (value of the token) can be read out with the C2xGetTokenInt function.

Optional Tokens

For received messages it has to be checked that the token was transmitted. When sending a message, the token is only transmitted if it was set.
