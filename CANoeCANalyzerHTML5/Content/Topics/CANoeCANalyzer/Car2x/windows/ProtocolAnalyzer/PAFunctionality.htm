Functionality

 

A Car2x frame can be structurally correct (interpretable) due to the large number of optional header and ASN.1 data elements, but the data can still be contradictory, incomplete or incorrect from the point of view of a specific Car2x use case. Individual guidelines/rules for the structure of the Car2x frame are distributed across several specifications and best practice documents from different standardization bodies and are also version-dependent in terms of content. Car2x Protocol Analyzer rule set(s) contain many of these guidelines/rules in a form that allows automatic passive verification.

The Car2x Protocol Analyzer does not modify any Car2x frame data and is completely passive (analysis only) from a communications point of view, it does not inherently send test stimulus frames or similar active artifacts that affect the Car2x communications.

Display of Rule Violations of a Car2x Frame

If one or more rules of the set of rules built into CANoe detects at least one rule violation in a concrete Car2x frame, the Car2x frame is marked in Trace Window with a yellow triangle symbol (warning) or red circle symbol (error) and background color of the frame. The background color of the marking can be set in CANoe Options|Windows/Blocks|Trace Window|Color Trace Window Events|WLAN|Error). In addition, the General section of the Detail view of the Trace Window displays the note protocol violation. However, the full reason and description of the rule violation are displayed in the Protocol Analyzer window itself due to the size of the note.

The assignment of Car2x frames in Trace Windows and the rule violations in the Protocol Analyzer window can be done in the following ways:

  • Once the CANoe measurement is finished, one of the purposes of window synchronization is to automatically select the first rule violation of a Car2x frame selected in Trace Window in the list of rule violations in the right part of the Protocol Analyzer window. Both Trace Window and Protocol Analyzer should be in chronological display mode.

  • To select the corresponding Car2x frame in the Trace Window from the list of rule violations in the right part of the Protocol Analyzer window, the Select Frame in Trace Window command is available in the context menu of each rule violation.

Displaying and Filtering the Rule Violations of a Car2x Station

You can use the following means to navigate more efficiently through a variety of the rule violations in the Protocol Analyzer window:

  • Fixed vs. chronological display mode of rule violations
    In the fixed display mode, only the rule violations that occurred last in time are displayed, defined differently. This greatly reduces the number of displayed rule violations if the same rule violations occur repeatedly. In chronological display mode, on the other hand, the rule violations are displayed in the order in which they occurred, including similar ones that differ only in the time stamp.

  • Disabling rules or rule groups
    If one or more rules or rule groups are not relevant or already analyzed by the user, they can be disabled by deactivating the check box in the left part of the Protocol Analyzer window. A disabled rule will not analyze Car2x frames and will not return rule violations until it is re-enabled. Accordingly, frames in the Trace Window will no longer be marked as containing errors if they do not contain any or if they contain errors/warnings exclusively from deactivated rules.

  • Filtering and sorting of rule violations
    The list of rule violations in the right part of the Protocol Analyzer window allows filtering and sorting of rule violations, e.g. filtering by station name. In the context menu of the column headers the display of unwanted rule violations can be suppressed, the filtering is done here based on concrete values in the columns of the list.

Rule violations can be reported by several active rules. To navigate to the rule definition directly, the Select rule command is available in the context menu of the rule violation.

Finding References To Standards and Requirements

In addition to a short justification of a rule violation (Rule column in the list of rule violations in the Protocol Analyzer window), a normative justification (a reference to the specification or standard) is available. It is displayed as the last line in the tooltip of a concrete rule in the rule overview in the left part of the Protocol Analyzer window.

Protocol Analyzer CAPL API

Additional custom rule violations can be created directly in CAPL code. For each rule violation (built-in or user-defined rule) a CAPL callback function is called where the contents of a rule violation can be added e.g. to a test report or used in a different way. See Protocol Analyzer CAPL API.
