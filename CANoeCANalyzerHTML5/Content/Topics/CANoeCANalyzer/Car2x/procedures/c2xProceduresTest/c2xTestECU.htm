Test a Car2x Control Unit

 

With the optionCar2x you can test an ECU in many ways. Some possibilities are described below.

Stimulation/analysis in HiL

Option 1 - Stimulating the ECU with WLAN / C-V2x packets.

With the option Car2x you can create scenarios using the Scenario Editor. The corresponding WLAN or C-V2x packets are generated from these scenarios. Using a VN4610 or third-party C-V2x hardware, these packets are sent to the ECU under test. Whether the processing of the data in the ECU was successful can be checked, for example, via CAN messages that the ECU places on the internal networks for further processing of the data. These CAN messages can be received via the CAN interface of the VN4610 and checked for correctness in a test environment. Car2x specific test functions facilitate the generation of tests at this point. A generated test report shows which tests were successful and which failed.

Option 2 - Stimulating the ECU with CAN messages

CAN messages are generated in the option Car2x and sent to the ECU via the VN4610 CAN interface. These CAN messages are intended to trigger the transmission of a specific WLAN / C-V2x packet, for example. Via the antenna of the VN4610 (or a third party hardware in case of C-V2x tests) it can be checked if the corresponding packet has been sent. A test environment can also automatically check whether the tests were successful. A generated test report shows which tests were successful and which failed.

Analysis in HiL

For a pure analysis in HiL, the setup is similar to that for stimulation. The only difference is that in the option Car2x no packets are created for stimulation. The antenna of the VN4610 (or a third party hardware in case of C-V2x tests) can be used to check what the ECU under test is transmitting. Typical parameters that can be tested here are e.g. the cycle time, the time stamps, the geo positions or the event codes. A test environment can also automatically check whether the tests were successful. A generated test report shows which tests were successful and which failed.

 

Stimulation in the vehicle

The special feature of a stimulation directly in a test vehicle is that the test vehicle has real GNSS data available. This requires that the corresponding scenarios, which are to test the vehicle, are always executed relative to the test vehicle. Proceed as follows.

  • The option Car2x must be operated in real mode.
  • A VN4610/C-V2x hardware (third-party) must be connected.
  • A GNSS receiver must be available and configured.
  • A scenario must be available.
  • The transmission behavior must be configured in the Car2x Network Explorer. Set the node attribute Use GNSS Sysvar to On for the ITS station that is to stimulate the test vehicle.

A test environment can also automatically check whether the tests were successful. A generated test report shows which tests were successful and which failed.

Analysis in the vehicle

For a pure analysis in the test vehicle, the setup is similar to that for stimulation. The only difference is that in the option Car2x no packets for stimulation are created or no scenario has to be created. Using the antenna of the VN4610 (or a third party hardware in case of C-V2x tests) it is possible to check what the test vehicle is transmitting. Typical parameters that can be tested here are e.g. the cycle time, the time stamps, the geo positions or the event codes. A test environment can also automatically check whether the tests were successful. A generated test report shows which tests were successful and which failed.
