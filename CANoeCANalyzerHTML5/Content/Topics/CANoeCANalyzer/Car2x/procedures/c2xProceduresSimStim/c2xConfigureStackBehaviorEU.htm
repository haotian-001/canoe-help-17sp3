Configure Stack Behavior (EU)

 

V2X Stack

V2X Stack means that CANoe automatically takes care of setting certain fields in the frame and calculating cycle times. The V2X stack for the EU application messages CAM, DENM, SREM and SSEM is currently implemented. The V2X stack for CAM and DENM is based on the Basic System Profile 1.5.2 of the Car2Car Communication Consortium. The V2X stack for SREM and SSEM is based on C-ITS Message Profiles 1.8 of C-Roads.

Configuration of the V2X Stack in the Car2x Network Explorer

How the V2X stack can be configured in the Car2x Network Explorer is described in Configuring the sending behavior.

V2X Stack and Scenario Editor

When the V2X stack is activated, by default fields are set in the frame. By using scenarios, the frames can be further completed with dynamic fields such as Latitude, Longitude, Speed and Heading. If attributes are created and set in the Scenario Editor that were previously filled by the V2X stack, the data from the Scenario Editor is applied.

V2X Stack and CAPL

When the V2X stack is activated, by default fields are set in the frame. By using CAPL, the frames can be further completed with dynamic fields such as Latitude, Longitude, Speed and Heading. This can be done e.g. in an OnPreTx CAPL callback. If fields are set in CAPL that were previously filled by the V2X stack, the data from CAPL is applied.

Filled Fields in the Frame - DENMs Described in the Basic System Profile

For each use case described in the Basic System Profile, the fields in the frame are set accordingly, depending on the cause code with different Traffic Class ID, Hop Limits, DistanceA, Relevance Distance, Relevance Traffic Direction, Validity Duration or Information Quality. The fields Stationary Since, Event Speed and Event Position Heading are not part of each Basic System Profile and are only set for the corresponding use cases. The exact values can be found in the Basic System Profile of the Car2Car Communication Consortium.

V2X Stack of DENMs That Are Not Described in the Basic System Profile

For some use cases there is no description in the Basic System Profile. Here CANoe makes realistic assumptions.

This applies to the following use cases:

  • Accident (Cause Code 2). The values from the Basic System Profile for Stationary Vehicle - Post Crash are assumed.
  • Roadworks (Cause Code 3). The values from the Basic System Profile for Stationary Recovery Work in Progress are assumed.
  • Hazardous Location - Surface Condition (cause code 9), Obstacle on the Road (cause code 10), Animal on the Road (cause code 11) and Human Presence (cause code 12). The values from the Basic System Profile for Stationary Vehicle are assumed.
  • Wrong Way Driving (Cause Code 14). The values from the Basic System Profile for Dangerous Situation are assumed.
  • Adverse Weather Warning - Extreme Weather (Cause Code 17). The values from the Basic System Profile for Adverse Weather Warning - Precipitation are assumed.
  • Slow Vehicle Warning (Cause Code 26). The values from the Basic System Profile for Emergency Vehicle in Operation are assumed.
  • Vehicle Breakdown (Cause Code 91). The values from the Basic System Profile for Stationary Vehicle Warning are assumed.
  • Post Crash (Cause Code 92) and Human Problem (Cause Code 93). The values from the Basic System Profile for Stationary Vehicle - Post Crash are assumed.
  • Dangerous Curve (Cause Code 96). The values from the Basic System Profile for Dangerous Situation are adopted.
  • Signal Violation (Cause Code 98). The values from the Basic System Profile for Pre-Crash Information are used.

Path History

The Path History is calculated automatically. This can be up to 800m long.

Cycle Time

The Cause Code is used to differentiate the cycle time with which the DENMs are sent. For Cause Codes not described in the Basic System Profile, realistic assumptions are made.

The following is an overview of the cycle times

  • 100ms: Wrong Way Driving, Collision Risk, Signal Violation, Dangerous Situation
  • 250ms: Slow Vehicle Warning, Special Vehicle (z.B. Emergency Vehicle)
  • 500ms: Dangerous End of Queue
  • 1000ms: Traffic Condition, Accident, Roadworks, Adverse Weather Warning, Hazardous Location, Rescue & Recovery Work in Progress, Vehicle Breakdown, Post- Crash, Human Problem, Stationary Vehicle, Dangerous Curve
  • 4000ms: Adverse Weather Warning – Extreme Weather Condition, Adverse Weather Warning – Visibility, Adverse Weather Warning - Precipitation

Update and Repetition

Basically, the Cause Code is used to distinguish whether a DENM is sent in update or repetition mode.

  • Update DENMs
    The Area Position Latitude / Longitude in the GEO-EH is updated, also the Event Position in the DENM. The Detection Time is set the first time the DENM is sent and remains constant. The Reference Time is updated at each transmission. This applies to:
  • Special Vehicle
  • Pre-Crash Information
  • Dangerous Situation
  • Wrong Way Driving
  • Slow Vehicle Warning
  • Signal Violation.
  • Repetition DENMs
    The Area Position Latitude / Longitude in the GEO-EH remains constant, as does the Event Position in the DENM. The Detection Time and the Reference Time also remain constant over the transmission time of the DENM. This applies to:
  • Traffic Condition
  • Adverse Weather Warnings
  • Rescue & Recovery Work in Progress
  • Dangerous End Of Queue
  • Stationary Vehicle
  • Collision Risk
  • Accident
  • Roadworks
  • Hazardous Location
  • Vehicle Breakdown
  • Post-Crash
  • Human Problem
  • Dangerous Curve

If you want to update a Repetition DENM manually in the scenario editor, you can do this by toggling the event keypoint from 0 (stop) to 1 (send) in the timeline. Disabling with 0 and re-enabling with 1 causes an update of the Repetition DENM. If provided in the Basic System Profile, an update DENM is sent after a specified time.
