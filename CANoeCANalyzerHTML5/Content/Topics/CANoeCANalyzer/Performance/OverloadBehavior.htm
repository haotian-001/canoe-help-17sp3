Behavior under Overload Conditions

 

In the course of a measurement, the network hardware registers frames on the bus and passes them via the paths designated in the data flow diagram to the evaluation and analysis blocks located at the far right. For this purpose, two program modules work closely together during a measurement. First the realtime process (RuntimeKernel.exe) retrieves information arriving at the hardware and delivers it to a buffer. In a second step, these data are read out by CANoe and are evaluated in the Measurement Setup. If the rate of messages registered by the hardware is so high that the main Windows program cannot follow along in its processing, the buffer overflows. However, as a user you are informed of this data loss immediately.

The Trace window signals load situations through separate events/lines:

  • Main Queue Load => Normal:
    Signals that a previously occurring load or overload situation (Main Queue Load => High and Main Queue Load => Data Lost) has been eliminated.
  • Main Queue Load => High:
    Signals the time from which a load situation is detected. The Trace window interrupts the display of newly arriving events starting from this time. However, the events are stored in the buffer of the Trace window. As soon as the load situation is eliminated, the display resumes as usual.
  • Main Queue Load => Data Lost:
    Signals the time from which a data loss came about due to the overload situation. Starting from this time, there is no longer any data in the buffers of the Trace Window until the next arrival of a Main Queue Load => High or a Main Queue Load => Normal.

The main queue events have a time stamp and are queued in the data stream in chronological order in continuous mode of the Trace window. In continuous mode, it is therefore possible to use the order to relate the described events to other events. In fixed position mode, only the last/current main queue event is displayed in each case. A chronological relationship to other events is only possible using the time stamp.

A data loss that occurs is also recorded in ASCII logging.

To observe the buffer between the realtime process and the main Windows program more precisely, the buffer has a fill level indicator. You can have the states of this indicator displayed in the Write window. To do this, set WriteLevel to at least the value WriteLevel = 3 in section [Environment] in the project file CAN.INI. Then, if the ring buffer overflows, the message Load transition: NORMAL->DATA LOST appears in the Write window and informs you of the data loss. After the overload situation has ended (possibly after a short burst on a bus) you are also informed as soon as a normal situation has been reinstated. You can recognize this from the message Load transition: QUEUE HIGH -> NORMAL in the Write window.

For test purposes you can induce overload situations yourself by holding the mouse button down over the title bar of the main window or by moving the window on the screen. When this is done, the data display of the main program is interrupted until you release the mouse from the title bar. However, work in the realtime process remains unaffected by these actions. As long as messages are being registered on the bus, the buffer will fill with data, but the data cannot be processed by the main program. Finally the ring buffer will overflow. As soon as you release the mouse from the title bar, you can observe the effects of this overflow in the individual windows.
