Low on Memory

 

The Low on Memory dialog is displayed by CANoe at the start of the measurement if only a small amount of virtual memory is still available for the program.

Background

The virtual memory is memory that the operating system (i.e., Windows) makes available to CANoe. All program data (e.g., DLLs), all configuration data (e.g., databases), and most measured data (e.g., for display in analysis windows) are stored in the virtual memory. If there is not enough virtual memory for all of the data, the program no longer functions correctly and crashes may occur, e.g., with error message bad_alloc. The size of the virtual memory is dependent on the operating system. CANoe normally receives up to 4 GB from a 64-bit operating system. However, if both the physical main memory of the computer is low and the Windows paging file is small, the virtual memory can also be smaller. CANoe does not use all of the virtual memory up to the maximum limit, but allows a certain safety margin of several hundred MB. The virtual memory must be differentiated from the main memory of the computer. On the one hand, a program can occupy a large amount of virtual memory, but use significantly less main memory, e.g., if the program has used data temporarily and deleted it again. On the other hand, if the main memory is smaller than the virtual memory, Windows may write parts of the virtual memory of a program to its paging file on the hard drive, which significantly reduces the operating speed. In addition, several programs can divide the same main memory area for their virtual memory, e.g., if they use the same program components (DLLs).

Causes of high memory use

If a large amount of virtual memory has been used, possible causes include:

  • Large databases (DBC, FIBEX, ARXML)
  • Large diagnostic descriptions (OCS, PDX)
  • Large CAPL programs (in particular, when using large, and possibly multidimensional, arrays)
  • Large buffer settings for measurement data, e.g., for the Graphics Windows

CANoe can also save a large data history, in which case the data are kept in the main memory in the default setting. You can change the settings for this in the Options dialog on the Data History page.

If CANoe identifies individual sources that contribute to high memory utilization, these are displayed in the Low on Memory dialog as soon as you click More Details. You can directly edit each source using the button to the right of the entries in the list.

Problem solving

If sufficient memory is still available for a small data history, you can start the measurement in spite of the low memory by simply confirming the Low on Memory dialog with OK. CANoe then switches over automatically to the small data history and the dialog is displayed again for this configuration.

If there is not sufficient memory available even for a small data history or if was already not possible to load all CAPL programs, the measurement cannot be started. You must then start by reducing the amount of memory used by the configuration. For example, you can remove diagnostic descriptions or databases you do not need from the configuration, reduce the size of buffers, and disable or modify CAPL programs. In CAPL, depending on the application, associative arrays can significantly reduce the amount of memory utilization compared to standard arrays, for example, if only a very few entries are actually needed in the array.

You can disable the check for available memory altogether and start the measurement despite having low memory. To do so, you must open the can.INI file and add the following in the [System] section:

MemoryUsageWarningLimit=0
MemoryUsageStopLimit=0

You then run the risk that you will run out of memory during the measurement.
