Performance Optimization

CANoe can be used as a simulation and testing tool for many use cases, including situations where data must be processed very quickly.

In such situations, it may be necessary to change settings or adjust the simulation to better align CANoe with the specific use case and thus avoid performance problems.

Possible Performance Requirements

The use cases that are relevant with regard to runtime performance, and thus the adjustments that may need to be made, can be grouped into the following basic categories:

  • Realtime Requirements

  • High Processor Load in Analysis

  • High Processor Load in Simulation

Realtime Requirements

In this case, the requirement is to respond to an event within a specified time during the simulation (low latency), or to perform an action with a specified accuracy at a specified time (high determinism).

Problems often manifest themselves in the failure to meet specified cycle times of frames or PDUs, or protocols in an ECU to enter an error state due to timeouts. You can also detect realtime problems via the Performance Monitor (system variable Simulation::TimerPrecision) or with the CAPL functions SetTimerAccuracyMonitoringActive/GetLastTimerAccuracyNS.

The cause of realtime problems can be a very high amount of data (read section High Processor Load in Simulation), but more common are other causes:

  • External programs on the simulation computer, e.g. virus scanner, other simulation participants, software management, etc.

    The Performance Monitor can sometimes reveal these causes. Tools included in Windows such as the Task Manager or the Resource Monitor as well as the free Sysinternals tools from Microsoft also offer a good possibility for analysis.

    The best way to avoid such disturbances is to run the simulation or even the realtime critical parts of it in an isolated environment, e.g. a VTP device. For a detailed explanation, see Realtime Execution (RT Setup) and Extended Real Time (ERT).

  • Using .NET in test modules or test units.

    When the test is started, the .NET assembly must be loaded and the .NET JIT compiler must first translate the IL code into binary code. With high realtime requirements, this can cause problems.

    Again, one possible solution is to use Extended Real Time (ERT) for the realtime critical parts of the simulation. Alternatively, you could use CAPL instead of .NET. Vector is working on other solutions.

  • Code in simulation or test that holds up execution for too long, for example, by writing data to or reading data from files, performing complex calculations, or waiting for external responses.

    You may be able to locate such code via code reviews. Measurements can also be helpful,

    • in CAPL e.g. with the class CAPLProfiler or the function GetSteadyClockCurrentTimeNS,

    • in .NET e.g. with the class System.Diagnostics.StopWatch.

    One possible solution to the problem is to keep data in memory and read it once when the measurement starts or write it when the measurement ends; this depends on the specific use case and the amount of data.

    Another solution is asynchronous programming in .NET or using appropriate CAPL functions (e.g. TcpSend). You can also move the processing completely to an additional thread (see WaitForTask) or execute it in an additional process with which you communicate via TCP or FDX.

    Note in .NET, however, that only a limited part of the CANoe API is available to you from additional threads.

High Processor Load in Analysis

Especially at high busloads it may happen that the simulation works fast enough, but the processing in the analysis requires too much time.

This situation can be recognized by the Load Indicator or by the fact that analysis windows are automatically switched off, as described in Behavior under High Load Conditions. The system variables Analysis::CPUUsageMainThread and Analysis::MainQueueLoad also reliably show the situation in the Performance Monitor.

To improve the performance of the analysis area, there are different possibilities:

  • Follow the notes under Working at High Busloads.

  • Reduce the data to be processed. Filter out the data as early as possible; also consider the option of not transferring the data to the analysis in the first place.

High Processor Load in Simulation

With particularly high data rates or very complex simulations, the processor performance may also be insufficient for the simulation.

This situation can sometimes be recognized by the fact that realtime requirements are not met, but more often by Tool Notifications, e.g. "can board: rx queue is full", "Realtime event processing was interrupted", "Measurement stopped due to realtime event processing overrun", or also "Disturbance of VT System communication detected". The system variable Simulation::CPUUsageMainThread can also be used to detect the situation in the Performance Monitor.

Programming

Sometimes the situation is caused by inefficient programming in CAPL or .NET. You may be able to identify the code in question by taking measurements, for example, in CAPL using the CAPLProfiler class or the GetSteadyClockCurrentTimeNS function, and in .NET using, for example, the System.Diagnostics.StopWatch class. Also, watch out for file operations, frequent string manipulation, or accessing simulation values using strings instead of using the names directly in the code. In some circumstances, it may help to generate some of the code with a generator rather than using generic code during simulation.

Also check whether you might be unnecessarily performing some calculations multiple times in the simulation.

Data Reduction

The best way to save computing time is to reduce the amount of data to be processed. Often, for example, it is not necessary to simulate the entire network, but only the communication relevant to the system under test. Especially for FlexRay, CANoe offers special support here. You can also use acceptance filters in the Network Hardware Configuration, e.g. for CAN. For Ethernet in network-based mode, you can select which ports you want to use in the simulation and measurement setup. This can be configured in the port configuration dialog. In addition, you can create an uplink filter in the Vector hardware configuration to reduce traffic by filtering specific Ethernet packets based on MAC, IP and TCP/UDP port.

External Communication

If you communicate with external simulation participants e.g. via FDX, Functional Mock-up Interface (FMI) or ASAM XIL API, especially try to reduce the communication rate. A more extensive but less frequent transmission is faster overall than a smaller but more frequent transmission.

Hardware Settings

Depending on the hardware used, it may help to change driver settings to reduce the interrupt load or the number of DPCs, for example.

Multi CANoe

If you have an overall system with a very large number of networks, it is also possible to split the remaining bus simulation among different CANoe instances – possibly even on several computers. The individual instances work together to perform simulations and tests in distributed form and can exchange data via FDX. Vector Support will be pleasedto assist you in setting up and configuring such a system.
