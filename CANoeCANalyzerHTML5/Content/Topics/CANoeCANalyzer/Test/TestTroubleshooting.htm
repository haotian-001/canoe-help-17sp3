Troubleshooting

 

  • Warning in the Write Window: "Test group was not closed – please use the function TestGroupEnd."
    A test group was opened in the test module via the TestGroupBegin function, but not closed via TestGroupEnd. The opened test group will be closed automatically at the end of the test module.
  • Warning in the Write Window: "Report information block missing for transfer of information pair with TestReportAddMiscInfo."
    An attempt was made to write an information pair to the report with TestReportAddMiscInfo, but no supplemental information block was created in which the information pair could be stored. Therefore a suitable supplemental information block was created automatically.
  • DTD not found
    The DTD file specified in an XML file should normally be located in the same directory as the XML file. If this is not the case, the DTD will not be found when the XML is read, which will lead to an error, for example when attempting to load an XML test module or when converting an XML logging file.
  • Failure of tests due to rounding errors
    Floating-point values are sometimes used, especially for the representation of physical measurement values. If floating-point values are transmitted as signals or generated from signals in the database on the basis of computational specifications that demand floating-point values, there are likely to be rounding errors. For this reason, such values should never be tested for equality to another value, but always compared in relation (<, >, etc. or <range> in XML, for example).
  • Overflow of the message queue while processing extensive test modules
    If CANoe is overloaded with the processing of data resulting from the measurement (e.g. logging test steps, recording communication in the logging file), an overflow of the message queue may occur. In this case, CANoe will post a warning; the logs and logging files will then be incomplete. If this occurs, the option SmoothPerformancePeaksAfterTestCases in CAN.INI can be used to enable a performance optimization and thus avoid this situation (see configuring test modules).
  • Test reports become too extensive
    In the Test Module Configuration Dialog and Configuring a Test Configuration you can activate/deactivate different classes of report information especially for the XML/HTML report format, which influences the size of the test report.
    When using the XML/HTML report format, the test reports can individually be adjusted to meet individual requirements by choosing the right XSLT stylesheet. In this connection, the supplied standard style sheet onepage.xslt can be modified as required. By changing the value of the variable TestStepMaxLevel the number of test steps that are included in the HTML test report can be reduced. The following rules are valid for automatic generated test steps: a variable value of 1 means that all test steps that are just informative will be ignored. Error messages, warnings and other important information will be included. If the value is 0 only test steps that provide error-related information are shown.
  • The Execute test modules sequentially context menu command of a folder in the Test Setup cannot be selected
    This menu command starts all the test modules in the given folder sequentially and therefore can only be selected when a measurement is running. Otherwise the menu command is disabled. Furthermore, the folder must contain only CAPL or XML test modules and must not include any sub-folders. See also sequential execution of several test modules.
  • Error message "no signal driver" in the Write Window
    In test modules it is possible to access signals directly. To enable Write access, however, a corresponding instance is required in the CANoe configuration that maps the signal to a message and sends the message. This instance is referred to as a Signal Driver and can be implemented in various ways. The error message above is output when an attempt is made to access a signal in Write mode for which no signal driver has been assigned. See also Signal oriented access.
  • CANoe cannot write the XML test report
    This can occur in particular if the directory to which CANoe is to write the test report does not exist.
    Another reason can be that a file with the same name already exists but is write-protected.
  • The title of the test case in the test report is incomplete
    The title of the test case must not contain parenthesis. Otherwise the title will be truncated at the first parenthesis.
  • CANoe crashes during creation of an HTML test report
    If a crash occurs when finishing a test module this may be due to too long description texts within the test module.
    Description texts in the test module - especially the configuration comment - should have less than 100 lines. Otherwise after finishing the test module during the conversion of the XML report into an HTML report the CANoe may crash.
  • Dialog/Warning in the Write Window: "Test report can not be written into file, as it is already used!"
    Two or more test modules try to access the same report file simultaneously. Therefore a report file will only be created for one of these test modules. Please change the report file in the test module configuration dialog to avoid the problem.
  • Path/name to XML Schema file in an XML test module contains whitespace(s)
    Whitespaces within path or filename have to replaced by %20.

Example

<testmodule xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                xmlns="http://www.vector-informatik.de/CANoe/TestModule/1.8"
   xsi:schemaLocation="http://www.vector-informatik.de/CANoe/TestModule/1.8 C:\My%20Projects\test%20module.xsd"
                title="Test module title" version="Test module version">
