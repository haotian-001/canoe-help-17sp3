Testing | Concept

 

Glossary: »Test

Test (rather testing with CANoe) describes CANoe's functional area for testing ECU's and networks.

  • Testing in terms of reviewing individual functionality or behavior of your Simulation can be found under the keyword Stimulation.
  • Testing in terms of "does my CAPL program what I want it to do" can be found under the keyword CAPL Programming.
  • Testing in the diagnostic tester region can be found under the keyword Diagnostics.

Architecture

Besides its well-known analysis and simulation components, another component was introduced in CANoe for testing. Test modules/test units can be started by the system or the user during measurement, and they each execute a test sequence.

For stimulation and examination purposes the test modules/test units can access

  • the complete remaining bus simulation
  • the connected buses (e.g. CAN, LIN, FlexRay, IP)
  • the digital and analog input and output lines of the Device Under Test using general purpose I/O cards or VT System via system variables
  • other external realtime systems (e.g. HIL systems and LabVIEW models) using the FDX interface
  • other external measurement systems (e.g. GPIB and Ethernet) using appropriate interfaces

An ECU may be tested by itself, or as part of a network consisting of various ECUs where the object of testing is called a System Under Test or SUT. CANoe's options are available to the test modules/test units , e.g. panels for user interaction or writing of outputs to the Write Window.

Testing ECUs and Networks

One of the main tasks of CANoe is to test ECUs and networks. Individual development steps are verified, for example, prototypes are tested and regression and conformity tests are performed. CANoe services the "System under Test" at all interfaces here. This guarantees the fullest possible test coverage.

  • With CANoe, sequential test sequences are implemented by test modules/test units, which are subdivided into test groups and test cases. The individual modules/units can be executed at any time during a measurement.

    • In XML modules, tests are put together from predefined test patterns and are easy to parameterize by input and output vectors.
    • CAPL and .NET test modules, on the other hand, are programmed and therefore exhibit a very flexible form of test flow control. The .NET test modules are developed conveniently in C# or VB.NET. The different description forms may be combined depending on requirements.
    • In test units you can also create test sequences – with several test cases – within test groups.

  • In parallel to test execution, you can check other system states such as conformance to the cycle times specified for individual messages. These constraints are automatically incorporated in the test evaluation.
  • The Test Service Library contains a collection of prepared test functions that make it easier to set up tests. They are used in the test modules/units and parameterized via the database. For example, it is possible to monitor the cycle times of messages, the reaction time of an ECU to receipt of a message as well as sending out of the diagnostic response and the validity of signal values and diagnostic parameters. To evaluate the quality of tested ECU, various statistical values are output, among them the number of reported deviations over the testing time period.
  • In executing a test module/test unit, an extensive test report is generated. Along with the names of the test cases that were executed and the individual test results, it is also possible to include user-defined information or automatically generated screenshots of different analysis windows in the report. CANoe writes the results to report file that can be further processed flexibly.
  • Setup
  • Test units are managed in so-called test configurations. A test configuration may contain 1…n test units, that will be executed sequentially. A CANoe configuration may contain any number of test configurations that may be executed parallel.
  • In the Test Setup, CANoe can manage any desired number of test environments that contain test modules. The test environments contain test modules and other function blocks for test execution, and they are saved independent of the system configuration. As a result, it is easy to use them in different projects.
  • Direct control of I/O hardware in CANoe makes it possible to use analog and digital ECU interfaces in addition to bus communication. Along with standard I/O components, the Vector VT System also offers a modular hardware system for comprehensively testing ECUs in the automotive field here.
