XML Test Module Structure

 

The use of text modules in XML is particularly expedient when a large number of test cases is to be generated from existing data. Since the XML files essentially contain the test parameters, the generators must simply convert the existing data into the required XML form. The fact that XML is a well-known and commonly used format, for which a large number of editors are available, is helpful in this regard.

The significant differences between the description of test modules in XML and in CAPL, in addition to the language used, are:

  • In CAPL, test modules are formulated procedurally. That is, how a test case should be executed is programmed. This is very flexible, and in principle permits the definition of any given test procedure. Although the CAPL language is task-oriented and is generally quite simple, some programming experience is required to create more complex tests.
  • Test modules are formulated descriptively in XML. That is, the XML file essentially contains information about what should be tested. The actual test procedures, that is the "how", are specified via test functions and control functions. The XML file contains the specifications required for the test functions/control functions, e.g., initial and target values, etc. The XML file therefore essentially parameterizes the tests.

The creation of a test module in XML is therefore easier, since the tester must only specify the required parameters. Conversely, there may also be cases in which the desired test procedures cannot be defined or can only be defined with difficulty with the means available in XML.

CAPL and .NET test cases can also be called from within a XML test module (see Call CAPL test case and Call .NET test case). For this reason, it is advisable to define all simple cases in an XML test module and to call CAPL/.NET test cases from within the XML test module only in complex cases.

An XML Test Module Consists of:

  • any number of test cases,
  • which may be grouped in test groups.

An XML test module, however, contains no explicit test procedure control. Instead, the test cases are processed in the sequence in which they are defined in the XML file. However, individual test cases can be excluded from the execution in the execution dialog.

Each test case is usually composed of individual test- and control functions. These are predefined test procedures that are parameterized and called in the XML file using suitable parameter values. They are processed sequentially, and each one contributes to the test case verdict.

You can define constraints and conditions in test cases as well as in test groups. These conditions then apply during the course of the entire test case. They apply to all test cases belonging to the test group in question, and they are checked in parallel to the execution of the test procedure.

The test case being executed aborts as soon as a test/control function fails or a constraint/condition is violated, and is marked as failed in the test report. The test module itself is not aborted; the test simply continues with the next test case.

In addition to the definition of the test cases, the XML file also contains information for the test report. A report is created with each execution of the test module. It includes precise information about the test procedure and the executed test/control functions, and not just the corresponding information from the XML file.

The description of the test module by the <description> element is adopted into the test report, which can be created automatically while executing test modules. To obtain line breaks (in form of <br /> tags) in this description in CANoe Test Report Viewer or the HTML browser, "\n" may be inserted at any place in the text.

The XML file of a test module may contain several variants of the test module. The variants may differ in the composition of the test cases (Example: a test case is executed in a variant but not in another one) or in the implementation of several test cases (Example: in one variant a signal has the value 5 in another variant the signal's value is set to 7). See page XML test module variants for further information.
