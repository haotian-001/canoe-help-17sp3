Variants

 

A single XML test module may contain several variants of one test. This makes it unnecessary, for example, to create separate XML files for ECUs that differ only in minor ways. For each variant, you can define which test cases are executed and how the test cases differ from one another.

Declaration and use of variants in test modules

To use variants in a test module, you must first declare the variants at the start of the XML file; there are no "default" variants.

Example 1

A certain test case of an existing test module shall only be executed in one variant, all other test cases always. This test case is now labeled as variant A. All other test cases don't need a variant label. Therefore the variants attribute has only to be added to this single test case. However, two variants have to be created: variant A, in which the test case is executed, and variant B, that includes all test cases without a particular label. In this case the test module doesn't need to use variant label B since this variant describes the "default" for all unlabeled test cases.

<testmodule title="Test" version="1.0">
  <variants>
    <variant name="A">Test with special test case</variant>
    <variant name="B">Test without special test case</variant>
  </variants>
  ...
  <testcase ident="72.1" title="Special test case" variants="A">
  ...
  </testcase>
  ...
  <testcase ident="80.3" title="Other test cases">
  ...
  </testcase>
  ...
</testmodule>

Example 2

<testmodule title="Central locking system test" version="1.0">
  <description>Description of the test</description>
  <variants>
    <variant name="A">Test for ECU XYZ Model 1</variant>
    <variant name="B">Test for ECU XYZ Model 2</variant>
  </variants>
  ...
</testmodule>

You need to assign a name and a description to each variant. The name goes in the name attribute of the variant element; the description is the text within the element. The name is used to identify the variants within the XML file. It should be short and simple, and may not contain any spaces. The variant description, which is displayed in the CANoe interface and the test report, may contain spaces.

To make a specific segment of the XML description valid only with certain variants, add the variants attribute to that element. The value for the attribute should be a list of the names of the variants in which this element is to be used. The names in the list should be separated by a space.

Example 3

<statechange wait="500" title="Velocity lower than the lock velocity" variants="A B">

This test function is only used in Variants A and B, i.e., it is executed only in these variants. If you have defined e.g. a Variant C in the test module and you run Variant C of the test module, this test function is ignored. This applies to all child XML elements.

Example

In the following example, the testcase element is not visible in Variant B, because the parent test group is defined only for Variant A:

  ...
  <testgroup title="Test cycle times of ECU XYZ" variants="A">
    <testcase ident="TC103" title="Lock by increasing velocity" variants="A B">
      ...
    </testcase>
  </testgroup>
...

Conversely, child elements may be defined in fewer variants than the parent element.

Elements that do not contain a variants attribute are active in all variants.

Note

Test cases and test groups that are inactive in certain variants are not displayed in the interface (execution dialog) when you select one of these variants. Consequently, you can't subsequently activate them for these variants.

The variants attribute can be used in the following elements:

  • testgroup, testcase, capltestcase, nettestcase
  • miscinfo, extendedinfo, sut, setup, comment
  • cansignal, linsignal, flexraysignal, envvar, sysvar, canmsg, mostmsg, mostsysmsg
  • signal, byte, signalref, byteref, param, node
  • value (serves to set the value of a variable (linked with "XML file: variables"))
  • at the highest level of all checks
  • at the highest level of all test functions and control functions

Executing Test Modules With Variants

If you declare variants in the XML file of an XML test module, the Execution dialog above the test case tree will contain an additional list box. This shows the descriptions of the different variants. Use this box to specify which variant you want to activate.

The tree shows only those test cases that are active in the selected variant. If you select a different variant, the XML file is read in again and the test case tree is re-populated. Please note that when you do so, the current selection of test cases is lost. You can still activate or deactivate test cases within a given variant. However, you cannot manually activate any of test cases that aren't visible for the selected variant in the XML file. These test cases are not shown in the test case tree.

If you haven't declared any variants in the test module, the list box is not shown.

When you run an XML test module that has variants, only those test cases that are visible in the test case tree are executed. Test cases that you have deactivated in the test case tree are not executed either, of course. It doesn't matter whether you start the test case via the Start button, via an environment variable or by another method. The variant selected in the execution dialog is always the one executed. When you compile an XML test module that has variants for the first time, the first declared variant will be the active one. This selection is saved to the test setup or the configuration.

You can also switch between test module variants via the CANoeCOM Interface (TestVariant property). When you subsequently run the test module, e.g. via the COM interface, the selected variant is executed.

Note

When you switch variants in the execution dialog, the XML test module is read in again. This means that any changes made to the XML file in the meantime are then applied. It may take some time to read in the data, depending on the size of the XML file.
