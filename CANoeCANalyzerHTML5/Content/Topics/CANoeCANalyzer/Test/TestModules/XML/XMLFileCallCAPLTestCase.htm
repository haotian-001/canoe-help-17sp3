Call CAPL Test Case

 

You can assign CAPL files to the XML test module as libraries in the Test Module Configuration dialog. Test functions called from the XML test module can be implemented in these CAPL files.

The CAPL test case libraries assigned here are structured like CAPL test modules, but they must not contain a MainTest routine. The call of event procedures is only performed during the runtime of the XML test module to which the CAPL file is assigned to as a library. Thereby all event procedures of the assigned CAPL libraries are considered. There is no independent test course and – like in CAPL test modules – the event procedures on start and on stop are not available. The test cases contained therein can only be called from the XML test module.

The user of such an XML test module will not see the difference between test cases that are implemented in CAPL or XML. CAPL test cases can, in this case, also be deactivated via the user interface.

Example

A test case programmed in CAPL is referenced in an XML test module in the following manner:

<capltestcase name="CAPL name of CAPL test case" title="Displayed name of the CAPL test case" ident="ID of the test case, e.g. number">
  <caplparam name="Parameter name" type="float|int|string|signal|envvar|sysvar">Parameter value</caplparam>
  ...
</capltestcase>

  • The name attribute indicates the test case name, used in the CAPL program.
  • The title attribute is optional; with this attribute the test case can get a title that is used instead of the CAPL name on the user interface and in the report. This procedure is particularly useful if the same CAPL command is called several times with different parameters from the XML test module.
  • The ident attribute is optional, too. It is used to assign a unique ID to a test case. This ID is also used in the XML and HTML test report.

If in the CAPL test case an additional name or ID (CAPL command TestCaseTitle) is set, this name and ID are used in the report. On the user interface the name is displayed that is defined with the attribute title. If the title attribute doesn't exist the name is used that is defined with the attribute name.

CAPL test cases are denoted by the keyword test case. They do not have a return type.

The only parameter types allowed are int for integers, float for floating-point numbers, string for character strings and signal for CAN/LIN/FlexRay signals. These are mapped to the corresponding CAPL types; for example, all integer CAPL types (long, int, …) are addressed by the int type in XML, while both CAPL types float and double are addressed by the double type in XML.
Beware of value overstepping!

The usage of environment variables or system variables as parameters is also possible (parameter types envvar, sysvar). The environment/system variable itself is handed over to the CAPL test case, not its current value. In CAPL you can find out and set the current value.
System variables of type Integer, Float and String (Integer and Float as arrays, too) are supported as well as environment variables of type Integer, Float, String and Data.

For the types int, float and string you can also use a variable rather than a fixed parameter value. The variable type must correspond to the CAPL parameter type.

Example for the Usage of Signals

<capltestcase name="TC1">
  <caplparam name="d" type="float">100</caplparam>
  <caplparam name="s" type="signal"><cansignal name="velocity" /></caplparam>
</capltestcase>

testcase TC1(double d, signal * s)
{
  $s = d;  // the value of the signal defined in XML is set to d
}

Example for the Usage of System Variables

<sysvardef type="int" namespace="ns1" name="sv1" default="0"/>

<capltestcase name="TC2">
  <caplparam name="sv" type="sysvar">
    <sysvar namespace="ns1" name="sv1" />
  </caplparam>
</capltestcase>

testcase TC2(sysvarInt* sv)
{
  if (@sv < 0)
  @sv = 0;
  // ...
}

Note

The parameter number of CAPL test cases is limited to 64.

If more values must be transferred or if the same value must be equally transferred to different called CAPL test cases, it can also be triggered by system variables.

For this reason, a system variable is defined and described in the XML test module and read out in CAPL test cases. e.g. via SysGetVariableInt.
