Parameters

 

In the definition of checks (for constraints and conditions), variables, test functions and control functions, various value specifications and the description of messages, signals, and environment variables are required. These are always defined in the same way.

Signals and Signal Values Are Defined As Follows:

<cansignal name="Signal name" msg="Message name" node="Node name" bus="Bus name">Value</cansignal>
<linsignal name="Signal name" msg="Message name" node="Node name" bus="Bus name">Value</linsignal>
<flexraysignal name="Signal name" frame="Frame name" node="Node name" bus="Bus name">Value</flexraysignal>

The specification of the associated message name, the node name or the bus name is only required if the signal name cannot be resolved uniquely by the incorporated CANdb databases.

The value itself can be specified by a corresponding number, variable, or an enum value. If sensible, when setting a signal value no relation is permitted, of course, the value can be described by a relation that is described more precisely by additional XML elements:

Note

  • Instead of Number an enumeration value of the database can be used.
  • All given signal values are always physical values.
    Physical value = raw value * factor + offset
    The values for factor and offset are defined in the database.
  • The signals, messages and environment variables stated as parameters actually have to be described in the related databases.
    This applies e.g. to numeric described CAN messages.

Syntax

Short Description

<eq>number</eq> (equal)

The signal value must correspond precisely to this number.
This is also the default assumption if the number is not described in more detail by a corresponding XML tag.

Note

Floating-point numbers should never be tested on equality!

<ne>number</ne> (not equal)

The signal value must be not equal to the specified number.

<lt>number</lt> (lower than)

The signal value must be smaller than the specified number.

<gt>number</gt> (greater than)

The signal value must be greater than the specified number.

<le>number</le> (lower or equal)

The signal value must be smaller than or equal to the specified number.

<ge>number</ge> (greater or equal)

The signal value must be greater than or equal to the specified number.

<range>
   <from>number</from>
   <to>number</to>
</range>

The signal value must be in the specified range, including the specified numeric values.
You can only use numbers or variables; values from value tables are not allowed.

<range>
   <base>...</base>
   <down_tolerance mode="absolute|relative">...</down_tolerance>
   <up_tolerance mode="absolute|relative">...</up_tolerance>
</range>

The signal value must be in the range that has been specified by a base value with upper and lower tolerance. The upper and the lower limit are part of the allowed range:
(base - down_tolerance) <= value <= (base + up_tolerance).

You can use numbers or variables of a numeric data type; values from value tables are not allowed. Variables will be evaluated at this point of time, when the function or check is executed.

Tolerances may be labeled by the mode attribute as absolute or relative; without a label they will be interpreted as absolute tolerances. Tolerances must always be positive values.
Relative tolerances are set as factors, e.g. 0.1 for 10% tolerance in the respective direction.
For signals with integer values possible decimal places in tolerance values will be cut, e.g. 12 -10%/+20% written as down_tolerance=0.1, up_tolerance=0.2, results in [12 - 1.2, 12 + 2.4] and the actual limits are [11, 14].

<one_of>
   Set of <eq> or <range> conditions
</one_of>

The signal value must fulfill at least one of the contained conditions. Only testing for identity or a value range is allowed as a sub-condition.
In <range> sub-conditions you can only use numbers or variables; values from value tables are not allowed.

Example

<one_of>
  <eq>2</eq>
  <range>
    <from>5</from>
    <to>8</to>
  </range>
  <eq>12</eq>
  <range><from>18</from><to>20</to></range>
<one_of>

Note

Note that no empty spaces are permitted between the elements (see Notes).

An environment variable is defined as follows:

Here too the elements named above apply for the relations of the specified numeric values:

<envvar name="Name of the environment variable">Value</envvar>

Please consider the following when working with environment variables of type Data:

  • They are given as a list of byte values. These byte values can be specified decimal or hexadecimal. The maximum length is 1 byte.
    Example: <envvar name="EnvData"><eq>0x12 0x34 0x56 0x78 0x90</eq></envvar>
  • Expectancies of this environment variable can only be specified using the following elements: <eq></eq> and <ne></ne>.
  • If an assignment from a variable or a check of a variable has to be performed, you have to use variables of the type string.

Instead of an environment variable, it is also possible to define a system variable:

<sysvar name="name of system variable" namespace="name space">value</sysvar>

System variables that were not created in the XML test module can be accessed only as read-only. This means setting values, e.g. in the initialize command, is only allowed for system variables that have been defined in the test module itself.

The name of the variable can also include the designation of a specific element of a Struct or Generic Array.

CAN, LIN, FlexRay and MOST messages/frames/PDUs are defined as follows:

<canmsg id="Name or ID of the CAN message" node="Name of the node" bus="Name of the bus" channel="CAN channel" dlc="Number of data bytes" dir="Send direction: Rx, Tx or TXrequest" rtr="Remote frame identification: rtr or nortr" type ="Standard or extended CAN identifier: std or ext" msgflags="Additional information: nerr or high voltage">
   Data byte values or signal values
</canmsg>

<linframe id="Name or ID of the LIN frame" node="Name of the publisher node" bus="Name of the bus" channel="LIN channel" dir="Send direction: Rx, TX" >
   Data byte values or signal values
</linframe>

<frframe flags="Flags (bitmask) from CAPL for updating" payloadlength="Length of the payload in bytes" bus="Name of the bus" dir="Send direction: Rx, TX or TXrequest">
   <frframe_symbolic_id name="Name of the FlexRay frame" node="Name of the Node"/>
   Data byte values or signal values
</frframe>

<frframe flags="Flags (bitmask) from CAPL for updating" payloadlength="Length of the payload in bytes" bus="Name of the bus" dir="Send direction: Rx, TX or TXrequest">
   <frframe_numeric_id slotid="Slot ID" cycleoffset="Base Cycle" cyclerepetition="Cycle Repetition Factor" channelmask="A, B or AB"/>
   Data byte values or signal values
</frframe>

<frpdu name="Name of the PDU" updatebit="0 or 1" updatecount="Number of the udates (-1 = infinite)" bus="Name of the bus" node="Name of the Node" dir="Send direction: Rx, TX or TXrequest">
   Data byte values or signal values
</frpdu>

<mostmsg fblock="FBlock ID" function="Function ID" optype="OpType ID" instance="Instance ID" name="Symbolic representation of the MOST message" channel="MOST channel" tellen="Number of data bytes" telid="Telegram ID (not with AMS messages)" dir="Send direction: Rx, TX or TXrequest" sa="MOST send address" da="MOST receive address" ams="Determination if it is an AMS message: ams or ctrl (ams is default)" spy="Determination if it is a Spy message: spy or node (node is default)">
   Data byte values or parameter values
</mostmsg>

<mostsysmsg rtype="RType ID" channel="MOST channel" dir="Send direction: Rx, TX or TXrequest" sa="MOST send address" da="MOST receive address">
   Data byte values
</mostsysmsg>

Here too specifications about node and bus can be omitted if the message can be found unambiguously in the databases. MOST messages are also resolved via the incorporated XML function catalogs, not via CANdb databases! MOST messages can either be defined by the declaration of a numeric ID for FBlock, Function, OpType, Instance or by the declaration of a symbolic name e.g. FBlock.Instance.Function.OpType.

Furthermore you can define CAN messages that are not contained in the database. If several buses exist in the context you must either specify the bus or the channel of the message. Additionally you have to specify the DLC of such a message. Without a database connection you cannot assign data signals to such a message.

The data bytes or data signals are described by a series of <byte> – as the case may be <signal> – elements. A mixture is not permitted. The relations presented above for the signals can be used for the values.

<byte pos="Byte position from 0">Value</byte>
<signal name="Signal name">Value</signal>

Various specifications can be omitted if, for example, diagnostic responses should be awaited for which only a portion of the attributes are defined fixed. A non-defined attribute is always defined as "arbitrary". But look out: this does not work in all cases; a message that should be sent must, for example, be specified completely.

<param> can be used for symbolic definition of parameter data in <mostmsg>:

<param name="parameter address">value</param>

In addition to direct value entries, conditions and variables can also be used here. Please see Symbolic Parameter Identification (MOST) a general description of symbolic parameter entries. When such entries are used in XML specifications, the following rules must be observed:

  • In order to use symbolic parameters, the message address must be entered completely and must be resolvable in the  XML function catalog. Parameter values cannot be assigned symbolically to a MOST message that is defined only in CANdb.
  • When defining more complex parameter structures, e.g. arrays from records, it is important to make sure that the specified parameter combination can be resolved, especially if a position value is explicitly specified (see example). Combinations that cannot be resolved are identified during parsing and flagged by means of a corresponding error message.
  • In contrast to all other numeric parameters, conditions such as <lt> or <gt> cannot be entered for position parameters, as these values specify the parameter structure.
  • Structure parameters such as Data cannot be set directly. Array elements are generated implicitly when a parameter value is set in the desired array element. Any missing array elements are then generated automatically and filled with blank values (see example).

Example of a Simple CAN Message

<canmsg id="Status" dlc="3" channel="1">
   <byte pos="0">1</byte>
   <byte pos="1">0xE8</byte>
   <byte pos="2">255</byte>
</canmsg>

Non-specified values here are taken from the database when sending. If this is not possible, then the test case reports an error at this point during execution. The same applies for other errors such as the specification of higher numeric values, for example.

Example of a Simple CAN Message (Without Database Connection):

<canmsg id="4" dlc="3" bus="CAN1">
  <byte pos="0">1</byte>
  <byte pos="1">0xE8</byte>
  <byte pos="2">255</byte>
</canmsg>

Example for a simple MOST Message:

<mostmsg name="CDC.1.MediaInfo.Error" bus="Device2" sa="0x101" da="0x101" ams="ctrl" spy="node">
   <param name="ErrorCode">0x7</param>
   <param name="ErrorInfo">010304</param>
</mostmsg>

Example for a MOST Message with a Complex Parameter Structure:

<mostmsg name="CDC.1.MediaInfo.Status" bus="Device2" sa="0x101" da="0x100" ams="ams" spy="node">
   <param name="Data.Record[6].MediaTitle">first CD title</param>
   <param name="Data.Record[5].MediaType">Audio</param>
   <param name="Data.Record[4].MediaFilesystem">UDF</param>
   <param name="Data.Record[3].FirstTrack"><gt>1</gt></param>
   <param name="Data.Record[2].LastTrack">6</param>
   <param name="Data.Record[1].PlayTime">123456</param>
   <param name="Pos">0</param>
</mostmsg>

Every time array indices are used, an array element is created unless one already exists, e.g., the example defined above consists of a total of 6 array elements, whereby only one parameter was set in each record.

Example for a MOST Message with a Parameter Combination that cannot be Resolved:

<mostmsg name="CDC.1.MediaInfo.Status" bus="Device2" sa="0x101" da="0x100" ams="ams" spy="node">
   <param name="Pos">0x0102</param>
   <param name="Data.Record[1].MediaTitle">first CD title</param>
   <param name="Data.Record[1].FirstTrack"><gt>1</gt></param>
</mostmsg>

ECU nodes are referenced by the node element:

<node name="Node name as specified in CANdb" bus="Bus name"/>

For MOST it is partially necessary to specify FBlocks or functions:

<fblock id="FBlock ID or Name" instance="Instance ID"/>
<function id="Function ID or Name"/>
