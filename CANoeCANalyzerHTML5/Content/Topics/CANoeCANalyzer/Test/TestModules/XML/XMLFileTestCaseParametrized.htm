Parameterized Test Cases

 

Test cases parameterized in XML can be used when a test module is to be used repeatedly for identical tests where the sequence of test functions/control functions executed does not vary, but the data used does (e.g. for checking different signals or comparisons with different must-values).

Template For Parameterized Test Cases

A parametrizable test case consists of a test case template that specifies the sequence of test functions/control function to be executed along with a set of parameters (paramdef) to be used in the sequence. These templates for test cases are defined at the start of the test module in the optional templates section:

<testmodule title="Testmodul Title">
  Test module preparation etc.
  <templates>
    <testcasetemplate name="Name of the test case template" version="optional version description">
      <paramdef name="Name of the parameter" default="default value (optional)"/>
      Contents are identical to XML test cases
    </testcasetemplate>
  </templates>
  Test groups and test cases
</testmodule>

Parameters defined via paramdef can be used at any given point in the test sequence, both as the content of XML elements and in their attributes. The parameter name is enclosed by $ signs in this case, as shown here:

<statecheck wait="100" title="check signal">
  <expected>
    <cansignal name="$t2-signame$">
      <eq>$t2-sigvalue$</eq>
    </cansignal>
  </expected>
</statecheck>

Note

  • Parameter names can consist of letters, numbers, underscores and hyphens.
  • Parameter values can only be used for a complete content/attribute. It is not possible to assemble these from predefined texts and embedded parameter references.
  • Using parameter references in the variants attribute of test case sequence elements is not allowed.
  • If the content of an XML element in a template test sequence is to consist of text enclosed by $ signs, but it is not a parameter, you should double one (or both) of the $ signs. You can do this with both XML element content and attribute values. Such a substitution can only be made inside of the test case template, not when passing such strings as parameter values.

Parameterized Test Case Instances

A test case template that is defined as described above can be used to create as many instances as desired; these instances can then be used at any given point in a test module just like normal XML test cases. A separate set of parameter values is specified for each test case instance. A series of test cases based on the same template is consolidated in "tabular" form within a common templateinstance element:

<templateinstances template="Name of the test case template">
  <testcaseinstance ident="Identifier of the test case, e.g. number"
    title="Test case title"
    variants="optional assignment to variants">
    <paramvalue name="Name of the parameter">value to be used</paramvalue>
  </testcaseinstance>
  further instances of the same template using other parameter values
</templateinstances>

Just like an XML or CAPL test case, each templateinstance contains a title and an ID, and can optionally be assigned to a variant. For all parameters defined in the test case template using paramdef, you can use paramvalue to specify the individual value to be used in a particular instance. If a paramvalue is not given for a parameter, the default value (default) is taken from the paramdef in the template. It is obligatory to specify a concrete value for each instance where a default value is not defined for that parameter in the template definition. Parameter values are used in a text-based manner, i.e., exactly as written in the XML file. It is not possible to nest other elements in the parameter values.

A test case instance works just like a normal XML or CAPL test case. It will appear as an individual test case in the Test Execution dialog and the test report.

Example

Parameterized test case template with parameters that can be used as attribute and element contents:

<templates>
   <testcasetemplate name="template2">
      <paramdef name="t2-signame" />
      <paramdef name="t2-sigvalue" default="0.0"/>
      <statecheck wait="100" title="check signal">
         <expected>
            <cansignal name="$t2-signame$">
               <eq>$t2-sigvalue$</eq>
            </cansignal>
         </expected>
      </statecheck>
   </testcasetemplate>
</templates>

"Tabular" set with several instances of the above test case template, with differing parameter sets:

<templateinstances template="template2">
   <testcaseinstance ident="tc-1.1” title=”signal check 1">
      <paramvalue name="t2-signame">SignalOne</paramvalue>
   </testcaseinstance>
   <testcaseinstance ident="tc-1.2" title="signal check 2">
      <paramvalue name="t2-signame">SignalTwo</paramvalue>
      <paramvalue name="t2-sigvalue">13.8</paramvalue>
   </testcaseinstance>
   <testcaseinstance ident="tc-1.3" title="signal check 3">
      <paramvalue name="t2-signame">SignalThree</paramvalue>
      <paramvalue name="t2-sigvalue">23.8</paramvalue>
   </testcaseinstance>
</templateinstances>

Note

You can use the vardef_bytester element if user input values are required in parameterized test cases. If the input is to be entered only once and then used for all instances, you can place it in the preparation section of the test module or in the enclosing test group and then read that value using a variable reference in the test case template. If the input is to be entered for each instance, you can place the vardef_bytester element in the preparation section of the test case within the test case template itself and then use it subsequently.
