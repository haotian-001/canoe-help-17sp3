Call .NET Test Case

 

In the test module configuration dialog, .NET assemblies can be added to the XML test module via the Modules tab. The assemblies contain test cases that can be called from the XML test module.

The test cases are defined as in .NET test modules, except that no Main function is required.

Example

using System;
using Vector.Tools;
using Vector.CANoe.Runtime;
using Vector.CANoe.Threading;
using Vector.CANoe.TFS;
using NetworkDB;
public class NETTestCaseLibrary
{
  [TestCase]
  public void SetSignal<T>(Double d) where T : Signal
  {
    RuntimeObject<T>.Value = d;
  }
}

In order to create the .NET test case libraries, you have to reference different assemblies. You need to include all the assemblies that are also required to develop complete test modules.

Event procedures are only active as long as the test case is active.

For the users of this type of XML test module there is no apparent difference between test cases in XML and .NET, as the .NET test cases can also be deactivated in the user interface.

The above test case programmed in .NET is referenced in an XML test module as follows:

<nettestcase name="SetSignal" title="TC 1" assembly="TestCaseLibrary" class=" NETTestCaseLibrary ">
  <netparam type="signal" name="T">
    <cansignal name="WindowPosition" />
  </netparam>
  <netparam type="float" name="d">1.1</netparam>
</nettestcase>

  • The name attribute indicates the name of the test case used in the .NET program.
  • The title attribute is optional.
    It can be used to give the test case a title for use in the user interface and the protocol.
    This is particularly useful when the same .NET test case is called several times with different parameters from an XML test module.
  • The class and assembly attributes are also optional.
    They can be used to provide a unique description of the test case.

The only permitted parameter types are whole numbers, floating-point numbers, character strings, signals environment variables and system variables. These are automatically mapped to the corresponding .NET types:

  • int: UInt64, UInt32, UInt16, Int64, Int32, Int16
    Careful with value overruns!
  • float: Double

  • string: String

    Note

    A variable (<var>) can also be used instead of a fixed value for the three types mentioned above. Their type has to correspond with the .NET parameter type.
  • signal: Vector.CANoe.Runtime.Signal
  • envvar: Vector.CANoe.Runtime.EnvironmentVariable

  • sysvar: Vector.CANoe.Runtime.SystemVariableBase

    Note

    In contrast to the value types (int, float, string), these types are declared as Generics in the test case. Constraints must be applied to these Generics in order to guarantee type reliability (in the example, where T: Signal). The specific types Signal, EnvironmentVariable or SystemVariableBase can be used for this.

The following definition is required if you want to use a test case for all three types:

[TestCase]
public void SetSignal<T>(Double d) where T : class, IRuntimeValue
{
  RuntimeObject<T>.Value = d;
}
