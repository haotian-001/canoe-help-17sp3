Call CAPL Test Function

 

In the test module configuration dialog, CAPL files can be assigned to the XML test module as libraries. Test functions called from the XML test module can be implemented in these CAPL files.

Event procedures are only called during the runtime of the XML test module to which this CAPL file has been assigned as a library. All event procedures are taken into account in the assigned CAPL libraries. There is no dedicated test run; also, like in CAPL test modules, the on start and on stop event procedures are not available. The included test functions can only be called by the XML test module.

Like test patterns, CAPL test functions are used within a <testcase> or in the <preparation> or <completion> sections. A test function programmed in CAPL is referenced in an XML test module as follows:

<testcase title=" Name of test case" ident="ID of test case">
  ...
  <capltestfunction
       name="CAPL name of test function"
      title="Display name for CAPL test function">
    <caplparam name="Parameter name"
               type="float|int|string|signal|envvar|sysvar">Parameter value</caplparam>
  </capltestfunction>
  ...
</testcase>

  • The name attribute indicates the name of the test function used in the CAPL program.
  • The title attribute is optional; it can be used to give the test function a title which is used on the user interface and in the log instead of the CAPL name. This can be particularly useful if the same CAPL test function is called from one XML test module multiple times with different parameters.

CAPL test functions are denoted by the keyword testfunction. They do not have a return type.

The only parameter types allowed are int for integers, float for floating-point numbers, string for character strings and signal for CAN/LIN/FlexRay signals. These are mapped to the corresponding CAPL types; for example, all integer CAPL types (long, int,…) are addressed by the int type in XML, while both CAPL types float and double are addressed by the double type in XML.
Beware of value overstepping!

The usage of environment variables or system variables as parameters is also possible (parameter types envvar, sysvar). The environment/system variable itself is handed over to the CAPL test case, not its current value. In CAPL you can find out and set the current value.
System variables of type Integer, Float and String (Integer and Float as arrays, too) are supported as well as environment variables of type Integer, Float, String and Data.

Example

Examples for the usage of the different parameter types can be found on page: Declaration and Transfer of CAPL Test Case and Test Function Parameters — Examples.

A variable can be used instead of a fixed parameter value for the int, float and string types. Its type must match the CAPL parameter type.

Example for the Use of Signals

<capltestfunction name="TF1">
  <caplparam name="d" type="double">100</caplparam>
  <caplparam name="s" type="signal">
    <cansignal name="velocity" />
  </caplparam>
</capltestfunction>

testfunction TF1(double d, signal * s)
{
  $s = d;  // sets the value of the signal defined in XML to the value d
}

Example for the Use of System Variables

<sysvardef type="int" namespace="ns1" name="sv1" default="0"/>

<capltestfunction name="TC2">
  <caplparam name="sv" type="sysvar">
    <sysvar namespace="ns1" name="sv1" />
  </caplparam>
</capltestfunction>

testfunction TC2(sysvarInt* sv)
{
  if (@sv < 0)
  @sv = 0;
  // ...
}

Note

The number of CAPL test function parameters is restricted to 64.

System variables can be used to provide a solution if more values need to be transferred or if the same value needs to be transferred to various called CAPL test cases.

For this purpose, a system variable is defined and written in the XML test module and read out in CAPL test cases using SysGetVariableInt, for example.

CAPL test functions do not have an own verdict. But within the implementation of the test function, the verdict of the test case can be affected by using the standard functions TestStepPass/TestStepFail.
