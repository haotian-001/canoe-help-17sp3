.NET Test Module Structure

 

Every .NET test module consists of several test cases and one test sequence control. The Main or StructuredMain function constitutes the test sequence control. In the simplest case, this merely consists of several test case calls. The test cases are always executed sequentially.

Test case execution is typically documented by means of test steps. The Vector.CANoe.TFS.Report class offers several functions for this purpose. Test steps can influence the verdict of a test module and, as with CAPL and XML test modules, are included in the report.

Test cases can be grouped together by arranging them into test groups. The test groups themselves can contain other test groups. These test groups can only be used for the test sequence control, and not within test cases.

Test cases are identified in .NET test modules by means of a method attribute; i.e., this attribute can be used to declare any function as a test case.

[TestCase("Title", "Description")]
public void MyTestCase()
{
  // do something
}

The following makes it possible to use one test case several times with different titles and descriptions:

[TestCase()]
public void MyTestCase(String title, String description)
{
  Vector.CANoe.TFS.Report.TestCaseTitle(title);
  Vector.CANoe.TFS.Report.TestCaseDescription(description);
  // do something
}

Special characteristics of structured test modules

The structure of a test module can be analyzed by executing a trial run of the StructuredMain() method. However, the actual implementation of the test cases is skipped in this case.

Within the StructuredMain() method

  • it is not permissible to query runtime variables,
  • it is not permissible to use Vector.CANoe.Runtime, Vector.Tools or Vector.CANoe.Threading methods,
  • some Vector.CANoe.TFS functions are not available, e.g. constraints, and
  • the test module should be designed in such a way that it always runs in the same sequence, i.e. test cases are not executed at random.

These limitations to not apply to methods that are flagged with the [Testcase] or [TestGroup] attribute.

If you want to execute test cases at random, or activate conditions or constraints at the same time, you need to consolidate these in a method that is flagged with the [Testgroup] attribute. These groups are skipped during analysis of the module. You can only activate and deactivate them (as complete groups) in the Text Execution dialog. For an example, see the CentralLockingSystem sample configuration.
