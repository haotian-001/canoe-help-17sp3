.NET Test Module

 

CANoe comes with two kinds of test modules:

  • Unstructured modules:
    Test cases are not shown in the Test Execution dialog until the module is running.
    These modules use the TestModule base class and implement the Main() function.
  • Structured modules:
    The structure of the test module is analyzed during compilation, and the test groups and test cases are displayed in the Test Execution dialog.
    The big advantage here is that you can deactivate test cases and test groups before running the module.
    Structured modules use the StructuredTestModule base class and implement the StructuredMain() function.

To program a .NET test module, you need to specify a .NET file as the source file for the test module. If this file does not exist yet and you then access the Edit function, a compilable file containing an unstructured .NET test module skeleton is automatically created. The recommended programming language is C#.

To create a structured module, replace TestModule with StructuredTestModule and Main with StructuredMain().

//Skeleton of a .NET test module
using System;
using Vector.Tools;
using Vector.CANoe.Runtime;
using Vector.CANoe.Threading;
using Vector.Diagnostics;
using Vector.Scripting.UI;
using Vector.CANoe.TFS;
using NetworkDB;
public class test : TestModule
{
  public override void Main()
  {
  }
}

Detailed information about the .NET test module structure can be found here.

Note

For the migration from .NET2 API to .NET4 API the following lines have to be deleted (if automatically inserted):

using ASAM.HILAPI.Interfaces;
using ASAM.HILAPI.Implementation;
using Vector.CANoe.ASAM.HILAPI;

You can also configure a .NET assembly directly. This is particularly useful if you don't want to pass on the source code. The assembly must contain exactly one class that is derived from Vector.CANoe.TFS.Testmodule, has a public default constructor and overwrites the Main method. This method contains the complete test sequence of the test module (similar to the Main method in CAPL).

So-called event procedures are used to process events in .NET test modules. In contrast to normal network nodes, event procedures are not activated during the entire measurement; instead, they are activated only when the test module is active.

A number of namespaces are available for programming .NET test modules:

  • Vector.Tools
    Contains general services such as Timer or output to the Write Window.
  • Vector.CANoe.Runtime
    Contains general functions for .NET programs in CANoe, such as event procedures.
  • Vector.CANoe.Threading
    Contains services for test module and snippet sequence control.
  • Vector.CANoe.TFS
    Contains services and functions that are only used in test modules (e.g. reporting).
  • Vector.Diagnostics
    Contains objects for the communication via diagnostic services. Available in test modules and snippets.
  • Vector.Scripting.UI
    Contains further user dialogs for testing purposes and to enter data.
