Test with Conditions

 

The functions of the Test Feature Set for CANoe permit the setup of tests according to different test strategies. The conditions are conceived as aids for tests according to the following pattern: for the system to be tested, conditions are formulated that the system must adhere to at all times (or at least in a defined period of time). These conditions are monitored parallel to the execution of the test procedure. In the test procedure, a series of actions are executed (for example, a type "vehicle cycle"). The test fails if the condition was injured at any time during the test execution.

With this type of test it is not necessarily required that test conditions are queried actively in the test sequences. The test cycles can be programmed as pure system usage (stimulus), while the test conditions to be checked are formulated as conditions.

Example

A transmission control system sends a signal cyclically that communicates the gear used. The cycle time of the signal may be subject to only small fluctuations even under load; larger fluctuations indicate a problem in the control unit.

For the test, a condition is set that describes the reliable cycle time of the relevant status message. The cycle time must adhere to the limits specified in the condition at all times.

cycleCheck = ChkStart_MsgRelCycleTimeViolation (GearStatus, 0.8, 1.2)
TestAddCondition (cycleCheck);

For the actual test execution, various sequences are run through. For example, shifting during slower and faster driving is simulated, quick change of different gears, quick braking, etc. For this a corresponding system status is represented to the control unit to be tested, for example by setting particular signal values or the connection of hardware inputs/outputs.

A section from such a test sequence might look as follows:

...
SetSignal (Engine::Speed, 120);
TestWaitForTimeout (3000);
SetSignal (Engine::Speed, 100);
...
putValue (IOLine1, 200);  // hardware I/O using environment variables
...

At the end of the test cycle, the condition, the test condition is deactivated once again.  If in the meantime a problem has occurred, this can be recognized from the verdict of the test case / test module / test unit and in the test log. The execution of the test sequence was not interrupted by this

TestRemoveCondition (cycleCheck);
