Introduction to Testing with CANoe

 

You can implement existing test scenarios that are usually sequential with a linear flow. This can be done for test modules (CAPL/.NET/XML) and test units. The former purely event-oriented execution paradigm has been expanded to include a sequential execution model for test modules / test units. The test procedures of the individual test modules / test units are executed in parallel, while the flow within each test module / test unit is executed sequentially.

Each test module/test unit has a beginning (start of the test sequence) and an end (end of the test procedure). Outside of the test runtime, there is the singular initialization phase on prestart that will be called on start of measurement. All other handlers will be called only, if the test module/test unit is currently running. Test modules, test units, and whole test configurations may be executed several times in succession during a measurement. The other familiar event procedures, which previously informed a CAPL node about the start and end of the measurement (on start, on end), are unnecessary in test module / test unit, because their functionality can be added simply to the test module sequence itself and are therefore unavailable.

The variables are set to their initial states at the beginning of the test module / test unit. The contents of a variable cannot — by definition — be transferred from one test procedure to another. Even variables that might have been retrieved during on prestart phase will be reinitialized, it is not possible to calculate something during on prestart and to store the results as simple CAPL variable for further usage.

In the prestart section, neither reporting nor the wait points are available. As a simple and rough guideline, all special test functions (they are prefixed with Test) cannot be used within the prestart section.

If XML test modules are created, the XML file contains the test parameters. The XML file basically consists of the specified test parameters. Programming of procedures is not necessary, since the tests are defined via a query from a parameterized, predefined test/control functions.

Different ways to input test parameters in CAPL test modules / test units include:

  • Fixed coding of parameters in CAPL
  • Read-in of a file at the beginning of the test module / test unit
  • Parameterization by means of environment variables (in which case the user is responsible for creating a defined initial state at the beginning of the test sequence. Environment variables are not automatically changed by the test functionality.)

A test module / test unit / test configuration may be started either automatically with the measurement start or interactively by the user. User inputs may be requested and processed during execution of a test module / test unit. User interaction is helpful if, for example:

  • the test sequence/SUT should be influenced by the user, or
  • observations obtained manually by the user are to be incorporated into the test.

Once a test has been started it usually terminates at the end of the test sequence. It can also be terminated by force by the user, or at the end of the measurement.

Test modules are usually defined in the Test Setup for Test Modules, and are not contained in the network data. They can be named using the context menu.
By default, a test module will be assigned to all buses (similar to a gateway node which has a connection to all buses). A test module can also be specifically assigned to one or more buses via the context menu, if necessary.

Test Units are usually defined in the Test Configuration dialog. The assignment of a test unit to buses happens already in vTESTstudio. A test unit is assigned automatically to all buses configured in the vTESTstudio project.

The user can activate/deactivate the test module / test unit / test configuration prior to the beginning of the measurement (recognizable by the corresponding check box in the test setup). A deactivated test module / test unit / test configuration cannot be started during a measurement. In addition, it's automatic start function is disabled.

Each execution of a test generates a Test Report.
