Test Module Execution Dialog

Context Menu: Execution…

You can use this dialog to control the execution of test modules.
The dialog can be accessed via the test module's context menu or by double-clicking the test module.

As long as the test module is not executed the XML test module provides a context menu to select the test cases to be executed. It contains the possibility to invert the current selection as well as the possibility to select all test cases with result passed or failed. The actions refer to the currently selected element. So you can easily choose e.g. all failed test cases of a test group of the previous execution. The current selection outside this context of the selected element remains untouched.

Note

The execution dialogs for CAPL and XML test modules differ slightly, because the test modules formulated in CAPL contain a CAPL routine (MainTest) which controls the execution. For this reason, it is not possible to manually select the test cases to be executed.

Furthermore for XML test modules with variants an additional box is shown on top of the Test Cases page where you can select the active variant.

Context Menu

  • Expand all
    The selected branch (with sub-branches) will be expanded.
  • Collapse all
    The selected branch (with sub-branches) will be collapsed.
  • Selection
  • Select all "failed" test cases
    All failed test cases of the selected branch (with sub-branches) will be selected.
  • Select all "passed" test cases
    All passed test cases of the selected branch (with sub-branches) will be selected.
  • Invert selection
  • Synchronize Windows
    Chronologically synchronized navigation from the Test Execution dialog to the Trace Window.

Buttons

  • Configuration and Options
    You can open the following dialogs:
  • Test Node Configuration
    Opens the Configuration dialog of the test module.
  • Options
    Opens the Options dialog of the test module.
  • Edit
    The following edit functions are available:
  • Edit test module
    The test module opens in the corresponding editor.
  • Edit Library
    The library opens in the CAPL Browser.
    This option is only available if the libraries have been added to the test nodes. Each library has a separate menu command.
  • Transformed XML Specification(XML test modules only)
    The XML test module is transformed into an HTML file with the help of the XSLT stylesheet specified in the test module configuration. This HTML file is then displayed in an HTML browser.
  • Debug
    Opens the program in the Debugger Window (default setting). In the Options dialog you can select Visual Studio as Debugger for .NET programs.
  • Test cases with result 'failed'
    Only test cases with result failed or test cases that haven't been performed yet will be shown.
  • Reset verdicts
    Resets the verdicts displayed in the Execution dialog on the Test Cases tab.
    In XML test modules, the verdict display is set to not known, while in CAPL test modules, the list of executed test groups and test cases is fully deleted. This does not affect the test report; it only affects the execution dialog display.
  • Open Test Report in CANoe Test Report Viewer
    If you selected the recommended Test Report Viewer Format in the global CANoe options, the following option is available:
  • Display the test report in CANoe Test Report Viewer.
  • Open Test Report in HTML Browser
    If you selected the XML/HTML test report format in the global CANoe options, the following options are available:
  • Display the HTML test report in an HTML browser.
  • Custom transformation
    Select the XSLT file in the following dialog. The result of the last pass of the test unit (stored in the XML file) will be transformed to HTML by using the selected XSLT and the HTML result is shown in the browser. The new HTML file is in the same folder as the XML file and the previously generated HTML file. to distinguish between the HTML files, the new one gets added the name of the used XSLT file.
  • As a further option up to five last used XSLT files are listed. If you select one of them, a new transformation is started as described under Custom transformation.
  • Start
    Starts the execution of the test module.
    This is possible only during a measurement. In addition, the test module must not already be running.
  • Pause
    Interrupts a test module that is running.
    The execution of the test module can be resumed with the Start button. Pauses are possible only after a test case, or in the case of an XML test module, after a test function or control function (see Setting on the Settings tab).
  • Stop
    Aborts a test module that is running.
    The verdict for an aborted test module is always fail.

    Note

    Because test modules are only aborted at the end of a test case, termination does not always take place immediately.
  • Search feature
  • Start search: <Ctrl>+<F>
  • Search forwards: <F3>
  • Search backwards: <Shift>+<F3>

Test Cases Page

A list of the test groups and test cases defined in the XML file is displayed in the XML test module execution dialog. The XML file is read in, at the very latest, when the execution dialog is opened. If errors occur then, they are described in the Write Window.

The test cases are listed in the order that they are defined in the XML file. The list shows:

  • the name of the test case
  • the result of the last test execution (verdict or "not executed")
  • check boxes for selecting individual test cases

When the test module is started, all of the selected test cases are executed sequentially in the order they are listed in. Test cases which are not executed have a result of "not executed", which is also noted in the test report.

The CAPL test module execution dialog only displays those test groups and test cases which have actually been executed, along with their verdicts. This list is only generated when the CAPL test module is executed.

When saving a configuration the selections (check box) of the test cases are saved, too. The next time the configuration is loaded the test cases that have been selected last time are selected, again; all other test cases are unselected. If errors occur on doing so, e.g. test cases have been renamed, moved or deleted, they are described in the Write Window.

In XML test modules, that contain variants, a list is shown on top of the page. There you can select the active variant.

The test cases tree can be folded/unfolded by using the +/- symbols or by double clicking the folders, whereas the former state may be rebuilt. Additionally you can use the context menu for easy folding/unfolding down to the lowest level.

During the execution of the test module the currently running test case is tracked; this means that it is selected in the list and the list scrolls so that the entry is always visible. This automatic tracking of the execution is stopped as soon as you select a test case or a test group with your mouse/keyboard or scroll manually during a test execution. Use the Track execution command from the context menu of the list to activate/deactivate the tracking during a test execution.

Test Observer Page

On this page the execution dialog all messages of the current test module are displayed while passing through. This visualizes the current test execution.

The information shown is a selection of the information that will be written to the test report.
