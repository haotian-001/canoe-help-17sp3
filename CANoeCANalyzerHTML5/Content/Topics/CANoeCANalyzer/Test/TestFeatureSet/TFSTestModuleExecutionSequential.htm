Test Module: Sequential Execution of Several Test Modules

Context Menu of a Test Environment: Test Module: Sequential Execution of Several Test Modules

As a rule, tests that belong together should be stored in a single test module. The test cases for different test areas can be arranged hierarchically as test groups within the test module. The test cases in a test module are executed sequentially and the results are recorded in a test report.

Starting Several Test Modules

In certain cases, there may be test modules that cannot be combined in a single test module. To avoid having to start each test module separately, CANoe offers the option of storing test modules in a given folder and, within certain limits, treating them as a unit.

By selecting Execute test modules sequentially from the context menu for this folder in the test setup it is possible to start the execution of all the test modules during a measurement. In this case, all the test modules in this folder are executed sequentially according to the order in which they are sorted in the folder. Deactivated test modules (check box in test setup) are skipped.

Important Note

  • This menu command is only available when the measurement is running.

  • Furthermore, the folder must contain only CAPL and XML test modules and must not include any subfolders.

The individual test modules in this folder remain independent of each other. The menu command simply provides a way to start the different test modules automatically in a defined sequence. The test modules can still be started separately, still run in separate, independent windows, and each test module generates its own test report.

Combined Test Report

Even if several test modules are executed sequentially as described above, each test module generates its own separate test report.

If you selected the XML/HTML test report format in the global CANoe options, it is possible to combine these different test reports in a single test report by using the Create test report menu command. This menu command merges the XML test reports by means of an XSLT script (the merge.xslt script located in the Exec32/report directory of the CANoe installation) to create a new combined XML test report and afterwards transforms it to HTML. If automatic numbering of test reports is activated, the respective est reports with the highest numbers are used.

The information for each test module is placed in a separate test group, while any extra information is simply copied. The header information about the device being tested (SUT), the test engineer responsible and the relevant test setup are taken only from the first test report, but this information should normally be identical for all the test reports in question.

Important Note

  • The option of creating merged test reports should be used cautiously.

    Since this function simply compiles the test reports stored on the hard disk, there is no guarantee that these test reports actually fit together regarding their content.

    It may happen that previous test reports from very different test runs are merged together in one test report.

    Furthermore, contradictory information (e.g. regarding the different test configurations or CANoe versions used) is not resolved, but simply compiled indiscriminately or extracted from the first test report.

After the XML test reports are merged, the new combined XML test report is converted to HTML format. This process is exactly the same as with the test reports for individual test modules. The name of the combined XML test report and the XSLT script used for transformation is specified in the configuration dialog, which can be accessed via Configure test report... in the context menu of the respective folder in the test setup.
