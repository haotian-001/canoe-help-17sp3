Constraints and Conditions

 

Using constraints and conditions you can check the adherence to conditions in a particular time interval. These checks occur parallel to the execution of the test case.

  • Constraints are used to monitor the test environment/configuration. They can ensure that the test environment/configuration does not assume a state during the test that hinders correct test execution and thus leads to a possibly-incorrect test result.
  • Conditions are used to check the system to be tested during the test procedure. This must often adhere to conditions that may not be injured during the test procedure. If these conditions are injured, then there is a failure of the test item and the test fails. See also Testing with conditions.

There is no functional difference between constraints and conditions regarding the verdict.

Any number of constraints and conditions for parallel monitoring may be used.

  • In XML test modules always complete test cases, test groups or test modules are monitored.
  • In CAPL test modules/test units Constraints and Conditions may be added and be removed from the active monitoring. This can occur for several test cases together on the level of the test module/test unit or within test cases. In the latter case, however, the monitoring is ended automatically with the end of the test case.

If a constraint or condition is injured, this is noted in the test report and it causes failure of the current test case (and thus failure of the entire test module/test unit). However, the test is not aborted. The other test cases of a test module/test unit are executed.

Constraints and conditions can be defined easily with the functions (checks) of the Test Service Library.

Example (XML)

<testcase ident="TC1" title="Check cycle time">
  <conditions>
    <cycletime_rel min="0.8" may="1.2">
      <canmsg id="StatusMsg" />
    </cycletime_rel>
  </conditions>
...cycle time of message StatusMsg will be checked ...
</testcase>

Example (CAPL)

checkID = ChkStart_MsgRelCycleTimeViolation (StatusMsg, 0.8, 1.2);
TestAddCondition (checkID);
  // cycle time of message StatusMsg will be checked
TestRemoveCondition (checkID);
