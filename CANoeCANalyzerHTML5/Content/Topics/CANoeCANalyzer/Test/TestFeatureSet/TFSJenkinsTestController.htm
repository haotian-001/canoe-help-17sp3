Jenkins as CANoe Test Controller

 

»Jenkins is a widely used continuous integration and delivery platform.

Using the CANoe COM Interface you are able to integrate CANoe jobs into your Jenkins infrastructure.

The following text provides basic information and hints on how to do so. It also provides a simple step-by-step example.

Important Note

For licensing reasons, the test setup described here is only permitted when testing real hardware!

This page focuses on Jenkins. But the general ideas are applicable to other continuous integration systems. Since the tool integration is based on simple operating system mechanisms, you can easily adapt the approach to other tools.

Integration Strategies

For several use cases a binary decision might be sufficient. This can be achieved by a simple batch script returning errorlevel information. This information can be used to decide whether the Jenkins job was successful. You can specify additional files produced by the simulation such as test reports and logging files to get a detailed view of the simulation result. The general result as well as the additional files can be directly accessed through the Jenkins web interface.

If your simulation is running tests, Jenkins provides a more specific view on JUnit test results. To benefit from this feature you can provide a simple transformation from CANoe test results to JUnit test results.

The following sections provide a step-by-step example on how to integrate a CANoe simulation into Jenkins.

Sample Configuration

CANoe provides a sample configuration Central Locking System, which can easily be adjusted for usage in Jenkins. Load Configuration

The script StartAutomaticTests.bat can be directly started without parameters: Open Folder

The file parametrizes the »VBScript AutomaticTest.vbs which controls CANoe:

  • it starts CANoe
  • runs the simulation and the test modules of CentralLockingSystemNET.cfg
  • reports information to a log file (AutomaticTests.log) and on the console
  • closes CANoe at the end
  • reports test success as errorlevel variable.

Try it: open a command prompt in the CentralLockingSystem directory and type Script\StartAutomaticTestsBinary.bat. It should report all test modules successfully executed.

Jenkins Configuration

Now you can use the StartAutomaticTests script in Jenkins.

Preconditions:

  • Install Jenkins on the computer (Jenkins server) executing the tests.
  • Install CANoe and drivers on the same computer (Jenkins server).

In Jenkins Administration Web UI:

  1. Create New item.
  2. Select ( * ) Freestyle project.
  3. Set a project name Execute CANoe sample.
  4. Open the Jenkins Advanced project options and set [ ] Use custom workspace to the CentralLockingSystem folder and set the CentralLockingSystem sample configuration folder.
  5. Add build step: Execute Windows batch command
    Set Command to file path Script\StartAutomaticTestsBinary.bat

Now, you can run the Jenkins job, and it reports success or failure of the CANoe tests.

To see the complete CANoe reports of test modules, you switch to the Jenkins job configuration and add post-build action: Archive the artifacts. The CANoe configuration places the test report files directly into the configuration directory. So you set Files to archive to **\*.vtestreport.

The next job execution will link the test reports as Build artifacts on the job execution page.

Advanced Options in Jenkins

Instead of copying the CANoe configuration to your Jenkins system, you can retrieve it from a source code management system (SCM).

Instead of running the tests manually you can use a trigger (e.g. on changes in SCM) or periodically or after other Jenkins jobs.

You can parameterize the test job to pass the parameters on job start, so different settings can be used. They can be passed to the execution script. You could also choose different CANoe configurations. The StartAutomaticTests.bat already contains parameters for the configuration file and the logging file.

You can set up a number of Jenkins slaves to execute tests in parallel or sequentially. Each machine needs a CANoe installation, too.

Use Jenkins JUnit-Plugin to Report Test Cases

The sample job developed (see Jenkins Configuration) reports success or failure in binary form. You can only see how many tests were executed and how many reported a failure by inspecting the test reports linked to the job.

This sample shows how you can directly and easily track the number of executed and failed test cases in Jenkins Web-UI.

So you create a new Jenkins job, using a different batch file, since you want it to always report success when the script is executed without errors.

This sample uses the Jenkins-XUnit-Plugin. Please install it using the Jenkins-WebUI.

In Jenkins Administration Web UI:

  1. Create New item.
  2. Select ( * ) Freestyle project.
  3. Set a project name Execute CANoe sample and evaluate test results.
  4. Open the Jenkins [Advanced project options] and set [ ] Use custom workspace to the CentralLockingSystem folder and set the CentralLockingSystem sample configuration folder.
  5. Add build step: Execute Windows batch command.
    Set Command to Script\StartAutomaticTests.bat.
  6. Add build step: Execute Windows batch command.
    Set Command to Script\XmlExportAll.bat.
  7. Add post-build action: Archive the artifacts Set Files to archive to **\*.vtestreport.
  8. Add post-build action to your Jenkins job.
    Select action Publish xUnit test result report.
  9. Add Custom Tool
    Configure Custom Tool Pattern to *.xml and Custom stylesheet to Exec64\testunit_report\xunit-impl\xunit.xslt.

The execution script StartAutomaticTests.bat will not report failing tests using the error level variable anymore. Instead you use an additional post process step to transform the CANoe test reports to a form that can be interpreted by the Jenkins-xUnit plugin (steps 7 and 8).

Now you get a nice graphical view of the number of executed and successful tests in the Jenkins Web-UI:

More Control

As an alternative to the xUnit transformation, you can also choose your own transformation processor or even decide to extract the information needed directly from the script controlling CANoe execution instead of using the CANoe test report.

Exec64\testunit_report\xunit-impl\xunit_msxml.bat calls xunit_msxml.vbs to transform the CANoe test report XML files to JUnit compatible format. It uses »MSXML, which is installed with CANoe, so it should be available on your system.

They can be used in Jenkins by appending a new build step after the script execution: Add build step: Execute Windows batch command. Command: Exec64\testunit_report\xunit-impl\xunit_msxml.bat reportNETTester.xml reportNETTester.junit.xml

View CANoe XML Test Report Files in Jenkins

If your CANoe configuration uses XML/HTML report files, you can view the HTML reports directly in Jenkins. In this case remove the build step XmlExportAll.bat, which transforms the *.vtestreport to *.xml and change the files to archive to from *.vtestreport to *.html:
  • This command is not necessary: Set Command to Script\XmlExportAll.bat
  1. Add post-build action: Archive the artifacts Set Files to archive to **\*.html.

Due to security reasons a "Content Security Policy" in Jenkins does not allow to display the HTML files including their embedded *.css styles. To have the reports displayed correctly you need to relax the Jenkins security policy:

Install the Jenkins HTML Publisher Plugin.
After that execute in the Jenkins Script Console the following line:

System.setProperty("hudson.model.DirectoryBrowserSupport.CSP", "sandbox; 'unsafe-inline';")

Extensibility

The sample script can be extended to use only specific or all test configurations of a CANoe configuration.

This sample uses the CANoe API to run test modules. It can easily be extended to run test unit, too.

It can switch on or off specific test configurations, test modules or single test cases.

It can also control the execution of test units.

It can do the JUnit transformation directly based on the report filenames inquired via COM API.

If you are running VN8900 standalone tests, you can also invoke and observe them using Jenkins. Check the help of the Standalone Manager for Command Line Interface options.

Limitations

CANoe requires an interactive session; therefore, the Jenkins agent cannot be run as a service in Windows.

Virtual Machines

Running CANoe in a virtual machine is possible.

  • Note that operation with Vector hardware may be affected by virtualization (e.g., higher latencies may occur).

  • Make sure CANoe can acquire a license and has access to the hardware required by the tests.
