Wait Points

 

CAPL

Typically stimuli for the SUT are generated within the test case followed by waiting for the reactions. The expanded CANoe execution model for testing permits real waiting for the most important events:

  • Receipt of a message with a specified ID
  • Writing of an environment variable with a specified name
  • Signaling of a user-defined text event
  • Signaling of events from the supplied TestServiceLibrary
  • Signaling of events from other Node Layer Modules
  • Expiration of a time period, and
  • AND / OR combinations of the above conditions

A wait point is exited (resume) once the specified condition or a timeout occurs.

The reason for a "resume" (i.e. the event that occurred) can be found in the return value of the wait instruction. If a message initiated the resume, the initiating message may be called in CAPL from immediately after the wait point until just before another wait point is executed.

.NET

As in CAPL, it is possible to wait for various events in .NET test modules. There are wait points for:

  • chronological conditions
  • bus signals, environment variables and system variables
    A value condition and timeout can also be specified.
  • custom event procedures with and without timeouts
  • composite conditions composed of the above two

Note

In .NET, wait points are resolved as soon as they are called if the condition is already met at that point in time.

Example of a custom wait point.

So-called criteria were used:

[Criterion]
[OnChange(typeof(MotorControl.OnOff))]
[OnChange(typeof(LightState.OnOff))]

public bool AreMotorAndLightsOn()
{
return MotorControl.OnOff.Value == 1 && LightState.OnOff.Value ==     1;
}

public void WaitForMotorAndLights()
{
Execution.Wait(AreMotorAndLightsOn);
}

Here, the event procedure is active only as long as the wait condition is active. This is specified via the Criterion attribute. Criteria can be used together only with OnChange procedures.

Note

A criterion can be used for both wait points and checks.
