Simulating a Slave [▲ back]

For testing I2C masters, CANoe offers the possibility to simulate I2C slaves. This allows you to stimulate the master with predetermined values and analyze its outputs.

Configuration of the Slave Simulation [▲ back]

For the simulation of an I2C slave by CANoe, it is necessary to set slave mode to simulated in the Sensor Configuration dialog.

Behavior of the Simulated Slave [▲ back]

The behavior of the simulated I2C slave is based on a simple model. It has a queue containing predefined replies to master requests. You can fill this queue by calling CAPL functions or modifying system variables with desired values. Once the master starts reading data from the slave, the simulated slave outputs the first reply from the queue. Once the communication is finished, the slave jumps to the next element in the queue. If there are no more elements in the queue, the simulated slave always replies using the last sent element.

Preparing the Slave Replies [▲ back]

There are two ways to define replies in the simulated I2C slave out of the CANoe:

  • Sending via system variable
    To do so, set the TX trigger member of a stimulation frame system variable, for example via CAPL or in a panel. The prerequisite for this process is that you have configured at least one suitable frame in the Sensor Configuration dialog.
  • Sending via CAPL function
    To do so, call the QueueFrame function on the namespace of the slave. You can transfer a byte array to this function which will be added to the reply queue of the slave.

Simulating a Master [▲ back]

For analyzing and testing I2C slaves, CANoe offers the possibility to simulate an I2C master. This allows you to stimulate the slave and also display and evaluate data sent by it in CANoe.

Configuration of the Master Simulation [▲ back]

For the simulation of an I2C master by CANoe, it is necessary to set the master mode to simulated in the Sensor Configuration dialog.

Behavior of the Simulated Master [▲ back]

An I2C master simulated by CANoe does not send any frames to the slaves on its own. Instead, you must actively initiate the sending process.

Sending Frames [▲ back]

The following options are available for sending frames to the I2C slaves:

  • Sending via system variable
    To do so, set the TX trigger of a stimulation frame system variable, for example via CAPL or in a panel. This will include the related frame into the sending queue of the master and send it to the slave. The prerequisite for this process is that you have configured at least one suitable frame in the Sensor Configuration dialog.
  • Sending via CAPL function
    To do so, call the QueueReadFrame, QueueWriteFrame or QueueCombinedFrame function on the namespace of the slave.

Analysis of the Communication between Real Devices [▲ back]

The communication between a real I2C master and a real I2C slave can be logged and analyzed it in realtime. The mode of the master and that of the slave must both be set to real in the Sensor Configuration dialog. The data received from the master and the slave will then be reported in CANoe via the corresponding system variables.

Representation of Frames in CANoe [▲ back]

Raw Frames [▲ back]

Whenever communication occurs on the bus, corresponding raw frame events will be generated. These events will be reported in the following way:

  • When the master creates a START or a REPEATED START condition on the bus, the event RawFrameStart will be generated.
  • After the slave address has been transmitted by the master, a RawFrameAddress event will be generated.
  • Then RawFrameReadData or RawFrameWriteData events will be reported for each data byte transmitted on the bus.
  • When the master creates a STOP condition, a RawFrameEnd event will be generated.

Note

You have to explicitly activate the generation of raw frame events in the Sensor Configuration dialog. Otherwise only configured frames will be reported.

Configured Frames [▲ back]

In addition to the raw frame events you can configure custom frames in the Sensor Configuration dialog. This allows CANoe to perform an interpretation of the raw frame data on signal level.

A configured frame usually consists of a write sequence followed by a read sequence. If required you may omit one of the two sequences. This way you can configure a frame that only has a write or a read sequence.

Each sequence can be up to eight bytes long and may contain an arbitrary number of signals. The signals will be available as system variables in CANoe and provide various convenience settings (e.g. automatic bit order interpretation, scaling with factor and offset).

To allow CANoe matching received raw data to configured frames, all frames must be uniquely identifiable. This can be achieved by making sure that:

  • Configured frames have different sequence lengths
  • Configured frames with identical sequences have constants (e.g. register addresses) with different values.

Low Level Events [▲ back]

In some cases it may be required to have more detailed information on the bus communication than the raw frame events provide. For this purpose you can enable the generation of low level events in the Sensor Configuration dialog. Once this setting is active, every transmitted data byte will be reported with an exact start and end time stamp.
