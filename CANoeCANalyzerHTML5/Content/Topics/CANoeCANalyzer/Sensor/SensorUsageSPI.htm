Simulating a Slave [▲ back]

For testing SPI masters, CANoe offers the possibility to simulate SPI slaves. This allows you to stimulate the master with predetermined values and analyze its outputs.

Configuration of the Slave Simulation [▲ back]

For the simulation of an SPI slave by CANoe, it is necessary to set slave mode to simulated in the Sensor Configuration dialog.

Behavior of the Simulated Slave in the Basic Mode [▲ back]

In the basic mode, the behavior of the simulated SPI slave is based on a simple model. It has a queue containing predefined replies to master requests. You can fill this queue by calling CAPL functions or modifying system variables with desired values. Once the master starts communication (chip select becomes active), the simulated slave outputs the first reply from the queue. Once the communication is finished (chip select becomes inactive), the slave jumps to the next element in the queue. If there are no more elements in the queue, the simulated slave always replies using the last sent element.

This mode is suitable for simulation of simple slaves which, for example send uniformly structured data in each frame. This mode can also be used for continuous value ranges.

Preparing the Slave Replies [▲ back]

There are two ways to define replies in the simulated SPI slave out of the CANoe:

  • Sending via system variable
    To do so, modify the value of a MISO_Stim variable, for example via CAPL or in a panel. The prerequisite for this process is that you have configured at least one frame with MOSI signal for the slave to be addressed in the Sensor Configuration dialog.
  • Sending via CAPL function
    To do so, call the QueueMisoData function on the namespace of the slave. You can transfer a byte array to this function which will be added to the reply queue of the slave.

Simulating a Master [▲ back]

For analyzing and testing SPI slaves, CANoe offers the possibility to simulate an SPI master. This allows you to stimulate the slave and also display and evaluate data sent by it in CANoe.

Configuration of the Master Simulation [▲ back]

For the simulation of an SPI master by CANoe, it is necessary to set the master mode to simulated in the Sensor Configuration dialog.

Behavior of the Simulated Master [▲ back]

An SPI master simulated by CANoe does not send any frames to the slaves on its own. Instead, you must actively initiate the sending process. The simulated SPI master automatically operates the chip select lines of the slave during transfer. This means that you do not have to set them explicitly to active or inactive.

Sending Frames [▲ back]

The following options are available for sending frames to the SPI slaves:

  • Sending via system variable
    To do so, modify the value of a MOSI_Stim variable, for example via CAPL or in a panel. This will include the related frame into the sending queue of the master and send it to the slave. The prerequisite for this process is that you have configured at least one frame with MOSI signal for the slave to be addressed in the Sensor Configuration dialog.
  • Sending via CAPL function
    To do so, call the QueueMosiData function on the namespace of the slave. You can transfer a byte array to this function which then will be written into the sending queue of the master for this slave.

Analysis of the Communication between Real Devices [▲ back]

The communication between a real SPI master and a real SPI slave can be logged and analyzed it in realtime. The mode of the master and that of the slave must both be set to real in the Sensor Configuration dialog. The data received from the master and the slave will then be reported in CANoe via the corresponding system variables.

Representation of Frames in CANoe [▲ back]

Generally, there two options for representing the received or sent SPI frames in CANoe:

  • User-defined frames
  • Raw frames

In both cases, a MISO and a MOSI system variable are created for each frame. These two variables are always simultaneously updated by CANoe when a corresponding frame is detected.

User-defined Frames [▲ back]

In the Sensor Configuration dialog, you can create one or several frames and define their structure for each slave. CANoe uses this information to analyze and correspondingly represent the incoming data. For example, signals within the frames are displayed as system variable members.

Raw Frames [▲ back]

If you have not configured frames, the incoming data is displayed as raw frames. This is also the case if the incoming data does not match any frame defined by you. In the raw frame, the MISO or MOSI data is represented as a simple byte array.
