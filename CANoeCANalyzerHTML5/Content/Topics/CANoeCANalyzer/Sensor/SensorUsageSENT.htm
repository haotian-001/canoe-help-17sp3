Overview of the SENT System Variables [▲ back]

If you have configured a SENT network in the Sensor Configuration dialog, CANoe automatically creates the corresponding system variables.

Simulating a Sensor [▲ back]

CANoe provides the opportunity to simulate SENT sensors for the testing of ECUs. This allows you to simulate ECUs with predetermined sensor values.

Configuration of the Sensor Simulation [▲ back]

For the simulation of a sensor with CANoe, it is necessary to set the sensor mode to "simulated" in the Sensor Configuration dialog.

Starting behavior of the Sensor Simulation [▲ back]

CANoe starts with the simulation of the sensor at the start of a measurement. First the sensor waits until the required supply voltage is applied by the ECU. This threshold can be defined in the Sensor Configuration dialog. As soon as the required voltage has been reached, this is reported by the PowerStatus system variable. The sensor is now active, but it doesn't send any data yet. The start time of the sensor is therefore simulated. The length of the start time can be set in the Sensor Configuration dialog.

Behavior of the Sensor in Case of a Voltage Loss [▲ back]

If a supply voltage loss occurs during the measurement, the sensor will stop sending. This error is reported in CANoe by the PowerStatus system variable.

As soon as the required supply voltage is reached again, the sensor will start with its normal starting sequence (see above). Also this event is reported by the PowerStatus system variable.

Sending of Simulation Data [▲ back]

CANoe provides different options to affect the data sent by the simulated sensor. These options are described in the following sections.

The hardware channel repeats the last sent frame until new data are set by CANoe.

  • Sending of signals via system variables
    CANoe creates a system variable for each signal that has been configured in the Sensor Configuration dialog. The signal values can be modified through, for example, a panel or a signal generator. The modification of the variables is transmitted to the hardware channel in which the new value will be initiated in the sent frame.
  • Sending of frames via system variables
    Similarly to sending signals via system variables, you can also send entire frames. To do this, modify the value of the frame simulation variables. This value is then transmitted to the hardware channel in which the new frame will be sent.
    The frame simulation variables provide access to the status and data regions of the frame. The CRC checksum is calculated automatically and therefore it cannot be set.
  • Sending of signals with CAPL
    The values of the sent signals can also be modified via CAPL. To do this, select the QueueValues method on the corresponding signal system variables. Alternatively, you can also use the CAPL function sensorQueueValues which expects the names of the signal system variables as first parameters. In both cases, an array of signal values is transmitted which will then wait in a queue.
  • Sending of frames with CAPL
    To send entire frames via CAPL use the QueueSentFrame or sensorQueueSentFrame function. A description of the frame is expected as individual parameter.
  • Sending of serial messages with CAPL
    The CAPL function QueueSerialMessage or sensorQueueSerialMessage allows you to prepare the sending of serial messages. On the one hand the message ID is transmitted as parameter. On the other hand a flag is transmitted that allows you to select the type of message:
  • Long ID: the ID is 8 bits long, the data are 12 bits long.
  • Short ID: the ID is 4 bits long, the data are 16 bits long.

    • The latter refers only to sensors that are used as enhanced serial messages. If the sensor is configured for short serial messages, 4-bit IDs and 8-bit data are always sent.
    The current data of the message are not transferred to the CAPL function. These data are taken from the signal system variables assigned to the corresponding message IDs.

  • Logging and replaying of the sensor communication
    The logging function of CANoe allows you to log the communication of a real sensor. This log can be later replayed by a simulated sensor. To do so, proceed as follows:
  1. Load or create a CANoe configuration with a real SENT sensor.
  2. Activate the Logging Block in the Measurement Setup of your CANoe configuration.
  3. Select BLF as output format.
  4. Start the measurement in order to allow the communication of the sensor to be logged.
  5. Create a new CANoe configuration with a simulated SENT sensor.
  6. In the configuration of the SENT channel dialog select the BLF logging file as data source.
  7. In the configuration of the simulated SENT sensor dialog activate the replay from the logging file.
  8. Select the names of the sensor whose logging file you want to replay.

    9. The data are sent from the logging file through the simulated sensor at the start of the measurement.

Simulating an ECU [▲ back]

CANoe provides the opportunity to simulate a SENT ECU for the analysis and testing of sensors. The data sent from the sensor can be displayed and evaluated in CANoe.

Configuration of the ECU Simulation [▲ back]

For the simulation of an ECU with CANoe, it is necessary to set the sensor mode to simulated in the Sensor Configuration dialog.

Behavior of a Simulated ECU [▲ back]

As soon as the measurement is started in CANoe, the simulation of the SENT ECU will begin. This allows the supply voltage for the connected sensor to be generated.

Analysis of the Communication between two Real Devices [▲ back]

The Option .Sensor allows you to log the communication between a real ECU and a real sensor and to analyze it in realtime. The mode of the ECU and that of the sensor must both be set to real in the Sensor Configuration dialog. The data received from the ECU and the sensor will then be reported in CANoe via the corresponding system variables.
