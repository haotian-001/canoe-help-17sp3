Network-based Access

 

New Concept for Ethernet VN Family since Firmware Version 11.1

Application note AN-IND-1-023_Ethernet_VN_Family_From_Firmware_Version_11.1.pdf describes how the concept behind the VN Ethernet interfaces was changed. These changes allow for greater flexibility in interface segmentation and in scaling both the number of network connectors and the simulation and analysis performance.

At the end of this page you will find two examples to illustrate the following terms.

Requirement

To work with the VN Ethernet Interface, you need a network-based configuration

You can activate the network-based access with the Access network via Ethernet ports option in Options dialog or by using the network-based templates.

Hardware Configuration

The following elements play a central role in the hardware configuration and are discussed from the CANoe perspective.

Network

In CANoe, it is no longer necessary to define a separate Ethernet network for each switch, TAP or direct connection. It is now possible to combine all nodes that can communicate with each other via layer-2 (L2) switches into one network.

Note

The network definition for the network-based access is a L2 network (i.e. the broadcast domain). All nodes can communicate with each other via L2-switches (VLANs not considered).

Give a meaningful name to an Ethernet network so that its use is unique. A network defined in CANoe and a network defined in the Ethernet device configuration automatically find each other when they have the same name.
Segments

The VN Ethernet Interface offers the segment types Switch and Link. The Link can be used as a TAP or direct connection. In most cases, the Switch can be used.

A Switch connects different network nodes (L2-Switch).

A TAP is usually used when you want to measure and analyze the network traffic on a certain cable in your network.

The cable is split in two and both ends are inserted into the TAP. Now the network traffic can be measured. The TAP consists of two ports (physical). When measuring the TAP, only one port needs to be added in CANoe.

A direct connection can be used if you want to simulate one node in CANoe (communication endpoint). The only difference with a switch is that the CANoe simulation can react to the status of the connection (link up/down, sleep/wake-up). If you do not want to use that, we recommend using a switch.

Ports

Ports are the access points for applications on the network. They can be used for simulation (read/write access) or for measurement (read access).

Give a meaningful name to a port in the Ethernet Device Configuration so that its use is unique.

You open the Ethernet Device Configuration via the Network Hardware Configuration/Vector Hardware Config dialog.

Simulation Ports

For simulation access, CANoe will use simulation ports.

To do this, create network nodes for the simulated participants in the Simulation Setup. These are then automatically displayed as simulation ports in the Port Configuration dialog. You can also create virtual ports as simulation ports in the Ethernet Device Configuration.

When creating the simulation port, CANoe will check first if the port is already available in the Ethernet Device Configuration. If so, it will use this port. If no port is available, one will be created automatically. If only one segment is available in the network, the port will automatically be linked to that segment. If multiple segments are available, CANoe will ask which segment the port should be linked to.

Measurement Ports

Real hardware is connected with measurement ports. They are add as physical or virtual ports in the Ethernet Device Configuration and displayed for the corresponding network in the Port Configuration dialog. To use them in CANoe, you must manually configure them as measurement ports in the Port Configuration dialog.

Simulated Mode

When starting the CANoe simulation in Simulated Bus mode, the simulation ports are connected via a single, simulated L2-switch. The measurement ports are available, but do not measure Ethernet pakets.

Changes

The following changes have been made:

  • The hwChannel selector is in CAPL no longer supported for the ethernetPacket object. Instead of the selector, you must now always specify the port starting with ethernetPort in the following notation:

    • ethernetPacket txPacket;
    txPacket.hwPort = ethernetPort::NetworkName::PortName;

  • The on ethernetPacketForwarded event procedure is no longer supported.
  • The TX/RX direction is based on the network interface perspective. This makes tests independent of the port (physical or virtual), i.e. whether the ECU is simulated or if a real network node is connected. As an indication of whether the events were generated by the CANoe simulation, the Sim column in Trace window can be used.
