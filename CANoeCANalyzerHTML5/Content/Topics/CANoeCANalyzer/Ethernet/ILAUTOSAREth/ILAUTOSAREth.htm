AUTOSAR Eth Interaction Layer (AUTOSAR/IP IL)

 

This interaction layer involves a simulation of AUTOSAR PDUs that are to be transmitted via Ethernet. For example, the IL implements the send behavior of the connections for this. This includes the collection mechanism of AUTOSAR PDUs, so that multiple PDUs are transmitted together in one Ethernet packet. In addition, the interaction layer supports the service discovery in order to activate the transmission of PDUs and to negotiate the connection data. The interaction layer you find under the filename CANoeILNL_AUTOSAR_ETH.vmodule in the Exec64 directory of your CANoe installation.

AUTOSAR PDUs with COM Signals

For AUTOSAR PDUs that contain COM signals, an additional interaction layer must be added in order to define the bus-independent portion of the send behavior. This second interaction layer might implement the cyclic sending of a PDU, for example. Without an additional IL, these PDUs are sent only at a signal change. A possible interaction layer for the bus-independent send behavior is supplied in the Exec32 directory. The name of this additional interaction layer is AsrPDUIL.vmodule.

AUTOSAR in Connection with SOME/IP Messages

SOME/IP messages can be simulated without AUTOSAR mechanisms. The SOME/IP Interaction Layer is supplied for this. On the other hand, the SOME/IP messages can be transported using AUTOSAR PDUs, which is supported by the AUTOSAR Eth interaction layer. The messages resulting on the network correspond to those of SOME/IP messages without AUTOSAR PDUs, because the resulting headers are the same. For this reason, the simulation of the SOME/IP interaction layer and the AUTOSAR Eth interaction layer may look exactly the same. However, since the latter implements the send behavior of the connections and thus the assembling of collections, the SOME/IP messages can be added in different Ethernet packets than in the SOME/IP interaction layer.

There are two other differences with regard to simulation of SOME/IP using AUTOSAR PDUs compared to simulation using the SOME/IP interaction layer.

End-to-end protection (E2E)

Another difference of the AUTOSAR Eth interaction layer compared to the SOME/IP interaction layer is the support of end-to-end protection. In the case of the AUTOSAR Eth interaction layer, this is implemented according to the AUTOSAR database. The CRC value is calculated and the sequence counter is increased before sending. Message Callback

The callback function OnAREthMessage is only called for received AUTOSAR PDUs if the headerID in the received header was explicitly assigned for a SOME/IP message. The identifier can be in the database or it can be made by manually adding events, fields, or methods. The manual addition takes place programmatically (via CAPL). Further information about this under SOME/IP IL: Providing a Service.

Programming with CAPL

The same CAPL function scope is available as for the SOME/IP interaction layer, whereby the function names have different prefixes. Instead of SomeIp the functions begin with AREth. However, to enable fast switching between the interaction layers, the AUTOSAR Eth interaction layer also supports all functions with the prefix SomeIp.
