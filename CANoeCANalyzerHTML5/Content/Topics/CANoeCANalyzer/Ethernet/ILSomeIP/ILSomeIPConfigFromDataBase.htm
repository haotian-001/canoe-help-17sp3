SOME/IP IL: Configuration from Database

 

You can configure a node in the Simulation Setup automatically from the database if:

  • The node was defined in a FIBEX database (since version 4.1) or AUTOSAR database (since version 4.2)
  • The database was assigned to the Ethernet network
  • The SomeIP_IL.vmodule file was assigned to the node as a component in the Simulation Setup

For an automatically configured node, all database objects (e.g., Service Instances, Eventgroups, Methods, Events, or Fields) are created and managed by the SOME/IP Interaction Layer (SOME/IP IL). CAPL code is no longer needed for this. Only the send behavior, for example, the cycle time of a SOME/IP service event notification, must be specified in CAPL (see also SomeIpSetProperty).

How to configure a node automatically from a database

Configure the node (step 2-6)

Start the measurement. If a receiver (real or simulated) exists for the setup node, the information required within the scope of the service discovery is now exchanged.

A few database objects can define CAPL callbacks in order to set or accept values of received or sent SOME/IP messages. If a node is configured automatically from the database, the values of these callbacks are set to 0. In this case, for example, the appropriate response to a received Method query is sent automatically. The content of the response is initialized with 0 by default and returns the value E_OK. You will learn how you can change this response behavior below.

How to register a SOME/IP callback

If you want to respond to a received Method query, you must register the missing callback in CAPL. Use the SomeIpRegisterCallback function for this. This also applies to all other available callbacks.

The SomeIpRegisterCallback function requires an object handle as the first argument. This object handle is created and managed in CAPL in the case of manual configuration of the node. In the case of automatic configuration, the SOME/IP Interaction Layer creates these handles for all provided or consumed objects. The following functions are available for calling these automatically created handles from the SOME/IP Interaction Layer:

  • SomeIpGetProvidedObjectHandle
  • SomeIpGetConsumedObjectHandle

In these functions, a symbolic qualifier is used to specify the needed object. This qualifier corresponds to the symbolic object name from the assigned database.

How to set the cycle time of an Event

The handle of an Event is needed, for example, to send the Event cyclically. Similar to the registration of a callback, the Event handle is obtained from the SOME/IP Interaction Layer using the SomeIpGetProvidedObjectHandle function and can then be used to set the cycle time using the SomeIpSetProperty function.

Symbolic database access with SOME/IP

SOME/IP identifies database objects using the following syntax:

[network name][package name::][service name::]method|event|field|event group name[[::major version[::minor version]]::instance id][::Reliable]

or for Service Instances

[network name][package name::]service name[[::major version[::minor version]]::instance id][::Reliable]

The names required for identification of the objects must each be separated by the character sequence "::". The complete qualifier is parsed from left to right. Note that the syntax is case sensitive. The parameters given inside square brackets (“[]”) are only needed when ambiguities exist, in order to explicitly identify a database object. In the case of ambiguities, the first object found in the database is returned.

Method, Event, and Field names are handled with higher priority than Eventgroup names. If a Method has the same name as the Eventgroup, the handle will be returned to the Method in this example.

Note

In AUTOSAR ECUs and System Extracts of System Descriptions it is not possible to provide any symbolic service names or event group names. CANoe overcomes this situation by creating every not-provided symbolic names itself.

Rules of creation: for services "sif_xxxxx" and for event groups "evg_xxxxx". xxxxx defines the service or event group identifier in decimal format.

Access to Field objects

The handles of Setter and Getter Methods of a Field are obtained by setting one of the following prefixes before the corresponding Field names.

  • set_ for Setter Methods
  • get_ for Getter Methods

Application endpoints

Service Instances can be provided or consumed under various application endpoints. A UDP endpoint (unreliable) and a TCP endpoint (reliable). In this example, the Service Instance of the TCP endpoint would be addressed by adding the suffix Reliable to the qualifier. An Unreliable qualifier does not exist and makes the qualification invalid. If the handle of the Service Instance that is provided/consumed via the UDP endpoint is needed, an additional suffix is not necessary in the qualifier.

Access without symbolic names

If the symbolic qualifier is unknown, the conventional CAPL functions can be called (for example, SomeIpCreateProvidedServiceInstance). These functions are always assigned with numeric values (such as Service ID and Method ID) and also return the handle to the database object.
