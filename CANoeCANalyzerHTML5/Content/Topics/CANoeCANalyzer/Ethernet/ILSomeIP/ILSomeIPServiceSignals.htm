Qualifying SOME/IP Service Signals

 

Note

Device parameters of a SOME/IP ECU that are transmitted by means of event notifications or field notifications will be referred to hereinafter as Service Signals. Service Signals always have a simple data type. Structure elements (for example, the root node of a complex data structure) are not considered to be Service Signals.

The signal values of simulated SOME/IP devices can be accessed using the basic functions (SomeIpGetValue... and SomeIpSetValue...) of the SOME/IP interaction layer.
A much more convenient way is to access the signal values using the $ notification contained in CAPL.

Service Signal

A Service Signal is always addressed by a qualifier. This qualifier is formed as follows:

{message qualifier} + {Signal path}

Message qualifier

Addresses the SOME/IP message in which the Service Signal is contained.
It comprises the following components, separated by the :: separator:

  • Database Name
  • Package Name
    Packages can, in turn, contain packages in the database. In this case, the names are also separated from each other by ::.
  • Service Name
  • Major Version
    Numeric value. If the Major Version is specified, the Instance ID must also be specified.
  • Instance ID
    Numeric value. If the Instance ID is specified, it must always be prefixed by the Major Version.
  • Parameter Name
    Name of the parameter for Events, or name of the Field.

{DB Name}::{Package Name}::{Service Name}::{Major Version}::{Instanz ID}::{Parameter Name}

The indicated qualifiers do not have to be specified. The network name can be omitted if the addressed Service Signals occur only in one database of the CANoe configuration. Likewise, the package name can be omitted if a service is contained only in one package.

Signal path

Addresses the Service Signal.
The signal path must be specified according to the underlying data type (simple data type or structured data type). The individual layers of structured data types are separated from each other by the . separator. A particular array index is accessed by placing the desired index in brackets ([Index]).

Note

In the CAPL Browser, you can insert Service Signals from the Symbol Explorer into your CAPL program using drag and drop. The complete access path for the Service Signal is then created. If the signal path contains an array, the indices for all dimensions of this array are automatically assigned with 0. If a different array element is to be accessed, the indices can be adapted subsequently.

Access to Service Signal

The examples given now are based on the Service Signals that are shown in the following screenshot of the Symbol Explorer. The device Device_1 offers two services (Service1 and Service2) with Major Version 1 and Instance ID 2. The services are defined in package PACKAGE1::PACKAGE2 of database DemoDatabase. These services contain both events and fields that contain the various Service Signals. Both simple and complex data types form the basis for these Service Signals. Use of the Service Signal Qualifier in CAPL

With the help of the qualifier, you can work with the signal values of a Service Signal in CAPL as follows:

Access to signal value

Use the $ notation to receive direct access to the signal value. This notation allows you to directly access Integer- and Float-type signals. For String-type signals, use the SetServiceSignalString or GetServiceSignalString function.

// set value of Service Signal 'Param1_CommonDataType'
$DemoDatabase::PACKAGE1::PACKAGE2::Service1::1::2::Event1::Param1_CommonDataType = 1;

// get value of Service Signal 'Param1_CommonDataType'
int value;
value = $DemoDatabase::PACKAGE1::PACKAGE2::Service1::1::2::Event1::Param1_CommonDataType;

Note

The example shows access to parameter Param1_CommonDataType, which is contained in Event1 of service Service1 (Major Version 1, Instance ID 2). The service interface is defined in package PACKAGE1::PACKAGE2 of database DemoDatabase.

Reaction to signal change/update

Using callback functions on serviceSignal and on serviceSignal_update, you can react to a value change or value update of the Service Signal in CAPL.
