Protocol Information [▲ back]

The data of an Ethernet packet (MAC Media Access Control-SDU Service Data Unit) can contain further protocols and the payload. The data must have a minimum size of 46 bytes and a maximum size of 1500 bytes. If less data are available a padding is necessary, more data have to be fragmented.

Following the payload the checksum (CRC) is sent.

Note

Preamble and Start Frame Delimiter are not part of an Ethernet packet in CANoe and are not evaluated.

Only with using VN5610, VN5610A or VN5640 the Ethernet checksum is displayed in the Detail view of the Trace Window and packets with incorrect checksum are displayed. If the IP network driver is used, packets with incorrect checksum will not be interpreted by CANoe.

Note

With Vector hardware, e.g. VN5610, Ethernet packets with a length of 14 to 1600 bytes (including the Ethernet header) can be sent, which are thus outside the minimum or maximum lengths of 60 to 1518 bytes.

Programming [▲ back]

The access to Ethernet fields of an Ethernet packet is performed in CAPL using the methods and selectors of the ethernetPacket object and in .NET using the methods and properties of the EthernetPacket-Objekte (see CANoe .NET API).

During the initialization of protocols, the EtherType field in the Ethernet header is set.

In addition to the fixed fields Destination, Source and EtherType, several VLAN tags may also be included in an Ethernet header according to IEEE 802.1ad (IEEE 802.1QinQ). CANoe supports up to three VLAN tags. The access to the VLAN tags of an Ethernet packet is possible in CAPL and .NET. The tags are numbered with a (zero-based) index. The innermost VLAN tag (the one that is next to the payload) has the index 0, the VLAN tag that is next to the innermost VLAN tag has the index 1 and so on. The VLAN tag that is next to the MAC addresses has the highest index.
