Protocol Information [▲ back]

The Hypertext Transfer Protocol (HTTP) is a stateless protocol that is used for data exchange. Client and server communicate with request and response messages. For this the transport protocol TCP is often used.

CANoe offers further possibilities to use HTTP in Configurations.

Header Format [▲ back]

The header of an HTTP message consists of an ASCII character string. The individual parts of the header are separated from each other by a carriage return and a line feed symbol (<CR><LF>, 0x0D 0x0A), making the structure of the header dynamically expandable.

Protocol Field

String Format

Start line

Indicates the message type. A Client request has the format of a Request line, a Server response has the format of a Status line.

Request line

<method><SPACE><request-target><SPACE><HTTP-version><CR><LF>

Contains the requested method, call-target, and the used HTTP version.

Some standard methods are:

Value

Meaning

RFC

GET

Requests the content of a resource.

RFC7231

HEAD

Requests only the meta information of a resource.

RFC7231

POST

Requests the processing of the transmitted content.

RFC7231

PUT

Requests the storage of the content of a resource.

RFC7231

DELETE

Requests the deletion of a resource.

RFC7231

CONNECT

Requests the opening of a TCP tunnel to forward subsequent packets blindly to the server.

RFC7231

OPTIONS

Requests the listing of the available options.

RFC7231

TRACE

Requests the mirroring of the message by the server (to detect modifications).

RFC7231

PATCH

Requests a partial modification of a resource.

RFC5789

The request-target is indicated in form of an (optionally modified) Uniform Resource Identifier (URI).

The HTTP version is indicated as HTTP/<major>.<minor>.

Status line

<HTTP-version><SPACE><status-code><SPACE><reason-phrase><CR><LF>

Contains the HTTP version, the status, and a human-readable explanation of the status.

The HTTP version is indicated as HTTP/<major>.<minor>.

The status code is given as a 3-digit decimal number and can be categorized by the first digit:

Status Code

Meaning

Description

1xx

Informational

The request was received.

2xx

Successful

The request was received, understood, and accepted.

3xx

Redirection

Further actions are necessary to process the request.

4xx

Client Error

The request contains invalid syntax or can not be processed.

5xx

Server Error

The server could not process an apparently valid request.

Header field

(optional)

<field-name>:<OPTIONAL WHITE SPACES><field-value><OPTIONAL WHITE SPACES><CR><LF>

A HTTP header can contain any number of these fields. Each header field has a name and a value, separated by a colon (":") and optional spaces.

End of Header section

<CR><LF>

Indicates the end of the header and signalizes the beginning of the message body that contains the payload.

Protocol Validation [▲ back]

Ethernet packets containing the HTTP protocol are analyzed and checked for consistency. Observed irregularities are displayed in the Protocol Validation column of the Trace Window.

Protocol Validation ID

Message

Description

[HTTP-001]

This is not a valid vanilla (no extensions) WebSocket frame

The present HTTP packet is part of a WebSocket connection but has an invalid format.

Note: WebSocket extensions are not checked.

Programming [▲ back]

Initialization

During initialization of the HTTP protocol, the field DestinationPort=80 in the TCP header is also set. The used port number can be configured in the Options dialog.

Note

The direct access to protocol fields is not supported.
