Protocol information [▲ back]

Cat PGT is a J1939-oriented protocol over Ethernet that enables a client (consumer) to request and receive parameter values (signals) from a corresponding server (provider) over a network.

HTTP PDU [▲ back]

The communication between the client and server is essentially based on the HTTP protocol, using embedded XML documents to some extent (XHTML). Since these documents can get very large, multiple packets (frames) are often necessary in order to transfer a complete document.

However, since further processing is only possible on complete documents, the individual frames must be collected and assembled. If a document (request/response etc.) has been received completely, then, in addition to the frame events, a so-called HTTP PDU event is also generated with the collected content and displayed in the Trace Window with a PDU symbol .

Note

  • The size of HTTP PDUs is limited to 64 kB, i.e. larger documents are truncated.
  • In this case, the PDU appears in the Trace Window with a yellow exclamation point, and a corresponding warning is displayed in the detail view.

PGT PDU [▲ back]

Many protocols are identified on the basis of so-called well-known ports. This typically also applies to HTTP (ports 80 and 8080) and can be specified in the Options dialog.

However, this does not apply to the UDP-based part of the communication. For this part, IP addresses and the UDP destination port are negotiated through the HTTP protocol and thus determined dynamically at runtime; therefore, the protocol assignment is not static.

If communication is not recorded from the beginning, it may happen that the corresponding HTTP PDU has not been received, so no PGT port is known. In this case, the port can also be specified through the Options dialog (entry <PGT>), e.g. for offline analyses.

XML validation [▲ back]

If HTTP XML documents are exchanged, their syntactic correctness (well-formedness) is verified.

However, for many XML documents, the underlying rules are specified in addition with an XML schema (xsd file) that is referenced in the documents themselves (e.g. through “noNamespaceSchemaLocation”). This allows a complete validation if the corresponding schema files are present and configured on the local computer

The configuration is performed through the corresponding CAN.ini:

[HTTP]
; provide a local schema for XML validation in HTTP xml documents
; XSDFile=
; alternatively provide the path to a local schema file hierarchy for XML validation
; XSDPath=
; the document schema path and the local path must follow the same relative path from a dedicated delimiter token
; XSDPathDelimiter=/Schemas/

 

If one XML schema reference suffices for validation, the local path can be assigned to the XSD file with the <XSDFile> parameter.

In contrast, if an entire hierarchy of XSD files is required, these must be present locally, corresponding to the relevant paths in the XML documents. The local path must match the referenced path in the document starting from a particular subtree (root node).

This common node is then entered as <XSDPathDelimiter>.

The local path to this root node is entered as <XSDPath>.

A verification failure is signaled by a yellow exclamation point on the PDU symbol, and an error description is output in the detail view in the HTTP message header.

PGT signals[▲ back]

PGT parameter values (signals) can be used in the following modules:

  • Trace Window
  • Data Window
  • Graphic Window
  • Symbol Explorer
  • CAPL (Measurement and Simulation Setup)
  • COM API
  • FDX

Detail view for HTTP and CATPGT PDUs [▲ back]

The following information is displayed in the detail view for HTTP PDUs:

  • General: PDU-type HTTP message, channel, and direction (Tx/Rx).
  • Header: The content of the HTTP header according to Topics/CANoeCANalyzer/Ethernet/Protocols/ProtocolHTTP.htm, and, if applicable, additional error descriptions concerning validation errors or a maximum content size overrun. For PGT messages, the associated session, group, and client IDs are typically available here.
  • Payload: The actual content of the HTTP payload, especially XML files, for example.

The following information is displayed in the detail view for CatPGT PDUs:

  • General: PDU-type PGT message, channel, and direction (Tx / Rx).
  • PGT header: Session, group, client and server IDs, sequence number.
  • Payload: Length and content of the byte sequence.
  • Sublines: The signal values transferred.
