Protocol Information [▲ back]

The Neighbor Discovery Protocol (NDP) (RFC 2461) replaces the ARP protocol on using IPv6. It is based on ICMPv6. The protocol is used for the following application areas:

  • Router Discovery: identify the available router within the network
  • Prefix Discovery: set the address prefixes for local and remote nodes
  • Parameter Discovery: set several parameters, e.g. MTU Maximum Transfer Unit or Hop limit

For this, the following message types are available:

Type

Name

Description

133

Router Solicitation

A host can request the router to send a Router Advertisement message.

134

Router Advertisement

With this message the router announce themselves in the network. The message is sent from the router cyclically or is requested with a Router Solicitation message.

135

Neighbor Solicitation

A node can request the network address of another node with this message, or check if a node is still available. It can be checked too if an address is already assigned.

136

Neighbor Advertisement

This message is the answer to a Neighbor Solicitation message. A node can send this message too to announce a network address change.

137

Redirect Message

This message is used by the router to show a host a better way to a destination node.

141

Inverse Neighbor Discovery Solicitation Message

A node can request the IPv6 address from another node by specifying the network address.

142

Inverse Neighbor Discovery Advertisement Message

This message is the answer to an Inverse Neighbor Discovery Solicitation message. The list with the IPv6 address is part of the options.

Programming [▲ back]

Initialization

During initialization of the NDP protocol, the Neighbor Solicitation message type is used and thus field Type=135 in the ICMPv6 header is also set.

Options

The NDP protocol specifies different message types with different protocol fields (see Header Format table). Using selectors (CAPL) or properties (.NET) of the appropriate message type, ethernetPacket.<protocol>.<field>.IsAvailable can be used to check whether the message is of the appropriate type, and ethernetPacket.<protocol>.<optional-structure>.Init can be used to set the message to the appropriate type. The Type field is also set in the ICMPv6 header to the appropriate value.

With nested selectors (CAPL) or properties (.NET), protocol fields of the appropriate message type are accessible.
