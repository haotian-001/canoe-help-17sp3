Protocol Information [▲ back]

The network protocol PTP (Precision Time Protocol) is used to synchronize time-consuming applications, eg. audio or video. PTP allows an accuracy in the range of nanoseconds. PTP is defined in IEEE 1588 and in a generalized form in IEEE 802.1AS (gPTP).

Protocol Validation

During the measurement the PTP communication between ports of time-aware end stations or bridges is observed. Detected irregularities are displayed in the Protocol Validation column of the Trace Window.

Protocol Validation ID

Message

Description

[PTP-001]

PTP message type not matching port role of sender: …

The PTP message type does not match the current port role of the source port identity. For example, a slave port is not allowed to send a Sync message.

PTP message type not matching port role of receiver: …

The PTP message type does not match the current port role of the port identity that is receiving this message. For example, a master port shall not receive a Sync message.

[PTP-002]

Sync while expecting Follow_Up

A Sync message is transmitted without a prior Follow_Up message following the last Sync from the source port identity.

Follow_Up while expecting Sync

A Follow_Up message is transmitted without a prior Sync message from the source port identity.

Former Pdelay_Req unanswered

The former Pdelay_Req of the source port identity was not answered by the port identity receiving it.

Pdelay_Req in between Pdelay_Resp and Pdelay_Resp_Follow_Up

A Pdelay_Req message is received in between a Pdelay_Resp and a Pdelay_Resp_Follow_Up.

Pdelay_Resp without prior Pdelay_Req

A Pdelay_Resp message is transmitted without a prior Pdelay_Req message from the requesting port identity.

Pdelay_Resp while expecting Pdelay_Resp_Follow_Up

A Pdelay_Resp message is transmitted, but a Pdelay_Resp_Follow_Up message is expected.

Pdelay_Resp_Follow_Up without prior Pdelay_Req and Pdelay_Resp

A Pdelay_Resp_Follow_Up message is transmitted without having been requested by a Pdelay_Req message from the requesting port identity and answered by a Pdelay_Resp message of the source port identity.

Pdelay_Resp_Follow_Up in between Pdelay_Req and Pdelay_Resp

A Pdelay_Resp_Follow_Up message is transmitted in between a Pdelay_Req and a Pdelay_Resp_Follow_Up.

[PTP-003]

Sequence ID does not match the one in the prior Sync which was …

The sequence ID field of a Follow_Up message contains a value that is not the same as this field’s value in the prior Sync message.

Sequence ID is not the incremented prior one which was …

The sequence ID field of a Pdelay_Req message contains a value that is not the increment of this field’s value in the prior Pdelay_Req message.

Sequence ID does not match the one in the prior Pdelay_Req which was …

The sequence ID field of a Pdelay_Resp message contains a value that is not the same as this field’s value in the prior Pdelay_Req message.

Sequence ID does not match the one in the prior Pdelay_Req and Pdelay_Resp which was …

The sequence ID field of a Pdelay_Resp_Follow_Up message contains a value that is not the same as this field’s value in the prior Pdelay_Req and Pdelay_Resp messages.

[PTP-004]

Port Identity not matching the one in the prior Sync which was …

The port identity field value of a Follow_Up message contains a value that is not the same as this field’s value in the prior Sync message.

[PTP-005]

Precise origin timestamp is less or equal than in last Follow_Up, setting rate ratio numerator to 1 and denominator to MAXUINT64

The precise origin timestamp field value in a Follow_Up message is less than or equal to this field’s value in the last Follow_Up. Backward time jumps are not valid so time progression is retarded to a minimum by assuming the rate ratio to reach zero.

Response origin timestamp is less than the request receipt timestamp carried in the former Pdelay_Resp message! Delay calculation impossible

The response origin timestamp field value in a Pdelay_Resp_Follow_Up message is less that the request receipt timestamp value in the former Pdelay_Resp message. Negative time progression is not permitted.

[PTP-006]

Equation for mean propagation delay would produce a negative result. A negative delay is impossible!

The equation for the mean propagation delay done after a complete Pdelay_Req – Pdelay_Resp – Pdelay_Resp_Follow_Up sequence would yield a negative result. This indicates an error as a negative delay is impossible.

[PTP-007]

Calculated doubled mean propagation delay is too big (> 2^53), precision is lost.

The calculated value for the doubled mean propagation delay is too big (> 2^53). Precision is lost because the corresponding 64 bit double data type (nanoseconds) cannot hold the exact value.
