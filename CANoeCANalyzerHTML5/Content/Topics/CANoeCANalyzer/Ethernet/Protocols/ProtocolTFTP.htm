Protocol Information

The TFTP (Trivial File Transfer Protocol) is a simple UDP-based file transfer protocol which supports the reading and writing of files. It is described by the RFC 1350.

It provides some standardized extensions (options) such as the block size (RFC 2348), FileTransferSize (RFC 2349) or TimeoutInterval (RFC 2349), but also a simple possibility to define further options (RFC 2347).

TFTP is packet-oriented and thus connectionless. The initiation is normally carried out via the well-known port 69, the further transmission, however, is done via the so-called Transfer Identifiers (TID) which correspond to the UDP ports of the opposite side and range from 1024 to 65535.

The data format depends on the respective Opcode, see below.

Protocol Validation [▲ back]

Ethernet packets containing the TFTP protocol are analyzed and checked for consistency. Observed irregularities are displayed in the Protocol Validation column of the Trace Window.

Protocol Validation ID

Message

Description

[TFTP-001]

 TFTP packet length is too small. Must be at least 3 bytes

The present packet is too short. A TFTP packet must consist of at least 3 bytes to be able to contain the header completely.

[TFTP-002]

 Invalid opcode; valid range: [1..6] The opcode (Feld Opcode field) of the TFTP layer is invalid. Only values between 1 and 6 can be interpreted.

Programming [▲ back]

Initialization

During initialization of the TFTP protocol, the fields SourcePort=69 and DestinationPort=69 in the UDP header are also set.

Options

The TFTP protocol specifies different message types with different protocol fields (see Optional Protocol Fields table). Using selectors (CAPL) or properties (.NET) of the appropriate message type, ethernetPacket.<protocol>.<field>.IsAvailable can be used to check whether the message is of the appropriate type, and ethernetPacket.<protocol>.<optional-structure>.Init can be used to set the message to the appropriate type. The Opcode field is also set in the TFTP header to the appropriate value.

With nested selectors (CAPL) or properties (.NET), protocol fields of the appropriate message type are accessible.
