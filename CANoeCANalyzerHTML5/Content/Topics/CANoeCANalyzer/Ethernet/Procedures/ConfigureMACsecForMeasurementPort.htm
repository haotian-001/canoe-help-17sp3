How to Configure MACsec for a Measurement Port

 

Prerequisites

MACsec can only be configured on measurement ports. Therefore, you need a network-based configuration in real mode.

Only one measurement port may be configured per network segment, otherwise the measurement start is prevented.

CANoe Configuration

Step by Step Procedure

  1. Configure your hardware so that the used network segment has only one measurement port. There is no restriction for the number of used simulation ports.

  2. Open the Port Configuration dialog via the Hardware ribbon tab|Port Configuration and activate the measurement ports that should be used.

  3. Open the Security Configuration dialog via the Simulation ribbon tab|Security Configuration

  4. Select the MACsec page.

  5. Select in the Security Profile selection list the MACsec profile that should be used.
    he available measurement ports are listed in the table.

  6. Activate Use Security and select a MACsec Port Configuration from the set profile.

  7. At the start of measurement, the MKA protocol will be started on the configured ports and the MACsec port is activated after the key has been successfully exchanged. Only then it is possible to communicate via the measurement port from the simulation. If no connection is established, check the used security profile.

MACsec Implementation

You can select the MACsec implementation in the Security Configuration dialog:

  • PreferHardware
    The hardware implementation is automatically executed if the measurement port is assigned to a hardware channel that supports MACsec encryption in hardware.
  • UseSoftware
    The software implementation is always used.
  • UseHardware
    If the hardware channel does not support MACsec encryption in hardware, then an error message is output and MACsec is disabled for the port.

The selection of the MACsec implementation affects the performance or possible bandwidth as well as basic properties of the Ethernet simulation.

Software Implementation

With the software implementation, MACsec frames are encoded/decoded entirely in CANoe using software crypto functions.

Frames are encoded in CANoe before transmission and decoded in CANoe on reception. This means that switch functionality configured on the hardware cannot be used, since each hardware port has its own secured channel with its own session key to the connected node. Via a switch, encoded MACsec frames are forwarded directly from a receiver port to a transmitter port, where they cannot be encoded.

For this reason, you may only configure one measurement port per network segment in the MACsec software implementation. To route frames from one measurement port to another, you must implement a gateway node with switch functionality in the CANoe simulation.

When using the software implementation, all frames are visible in the Trace Window, i.e. the original Ethernet frames generated by the simulation model and the encoded MACsec frames at the measurement ports.

Hardware Implementation

In the hardware implementation, encoding/decoding is done via the Ethernet PHY (installed behind port) of the Vector network interface.
Frames are encoded on the hardware before physical transmission and decoded on reception.

Therefore the switch functionality can be used normally on the hardware. You can use any number of measurement ports on one segment. The performance or bandwidth is basically higher than with the software implementation and similar to the operation without MACsec.

When using the hardware implementation, the encoded MACsec frames are not visible in the Trace Window because they are only transmitted between the PHYs ("on the wire"). Only the original or decoded frames are displayed. To make the MACsec frames visible anyway in this case, you can use an additional tap connection on the network interface through which the frames of the measurement port are routed.

A mixed configuration (hardware/software) on a network segment is not possible and is prevented by CANoe (error message).

Bypass Filter

Bypass filters allow you to define Ethernet frames that should exceptionally bypass the MACsec block

  • Egress
    Ethernet frames that match the filter are not encoded with MACsec but sent as a regular Ethernet frame instead.
  • Ingress
    Ethernet frames that match the filter are passed into the simulation as a non-MACsec encoded frame when received.

Without defined filters, all frames are encoded with MACsec. Frames that are not MACsec encoded are discarded by default on reception.

Typical use cases are:

  • Certain VLANs are to be excluded from MACsec operation.
  • Certain EtherTypes (e.g. gPTP) should not be MACsec-encoded.
  • Individual nodes should be excluded from MACsec operation by MAC address.

How to configure a bypass filter in the Security Configuration dialog

  1. click [+] in the Network column or click the MACsec Bypass Filter cell to open the Bypass Filter view.

  2. Press <Ins> to create a new row in the filter view. The new entry has the filter type Ether Type.

  3. In the Bypass Filter Type column, select the filter type.

  4. Click in the Bypass Filter Data cell to open the configuration of the data.

  5. In the Direction column, select in which direction the filter should act.

    • Ingress: incoming frames only
    • Egress: outgoing frames only
    • Both: both directions

  6. Using the context menu of a row you can

    • delete the current line
    • create a new entry above the current line
    • create a new entry below the current line
    • move the current row up by one position
    • move the current row down by one position

The effect of the bypass filters is independent of their position in the table. The filters are OR-linked. A frame that matches one of the filters is bypassed at the MACsec block.

Note

  • Ethernet frames with type 0x888e EAP over LAN are sent or processed independently by CANoe and the Vector Security Manager if the MACsec configuration is enabled. These frames are used for the MACsec Key Agreement (MKA) and are handled specially in CANoe.
  • The availability of the bypass filters may be limited in the hardware implementation. Not all filter types are supported depending on the hardware PHY used, and the number of filters is limited. Failed transfers of bypass filters to hardware are output to the Write Window.
  • In the software implementation, basically all filter types are supported in any number.
