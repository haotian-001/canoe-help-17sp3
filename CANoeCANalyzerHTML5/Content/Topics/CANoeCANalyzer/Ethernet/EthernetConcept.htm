Ethernet | Concept

 

Overview

The option .Ethernet supports the basic concepts of CANoe and extends them with Ethernet-specific functions.

With Ethernet, categorization of protocols in layers is strongly pronounced. Option Ethernet supports you with the analysis,simulation and test at all levels, providing multiple functions.

Link Layer

Ethernet packets are exchanged at this level. Ethernet switches guide an Ethernet packets to the addressed receiver using MAC addresses. Virtual networks (VLAN) can be used to form domains.

CANoe processes Ethernet packets as Ethernet packet events. Via a network interface, for example VN5610, Ethernet packets can be received and also sent by an Ethernet link.

CANoe allows you to evaluate an Ethernet packet from the Ethernet level to the application protocol. For example, all protocols of an Ethernet packet are displayed in the Detail View of the Trace Window. For stimulation, Ethernet packets with the required protocol information can be generated and sent. The Ethernet Packet Builder or CAPL can be used for this purpose.

Programming

The CAPL functions of the link layer can be used for analysis, stimulation, test and fault injection. For the simulation, CAPL functions of the internet and transport layer or the application layer should be used in order to make the generation of CAPL programs easier.

on ethernetPacket can be used to receive and evaluate individual Ethernet packets. Selectors allow the protocol fields to be read. The ethernetPacket object is used to generate Ethernet packets and output( ethernetPacket ) is used to send them.

To generate busload, ethStartPacketGenerator can be used.

If AUTOSAR I-PDU are transmitted via Ethernet, ethernetPacket::PDUCountand ethernetPacket::GetPDUcan be used to read a PDU.

The handler on ethernetStatus can inform a CAPL program if an Ethernet link is established or removed. The Eth object or the Ethernet statistics system variables can be used to query the Ethernet link state at any time.

In test configurations/modules, you can wait for selected Ethernet packets with testWaitForEthernetPacket.

Internet and Transport Layer

IP addresses allow Ethernet packets to be delivered to receivers across networks. These can also be virtual networks in which an ECU uses different IP addresses.

Transport protocols, such as UDP and TCP are used to transmit data of the Ethernet packets to the addressed application of a ECU. The fragmentation of data which must be distributed over several Ethernet packets is also ensured at this level.

In CANoe, a simulated node can access its own TCP/IP Stack. The TCP/IP Stack contains the network configuration, such as IP addresses and VLAN. The network settings can be made manually or via a database. The Ethernet IG or the TCP/IP API in CAPL/.Net can be used to transmit data via TCP or UDP.

For windows and blocks in the Measurement Setup, CANoe monitors the Ethernet packets and collects information across several Ethernet packets to be able to show them in a protocol-specific way. This data is used to generate additional events which are shown, for example in the Trace Window as separate lines. This is how AUTOSAR PDUs, SOME/IP events, DoIP and XCP messages are processed in CANoe.

Programming

Using the TCP/IP-CAPL functions, UDP and TCP sockets can be created and used for communication. The simulated node or the test module is always an active terminal station. To establish a connection, the IP addresses and port numbers of the local and remote endpoints must be specified. The configured TCP/IP stack establishes the connection. Afterwards, the CAPL program can send the user data via UdpSendTo or TcpSend and receive them in CAPL callback functions.

The TLS/IP-CAPL functions also allow the establishment of encrypted connections.

Application Layer

At this level, transmitted data is interpreted by the application. The meaning of the data can be determined by the application protocol or by the definition of a database.

CANoe supports various application protocols which are used in the automotive industry. To show signals in the Graphics Window, Data Window, and in panels, the definition from an ARXML or Fibex database is necessary.

Various interaction layer VMODULE files are available for simulation. They can be assigned to a simulated node, expanding the node with the application-dependent functions. The interaction layer VMODULE files receive their configuration from the database.

Programming

For the different application protocols, individual functions are made available in CAPL.

on pdu and on signal can be used to evaluate AUTOSAR I-PDUs or signals. It is even possible to read out signal values with $<signal name>. An ARXML database should be used to be able to use the symbolic designators of the PDUs and signals. In order to be able to send AUTOSAR I-PDUs, the interaction layers AsrPDUIL2 and CANoeILNL_AUTOSAR_ETH must be integrated. They are completely configured from the assigned database node. The transmission of PDUs can be triggered by triggerPDU and the $<signal name>.

Values from SOME/IP events and notifications can be received with an on serviceSignal and read with a $<service signal name>. An ARXML or Fibex database must be integrated. To be able to send SOME/IP events, methods or notifications, the CANoeILNL_AUTOSAR_ETH or SOMEIP_IL must be integrated. The assignment with a $<service signal name> can trigger the transmission of an event. For methods, other CAPL functions are available.

The AUTOSAR UDP Network Management can be simulated using the AUTOSAR UDP NM Interaction Layer.

The AVB Interaction Layer can be used to send and receive audio and video streams via AVTP and RTP.
