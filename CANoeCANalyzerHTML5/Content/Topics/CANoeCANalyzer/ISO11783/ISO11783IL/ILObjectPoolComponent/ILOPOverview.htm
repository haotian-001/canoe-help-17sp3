Object Pool Component

 

General

List of Functions

The object pool component is a part of the ISO11783 Interaction Layer (ISO11783 IL) and is needed to implement communication with a virtual terminal and one or more auxiliary ECUs. This functionality is constructed based on an object pool, which is described in a special file (ISO11783-6).

Functionality

Object Pool CAPL API
  • Connection management with the virtual terminal (initialization procedure, send maintenance message)
  • Loading an object pool file (*.iop or *.iopx)
  • Export a loaded object pool to an IOP or IOPX file
  • Modification of the local object pool
  • Changing and Transmitting Object Attributes
  • Changing and Transmitting Object Values
  • Evaluating messages from the VT
  • Managing auxiliary inputs and auxiliary functions

The Object Pool API of the ISO11783 IL offers another way to change objects in the object pool or to query their state. Various functions are available for doing this:

  • Iso11783IL_OPChange...
  • Iso11783IL_OPSet...
  • Iso11783IL_OPGet...

Operating Principle

For each node in the simulation, there is a definition right in the database on whether there is an object pool description, and if so which file contains it.

If the object pool file is configured in the database or in the Simulation Setup, CANoe automatically generates suitable system variables for all objects when the measurement is started. This is done as follows:

  1. The ISO11783 IL automatically reads in the specified file, analyzes it and generates a node object pool. After that, the ISO11783 IL generates a set of system variables, each of which corresponds to an individual attribute of an object from the node object pool.
  2. Based on the status message of the Virtual Terminal, the ISO11783 IL automatically detects the virtual terminal after sending of the status message and sends a copy of the read-in node object pool to the virtual terminal.
  3. Each change to a system variable is automatically transmitted to the relevant object in the node object pool. In the reverse case, this applies: each change to an object from the node object pool is automatically sent to the relevant system variable. After that, a VT message is used to automatically send this change to the relevant attribute of an object in the VT object pool. In this way, the user can adapt the image that is displayed by the virtual terminal without having to write a single line of code.
  4. In addition, each user input in the virtual terminal, which is sent to the simulated node with a VT message, is transmitted to the relevant system variable and to the relevant object in the node object pool. In this way, the user can check all user inputs in the virtual terminal without having to write a single line of code.

Note

Only one-dimensional system variables are permitted in CANoe. Therefore, system variables are only generated for one-dimensional attributes (e.g. "color", "type", "length" etc.). No system variables are generated for multidimensional attributes (e.g. "child position" and "size").

Alternatively, an object pool file can be loaded and transferred at runtime using the Iso11783IL_OPLoad function. But then no system variables are generated for the objects.

The object pool component supports loading and saving of object pool files in binary IOP format as well as a proprietary XML format (IOPX). This allows changes in the object pool to be easily made in a text editor.
