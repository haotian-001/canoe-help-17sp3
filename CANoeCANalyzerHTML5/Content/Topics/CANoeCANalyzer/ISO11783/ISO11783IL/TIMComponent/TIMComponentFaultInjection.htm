Fault Injections

 

You can influence the communication between a simulated TIM participant with a TIM counterpart in various ways. E.g. you can use invalid certificate files or you can modify the TIM server status notification (with CAPL function Iso11783IL_TIMSetServerStatus). Furthermore, you can influence the standard communication procedure between the client and server.

Influence the Standard Procedure

Sometimes it is necessary to influence the standard procedure: certain steps should be skipped or executed in a different order, e.g. to inject an error or to avoid an implementation error.

The states of the internal status machine of a TIM server/client are defined on page Server States and Client States. In the standard run, the simulated node runs through all its states automatically. However, the user has the option of only allowing the process flow up to a certain state. When the state is reached, the simulated node stops (except for the status notification – it is still sent), and the user can intervene manually: e.g. after a pause to resume the automatic sequence, to send one or more notifications or to resume the automatic sequence with another state.

Example for the TIM Client

In state X, an event occurs that leads to a change to state Y.

In state Connection Version request is sent, the notification TIM_ConnectionVersionResponse of the server is received. If it contains a valid connection version, the system switches to state Connection Version Valid.

2

State Y becomes the new state.

The new state is now Connection Version Valid.

3

The simulated node executes an action.

The client now sends a Functions Supported Request to the server and changes to the Supported Functions Request is sent state.

If the standard procedure is influenced by a control node, this can look like this:

 

Description

Example for the TIM Client

1

The control node specifies that the simulated node should stop in a certain state Y.

The control node specifies that TIM client should stop in state Connection Version Valid.

2

In state X, an event occurs that leads to a change to state Y.

In state Connection Version request is sent, the notification TIM_ConnectionVersionResponse of the server is received. If it contains a valid connection version, the system switches to state Connection Version Valid.

3

State Y becomes the new state.

The new state of TIM Client is now Connection Version Valid.

4

The simulated node does not execute an action.

The TIM client remains in the state Connection Version Valid.

5

The control node performs an action.

The control node sends a TIM_FunctionsSupportRequest to the TIM server.

6

The control node specifies that the simulated node shall resume with the current or with another state.

The control node specifies that TIM client shall resume with state Function Supported Request is sent.

To influence the standard behavior, you can use following CAPL functions:

Procedure

CAPL Functions

First you can make the state of a simulated node available. For this you can link the internal state to a system variable.

Iso11783IL_TIMConnectSysVarToState

Then you define the state up to which the simulated node should run autonomously.

In a test, you can wait until the state is reached (e.g. using TestWaitForSysVar). In a simulation, you can use the event procedure on sysvar.

Iso11783IL_TIMFreezeConnection

After performing certain actions, you can continue automatic run with a certain state.

Iso11783IL_TIMContinueConnection
