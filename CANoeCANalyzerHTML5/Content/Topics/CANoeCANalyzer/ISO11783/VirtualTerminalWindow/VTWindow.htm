Virtual Terminal Window

 

The Virtual Terminal Window provides a ready-to-use virtual terminal instance without any additional CAPL programming. VT window offers the possibility to visually control the data masks as well as the contents of object pools. In addition, the ISOBUS Shortcut Button (ISB) is supported.

Insert the Virtual Terminal Window [▲ back]

To activate the Virtual Terminal Window you have to open the context menu in the Simulation Setup by right-clicking on the bus symbol. Pressing command Insert Virtual Terminal inserts a Virtual Terminal node. By double clicking it the window is opened.

Layout of the Virtual Terminal Window [▲ back]

In the top part of the window the symbols of the Working Sets, which has sent by an object pool to the Virtual Terminal, are displayed. The active Working Set is selectable here. The context menu of the window part offers the following commands:

  • Delete Working Set
    The Working Set is deleted from Virtual Terminal.
  • Add predefined Working Set
    A new Working Set is created for observing a Working Set in passive mode.
  • Add predefined Working Set with Object Pool
    To observe a Working Set in passive mode, a new Working Set is created and the object pool to be used is defined.
  • Export Object Pool...
    The actual object pool of the Working Set is saved to an IOP file.
  • Import Object Pool...
    An object pool is read from an IOP file and displayed. The imported object pool can only be monitored.
  • Remove stored versions
    Removes the stored object pool versions of the Working Set.

In the middle part of the window there are some property pages:

  • Page Data Mask
  • Page Object Pool
  • Page Working Set Details
  • Page Auxiliary Function
  • Page Configuration

The expandable information view User-Layout Masks below shows the User-Layout data masks of the Virtual Terminal. You can set the number of the masks on the Configuration page. There you have to set the Virtual Terminal version 4 or higher due older version don't support User-Layout data masks. If the object pools of the Working Sets contain Window-Mask or Key-Group objects, you can select them in the context menu of a User-Layout data mask.

At the bottom the expandable information view Information displays the messages issued by the Virtual Terminal. When selecting a line in the list and pressing <F1> the help will show a more detailed description. When the measurement is stopped the windows synchronization is activated with the button [ ]. The marker is set to that instant of time at which the current selected message occurred.

The status bar of the window shows the operating mode of the Virtual Terminal and allows the change of the mode. If enabled on page Configuration the ISOBUS Shortcut Button [ISB] deactivates functions of other ISOBUS participants (if they support the ISB). The button [Ack] acknowledges an alarm mask. The output of acoustic signals can be enabled and disabled with the speaker symbol too. The loupe symbol fits the size of the data and softkey mask of the selected Working Set to the size of the Virtual Terminal Window. If the Virtual Terminal is in passive mode it always displays the active Working Set. This behavior is deactivated, when the user selects another Working Set. Further you can deactivate or reactivate this behavior via the icon. You can remove stored object pools versions by clicking on .

Operating Modes [▲ back]

The Virtual Terminal can be in the following operating modes:

  • Offline
    The Virtual Terminal is not active, it can be activated and deactivated during the measurement.
  • Online
    The Virtual Terminal is active and sends status messages.
  • Passive
    The Virtual Terminal is in passive mode.

The right mouse button opens a context menu which allows the activation and deactivation of the Virtual Terminal.

Passive Mode [▲ back]

In passive mode, the CANoe Virtual Terminal observes bus traffic and mirrors the display of a real, active Virtual Terminal. No messages are sent, only the messages sent to and from the active Virtual Terminal are observed and evaluated. The information transmitted in the messages is used as follows:

  • The Address Claimed, Language Command, VT Status, Get Memory response, Get Text Font Data response and Get Hardware response messages contain properties of the active Virtual Terminal, which are mapped in corresponding areas on the Configuration page.

  • The messages Get Number of Soft Keys response and Get Hardware response contain information relevant for the representation of the object pools. You can see the results on the Data Mask page.

  • In addition, Address Claimed and WS Maintenance messages sent by each Working Set describing the respective Working Set are evaluated. The information obtained from this(address, device name and VT version) is displayed on the Working Set Details page.

You can activate the passive mode on the Configuration(Virtual Terminal Properties | Passive Mode | Enable passive mode check box). With further settings you can define whether either a Virtual Terminal with the specific address, the Virtual Terminal seen first or the Virtual Terminal with a specific Function Instance should be watched.

Of course, the Virtual Terminal Window must also know the contents of the respective object pool in order to display them. Different use cases are described in the following table.

Use Case

Preparation

The Virtual Terminal Window observes the entire communication of the Working Set from the beginning. It therefore sees both the Address Claim, WS Master and WS Maintenance messages of the respective Working Set and the following transmission of the object pool.

No preparation necessary: Virtual Terminal Window receives the object pool directly from the Working Set.

The object pool is loaded via the Load Version command and the message with Load Version is present on the bus during the measurement.

For the object pool to be loaded from the Virtual Terminal window, the IOP file with the appropriate name (either <version name>.iop or <device name>_<version name>.iop) must be stored in the directory specified on the Configuration page in the Version Management area.

The object pool is loaded via Object Pool Transfer or Load Version, but the corresponding messages are not present on the bus during the measurement. This is the case if the CANoe measurement was started at the time when the transfer or loading of the object pool was already completed.

In the Virtual Terminal Window, add a predefined Working Set and define an IOP file that contains the matching object pool. Adding the working set is done from the context menu at the top of the Virtual Terminal window and defining the object pool on the Working Set Details page in the Passive Mode Settings area.

If a problem is detected while displaying the object pool, the Virtual Terminal window freezes the state of the object pool. The Object Pool can thus still be examined even if the real Virtual Terminal has already deleted the Working Set with its Object Pool.

Interaction with the Virtual Terminal Interaction Layer [▲ back]

If the Virtual Terminal is also to be influenced via a CAPL program, the Virtual Terminal can be coupled with a simulation node that uses the Virtual Terminal Interaction Layer (VT IL). The Virtual Terminal can be controlled with the help of a test unit, for example. However, it can also be operated via the user interface if necessary.

The coupling of the Virtual Terminal with a simulation node you can establish via the following steps:

  1. In the Virtual Terminal Window open the Configuration page.
  2. In the Device Name area activate the Use node from database option.
  3. Select the node from the Node list. (Node that is used by the corresponding simulation node).

The Virtual Terminal Window that is coupled in this way to a simulation node, configures the corresponding simulation node: All parameters that are set on the configuration page of the Virtual Terminal Window are made available to the simulation node. That happens before the start of any measurement, and in particular before the CAPL methods on preStart and on start are called. These settings can be overwritten for the simulation node with the VTIL_SetNodeProperty method, which is then called in on preStart or on start, for example. This coupling option is not available in passive mode.
