Configuration

 

Überblick

Step by Step Procedure: Workflow for SIL Tests

  1. Describe the functional system interfaces of the software under test as distributed objects (DO).

  2. For these distributed objects a SIL Adapter for data exchange with CANoe is generated.

    The adapter is provided as a set of source files and libraries.

  3. Instrument the SUT source code using the SIL Adapter in the development environment.

  4. As soon as the target hardware is running and connected to a hardware debugger, you can read and write the values of the software under test from CANoe.

The configuration of a software-in-the-loop setup on real hardware with a hardware debugger connection to CANoe requires a configuration as shown below.

The interface description is created with the Vector Communication Description Language (vCDL) and integrated into the CANoe configuration via the Communication Setup. In vCDL, also a YAML file is referenced, which describes the hardware debugger specifics. The debugger itself must be configured and started outside of CANoe.

Configuration scheme for the connection of hardware debuggers to CANoe:

Description of the Functional System Interfaces

You can define physical environments and software environments to stimulate the application under test and verify its behavior.

You define this environment usingdistributed objects (DO) described by the Vector Communication Description Language (vCDL) in text form. You can edit the vCDL file using the vCDL Editor.

To ensure that the distributed objects are included in the generation of the C SIL Adapter, you must set the binding of the distributed objects to HwDebugger.

You can use the HwDebuggerBinding::IDebugger interface to define a distributed object that allows access to the hardware debugger infrastructure, for example, to programmatically query the online status of the hardware debugger. The description of the IDebugger interface can be found in the system vCDL file <CANoe Installation>/Bindings/Vector.CANoe.FunctionBus.HwDebuggerBinding.DataModel.vcdl.

Example: vCDL with HwDebugger Binding (One Debugger in the Test Setup)

version 1.4;

import module "HwDebugger";

[Binding="HwDebugger"]
[HwDebuggerBinding::ConfigFile="..\\Environment\\config.yaml"]
refine defaultnetwork;

namespace Calculator
{
  interface ICalculator
  {
    consumed method int32 Add(int32 x, int32 y);
  }

  ICalculator Calculator;

  HwDebuggerBinding::IDebugger debugger;
}

System Setup in CANoe

You can import a vCDL file into CANoe via Communication Setup. After successful import, the environment objects are available in the CANoe configuration and can be used for panels, models, Windows (Analysis) and automated tests.

The debugger-specific settings such as type and connection to the debugger software are managed in a separate YAML file.

Example: YAML Debugger Configuration (One Debugger in the Test Setup)

version: 1.0

debuggers:
  MyDebugger:
    type: isystem
    node: localhost
    port: 3515

For generation, refer to C SIL Adapter.

Several hardware debuggers can also be used simultaneously in a test setup. In this case, the hardware debugger instances must be defined in the YAML debugger configuration and the distributed objects must be assigned in the vCDL file.

Example: vCDL with HwDebugger Binding (Several Debugger in the Test Setup)

version 1.4;

import module "HwDebugger";

[Binding="HwDebugger"]
[HwDebuggerBinding::ConfigFile="..\\Environment\\config.yaml"]
refine defaultnetwork;

namespace Calculator
{
  interface ICalculator
  {
    consumed method int32 Add(int32 x, int32 y);
  }

  [HwDebuggerBinding::DebuggerName="debugger1"]
  ICalculator calculator1;

  [HwDebuggerBinding::DebuggerName="debugger2"]
  ICalculator calculator2;

  [HwDebuggerBinding::DebuggerName="debugger1"]
  HwDebuggerBinding::IDebugger debugger1;

  [HwDebuggerBinding::DebuggerName="debugger2"]
  HwDebuggerBinding::IDebugger debugger2;
}

Example: YAML Debugger Configuration (Several Debugger in the Test Setup)

version: 1.0

debuggers:
  debugger1:
    type: lauterbach
    node: localhost
    port: 30000
    reset: false
    run: false

  debugger2:
    type: isystem
    node: localhost
    port: 5315
    reset: false
    run: false
