MOST Hardware Configuration

Ribbon: Hardware Tab|Network Hardware|MOST

The Network Hardware Configuration dialog contains three areas for each activated MOST channel:

MOST

HW Type

Select the desired hardware type and speed grade (MOST50 or MOST150) for each MOST channel.

Additional Services

Here you can set which services should be activated in CANoe:

  • None

CANoe behaves entirely passive; all bus accesses must be performed by the user or be initiated by simulation nodes or Replay Blocks.

  • AMS

The AMS Service is activated; fragmented transmissions may be performed, and when they are received the individual control messages of a segmented transmission are reassembled into an AMS message. CANoe continues to remain essentially passive; only segmented transmission errors are communicated to the sending node per the MOST specification.

AMS must also be activated if the AMS Combiner should be active in offline mode or when importing logging files into a Trace Window.

  • AMS+AS

Besides AMS the Application Socket (AS) is also activated. Depending on the node mode that is set for the specific channel, Master or Slave, CANoe is now active in the sense of NetServices.
As a Master node the integrated MOST NetworkMaster polls existing function blocks of the connected devices and propagates the configuration status.
Configured as a Slave the CANoe Application Socket answers requests from a MOST NetworkMaster via the integrated NetBlock.
See also MOST Simulation Concept

  • MOST System Viewer and MOST FBlock Monitor Support

The data collector of the MOST System Viewer and MOST FBlock Monitor is activated/deactivated. If activated, any information about:.

  • the attached devices and busload is gathered and displayed in the MOST System Viewer
  • the function blocks is gathered and displayed in the several views of the MOST FBlock Monitor.

Note

The data collector can only be activated for one MOST channel. The Control Spy should also be activated on this MOST channel in order to obtain as much information as possible.
  • Retry detection threshold

If an ECU sends the same message multiple times, the message is classified as a Retry as long as the interval between transmissions is smaller than the threshold value specified here. The message is then counted in the Retry statistics of the MOST System Viewer. If the interval is greater than the threshold value, it is assumed that the multiple transmissions have to do with the internal state of the device or with a new request.

  • Fault injection

This switch activates the fault injection function of the Application Socket and the MOST NetworkMaster. If this switch is turned off, CAPL fault injection functions have no effect.

Advanced Options

These are used for refined configuration of the services offered by CANoe.

  • Poll configuration status after measurement start

When the CANoe measurement is started and the MOST Ring is already in operation, the MOST NetworkMaster might not report the current configuration status. If this happens, 5 s after the measurement start CANoe automatically sends an appropriate request to the MOST NetworkMaster.

  • Address for polling the Configuration Status (hex)

Address of the MOST devices, which contains the MOST NetworkMaster.

  • Address Handler

Typically messages with a destination address of 0xFFFF are sent to the right address by NetServices based on the FBlock.InstanceID. If this is not desired, and the user actually wishes to have the message sent to the destination address 0xFFFF on the bus, the Address Handler can be deactivated to allow this.

  • Automatic address calculation

Per the MOST specification the node address is derived from the node's position on the MOST ring. Therefore, the node address set in the Configuration Dialog should only be understood as the initial node address that is modified to the actual node position when the ring is started up. If this is not desired, i.e. if the configured node address should be retained over an entire measurement, address calculation can be disabled here.

Interface

Configure the connection parameters for the several network interfaces here.

  • VN2640 (MOST150)
    No additional parameters are needed to connect to the VN2640 via USB.
    You just need to verify that a VN2640 is allocated to the relevant MOST channel in the Vector Hardware Configuration dialog.
    By pushing the Twinkle button the Power LED of the VN2640, which is attached to this channel will twinkle.
    Select Passive Start of Measurement to skip the configuration of the transceiver chip at the start of measurement.
    Thus the MOST interface remains in the same state as after the last measurement and the MOST ring is not influenced during measurement start.
  • OptoLyzer MOCCA compact 150c (MOST150)
    Here the coax transmission type has to be set according to the wiring at the interface. The wiring alternatives are described in the manual of the interface.
    To get CANoe connected to the PCI card, the MOST Tool Base Service (ToolBase.exe) must be active. This service is part of K2L's OptoLyzer MOCCA bundle.
  • OptoLyzer G2 3150o (MOST150)
    Enter the IP address of the OptoLyzer and choose the network adapter to which it is connected. In case the OptoLyzer is already connected you can determine the IP address by pressing the [Search...] button.
  • OptoLyzer G2 3050e (MOST50)
    Enter the IP address of the OptoLyzer and choose the network adapter to which it is connected. In case the OptoLyzer is already connected you can determine the IP address by pressing the [Search...] button.
  • OptoLyzer MOCCA compact 50e (MOST50)
    No special settings are necessary.
    To get CANoe connected to the PCI card, the MOST Tool Base Service (ToolBase.exe) must be active. This service is part of K2L's OptoLyzer MOCCA bundle.

Setup

MOST Settings

  • HW mode: Master, Slave, Spy (only with OptoLyzer), Bypass (activates the bypass on measurement start

    • MOST150: The switch Static defines, whether the network interface keeps the MOST signal on at its output as Timing Master against the MOST specification, even if no input signal is detected.

    MOST50: As Timing Master the interface behaves according to the MOST specification, i.e. it sets the MOST signal state (on/off) at its output, corresponding to the input signal.

  • Addresses: Node address, group address

    • MOST150: MAC address of the MOST Ethernet node.

  • Send parameters: Low Level Retry parameters (XTIM, XRTY)

    • MOST150: Additionally the retry parameters and priority of the asynchronous channel.

    MOST50: Currently the retry parameters cannot be set.

  • Bandwidth parameters: SBC, frequency

If the node is configured as a Master the frame rate and SBC can be selected.

Note

During the measurement you can adjust several of this settings using the context menu of the MOST State Window.

Hardware-specific

  • VN2640, OptoLyzer MOCCA compact 50e, OptoLyzer MOCCA compact 150c, OptoLyzer G2 3050e and OptoLyzer G2 3150o

All MOST interfaces permit separate activation of Control Spy and Asynchronous Spy, as well as the Ethernet spy for MOST150.

The Control Spy delivers all Control messages to the application as CtrlSpyFrames independent of addressing. These Spy messages are also delivered for any messages that might have been received in the Node mode.

  • Apply analysis to...

Simultaneous activation of Node mode and Control Spy node for a channel causes duplication of messages in the Measurement Setup, e.g. in the Trace Window or logging file. This might be undesirable in some cases.

The easiest way to avoid this redundancy is to choose whether only Spy messages (recommended), only messages received in Node mode, or both types of messages should be processed. This filtering only applies to the control channel and only to the Measurement Setup. Nodes in the Simulation Setup are not affected by this.

Note

For MOST50 and MOST150 Vector strongly recommends due to technical reasons to configure the node and the spy of a connected OptoLyzer G2 on different channels. This is already pre-configured in the MOST50 and MOST150 template.

AMS/MHP

AMS

Long AMS messages are transmitted by sending a large number of individual Ctrl frames. This can lead to an overflow of transmission queues in some receiving devices. To prevent such an overflow, CANoe ensures a minimum interval between the individual Ctrl frames. The configured value (in ms) means that at least this time is waited before the next frame is sent. Especially for values < 5ms, please note that the AMS service also waits for the send receipt of the previous frame before sending the next frame. Hence, the effective send delay is the maximum between the configured interval and the latency of the network interface. For Vector interfaces the latency can be changed in the dialog Vector Hardware Config (button [Driver…]; interface setting: Receive latency).

MOST50 / MOST150: With MOST50 / MOST150, long AMS transmissions can be introduced by means of a telegram with TelID=4 and the transmission data volume as parameters. You can activate this mode here and determine the data volume above which such an introductory telegram is to be sent.

The Apply to all active MOST channels option can be used to determine whether the minimum interval specified for this MOST channel should be applied to all MOST channels.

Hardware specific

CANoe transmits AMS messages by sending individual CMS messages.

OptoLyzer G2 3150o, OptoLyzer G2 3050e, OptoLyzer MOCCA compact 50e: For these interfaces you can specify that the interface's internal AMS service is to be used. Please note, however, that the interface does not send Tx acknowledgments to CANoe for the individual CMS messages and that the size prefixed mode cannot be activated. Based on the AMS implementation in the interface's firmware, use of the interface's AMS service can yield increased performance.

MHP

With Receiver active, you activate/deactivate the automatic MHP receiver. This option has no effect on simulations using MOST High DLL.

With Accept connections, you specify how many connections the automatic MHP receiver can accept simultaneously. If the number of active connections for the node address of the connected HW interface exceeds the set amount, all other connection requests are actively refused. This is done by responding with a Start Connection message that has a higher priority than that of the request (according to MOST High Specification).

You can use Receiver buffer size to change the buffer size in order, for example, to simulate a device with an unlimited receive buffer.

Application Socket

  • Active:
    Activates/Deactivates the Application Socket.
  • Services:
    MOST NetworkMaster    , MOST PowerMaster, AddressHandler, NetBlock, NotificationService, CommandInterpreter:
    Activates/Deactivates the particular function block/service of the Application Socket.
  • Typical Master/Slave services:
  • Sets the typical services of a master node/slave node.
  • Master:
    All function blocks / services are activated.
  • Slave:
    All function blocks / services except MOST NetworkMaster and MOST PowerMaster are activated.
  • Automated query of the Central Registry
  • active:
    The Central Registry is being queried automatically after each Config (Ok/New).
  • inactive:
    You can decide whether to query the Central Registry (e.g. via CAPL).
