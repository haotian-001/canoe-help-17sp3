Differences to MOST25

 

Light & Lock, resp. MOST signal

With MOST50, data is transmitted via electrical lines. However, just as with the optical MOST25 and MOST150 systems, MOST50 also binds the individual devices into a ring using point-to-point connections. In the case of MOST50, as with the others, this ring contains exactly one selected device that acts as the timing master and MOST signal source. All the other devices synchronize to this MOST signal. This signal can, just like the light in the case of MOST25, either be present or not. The light/no light terminology is retained in CANoe/CANalyzer, which means that the MOST signal is on or off. In addition, slave devices can either be synchronized to the timing master (Lock) or not (Unlock). These cases are handled in the same way for all most systems.

Setup: Node & Spy

For technical reasons, even though the OptoLyzer G2 3050e supports node and spy at the same time, we strongly recommend that you assign these two event sources (node/spy) to separate MOST channels in CANoe. In this way, two MOST channels can use the same OptoLyzer device.

The OptoLyzer G2 3050e does not currently support the software synchronization of spy channel time stamps with time stamps of other channels. If you need to synchronize the spy, you can connect a sync line (see Installation OptoLyzer G2 3050e) to the OptoLyzer and the network interfaces of the other channels.

Current limitations of the OptoLyzer G2 (firmware 1.5.0)

  • Asynchronous channel: The node only supports transmission and receipt of packets no longer than 48 bytes.
  • Software synchronization of spy channel
  • The OptoLyzer G2 3050e does not currently support the software synchronization of spy channel time stamps with time stamps of other channels. If you need to synchronize a spy, you can connect a sync line to the OptoLyzer and the network interfaces of the other channels.
  • Stress functionality: Busload stress functionality is currently not supported. To generate a busload, you can send timer-controlled messages from a CAPL node.
  • Control channel: It is not possible to set the retry parameter of the INIC.
  • Spy: The spy channel does not issue notifications of SBC changes or node positions.
  • The OptoLyzer does not support static timing master functionality (i.e. the timing master always adapts its output signal to the input signal)

Control channel

  • Control Messages now have a variable length between 0 and 17 bytes

With MOST50, control messages have a variable length when they are transmitted. The maximum length still is 17 bytes. This means that no higher values can occur in any existing CAPL code that evaluates the DLC of mostMessages.

Currently, control channel transmission only supports messages with a value of TelLen in the data area, which corresponds to the real length of the CMS message. Thus, sending erroneous CMS messages with invalid TelLen values is not possible.

  • No CMS acknowledgments for segmented AMS transmissions

With MOST25, each CMS frame of a segmented AMS transmission is either received by the node or reported as Tx acknowledgment to the node; the OptoLyzer G2 3050e as node only reports the AMS messages. This leads to an absence of CMS events in the Trace Window. Analysis nodes working on CMS might be impaired, which makes using the spy mode for logging purposes mandatory.

  • MOST50 does not support system messages

Existing CAPL code that allocates synchronous channels via system messages either to stream data or to stress the system with a full allocation table will not work any more. The handling of the synchronous channels is significantly different to that of MOST25.

Existing CAPL code that remotely reads the registers of a MOST transceiver for analysis purposes will no longer work. Access to the transceiver of a device can be gained by sending normal CMS or AMS messages to the INIC function block of the relevant device.

  • AMS – "size-prefixed transfer"

The MOST Specification 3V0 introduced "size-prefixed-transfer" for transmission of segmented AMS messages. A segmented transmission is thereby started by sending a telegram with TelID = 4 and two user data bytes. The user data contains the length of the whole AMS message. This allows the receiver to allocate the appropriate amount of buffer memory for reception of the message. Size-prefixed transfer can be activated and deactivated in the hardware configuration dialog. Size-prefixed transfer is deactivated by default.

Packet channel

The OptoLyzer G2 3050e currently (firmware 1.5.0) only supports packets of up to 48 bytes. Please take this into consideration when transmitting packets from CAPL, using the MOST IG and setting NDF parameters in the MOST High Protocol.

The OptoLyzer also limits throughput from the application to the asynchronous channel. The AIR parameter of the MOST High Protocol therefore needs to be configured to maintain bigger distances between single packets than was the case with MOST25. The optimum value for AIR depends on the packet size used.

Pre-emptive acknowledgment

In order to reduce busload, the MOST50 transceiver terminates message transmission in the early stage as soon as message reception is foreseen to fail (= pre-emptive acknowledgment). When spying MOST50 traffic, these aborted transmissions are reported as message fragments. The information a fragment carries depends on its position in the MOST frame, e.g. a very early abortion of a send attempt may lead to a fragment in which not even the source address field was put on the bus, and which therefore cannot be spied.

Audio routing

In contrast to MOST25, where the synchronous channels are managed with the help of allocation and deallocation messages and where an allocation table indicating the occupation of each synchronous channel can be observed on the bus, this management is transparent with MOST50.

Here the MOST controller only delivers information on which labels exist and what bandwidth they offer. For synchronous sources, allocation and subsequent routing to a label are combined in a single step.

Both differences are accounted for by an adapted view of the MOST Audio Routing Window.

MOST IG

CANoe is able to determine the speed grade of the configured MOST channel and allow either 17 bytes or up to 45 user data bytes when sending via CMS.

When sending on MOST50, however, the value of TelLen determines the number of bytes sent. If TelLen exceeds the number of entered bytes in the hex dump view, the messages will be extended with zero bytes. If TelLen is smaller than the number of entered bytes in the hex dump view, only TelLen bytes will be sent, although the configured bytes will remain in the hex dump view.

Application socket

CANoe's application socket was enhanced for MOST50 according to the MOST Specification 3V0.

  • NetBlock

The NetBlock function block provides the following new functions:

  • ShutdownReason (0x009)
  • ImplFBlockIDs (0x012)
  • EUI48 (0x013)
  • RBDResult (0x405)

As all these functions deliver device or manufacturer specific information, CANoe NetBlock does not fully implement them. CANoe will merely respond with ‘FktID not available’ error messages to the caller.

Responding with error messages, simulating behavior or intentionally introducing erroneous behavior is easily done by programming a pattern in a CAPL node, as shown below.

Example CAPL

on mostAmsMessage NetBlock.ShutdownReason.Get
{
   // CANoe does not report an error (“FktID not available“)
   // Report message variable
   mostAMSMessage NetBlock.ShutdownReason.Status msg;

   // initialize report message
   MostPrepareReport(this, msg);
   // device specific evaluation of the last shutdown reason
   // msg.SSOCUStatus = ...
   // ...
   // send report
   msg.InstanceID = MostGetNPR(MostGetChannel());
   output(msg);
}
  • MOST NetworkMaster
  • New function: SystemAvail (0xA10)

The MOST NetworkMaster function block has one new function: SystemAvail (0xA10). This function is handled in the same way as the new NetBlock functions (see example in chapter 3.7.1).

  • Obsolete function: Boundary (0xA03)

The MOST NetworkMaster no longer provides the Boundary (0xA03) function.

  • Number of retries to resolve a duplicate functional address

When resolving duplicate functional addresses (FBlockID + InstID), the MOST NetworkMaster tries to change the InstID of a function block. If the first attempt to change the InstID fails, the MOST NetworkMaster attempts to change the InstID again using another value. The number of attempts made can be configured in the CAN.INI file:

[MOSTApplicationSocket]
; Default: 20 attempts
NMAttemptsChangeInstID=20

  • MOST PowerMaster

The MOST PowerMaster uses the Shutdown method of the NetBlock of each device for the shutdown procedure. Ack-OpTypes must now be used for methods. The MOST PowerMaster thus sends the following message to all devices during the shutdown procedure:

NetBlock.Shutdown.StartAck(SenderHandle, Control)

If a device wants to suspend the shutdown procedure, it has to respond with the following message to the MOST PowerMaster:

NetBlock.Shutdown.ResultAck(SenderHandle, Suspend)

The MOST PowerMaster checks whether the received suspend message refers to the current shutdown procedure by evaluating the <SenderHandle> parameter.

"InitReady" versus "NetOn"

The NetOn event was renamed to InitReady. The meaning remains the same, however. The Trace Window displays this information in the disassembly column and the detail view. In CAPL there is an additional event procedure, OnMostInitReady. To assure the correct operation of existing CAPL programs, the OnMostNetOn event handler is still called.

Important Note

As soon as this event occurs, both event procedures are called, in case they are both implemented in CAPL.
