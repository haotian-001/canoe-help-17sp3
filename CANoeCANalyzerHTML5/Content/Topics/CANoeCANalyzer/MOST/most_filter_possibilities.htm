Filter

 

CANoe.MOST offers a range of possibilities for filtering the data displayed in the Trace Window or recorded in a logging file. This page helps you select an appropriate filtering option based on a series of typical use cases.

Use cases:

Observation only, possibly with simulation (e.g. via IG) as a secondary requirement

  • Set up node/spy filtering in the Hardware Configuration dialog so that the Analysis branch only shows spy messages. If you don't want this, you can add a MOST application filter to the test setup instead. Filtering takes place via insertion of a line that simply selects the direction attribute between node and spy messages. All other attributes are left with wildcard "*" for this.
  • Configure the MOST filters in the Trace Window configuration so that all CMS messages, but only segmented AMS messages, are displayed. This results in the most compact Trace Window display.

Restriction to messages from one bus system

Insert a channel filter in the test setup

Restriction to messages from one device, function block or use case

Insert a MOST application filter in the test setup and configure it using wildcards (*)

Computer with low performance capacity or with many bus systems attached

  • If access to an asynchronous channel is not needed, you should deactivate this feature in the Hardware Configuration dialog.
  • If the MOST FBlock Monitor is not needed, you can switch off the corresponding service in the Hardware Configuration dialog.
  • If MHP observer events are not needed, deactivate the observer via its configuration dialog. In contrast to this, filtering events in the Trace Window's configuration dialog reduces processing time only slightly.

Long term recording or logging on computers with limited hard disk space or low hard disk data transfer speeds

  • Use the BLF logging format (lowest disk space requirement and highest performance, most detailed recording)
  • Limit the events recorded using the MOST application filter (messages, system events)
  • If needed, switch off internal events (e.g. bus statistics) in the logging block
  • If needed, insert an environment variable filter in the Measurement Setup in order to limit file contents to bus events.

Recording to a log file: With sufficient hard disk space and performance

Set up node/spy filtering in the Hardware Configuration dialog so that the Analysis branch processes both spy and node messages. In some instances, these two sources may deliver mutually required data (e.g. acknowledgment data); these messages can still be filtered later on during the offline analysis.

Simulation or partial simulation of nodes in CANoe

Configure the MOST filters in the Trace Window configuration so that internal messages are displayed. This allows you to observe the interaction between multiple simulation nodes (e.g. the function block and controller, and the MOST NetworkMaster).

Use of CAPL analysis nodes in the test setup

  • These should react to AMS messages with on mostAmsMessage as this guarantees access to all data, even for segmented transmissions.
  • Configure the Trace Window so that all AMS messages are displayed in order to reconstruct the behavior of the CAPL node.
  • If you want the CAPL node to work as a Pass Filter by default, the handling routine

on *
{
  output(this);
}

has to be implemented. If there are messages that should not be passed on, you will need to implement a separate handling routine that does not forward the messages using output(this).

Restriction to error situations that only occur when several conditions coincide (dynamic filtering)

Filtering and triggering can occur in CAPL nodes in the test setup. This lets you automatically query any given conditions and chronological relations. The relevant CAPL nodes function as blocking filters by default, which means that every message processed by a handling routine must be forwarded explicitly using output(this). If you want the CAPL node to work as a Pass Filter by default, the handling routine:

on *
{
  output(this);
}

has to be implemented. If there are messages that should not be passed on, you will need to implement a separate handling routine that does not forward the messages using output(this).

Notes

  • Combiner and observer events are not written to the logging file: AMS messages, MHP packets and MHP protocol errors are reconstructed when the logging file is reloaded.
  • If you are using OptoLyzers, you can choose whether the Trace Window displays only node messages or only spy messages for the corresponding channel. If used as a node and as a spy, two OptoLyzers are required; you can easily separate their messages using a Channel Filter before the Trace Window.

Filtering possibilities:

  • Hardware settings (deactivated asynchronous channel, choose between node and spy)
  • MOST Filter configuration page of the Trace Window (select messages and event types to be displayed in the Trace Window: AMS/CMS, packets, internal/external, register events) and MOST analysis filter.
  • Predefined filter of the Trace Window (select MOST High Observer messages to be displayed in the Trace Window) and MOST analysis filter.
  • Analysis filter in the Trace Window (temporarily reduces display in the paused Trace Window to selected message types) and MOST analysis filter.
  • MOST Application Filter (universally usable filter for messages, packets and hardware events)
  • CAPL nodes (dynamic event filtering)
  • Channel filter (select channels that pass events, select bus systems)
  • MOST Filter (MOST message and event types: AMS/CMS, packets - recommendation: use the MOST Application Filter)
  • Multibus filter (MOST messages - recommendation: use the MOST Application Filter)
