LIN Communication (Network Errors and Network Events)

 

LIN Communication

The LIN protocol is implemented in software, in general. As a low-cost protocol, its implementation is on the same controller as the application and it dispenses with explicit error detection and handling.

For analysis of the LIN communication, however, detailed interpretation of the communication behavior is essential. To detect any network errors and the network status, CANoe provides an extensive event structure and a unique interpretation.

LIN Network Errors

Error Type

Description

Send Error

A send error occurs when no Slave responds to a send request from the Master (internal or external).

A send error event includes information about the send identifier.

Note

Send errors for the diagnostic frame SlaveResp (Frame Id = 0x3d) and for event-triggered frames do not violate the LIN protocol.

Receive Error

A receiving error may have a wide variety of causes.

An external Master can cause a receiving error by

  • transmitting synch break that is too short or
  • assigning an incorrect parity to the frame identifier.

A receiving error can also be caused

  • by a slave sending an illegal character during a Bus Idle phase (e.g. because it did not finish transmission quickly enough and the checksum byte of the response was sent during the Bus Idle phase).
  • due to a faulty (dominant) stop bit (i.e. framing error).
  • if the LIN hardware receives a character that is different from the character sent during transmission.
  • at the start of a measurement (in a correctly functioning system), if the LIN hardware only receives part of a frame.

If a receiving error is detected for a data byte, it should be noted that the first byte is data byte 1 and not data byte 0 i.e. the data bytes begin with the index 1.

Checksum Error

A checksum error occurs when a slave sends an incorrect checksum value for a frame response that is otherwise correct.

Synchronization Error

This error can occur when an external Master is used.

Synchronization errors occur if the LIN hardware cannot synchronize with an external Master. This might happen if the baudrate actually used by the Master deviates by more than 15 % from the baudrate specified by the LIN hardware. In this case the baudrate value should be modified (see Configuring baudrate on LIN channel).

This error may also occur if the Master transmits an invalid or corrupted synch field. The synchronization error messages display 4 time differences (in microseconds); each represents a time span between two edges. Depending on when the LIN hardware detects an invalid synch field, up to three of the last values may be zero. For comparison, if the synchronization was successful, the time differences should be equal 2 bit times.

Dominant Signal (Error)

This error event occurs when a long dominant signal is received, which is not a valid wake-up request nor a valid synch break or a valid byte.

Spike (Error)

This error event occurs when a short dominant signal is received. The spike must be shorter than one bit and must not be a valid wake-up request. This event occurs only with LIN 2.x.

LIN Network Events

Event Type

Description

Scheduler Mode Change Event

This status event is only displayed when the LIN hardware scheduler generates send request from a new table for the first time. In particular this event may appear on starting a measurement.

Sleep Mode Event

A Sleep mode event occurs…

  • at the start of a measurement in order to report the initial state and
  • every time the mode (wake-up or sleep) of the LIN hardware changes.

The reason for the mode change is reported in the Data column of the Trace Window.

Note on "wake-up due to bus traffic"

A regular slave is usually not capable of receiving or answering the frame that caused the slave to leave the sleep mode.

Wake-up Signal Event

A wake-up signal appears when a wake-up request frame is received.

If the master mode of LIN hardware is activated, the wake-up frame does not automatically force the LIN hardware to leave the Sleep mode. The exact behavior of LIN hardware while in master mode can be set by the linSetwakeupCondition() CAPL function. With this function, a slow booting master can be simulated.

Baudrate Change Event

This message is only displayed when an external Master is configured.

The LIN hardware reports with this event at the start of a measurement and whenever the baudrate changes by more than 0.5 % during a measurement. If this event is displayed, then the LIN hardware is synchronized to the baudrate of the external Master.

When a Master is simulated this event is reported each time the baudrate is manipulated via corresponding CAPL function.

Checksum Model Event

This status event is only displayed when an external Master is configured and the LIN hardware successfully read the data of an unknown frame. If the data and the checksum fit to a checksum model, this checksum model is set as the expected model for this frame.

DLC Event

This status event only occurs when an external Master is configured and the LIN hardware successfully determined a data length of a new frame. This value is set as the expected data length for the future occurrences of this frame.

An error will be reported if the same frame is received with a different data length.

If the Master mode of the LIN hardware is activated and a frame's DLC is not specified, then the DLC is determined using the frame's identifier.

Event Triggered Frame Event

An event-triggered frame event will occur for frames defined as event-triggered frames in the LIN Description File (LDF). In case of a collision (i.e. more than one Slave tried to respond) or no Slave response, an event-triggered frame event will be displayed in the Trace Window rather than a receive or transmission error, since both cases are allowed by the LIN protocol for event-triggered frames.

If only one Slave responds to an event-triggered frame header or a collision resolution is in progress, then the event-triggered frame event will appear in the Trace Window in addition to the event-triggered frame.

The event-triggered frame event provides an interpretation of event-triggered frame slot:

  • No Response
    No Slave responded.
  • Collision
    More than one Slave tried to respond.

  • Collision Resolution
    The Master is carrying out a collision resolution i.e. headers are being sent for each associated frame using the event-triggered frames slot.

    Note

    This event is obsolete and can occur up to CANoe v5.2 only.
    Different states of event-triggered frame are reported as follows:

    • No Response
      Reported as Transmission Error, which doesn't violate LIN protocol.
    • Collision
      Reported as Receive Error, which doesn't violate LIN protocol
    • Collision Resolution
      Reported as a normal LIN frame event.
  • Single Response
    Only one slave responded.

LIN Disturbance Event

A disturbance event is triggered when bits are inverted on the network with CANoe.

The level of the disturbance (dominant or recessive), the byte position and bit position, and the length of the disturbance are specified.

LIN Error Interpretation: Events

For receiving errors in particular, it is not easy to find out the actual cause of the incorrect behavior. Receiving errors often result from timing violations of the LIN protocol. To facilitate finding out the cause of an error, CANoe provides an interpretation of previous errors. All data received are thereby displayed in the Trace Window.

Event Type

Description

LIN Error Interpretation
(slow response)

If a LIN node responds too slowly (the allowed frame length is exceeded by more than 40%), CANoe interprets previous receiving errors as slow response.

The received data are displayed in the Trace Window for further analysis.

LIN Error Interpretation
(short response)

If a LIN node responds with a response that is too short (for example, due to a bit error), CANoe interprets previous receiving errors as short response.

The received data are displayed in the Trace Window for further analysis.

LIN Error Interpretation
(unexpected wake-up)

If a byte is sent unexpectedly during bus idle, this may entail an unexpected wake-up signal in the following cases:

  • Byte value: 0x00, 0xC0, or 0x80 ( LIN 1.x).
  • The byte has a dominant phase that is between 150 µs and 5 ms (LIN 2.x).

In this case, CANoe signals an Unexpected Wake-up.

The length of the wake-up signal is displayed in the Trace Window.

LIN Error Interpretation: Global Messages

If multiple error messages occur during communication, incorrect configurations may be the cause. CANoe analyzes the communication errors heuristically and outputs a warning in the Write Window that allows inferences to be drawn about the potential problem.

Event Type

Description

Possible conflict with external master on LIN <channel> detected at <time>.

If a LIN Master is being simulated with CANoe and a real LIN Master is simultaneously located on the connected network, collisions will occur.

CANoe detects such collisions by a heuristic interpretation of the communication errors on the network.

Possible incorrect baudrate setting on LIN <channel> detected at <time>.

If the baudrate configured in CANoe differs from the baudrate used on a connected real network, then a communication error on the connected network is interpreted.

CANoe detects this situation by a heuristic interpretation of the communication errors on the network.

Short delimiter (less than 2 bit times) on LIN <channel> detected at <time>.

Invalid break delimiter was detected. The valid break delimiter is to have a length of at least one bit time.

The LIN Consortium recommends delimiter lengths of at least 2 bit times.
