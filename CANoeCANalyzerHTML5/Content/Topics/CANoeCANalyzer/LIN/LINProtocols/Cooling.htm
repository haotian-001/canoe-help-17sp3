COOLING Protocol (LIN)

 

Introduction

This document describes the results of an analysis made by Vector according to the COOLING-specification Cool_lin_1.0._F (released 2003-10-07) using CANoe 5.0 SP3 with the LIN option.

Database

An LDF database is not suitable for defining COOLING networks, since for each frame only one sender/publisher can be defined. Vector's DBC format, however, permits the definition of multiple senders/publishers and also allows the additional COOLING attributes (e.g. node address) to be defined. Vector has therefore prepared a DBC database example of COOLING customers. If necessary, this database can be edited by customers using the CANdb++ Editor provided with CANoe.

Bus Monitoring

Using a DBC database for COOLING, all COOLING frames can be monitored and displayed symbolically in the Trace Window.

  • Address Filtering
    Instead of displaying all COOLING frames in the Trace Window, it can be useful to display the frames for each COOLING slave in separate Trace Windows. This can be achieved by using a very simple CAPL program, which filters the COOLING frames according to their first data byte i.e. the slave’s address. An sample configuration has been created for COOLING, which demonstrates how to do this.
    Please note that the filter cannot currently be used for this purpose.
  • Addressing Service
    The current version of the COOLING specification permits long synch breaks for auto-addressing COOLING-slaves. For the baudrate 19200 this requires a non-LIN-conform synch break length of 36 bit times. In this case, CANoe will display the frame as a receive error in the Trace Window. It is however to be expected that the COOLING consortium will change this sync break length to be within the ranges permitted by the LIN specification.
  • Sleep Service
    CANoe is currently not capable of recognizing a COOLING sleep service. This means that in the Trace Window no Sleep mode event will be displayed if the COOLING master performs a sleep service.

Master Simulation

Using CANoe it is possible to simulate a COOLING master. The COOLING sample configuration includes a COOLING master implemented using CAPL. This implementation of a COOLING master should be capable of performing all COOLING services. The following services have been tested with stepper motors from Valeo: ACT_CFR_ADDR, ACT_CFR_INI, ACT_CFR_MOD, ACT_CFR_SET, ACT_DFR_PRG, ACT_RFR_PRG, ACT_CFR_PRG, ACT_RFR_STA, ACT_DFR_STA, ACT_RFR_ISSRx.

  • Addressing Service
    It is possible perform an addressing service, provided that the synch break length conforms to the LIN specification. The current version of the COOLING specification requires for the baudrate 19200 a non-LIN-conform sync break of 36 bits. Such a synch break cannot currently be sent using CANoe. As already mentioned, it is to be expected that the COOLING consortium will change this synch break length to be within the ranges permitted by the LIN specification.
  • Sleep Service
    Although it is possible to perform a sleep service, the Trace Window will not recognize this service as a Sleep mode event.

Slave Simulation

COOLING services require two LIN frames. The first LIN frame (demand-frame) addresses the slave and the second LIN frame contains the service and its data. A COOLING slave should only respond to the second LIN frame if it was correctly addressed by the first frame. This means that in order to simulate a COOLING slave, CANoe must be capable of activating and deactivating Slave responses during a measurement. This is now possible with CANoe 5.0 SP3 using the CAPL function LINSetRespCounter (requires LIN firmware version 1.14 or higher). The COOLING sample configuration also demonstrates how COOLING slaves can be simulated using CANoe.

  • Timing Limitations
    Since CANoe requires a certain amount of time to update the response data, COOLING services, which are performed at high baudrates, may fail. In particular if both LIN frames belonging to a single COOLING service are sent immediately after another.
  • Auto-Addressing
    Auto-addressing cannot currently be simulated using Vector's LIN hardware.

Remaining Bus and Network Simulation

Using CANoe it is possible to simulate a complete COOLING network or perform a remaining bus simulation. The COOLING sample configuration simulates both a COOLING Master and COOLING Slaves.

COOLING Sample Configuration

A COOLING sample configuration and DBC database template is available to CANoe customers either from the member's area of the Vector homepage or via support.
