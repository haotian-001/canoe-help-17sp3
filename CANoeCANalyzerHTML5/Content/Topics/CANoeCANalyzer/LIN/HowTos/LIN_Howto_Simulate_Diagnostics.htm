How to Simulate a LIN Diagnostic Tester

 

To simulate a diagnostic tester on a LIN bus perform the following steps:

  • Add a diagnostic description or a basic diagnostic device to your configuration.
  • For that purpose open the diagnostic configuration dialog via Configuration|Diagnostics/ISO TP Configuration…
  • Assign a diagnostic description file or a basic diagnostic device to your LIN bus (represented with an own tab). You may also add an generic diagnostic description which is shipped with your installation.
  • After adding a diagnostic description file or a basic diagnostic device the dialog Diagnostic Description Settings opens.
  • Name your control unit in the ECU control.
  • Select the slave node under test via the Function/Node control.
  • The slave node must be described in an LDF file which is assigned to the configuration and must have assigned a unique NAD.
  • When leaving the dialog the Diagnostic Consoles are opened. The consoles allow to send diagnostic services directly.
  • Add the master node in the Simulation Setup. This is necessary to set the LIN hardware into master mode and to start the scheduling.
  • Instead of the Diagnostic Consoles it is also possible to use a test module and to send diagnostic requests via CAPL. If you work with a basic diagnostic device configure your diagnostic services via the Basic Diagnostic Editor.

To simulate a diagnostic server on a LIN bus perform the following steps:

  • To simulate a diagnostic server the node must implement a TP layer.
  • Add the LINtp DLL via the context menu of the node within the modules page. The LINtp DLL implements the LIN Transport Layer.
  • The LINtp DLL is located in  the Exec32 directory of your installation.
  • You have to implement the following callback interface in your CAPL program. (New functions in Callback function in the CAPL Browser):

Interface

Callback Functions

Description

Generic

_Diag_DataRequest (byte data[], dword count, long flag)

Data transmission is started. Layer7 commits the data to the transport Layer (Layer 4). This function encapsulates the used transport Layer.

Generic

_Diag_SetChannelParameters ()

Performs the channel settings. E.g. sets the NAD of the tested slave node.

Generic

_Diag_SetupChannelReq ()

Opens the diagnostic channel.

LIN

LINtp_DataCon (long count)

If data were received successfully this function routes the data to the diagnostic layer.

LIN

LINtp_DataInd (long count)

This function is called when data are received. The assembled data can be fetched.

LIN

LINtp_ErrorInd (int error)

Called when an error occurs in the TP layer.

Example

_Diag_DataRequest( byte data[], dword count, long flag)
{
LINtp_DataReq(data, count, gNodeAddress);
}

_Diag_SetChannelParameters()
{
gNodeAddress = diagGetCommParameter("CANoe.AddrExt");
}

_Diag_SetupChannelReq()
{
Diag_SetupChannelCon();
}

LINtp_DataCon(long count)
{
Diag_DataCon(count);
}

LINtp_DataInd( long count)
{
BYTE data[4095];
LINtp_GetRxData(data, count);
Diag_DataInd(data, count, 0);
}
 

LINtp_ErrorInd( int error)
{
Diag_ErrorInd(error);
}
  • You can use this example via copy and paste in your implementation.
  • Use the CAPL diagnostic API to program your diagnostic responses.
