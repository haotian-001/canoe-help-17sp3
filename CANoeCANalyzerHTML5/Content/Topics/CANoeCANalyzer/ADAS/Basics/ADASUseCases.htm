Stimulation of ADAS algorithms with simple scenarios

For a first test of an ADAS algorithm it is often sufficient to stimulate it with a simple scenario. The Scenario Editor offers the possibility to create exactly such simple scenarios. An ego vehicle and other vehicles can be placed on a self-created route. Freely configurable sensors can be added to the ego vehicle. After importing the created scenario with the Scenario Manager, all vehicles that detect these sensors are made available as Detected Objects in CANoe. They can thus be analyzed in the Scene Window, for example, or passed on to the ADAS algorithm under test using CANoe SIL Adapter Builder. It should be noted that scenarios created in this way do not offer any track accuracy, no models are stored for the driving physics, and no closed-loop tests are possible. On the other hand, they can be created very quickly and easily.

Stimulation of ADAS algorithms with complex scenarios

For a complete software-based test of an ADAS algorithm, complex scenarios are required that, among other things, provide lane accuracy, have stored models for the driving physics and are closed-loop capable. Such scenarios can be created e.g. with the simulation environment DYNA4. Likewise, any other simulation environment can be connected. This is made possible via documented interfaces such as an S-Function in the Matlab interface or a C-API interface. The prerequisite is that the simulation environments provide data in ASAM OSI format. After the simulation environment has been integrated, all vehicles that the sensors of the Ego vehicle detect are made available as Detected Objects in CANoe. They can therefore be analyzed in the Scene Window, for example, or passed on to the ADAS algorithm under test using CANoe SIL Adapter Builder.

Testing ADAS Algorithms

The main use case of the CANoe ADAS Feature Set is testing ADAS algorithms. To do this, the ADAS algorithm does not have to be running on an ECU and must have functioning network communication. The ADAS algorithm can be stimulated directly in the development environment. The CANoe SIL Adapter Builder (SAB) can be used for this purpose. The Detected Objects created from the stimulation scenarios are passed on to the development environment via the SAB. There they can be used to stimulate the ADAS algorithm. Potential results of the stimulation (e.g. HMI warnings or automatic driving interventions) can also be passed back to CANoe via the SAB. Test units or test modules running in CANoe can monitor these results automatically. Test creation is facilitated by an ADAS-specific Test Feature Set API.

Analysis of sensor data via network communication

For the analysis of a real sensor (e.g. a camera or a radar) that delivers pre-processed objects over vehicle networks, the network communication can be mapped to DOs. Then, they are available in the Trace Window and Scene Window for better analysis.

Stimulation of ADAS algorithms via network communication

For the stimulation of an ADAS algorithm with a real sensor (e.g. a camera or a radar) that delivers pre-processed objects over vehicle networks, the network communication can be mapped to DOs. Then these Detected Objects can be passed on to the ADAS algorithm under test using CANoe SiL Adapter Builder.

Analysis of Logfiles

Log files that already contain ADAS DOs (e.g. from logged measurements) can be imported again in offline mode. The data can then be analyzed e.g. in the Trace Window and in the Scene Window.

Analysis of scenarios

If the simulation environment generates both ground truth and perception data, the scene window can be used to compare whether all objects are where they are expected to be. Additional filter options of the Scene Window allow to show/hide the ground truth or perception data. Color gradations can be used to easily distinguish between the different object types.
