Electrical Error Stimulation Port

 

General

Starting with version 8.5, CANoe supports the EES (Electrical Error Stimulation) port of the ASAM XIL API.

This port enables you to generate electrical errors using a standardized interface. A variety of possible error types, such as interrupts and short circuits, are available to you for this.

Hardware Requirements

A Vector VT System is required to use the CANoe EES port. This must be equipped with at least one VT1004 or VT2004. Other VT modules are not supported at present. Each connected VT1004 or VT2004 provides the possibility of disturbing four different signals independently of one another.

The CANoe EES port thus supports the following error types:

  • Short circuit of one or more signals to Gnd, UBatt, or one of two potentials.
  • Interruption of one or more signals.

These errors can be enabled in two ways:

  • Statically (i.e., permanently active)
  • Dynamically (i.e., only active for a certain time)

The following errors are not supported:

  • PWM-controlled errors (Loose Contact errors)
  • Errors with resistance (Resistor errors)
  • Errors between two signals (Pin 2 Pin errors)
  • Errors with load (With Load errors)

A detailed description of the different error types can be found in the documentation of the XIL API standard under: https://www.asam.net/standards/detail/xil/.

Both manual triggers and software triggers are available to you in the CANoe EES port. You can formulate software triggers in the form of ASAM GES conditions. This allows you to evaluate CANoe system variables, as well. Because hardware signals can also be read out in this way, hardware triggers are not supported explicitly by CANoe.

Connection of the Hardware

You can inject a signal in the original load a and original load b connections on each channel of a VT1004 module. Analogously, the signals for the VT2004 are connected to the original sensor a and original sensor b connections.

You can apply up to four potentials to the bus bar connections of the VT1004or VT2004. In doing so, you must adhere to the following assignment:

  • Bus Bar 1A -> Gnd
  • Bus Bar 1B -> UBatt
  • Bus Bar 2A -> Potential 1
  • Bus Bar 2B -> Potential 2

Note

Adherence to this assignment is important for the following reason:

If, for example, an error of type Shortcut to Gnd is generated, CANoe switches the RelayBusBar1A relay. This produces a short circuit between the original load a and bus bar 1a connections. If the expected potential is not connected there, however, it may result in undesired behavior or even hardware damage.

Configuration in CANoe

In order to use theCANoe EES port, you must perform the following configuration steps in CANoe:

Step by Step Procedure

  1. Activate XIL API support in the Options dialog under Extensions|XIL API & FDX Protocol.
    In most cases you can keep the preassigned UDP Port 3030.
    If you want to use an MA port in addition to the EES port, also activate FDX support here and specify a corresponding description file.


  2. Open the VT System Configuration dialog (Hardware ribbon tab) and use the Adapt to Connected HW button to search for connected hardware.
    All connected VT1004 and VT2004 modules are then listed.
    Assign descriptive names to the channels of the VT modules, e.g., corresponding to the required EES port signals.

  3. Open the VT System Control Window (Hardware ribbon tab).
    This window displays the current state of the VT System channels.
    Here you now connect all ShortCircuit relays of the utilized channels.
    Save this state as the start state of the respective channel (button Save current state as start state).
    This saved state is now produced automatically at each measurement start.

Structure of the EES Port Configuration File

To configure the EES port on the tester side, you need a special configuration file in XML format. This contains the following information:

  • A mandatory connection element. This holds:
  • the IP address of the computer on which CANoe (or more precisely the RT portion of CANoe) runs
  • the XIL API port configured in CANoe (usually 3030)
  • An optional list of software triggers that are to be available to the EES port.

The following example shows what this type of configuration file might look like:

Example

<EESPort>
  <Connection ip="127.0.0.1" xilPort="3030"/>
  <Triggers>
    <Trigger name="Error Set 1" condition="5 > 1"/>
    <Trigger name="Error Set 2" condition="Triggers::Trigger1 &gt; 0.5"/>
    <Trigger name="Error Set 3" condition="Triggers::Trigger1 &gt; 1.5"/>
  </Triggers>
</EESPort>

You formulate the trigger conditions corresponding to the ASAM GES syntax. Note that all special characters must be written in an XML-compliant manner (thus, for example, &gt; instead of >). In addition, the name of a trigger must match the name of the error set that it is to trigger.

Naming of VT Channels and EES Signals.

The signal names that can be used when configuring EES port errors correspond to the names of the VT System channels in CANoe. For example, if you assign the name Signal1 to a VT channel in CANoe, you can also address this channel in the EES port with Signal1.

Interaction with other XIL API Ports

You can operate the EES port together with other XIL API ports supported by CANoe without any problem. This may even be necessary since the EES port only generates errors when the CANoe measurement is running. Because this port does not provide the option of controlling the CANoe measurement, however, additional use of an MA port is advisable. You can use this port to not only start and stop the CANoe measurement but also to set triggers and read out variables, for example.
