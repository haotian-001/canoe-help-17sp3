ASAM XIL API

Ribbon: Environment Tab|Tool Couplings|XIL API/FDX

General

The ASAM XIL API is a standard that specifies software interfaces for access to test systems. This API allows different test systems to be programmed in a standardized manner. Furthermore, investments in test scripts remain protected even when a change is made from one test system to another.

The following XIL API standard versions are supported by CANoe:

  • XIL API 2.0.1:
  • Supported starting with CANoe 8.5.
  • Discontinued starting with CANoe 15 SP3
  • XIL API 2.1.0:
  • Supported starting with CANoe 11.0 SP3
  • XIL API 2.2.0:
  • Supported starting with CANoe 15 SP3

Architecture

A Vector XIL test system with XIL API interface is structured as follows:

It consists of the following components:

  • A test application provided by the user.
    This performs tests that are implemented against the official ASAM XIL API interfaces.
  • The implementation of the XIL API interfaces provided by Vector.
    This can be configured using simple configuration files.
  • Vector CANoe
    That enables access to the hardware and provides comprehensive simulation possibilities.
  • Hardware for interaction with the system to be tested.
    Besides the Vector hardware, many interfaces of third-party providers are supported.
  • The system to be tested.

XIL API Installation

The components of the XIL API consist of the ASAM AE XIL API standard assemblies form ASAM e.V. and additional of the XIL API Implementation for CANoe. This implementation is made available in the form of a .NET component by Vector Informatik GmbH. It contains all implementations of the XIL API ports currently supported by CANoe (e.g., the Model Access port).

With the execution of the Vector CANoe XIL API setup both components are installed.

Installing the XIL API for CANoe

In order to use the XIL API implementation for CANoe, you need to install the required assemblies on your computer. To do so execute the setup: Start Setup

During installation the following file will be created:

C:\ProgramData\ASAM\XIL\Implementation\Vector.CANoe.imf

This file is required by the XIL API test bench. It is used to locate and instantiate the CANoe XIL API at runtime.

Test Bench Initialization

XIL API test benches are instantiated by calling ITestBench.CreateVendorSpecificTestbench.
To create the CANoe test bench you have to use the following parameters:

  • vendorName: must be set to Vector
  • productName: can be either CANoe32 or CANoe64.
    This settings depends on what architecture your XIL API client application is using.
    It does not correspond to the CANoe version being used.
  • productVersion: the version of the XIL API interface to use (2.1.0 or 2.2.0).

Backend Selection

CANoe offers two different backends for its XIL API implementation: FDX and LightRPC. You can choose between these backends implicitly via the port configuration files. Whenever a FDX configuration block is found in the file, FDX is used as the backend. If you omit the FDX configuration block, LightRPC will be used by default.

LightRPC Backend

LightRPC is a proprietary network-based protocol. It allows for a fast data exchange between CANoe and a client application. The main benefit of the LightRPC backend is that requires much less configuration effort compared to the FDX backend. You do not need to specify the exchanged variables and signals beforehand. All information will be retrieved automatically during the runtime.

Using the LightRPC backend is recommended for most users. It is easy to set up and provides high data throughput and low latencies.

FDX Backend

CANoe FDX (Fast data eXchange) is an open network protocol for fast data exchange with CANoe. The main benefit of the FDX backend is that it provides even higher data throughput than the LightRPC backend. It produces less communication overhead and can thus read/write more values per second. Also the FDX backend supports advanced data types (e.g. maps) that are not available when using the LightRPC backend.

Using the FDX backend is recommended when you want to transfer huge amounts of data between CANoe and your XIL API client.
