Model Access Port

 

General

Starting with version 8.2, CANoe supports the MA (Model Access) port of the ASAM XIL API.

This port enables you to read and write model variables using a standardized interface. In addition, the option exists to stimulate variables with Signal Generators or to record them.

Function Overview

Starting with CANoe version 8.5, the complete scope of functions (reading, writing, Signal Generators, and capturing) of the XIL API MA port is supported.

Configuration in CANoe

To use the CANoe MA port, activate XIL API support in the Options dialog. The corresponding setting can be found under Extensions|XIL API & FDX Protocol. In most cases you can keep the preassigned UDP Port 3030.

By default the LightRPC XIL API backend is used. If you wish to use the FDX backend instead, you need to enable FDX in this dialog, too. You also need to specify a corresponding FDX description file.

Details for creating an FDX description can be found in Section FDX Protocol.

Structure of the MA Port Configuration File

To configure the MA port on the tester side, you need a special configuration file in XML format. This file contains the following information:

  • A mandatory connection element.
    This holds the IP address of the computer on which CANoe (or more precisely the RT portion of CANoe) runs, as well as the XIL API port configured in CANoe (usually 3030).
  • An optional application control.
    This element can be used to automatically start CANoe and load a configuration file when the MA port is initialized.

The following example shows what this type of configuration file might look like:

Example

<MAPort>
  <Connection ip="127.0.0.1" xilPort="3030"/>

  <ApplicationControl controlMode="COM" startCANoe="true" loadConfig="../MyConfig.cfg" /> </MAPort>

Additional Configuration for the FDX Backend

If you wish to use the XIL API FDX backend instead of the LightRPC backend, you need to include a FDX configuration block. This block contains the following information:

  • An FDX element.
    This element stores the FDX port configured in CANoe (usually 2809).
  • A list of FDX description files.
    These files contain information about the variables and signals available in the CANoe configuration.
  • A list of FDX ReadGrops and WriteGroups that are to be available to the MA port.

The following example shows what this type of configuration file might look like:

Example

<MAPort>
  <Connection ip="127.0.0.1" xilPort="3030"/>
  <FDX fdxPort="2809">
    <DescriptionFiles>
      <DescriptionFile name="Fdx.xml"/>
    </DescriptionFiles>
    <ReadGroups>
      <ReadGroup id="1" readMode="cyclic" cycleTime="100"/>
    </ReadGroups>
    <WriteGroups>
      <WriteGroup id="2"/>
    </WriteGroups>
  </FDX>
  <ApplicationControl controlMode="COM" startCANoe="true" loadConfig="../MyConfig.cfg" /> </MAPort>

The following points must be taken into consideration for the configuration:

  • You can specify the names of the FDX descriptions both as absolute paths and as paths relative to the Vector.IMF file.
  • The indices of the ReadGroups and WriteGroups must correspond to the IDs in the FDX descriptions.

Configuring ReadGroups:

  • If the readMode is set to cyclic, the group will be transmitted cyclically from CANoe to the XIL API application.
  • You can provide a cycle time (in ms). If no cycle time is specified, a default value of 1 ms is used.
  • Alternatively the readMode can be set to onDemand. In this case the group will only be transmitted if a read operation is invoked on the MA Port object.
  • If you want to use XIL API Capturing, it is required to use cyclic transmission.

Using the ApplicationControl element:

  • This element can be used to automatically start CANoe and load a configuration file when the port is initialized.
  • To start a locally installed CANoe via COM, set the attribute controlMode to COM, the attribute startCANoe to true and the attribute loadConfig to the path of the CANoe configuration to load.

E.g. <ApplicationControl controlMode="COM" startCANoe="true" loadConfig="../MyConfig.cfg" />

  • To control CANoe running remotely in standalone mode set the attribute controlMode to standalone, the attribute startCANoe to true, the attribute port to the port of the CANoe runtime kernel and the attribute loadConfig to the local path of the CANoe configuration to load.

E.g. <ApplicationControl controlMode="standalone" startCANoe="true" loadConfig="MyRemoteConfig.rtcfg" />

Access to Variables

The XIL API allows access to all essential variable types within CANoe. These include: System variables, environment variables, XCP variables, bus signals and service signals.

When using the LightRPC XIL API backend no further configuration is required to get access to these values. For the FDX backend additional configuration is necessary.

Variable Access Configuration for the FDX Backend

To make a value available, you need to include it in the FDX description. If the variable is only to be read via the XIL API, it is sufficient to assign it to a ReadGroup. This applies analogously to the writing of variables and the WriteGroups. If a variable is to be read and written, it must be assigned to both a ReadGroup and a WriteGroup.

The variable can then be addressed within the XIL API application using the variable name (including namespace).

Example

Given the following FDX description:

<canoefdxdescription version="1.0">
  <datagroup groupID="1" size="4">
    <item type="int32" size="4" offset="0">
      <sysvar namespace="TestVariables::Inputs" name="MyVariable"/>
    </item>
  </datagroup>
</canoefdxdescription>

Here the variable MyVariable can be read out as follows using the XIL API:

var value = mMaPort.Read("TestVariables::Inputs:: MyVariable") as IIntValue;

The same name conventions also apply to the use of variables in Signal Generators and for capturing.

Interaction with other XIL API Ports

You can operate the MA port together with other XIL API ports supported by CANoe without any problem.
