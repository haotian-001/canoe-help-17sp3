CAPL code execution modes for improved realtime characteristics

 

General

To improve the realtime characteristics of a CANoe remaining bus simulation, you can use special execution modes for CAPL code on selected network interfaces.
If a CAPL node is executed in one of these modes, a restricted set of CAPL functions is available for this node.

Configuration

The CAPL node's configuration dialog contains an option to specify where the node is to be executed.
The consistency check to make sure that the allocated buses are all connected to a single, appropriate network interface is not carried out until the measurement starts (realtime mode).

CAPL-on-Board

Overview

The network nodes of a CANoe remaining bus simulation can be executed directly on a standard Vector network interface. This has the advantage of lower latency times, especially with USB devices. It thus provides a simple way to execute code under realtime conditions.

Differences in Comparison to Distributed Mode / VN8900 Standard Operation

  • Computer connectivity and power supply via USB
  • More affordable than an RT box or rack
  • The runtime environment offers fewer functions than distributed mode

CAN FD

Effective with CANoe 8.2 SP3 CAN FD is also supported on the devices VN1630/VN1630A, VN1640/VN1640A. For these devices a new driver is required.

FlexRay

All FlexRay frames to be sent by the CAPL program are submitted on the computer at the start of a measurement. Retrospective submission via CAPL code is not supported on the device (FrSetSendFrame cannot be used in the CAPL on preStart handler).

  • Trigger Input
    Pulses on the trigger input of the VN interface can also be received with CAPL-on-Board. See "FlexRay Configuration: VN Interface Trigger I/O".
  • I/O
    The I/O lines of the network interfaces VN1630/VN1630A or VN1640/VN1640A can addressed directly by CAPL-on-Board.
  • Automatic Calculation of CRC Signals and Message Counters
    CRC signals and message counters can be calculated automatically with the aid of a special CAPL-on-Board DLL. With each data update, the CRC signals and message counters that are defined in the database are set appropriately. The DLL is included in the relevant OEM-specific FlexRay add-on package.

Important Notes

  • The reception latency of CAPL-on-Board is essentially much lower than with CAPL programs run on a computer.
    However the execution time for CAPL code on an interface is considerably greater.
  • Each code that is executed on the interface increases the latency for receiving events.
    This means that executing very large CAPL event handlers on the interface has a negative impact in terms of reaction delay.
