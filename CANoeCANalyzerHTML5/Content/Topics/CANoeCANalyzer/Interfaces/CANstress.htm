CANstress Access

 

As of CANoe 5.0 it is possible to control CANstress via CAPL functions.

Notice!

For using CAPL functions for CANstress knowledge with handling CANstress is required. Please read the CANstress help or documentation.

CANstressCAPL functions can only be executed in a test module. Within a normal network node the CAPL CANstress functionality can't be used.

Mode of Operation

Since CANoe Version 7.5 SP2, the communication between CANoe and CANstress devices takes place exclusive via DLLs that are part of CANoe. There is no longer access to any installed CANstress software.

Conditions

In order to be able to use this functionality, the following conditions must be fulfilled:

  • To use the CANstress_NL.DLL within a test module, this DLL must be specified as a module for the configuration of a test module (tab: Module)
  • In certain cases, a configuration of the CANstress integration is required. This is described in more detail below.

Configuration

In certain cases, e.g., when multiple CANstress devices are used in parallel, access to CANstress devices must be configured. The configuration can be made via a page in the global options of CANoe. Open the CANstress page in the Options dialog (Extensions|CANstress). The display of the page may be somewhat delayed while CANstress devices are searched for.

CAPL functions are synchronous. This means that a function will only be returned once the command has been processed in full. Unlike in previous versions, the occurrence of an error will no longer lead to the measurement being aborted. Instead, the test will continue messages will appear in the Write Window and in the test report informing the user of the situation. Return codes can be used for an optional application in the control flow.

Like the operating principle in CANstress (i.e., direct control of the CANstress hardware via the CANstresssoftware), when changes are made to the configuration, a download needs to be performed before these changes will be applied. Configuration downloads are executed using the CANstressStart call. If the CANstress device is already active, the download will take place immediately. However, the functions used to change the configuration of CANstress devices cannot activate or deactivate the device.

After completion of a test module, all interferences with CANstress devices that were used from this module are stopped, and the triggers of the devices are switched off, i.e., prevents additional disturbances from being made from these devices. This will prevent them having any further impact on the test. At the end of the measurement, the same occurs if test modules are still running. This will prevent them having any further impact on the CAN bus. Changes to CANstress configurations are not saved.

Use in Test Groups and Test Cases

When using these functions, please note that activated CANstress devices will remain active until they are explicitly deactivated or until the measurement is complete. To prevent test groups or test cases inadvertently impacting one another, a CANstress device should always be deactivated on the level on which it was activated by means of a CANstressStop or CANstressQuit call.

Changes to Troubleshooting Techniques

In older versions CANstress NL DLL, CANstress device access errors would always lead to the measurement in progress being aborted immediately. However, since Version 2.0, an error message is simply output and the measurement will continue. As well as appearing in the Write Window, the message is recorded in the test report (if this has been activated). The result of the test currently in progress will automatically be set to Failed in the event of an error.

Note on Accessing CANstress Devices in Test Units from vTESTstudio:

You can access CANstress devices also from test units. Just keep in mind that the vTESTstudio system environment has to be configured correctly. When configuring the database there has to be at least one CAN network with the same name as a CAN network in the CANoe configuration. The easiest way to ensure this is to import the system environment from the CANoe configuration. However, the CAN network can also be created manually vTESTstudio. If the system environment isn't configured correctly when creating a test unit, you'll get the following error message when executing CANstress commands:

CANstress_NL: Function-Call 'CANstressStart' failed! Test Control API not found! Probably you are trying to call

CANstress CAPL functions in an invalid context. CANstress CAPL functions can only be called in the context of test code in a test module/test unit.
Hint for test units: Make sure a CAN network exists in your CANoe configuration and import the CANoe environment to your test project.
