Automation Interface

 

Via the automation interface, the web services provided by the Vector Tool Platform and the installed applications can be used. The automation interface is provided as a REST API and can therefore be integrated universally.

Note

Prerequisite is the communication connection of the VTP device via Ethernet.

Activation

To activate the automation interface, proceed as follows:

  1. Either open the Vector Platform Manager or open the Vector Tool Platform in CANoe via the ribbon tab.

  2. On the VTP device, start the Web API Server via Tool Platform|Web API Server|Status.

You can get an overview of the functions and the help for the automation interface either by directly entering the IP address of the VTP device into the browsers address bar or selecting one of the entries provided in Tool Platform|Web API Server|OpenAPI.

Select Automation Target and Perform Test

There is an automation interface for the VTP device itself and, depending on the plugin installed, one for it as well. This is the case with CANoe. You can choose between the different automation targets via drop-down list in the upper right corner.

The documentation shows the data structure needed to execute a command and the corresponding return value. For tests, simple actions can be tried out directly in the documentation interface.

Note

CANoe installations smaller than version 15 on a VTP device do not provide an automation interface for the CANoe/CANalyzer plugin. Therefore, the CANOE option is not displayed in the automation interface documentation in the web browser. A subsequent installation of the automation interface into older CANoe/CANalyzer versions is not possible.

OpenAPI Documentation

The REST API of the automation interface has an online documentation according to the OpenAPI standard (Swagger UI). You can access the index page of the online documentation by opening the IP address of the VTP device in a web browser.

There are already a lot of open source tools that can be used to automatically process an OpenAPI description. For example, you can have the client code generated directly from the API description (swagger.json) in many common programming languages (see for example: https://editor.swagger.io/).

It is recommended to use the respective API description file (swagger.json) directly via the link in the online documentation, because this way the API description exactly matches the version installed on the hardware used. However, in case you do not have access to the hardware to be used, you can find a version of the JSON description files for the Vector Tool Platform (VTP.json) and for CANoe (CANoe.json) in the DOC folder of the Vector Platform Manager installation directory.

Note

The OpenAPI documentation in the web browser displays various cURL code snippets after the execution of a command (through [Try it out] and [Execute]). These code snippets are intended for copy and paste usage by the end user. These commands are generated automatically through the Swagger UI component and do not equal the mechanism that the browser has used to execute the REST command.

The implementation of the external component Swashbuckle.AspNetCore does require PUT or POST requests to have a content length header. If a PUT or POST request does not provide such a header, it will be rejected by the webserver. The auto generated cURL code snippets of PUT and POST requests without content in the requests body do not contain a content length header. In this case, the content length header has to be added manually to the corresponding code snippet:

-H "Content-Length: 0"

Use of the REST-API

Example

  1. Select your VTP Device in the Platform Manager and update the Vector Tool Platform.

  2. Configure the VTP Device with an IP address under Tool Platform|Network settings (in the example: 192.168.100.100).

  3. Activate the Web API server under Tool Platform|Web API Server|Status.

  4. Enter the IP address of the VTP Device in the address bar of your web browser.

  5. The Swagger UI of the REST API is displayed in the web browser.

  6. In the header, you will find a selection of API definitions available on the VTP Device. Each application can provide a definition of its REST API when it is installed on the VTP Device. If no additional application is installed, then only VTP is available. Select the VTP definition.

  7. Expand the DeviceInfos group and click on Get /api/vtp/v1/DeviceInfos/DiskInfo.

  8. A description of the parameters of the request and the possible responses will be displayed.

  9. Click on the [Try it out] button.

  10. Since this request has no parameters, you can click directly on the [Execute] button.

  11. Now the request is sent to the Web API server and the response received from the server is displayed.

  12. If successful, code 200 and a respose body in JSON format is returned. The JSON object contains a list of the partitions available on the VTP Device and their current occupancy:
    200 Response body
    {
      "partitions": [
         {
            "partitionName": "System",
            "used": 7705444352,
            "free": 3031969792
          },
          {
            "partitionName": "Data",
            "used": 1424318464,
            "free": 2849673216
          }
        ]
    }

  13. To query the occupancy of the system partition with a Powershell call, open a command prompt and enter the following command with the corresponding IP address of your VTP Device:
    powershell -C "(Invoke-Webrequest -Uri 192.168.100.100/api/vtp/v1/DeviceInfos/DiskInfo -Method Get -UseBasicParsing | ConvertFrom-Json).partitions.Where({$_.partitionName -eq 'System'})"

  14. Output in case of success:
    partitionName used free
    ------------- ---- ----
    System 7705444352 3031969792
