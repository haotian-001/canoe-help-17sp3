Customizing for Multitasking Environment

 

In this page all steps are described which are necessary for running the Simulink model in multitasking mode. In this mode, each sample time of the model is a separate task. The generated DLL emulates a simple processor where these tasks can preempt each other.

This simple emulated processor has only one core, therefore tasks can not be executed in parallel. A performance gain from parallelization is not to be expected from using this multitasking mode.

System Requirements

  • MATLAB R2014a or newer is required.
  • One of the compilers listed in Compiler Configuration, besides Linux which is not yet supported.
  • DLLs built with Multitasking support require a CANoe version starting with 8.5.

Simulink Settings

To activate the multitasking mode in a Simulink model, open the Configuration Parameters dialog of your Simulink Model, go to the Solver page, and set the following settings:

  • Periodic sample time constraint: Specified
  • Sample time properties: Set your sample times here
  • MATLAB R2014a – R2016a: Tasking mode for periodic sample times: MultiTasking
  • MATLAB R2016b or newer: Treat each discrete rate as separate task: on

For details about how to set the sample time properties, see the Simulink help. (You can open it with a right click on the item in the dialog, then click What’s This?).

Behavior of the Multitasking Mode

A DLL built with the activated multitasking mode contains a simple emulation of a realtime operating system. This operating system exchanges data with CANoe at a rate of 100 µs. Thus, the accuracy of the determination of any timing behavior is limited by this value. Furthermore, as the OS is only emulated, timings can have a significant jitter due to the underlying Windows OS, which is no realtime OS.

The Scheduler

The DLL uses a simple preemptive scheduler. Note that this scheduler runs on the emulated OS, i.e., an interrupt means an interrupt in the emulated OS, not an interrupt in the Windows OS.

  • The scheduler runs a cyclic timer at the base sample rate of the model. Each time the timer fires, it is determined which tasks are due, and these tasks are enqueued. The timer routine runs at interrupt level, i.e. it runs with minimal jitter and can preempt any other task.
  • The scheduler always activates the task with the highest priority. If a task with higher priority than the currently active task is enqueued, the currently running task is preempted and the newly enqueued task is executed.
  • In the timer routine, if a task is due and the previous computation of that task is not yet finished, an overrun occurs. The behavior of the scheduler in this case can be configured in the CANoelib Parameters block. For possible settings, see the next section.

Task Overrun Strategies

The following overrun strategies can be configured in the CANoelib Parameters block.

  • Stop on overrun
    Whenever an overrun is detected, the overrun counter of this task is incremented and the simulation is aborted.
  • Enqueue tasks on overrun
    Whenever an overrun is detected, the overrun counter of this task is incremented and the task is enqueued. As soon as the previous instance of this task is completed, the enqueued task will be started. The maximum number of enqueued instances of each task can be set by the setting Max queued task instances. If more instances of one task are enqueued, the simulation is aborted.
  • Ignore overruns (default)
    Whenever an overrun is detected, the overrun counter of this task is incremented, but the task is not enqueued.

System Variables Created by the DLL

A DLL built with the activated multitasking mode creates two new system variables ActiveTask and TaskInfo. Both system variables are in the namespace <modelName>::Multitasking.

  • ActiveTask
    This system variable shows which task is currently active. Task 0 is the task running at the base sample time, Task 1 is the next slower task, and so on. As a higher sample time also means a higher priority, this means the lower the task number, the higher its priority. If no task is active at all, the system variable is set to -1. A value of -2 means that the emulation is currently in an interrupt. This can be the case, for example, if the model was triggered using a CAPL function (see also CAPL Call) block.
    The system variable is polled by CANoe every 100 µs. Therefore, if a task is active for less than this time, the system variable change might not be seen in CANoe, i.e., the task seems to be inactive according to the system variable.
  • TaskInfo
    This system variable is an array that provides information about each task. Each array element contains a structure with the following information.
  • TaskState
    The current state of the task.
  • 0 Idle: The task is inactive, waiting to be triggered.
  • 1 Ready: The task has been triggered but could not yet start due to a higher prior task that is currently executing.
  • 2 Running: The task has started its execution. The task will remain in this state until the execution is finished, regardless of whether it is suspended by a higher prior task or not.
  • CallCount
    Counts how often the task was executed. This value is incremented whenever the execution of the task is finished, i.e., the task state changes from Running to Idle.
  • OverrunCount
    Counts for each task the number of overruns. Whenever a task is due and its previous execution is not finished yet, the overrun counter for this task is incremented by 1.
  • OverrunQueueCount
    Shows the size of the overrun queue if the overrun strategy queue tasks is selected. The overrun queue count is incremented whenever an overrun is detected. When the previous instance of the task finishes its execution, the task is immediately scheduled again and the overrun queue count is decremented.
  • TurnaroundTimeLast
    The turnaround time in microseconds of the last completed execution of the task. The turnaround time starts when the task is triggered (i.e., on the state change from Idle to Ready), and ends when the task execution is finished (i.e., on the state change from Running to Idle).
  • TurnaroundTimeMin
    The minimum turnaround time measured for this task.
  • TurnaroundTimeMax
    The maximum turnaround time measured for this task. Note that due to the fact that CANoe is not running on a realtime OS, this value can be unexpectedly high. If the CANoe task is suspended by the Windows OS due to a high-prior OS task, the computation time of that OS task will be added to the turnaround time.
  • TurnaroundTimeAvg
    The average turnaround time measured for this task.
  • CycleTime
    The time between the last two triggers of the task.

Signal Output

When using Signal Output or Frame Output blocks, one should be aware that the threading model only contains the computation of the signals, not the sending of the frames over the bus. This is still be done by the CANoe interaction layer, which runs in the CANoe simulation thread. That is, if a signal is configured to be send cyclic, and the computation of the signal is not yet completed when the send timer is triggered, sending of the signal will not be delayed. Instead, the interaction layer sends the old value of the signal again.

Note: Further Differences to the Singletasking Mode

In singletasking mode, the DLL is always unloaded at measurement end. In multitasking mode, this is only true for DLLs which are built with the Debug build option set. This means that if you try to rebuild your DLL in release build while a configuration containing that DLL is opened in CANoe, your build will fail because the DLL is still locked by CANoe. You can unload the DLL by either closing CANoe or opening another configuration in CANoe. For a rapid development, it is recommended to activate the Debug build check box while developing the model. Once the model is completed, deactivate this check box and rebuild your model once more in order to get the best performance while executing the DLL.

Limitations of the Multitasking Mode

The Export CAPL Functions option is not supported in multitasking mode and must be deactivated in the code generation options.
