GPIB

 

GPIB is a protocol for networking external instruments. These instruments include measuring devices, power supplies and function generators. This standard was originally developed by Hewlett Packard under the name HP-IB. Later HP-IB was renamed GPIB (General Purpose Interface Bus) by the IEEE Committee as. Other equivalent names are IEEE 488 and IEC 625.

IEEE

Since the specification does not have anything to say about device functionality, almost any instrument can be controlled with IEEE 488.

The specification only defines an interface; it describes how signals are read from the bus and written to the bus.

The entire bus is controlled by a Controller. This controller is generally the GPIB card installed on the host computer.

Listener, Talker and Controller

A GPIB device is defined by three functional device types (Listener, Talker, Controller).

Some controllers contain more than one type. A spectrum analyzer, for example, expects commands to sample the frequency band, and it then reports the peak value to the Controller. It is therefore both a Listener and Talker.

  • The specification allows a maximum of 15 devices per bus.

  • A minimal system consists of one Controller and one Talker or Listener.

Supported Devices

The GPIB interface is based on the API defined by the NI-488.2 for Windows (Version 2.1) software, so that the corresponding adapters made by National Instruments are supported.

Examples of supported and tested devices:

  • National Instruments GPIB-USB-B with software NI-488.2 for Windows, Version 2.1

Message Types

  • Device-specific messages ("Data")

    represent device-specific information, programming instructions, measurement results, device states and files.

  • Interface-Based messages ("Commands")

    represent commands for bus management, e.g: bus initialization, device addressing, setting of device mode, etc.

Usage

GPIB functionality is integrated by means of CAPL functions.

Some GPIB functions are synchronous (blocking). These functions were relocated to a separate thread to avoid interrupting a running measurement. This thread permits asynchronous processing of these calls. In concrete terms this means that there is no CAPL function that can be used to read results directly from a GPIB device. The response of a GPIB device is prepared in the callback function GPIBResponse. This function is called automatically as soon as the response from the GPIB device is available. The user implements this callback in CAPL.
