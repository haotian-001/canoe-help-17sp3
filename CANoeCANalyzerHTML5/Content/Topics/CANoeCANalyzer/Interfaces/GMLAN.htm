GMLAN Extension

 

GMLAN Extension in CANoe

ID Format

GMLAN 3.0 uses a special format for message IDs:

  • Solely extended IDs are used.
  • The ID consists of the fields Priority, Parameter ID (PID) and Source Address.

If a bus has a database with GMLAN messages assigned to, no extended IDs of another type can be used on this bus. Messages with standard IDs are still allowed.

GMLAN Database

If the UseGMParameterIDs bus attribute is defined, all messages of this database are interpreted as GMLAN messages.

Trace Window

  • The columns Priority and Source Address can be displayed. Therefore you have to select the GMLAN bus system in the Column Selection part of the Trace Window configuration. The GMLAN bus system is only available if a GMLAN database exists.
  • The parameter ID (PID) is displayed in the ID column.
  • In fixed (display) mode all messages with the same PID and source address are displayed in a single line.

ASCII Log

In every line that contains a GMLAN message Priority, PID and Source Address are added at the end of this line.

Message Filter

The table on the right side of the dialog shows Priority, PID and Source Address; Priority and Source Address can be changed there. You can enter either a value or  the wildcard '*'.

Visual Sequencer

If a GMLAN database is used, the following extensions for Visual Sequencer Commands are available:

  • Send GMLAN Frame command is available
  • Wait For GMLAN Frame command is available
  • Set command can set values from GMLAN signals

Symbol Selection

Messages are solely identified by their Parameter ID. Priority and Source Address are set to 0. If you select the message via the node and is the attribute SourceId defined for this node, this source address is being used.

Signal Display in the Graphics Window, in the Data Window and in Panels

If you select the signal in the symbolic Signal Selection dialog via a network node and is the attribute "SourceId" defined for this node, this signal is only displayed for messages with this specific Source Address. Otherwise this signal will be displayed independent of the source address.

In the Symbol Selection dialog the source address is added to the message name.

CAPL

If you use a GMLAN database the following features are available:

Message Handler "on gmLanMessage"

The message handler can be set up…

  • …for a single message
    (thereby source address and priority are ignored)
  • …for all messages
    (on gmLanMessage *)
  • …for a message range

GMLAN Message

You can define a GMLAN message as follows:

gmLanMessage Airbag_Impact_Data msg ={SA =0x44, PRIO= 0x7};

Source address and priority are set to "0" (default) as long as nothing else is specified by the selectors SA and PRIO.

Note

In the same way as with simple CAN messages, the CAN identifier is not copied when one GMLAN message is assigned to another GMLAN message. This means that the PID, priority and source address are not set and will need to be copied separately if required.

on gmlanmessage Battery_Voltage 
{
  gmlanmessage Battery_Voltage msg;
  msg = this;   // SA and PRIO are not copied
  gmLanSetSourceId(msg, gmLanGetSourceId(this));   // Copy source address
  gmLanSetPrio(msg, gmLanGetPrio(this)) ;   // Copy priority
  ...
}

If the complete CAN identifier (with source address and priority) needs to be copied, you will have to create the destination message as a wildcard message.

on gmlanmessage Battery_Voltage 
{
  gmlanmessage * msg;
  msg = this;   // PID, SA and PRIO are copied
  ...
}

Message Selectors

Message selectors are used to create a message buffer.

When defining a GMLAN message you can specify

  • source address with selector SA
  • priority with selector PRIO
  • parameter ID with selector gm_pid
    (numerical message specification; with symbolic message specification the parameter ID is included)

Functions to modify message buffers

gmLanSetPID(gmLanMessage msg, long v); );
//changes the parameter ID of the "msg" message to "v"

gmLanSetSourceId(gmLanMessage msg, long v);
//changes the source address of the "msg" message to "v"

gmLanSetPrio(gmLanMessage msg, long v); );
//changes the priority of the "msg" message to "v"

long v = gmLanGetPID(gmLanMessage msg);
//Access to the parameter ID of the "msg" message

long v = gmLanGetSourceId(gmLanMessage msg);
//Access to the source address of the "msg" message

long v = gmLanGetPrio(gmLanMessage msg);
//Access to the priority of the "msg" message

Sending a message

You can send a GMLAN message with the output(msg) method.

Example

A GMLAN message is sent with output(msg).

gmLanMessage Airbag_Impact_Data msg1 ={SA =0x44, PRIO= 0x7};
output(msg1);
write("pid: 0x%x, source: 0x%x, prio: x%x",gmLanGetPID(msg1),gmLanGetSourceId(msg1),gmLanGetPrio(msg1));

Example

A GMLAN message is modified and sent with output(msg).
The parameter ID is set with selector gm_pid.

gmLanMessage * msg1 ={SA =0x44, PRIO= 0x7, gm_pid=0x11};
gmLanSetSourceId(msg1,0x55);
gmLanSetPID(msg1,0x11);
gmLanSetPrio(msg1,0x5); 

output(msg1);

write("pid: 0x%x, source: 0x%x, prio: x%x",gmLanGetPID(msg1),gmLanGetSourceId(msg1),gmLanGetPrio(msg1));

Creating a GMLAN message ID

The gmLanId function can be used to create a GMLAN message ID. To specify the source address is concerned, the message ID is either entered directly or defined using transmission nodes. The GMLAN message ID can be used, for example, to wait for a GMLAN message with the TestWaitForMessage function.

GMLAN Extension for CANoe Test Feature Set

CANoe supports testing of GMLAN messages and GMLAN signals. Below is a summary of points to take note of.

GMLAN Support in CAPL Test Modules

There are two ways of using GMLAN messages in CAPL test modules.

Some functions use the gmLanMessage CAPL type.

Others work on an ID basis. This is particularly the case with functions from the Test Service Library (TSL) as well as the Wait and Join functions for GMLAN messages.

Example

Use of a gmLanMessage

gmLanMessage Comfort::Console_1 gmMsg;

TestCaseComment("GMLAN Message: ", gmMsg);

Use of a gmLanMessage 

gmLanMessage Comfort::Console_1 gmMsg;

TestCaseComment("GMLAN Message: ", gmMsg);

The gmLanId function creates a GMLAN Extended ID based on the values specified in the database for PID, Prio and Source Id. This can be done using the send node or by specifying the Source ID directly.

There are two ways of using GMLAN signals in CAPL test modules.

Some functions work with a separate parameter that specifies the send node of the signal. These functions can be recognized based on the ..:ByTxNode postfix.

Other functions offer the option of specifying the send node directly by means of the signal parameter. This applies in particular to the signal based checks from the TSL.

Example

Validation of a signal value

checkSignalInRangeByTxNode(Message::Signal, Node, 2, 4);

Setup of signal value monitoring 

ChkStart_MsgSignalValueRangeViolation(Node::Signal, 2, 4);

GMLAN Support in XML Test Modules

All constraints, conditions and test patterns that can be used in a CAN environment will also work for GMLAN. Please note that it is necessary to specify the send node for all GMLAN messages and GMLAN signals. This means that in <canmsg> and <cansignal> elements  the node attribute has to be included. Otherwise it is not possible to set the Source ID correctly in the sent message. The ID used is the PID of the GMLAN message. The message type has to be set to ext.

Example

cycle time monitoring of a GMLAN message

<cycletime_abs min="200ms" max="300ms">
  <canmsg id="Console_1" node="Console"/>
</cycletime_abs>

ID based definition of a GMLAN message

<canmsg id="0x1a0" node="Console" type="ext"/>

GMLAN Signal

<cansignal name="Mirror_l2l" node="Console">
