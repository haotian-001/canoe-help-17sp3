Extended Real Time (ERT) as of CANoe 17

 

With CANoe 17, the Extended Real Time System has been fundamentally revised. At the start of the simulation, a check is made for correct settings. If there is a need for action for adjustments, notifications are output in the Write Window.

Note

The realtime module VT6051A is excluded from the ERT change and remains at the status of CANoe 16.

If necessary, adjustments must be made in the following three areas:

1. MATLAB Modules

Starting with CANoe 17, MATLAB modules for Extended Real Time must be compiled with a new target:

  • At least Matlab Integration Package version 8.2, which is included with CANoe 17, is required for compilation. This Matlab Integration Package contains the required compiler, but it must be explicitly selected during installation. Updating from an older Matlab Integration Package is therefore unfortunately not possible. See Compiler Configuration for more information.

  • The target cn_linux.tlc must be used, see HIL Mode - Simulink Configuration for details.

  • In the CN code generation options the check box Build Extended Real-Time DLL must be selected as usual.

Note

When changing the target, note that the options set in the CN code generation options are not automatically transferred to the new target.

2. Consistent Use of vModules

Up to CANoe 16 it was possible to specify nodelayer DLLs directly for ERT network nodes. Optionally, vModules could be used to reference modules. As of CANoe 17, it is mandatory that modules be referenced only via vModules.

Figure 1: Use of vModules

3. Use of CAPL Scripts

Many CAPL scripts were created using the Windows character set table. The first line of a CAPL script then contains, for example, the comment: /*@!Encoding:1252*/. The character encoding of CAPL scripts must now be changed to UTF-8.

To do this, a change must be made in the CAPL browser to change the code page setting.

After that, the CAPL scripts must be recreated and saved. The CAPL scripts in UTF-8 format should then contain the following comment in the first line: /*@!Encoding:65001*/.

Figure 2: Character encoding in the CAPL browser

Note

If errors occur during compilation of CAPL scripts when accessing data that do not match the data types, this may be due to the conversion of internal data types. This case can be solved by using a different method.

For example, it may be necessary to replace the sysGetVariableLongLong function with sysGetVariableInt.
