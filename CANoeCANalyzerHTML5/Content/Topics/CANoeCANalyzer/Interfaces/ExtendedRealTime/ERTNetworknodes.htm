ERT Network Nodes

 

ERT network nodes are executed in the Extended Real Time area of a supported Vector Hardware.

Configuration

Before adding ERT network nodes to a CANoe configuration you must enable the option Activate ERT network nodes in the CANoe Options dialog|Performance|Extended Real Time (ERT). After that you can add ERT network nodes to the Simulation Setup e.g. via the context menu of the bus.

Configurations containing an ERT network node can only be executed under the following conditions:

  • Execution on Vector Hardware that supports ERT
  • The distributed modemust be activated
  • Real Bus Mode must be activated

Supported Features

ERT network nodes support following features:

  • CAPL programs: If a CAPL program is attached to an ERT network node it is automatically compiled for and executed in ERT. For a list of supported CAPL features in ERT see CAPL in ERT.
  • MATLAB/Simulink models: MATLAB DLLs compiled for ERT can be added in the node configuration dialog on the Components page.
  • CANoe Interaction Layer: If a Network node from the database is configured, the Interaction Layer is automatically taken from the database. In the node configuration dialog on the Components page DLLs with CANoe Interaction Layer can also be added manually.

Execution Concept

Every ERT network node runs as a realtime thread in the Extended Real Time part. The priorities of the threads can be controlled in the node configuration dialog on the ERT page.

Events from ERT supported hardware (e.g. VN8972) are first processed in the Extended Real Time part. System variables used to communicate between ERT network nodes and other ECU nodes require about 1 millisecond for synchronization.

Data Consistency

ERT nodes support two different settings for handling data consistency. This can be controlled by choosing either Write-through or Snapshot in the node configuration dialog on the ERT page. The setting has an influence on how data is processed, e.g. in CAPL handlers or MATLAB steps.

  • Write-through: This mode is optimized for speed. All data written by the task is directly updated. If data (e.g. a system variable) is read multiple times in the same handler it might have changed in between. There is no guarantee that two dates (e.g. two signal values from the same frame) are consistent.
  • Snapshot: This mode is optimized for data consistency. It guarantees that all data used in a handler is in a consistent state. Data written will be updated at the end of the handler and all read data is updated at the beginning of the handler.

Timing Variables

ERT provides system variables for time measurements of CAPL timers. The variables can be activated in the dialog Node Configuration on the ERT page if a CAPL script is selected in the node specification.

  • DeltaTime_<CAPL script name>: The time between two consecutive timer calls. This corresponds to the cycle time if a cyclic timer was set but can also be the time between two different timers in the same CAPL script.

  • ExecutionTime_<CAPL script name>: The runtime of the last timer handler that was executed. Note that if more than one timer is set, this value might not be meaningful.
