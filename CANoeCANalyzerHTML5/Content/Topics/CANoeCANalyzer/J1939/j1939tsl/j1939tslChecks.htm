Checks

 

Note

The following checks are part of the J1939 Test Service Library (J1939TestServiceLibraryNL.dll), which is no longer being developed and should therefore no longer be used. Instead, you can use the checks described on page Testing in a J1939 Network.

Checks observe and validate procedures in the network. A check must be setup in a CAPL or XML test module and runs in parallel to test cases. If a check detects a violation of an observed condition, a message is written to the test report and the test verdict is set to failed.

In CAPL a check can be configured to call a callback function or trigger a wait function, if a violation of an observed condition is detected.

Checks are passive, this means no messages are sent or environment variables are modified by the check.

Use Checks in XML

A check can be embedded in <constraint> XML elements and they are active during the test case or a test group.

Use Checks in CAPL

The set up of a check follows these steps:
  1. Create a check with J1939TestChkCreate_ function
  2. Start the check with J1939TestChkControl_Start
  3. After the test, the check can be stopped with J1939TestChkControl_Stop. Resume the check with the Start-Function.
  4. With J1939TestChkControl_Destroy the check is deleted.

If a violation of the observed condition occurs during an active check, a CAPL callback function is called. The CAPL function must be specified at the creation of the check. The function must have the following parameters: void onCheckViolation( dword aCheckId )

In the CAPL function the violation can be handled, e.g. the statistic functions can be used to get more information about the violation.

Note

The checks of the J1939 Test Service Library must be used with the J1939TestChk functions. The checks cannot be controlled with the functions of the standard CAN Test Service Library.
