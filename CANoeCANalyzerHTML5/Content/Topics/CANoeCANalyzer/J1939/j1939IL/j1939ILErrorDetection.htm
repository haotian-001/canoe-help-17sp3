Error Detection

 

In some cases, it is necessary to take measures to detect interference from transmitting ECUs.

The J1939 DigitalAnnex defines a large number of messages that also contain a checksum and a message counter in the user data.

The J1939 Interaction Layer supports you in automatically setting the message counter and the checksum for these messages.

To use this functionality you have to configure the database.

  • Make sure that the FsJ1939UseCrcAndCounter attribute is defined correctly.

  • Define the calculation formula in the form [A-E][a-f] (e.g. Bc) for each message that the Interaction Layer is to provide with the checksum and message counter. Take the calculation rules from the table below. You can combine (with the exception of overlaps) the calculation of the checksum and the position of the message counter as you wish.

Note

CANoe has already supported the calculation of checksums in earlier versions. However, the function was only available to PGs with PGN 0, 1024, 2816, 61469 and 61483. You can still use this functionality. However, if you add the FsJ1939UseCrcAndCounter message attribute to your database, only the new implementation will be used. The following applies: only for those messages where the attribute has a value in the form [A-E][a-f], checksum and message counter are set.

Note

Up to and including CANoe 15, five messages (0, 1024, 2816, 61469, 61483) have already been handled by the interaction layer. In the meantime, the Digital Annex contains over 200 messages that must be provided with a checksum and a message counter. For this reason, the support of the interaction layer has been changed. The messages with PGN 0, 64899 and 64932 are now provided with the calculation rule Ef. Messages with PG 1024, 2816, 61469 and 61483 are assigned Cd.

Fault Injection

For fault injection you have two possibilities. With the function J1939ILManipulateMessage you can overwrite the checksum and the message counter of the interaction layer. This function is also available in the test environment. If you do not depend on the test environment and only want to manipulate messages in the simulation node, you can use the J1939ILOnTxPrepare function as a second option, which offers you more flexibility than the first variant.

Manipulation of Data with J1939ILManipulateMessage

The fault injection by means of J1939ILManipulateMessage has the advantage that you can use it in the test environment.

// Manipulate the message with PGN=0xF123 (DCDC40S)
// Checksum is located in the byte 7, counter is located in the 4 highest bits of the byte 6
on start
{
  qword filterMask, filterValue, dataMask, dataValue;

  //Apply filter to bytes 0, 1, 2, 3
  filterMask = 0x00000000FFFFFFFFLL;
  //Expected values are 0x01, 0x02, 0x03, 0x04
  filterValue = 0x0000000004030201LL;
  //Manipulate bytes 6 (only 4 highest bits) and 7
  dataMask = 0xFFF0000000000000LL;
  //Set 4 highest bits of the byte 6 to 0x4 (wrong counter) and byte 7 to 0xFA (wrong checksum)
  dataValue = 0xFA40000000000000LL;
  //Manipulate data if the values of bytes 0, 1, 2, 3 meet the criteria
  J1939ILManipulateMessage( 0xF123, 0xFF, filterMask, filterValue, dataMask, dataValue);
}

Manipulation of Data with J1939ILOnTxPrepare

Long J1939ILOnTxPrepare(pg * txPG)
{
  byte msgCounter;
  switch(txPG.pgn)
  {
    // Manipulate PG DCDC40S (PGN=0xF123)
    // Checksum is located in the byte 7, counter is located in the 4 highest bits of the byte 6
    case 0xF123: // PG has validation scheme 'Aa'
      msgCounter = txPG.byte(6) >> 4; // read current message counter value
      msgCounter = msgCounter << 1; // manipulate counter
      txPG.byte(6) &= ~(0xF0); // clear old message counter bits
      txPG.byte(6) |= (msgCounter << 4); // set new message counter bits
      txPG.byte(7) = txPG.byte(7) >> 1; // manipulate checksum value
      break;
   default:
    break;
  }
  return 1;
}
