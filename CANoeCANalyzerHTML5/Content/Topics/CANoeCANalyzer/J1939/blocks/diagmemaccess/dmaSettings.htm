Settings

 

This page is divided into various areas, which will be described below:

Tester

Here the address of the Memory Access Window and the channel on which the messages are sent/received is specified. In CANoe the CAN channel is assigned in the Simulation Setup and it cannot be changed here.

If the Use address claim check box is activated, then on measurement start, the Memory Access Window tries to log onto the network with the specified address (Address Claiming). If this fails, then no messages can be sent. In this case, a new address must be selected and the measurement started anew. Additionally the J1939 device name must be specified. Open the appropriate dialog with the button [Device Name...] and enter the values that should be used for the device name.

Node

In the selection field, the address of the ECU is selected whose memory should be accessed, or whose transmission of broadcast messages should be started or stopped. This address can be changed even during measurement (it does not influence the cyclical reading of objects).

Objects

Objects that should be read or written are identified using a number (ID). These numbers are interpreted depending on the so-called SPACE. J1939 specified previously only the SPACE for SPNs (value 0).

The value entered here is used as default value for new objects on pages Manual direct memory access and Manual object-oriented memory access.

Used Ports (DM13)

Here you can select the ports on which the transmission of broadcast messages is started or stopped.

Authentication

During access to the memory, an ECU can have implemented various security mechanisms. In this area it is possible to enter a password or UserLevel. If the ECU supports the Seed & Key procedure, then after receipt of a numeric code (Seed) and answer code (Key) is sent back, which is specified as a hexadecimal value.

The key can be entered as fix value or calculated dynamically in two ways:

  • At runtime the key can be calculated with the help of two system variable or two environment variables (environment variables with CANoe only). In a CAPL program, the resulting key is calculated and assigned to the system or environment variable for the key.
    If system variables are used the data type must be Integer array, the data type of environment variables must be Data.

Example

on envVar envSeed {
  byte buf[1785];
  long copiedBytes;
  copiedBytes = getValue( this, buf );
  CalcAndSendKey( buf, copiedBytes );
}

on sysvar_update sysvar::SampleNamespace::svSeed
{
  byte copiedBytes [1785];
  dword length, i;
  length = SysGetVariableArrayLength(this);
  for (i = 0; (i < length) && (i<elcount(copiedBytes)); ++i)
  {
    copiedBytes[i] = @this[i];
  }
  CalcAndSendKey(copiedBytes, length );
}

void CalcAndSendKey( byte seed[], dword len ) {
  byte newKey[1785];
  dword i;
  // in this example we only copy the seed and use it as new key
  for (i=0; i<len; ++i) {
    newKey[i] = seed[i];
  }
  // set environment variable
  putValue( envKey, newKey, len );
  // set system variable
  SysSetVariableLongArray(sysvar::SampleNamespace::svKey, newKey, len );
}
  • A Seed & Key DLL can be specified that calculates the key.

If the key could be calculated it is sent back to the ECU.

Boot Load

Here you can specify a system variable or environment variable (with CANoe only) that is set to 1 if a boot load was executed successfully, that is, if data was written to the ECU and the execution in the ECU was continued to the specified address. In this case another external (CAPL) program can communicate with the ECU with the help of the Parameter Group Boot Load Data (DM17) (no particular protocol is provided for this).

The system variable and environment variable must be of data type Integer.
