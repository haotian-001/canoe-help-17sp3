Send and Receive Parameter Groups with CAPL

 

Send with the CAPL Function output

For sending J1939 parameter groups (PG) you may use the CAPL function output. You can use the TM selector to control whether a PG is to be sent with a Multi-PG or via the FD transport protocol or whether CANoe determines the transmission type itself, depending on the length of the PG. Furthermore you can define assurance data with the help of the selectors ADT and ADS (transmitting by means of transport protocol) as well as the selectors TOS and TF (transmitting by means of Multi-PG).

When using the CAPL function output there are a few things to keep in mind, which are explained below.

Transmission Type

According to J1939-22, a parameter group can be transmitted either as C-PG within a Multi-PG or via the FD transport protocol. In addition, the address-claimed message can be sent directly as a CEFF frame or FEFF frame. With the help of the TM selector you can directly determine which transmission type is used:

  • TM=0: Parameter group is sent as CEFF or FEFF (depending on the selector FDF).
  • TM=1: Parameter group is sent as C-PG within Multi-PG.
  • TM=2: Parameter group is sent with the FD transport protocol.
  • TM=3: Transmission type is determined by CANoe based on the length of the PG.

Assurance Data

In some use cases it is necessary to analyze or evaluate the assurance data. This is done via the selectors TOS (Type of Service) in combination with TF (Trailer Format) for C-PGs oderADT (Assurance Data Type) and ADS (Assurance Data Size in Bytes) for transmissions with the transport protocol. All four selectors are of type integer.

Examples

In the following, the sending and receiving of a parameter group with or without assurance data is considered.

Example — Send with Output

If a PG is created in CAPL via a message from the database or as pg *, ads and adt have the value -1, i.e. no assurance data is transmitted. When sending such parameter groups, CANoe can independently decide whether Multi-PG or FD.TP is used.

void SendMyPG()
{
  pg CPGWithDlc50 myPg; // FDF=1, TM=3, ADT=-1, ADS=-1
  … // define payload
  output(myPG); // result: myPG will be sent with Multi-PG
}

If assurance data is now to be added to the parameter group, the PG must already be enlarged when it is created so that there is space for the assurance data at the end of the PG.

Here is an example of how to define a C-PG with 50 bytes and 8 bytes of assurance data so that it is then transmitted with a Multi-PG (the message CPGWithDlc50 has the attribute J1939PgAppearanceOnBus="Default" defined in the DBC):

void SendMyPG()
{
  pg CPGWithDlc50 myPg {dlc = 58}; // FDF=1, TM=3, ADT=-1, ADS=-1
  myPg.tm = 1; // enforce Multi-PG
  myPg.tos = 1;
  myPg.tf = 6; // functional safety, 64 Bit (Multi-PG-specific coding)
  … // define payload
  myPg.qword(50) = assurancDataWith8Bytes; //set Asssurance Data (8 bytes) at the end of payload
  output(myPG); // result: myPG will be sent with Multi-PG
}

If the same C-PG is now to be transmitted with 12 bytes of assurance data, the length of the PG increases to 62 bytes. The PG must now be transferred with FD.TP instead of Multi-PG. The same applies if a short parameter group with more than 8 bytes of assurance data is protected. In both cases the transfer must be forced with FD.TP and the length of the assurance data must be specified explicitly:

void SendMyPG()
{
  pg CPGWithDlc50 myPg {dlc = 62}; // FDF=1, TM=3, ADT=-1, ADS=-1
  myPg.tm = 2; // enforce FD.TP
  myPg.adt = 2; // functional safety (FD.TP-specific codung)
  myPg.ads = 12; // 12 bytes (FD.TP-specific codung)
  … // define payload
  for (i=0, i<12, ++i) // set assurance data at the end of payload
  {
    myPg.byte(i+50) = assurancDataWith12Bytes(i);
  }
  output(myPG);
}

Example — Send with InitializeMultiPG and AddContainedPG

Using the CAPL function output has one major limitation: it is not possible to send multiple C-PGs with only one Multi-PG. To overcome this limitation there are the CAPL functions J1939InitializeMultiPG and J1939AddContainedPG.

In the following example, a Multi-PG with three C-PGs (pg1, pg2 and pg3) is sent. The C-PG pg3 is expanded by assurance data. Since for C-PGs the length of the assurance data is implicitly defined by the selector ADT, selector ADS is ignored.

void SendMultiPG()
{
  message * multiPG;
  pg Msg1 pg1 = {dlc = 10}; // Msg1 is defined in the attached DBC, Default values tos = -1, tf = -1
  pg 0x100 pg2 = {dlc = 15}; // Default values tos = -1, tf = -1
  pg * pg3 = {dlc = 24};//16 bytes for payload and 8 bytes for Assurance Data
  pg3.tos = 1;
  pg3.tf = 5; //functional safety, 64bit

  // Here a 29 bit CAN FD message is initialized as multiPG
  // for the source address 0x23, destination address 0x45 and prio 5:
  J1939InitializeMultiPG(multiPG, 0x5, 0x45, 0x23);

  … // Init payload of pg1
  J1939AddContainedPG(multiPG, pg1); // Add pg1 as a C-PG to the multiPG

  … // Init payload of pg2
  J1939AddContainedPG(multiPG, pg2); // Add pg2 as a C-PG to the multiPG

  pg3.PGN = 0x100;
  … // Init payload of pg3
  pg3.qword(16) = 0x1122334455667788ll; //Put Assurance Data at the end of payload
  J1939AddContainedPG(multiPG, pg3); // Add pg3 as a C-PG to the multiPG

  output(multiPG);
}

Example — Receive PGs

When a C-PG is received in a Multi-PG, the selectors TOS and TF are set. The resulting length is also set in ADS. If, on the other hand, a PG is received by means of FD transport protocol, the selectors ADT and ADS are set.
