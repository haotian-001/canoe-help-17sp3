Transport Protocol

 

General

Messages longer than 8 bytes for J1939-21 (Classic CAN) or 60 bytes for J1939-22 (CAN FD) are too large to fit into a single CAN frame. Therefore, they must be split by the sender into individual packets, each of which can then be sent with a CAN frame. The receiver must reassemble the individual fragments in the original order. For this purpose, a set of rules is defined in the J1939 standard: a so-called transport protocol.

The standard describes one transport protocol each for connectionless (broadcast) and connection-oriented (1 to 1) transmission. According to J1939-21, the transport protocols support a maximum transmission length of 1785 bytes (max. sequence counter: 255, 7 bytes per TP.DT message). For J1939-22 (CAN FD), the upper limit is 16 777 215 bytes (connection-oriented) or 15 300 bytes (connectionless). The significantly larger maximum transmission length has become possible because, on the one hand, CAN FD frames are larger than CAN frames and, on the other hand, with the fact that 3 bytes are used for data packet numbering instead of 1 byte.

Connectionless – Broadcast Announce Message (BAM)

The Broadcast Announce Message (BAM) transport protocol is used for global transmission of parameter groups. The sender alone controls the flow control. The message is always sent to all nodes. A receiver cannot intervene in the communication. If a message is missed, the receiver cannot signal this. He may have to wait for a new message.

Since with the BAM protocol the receiver has no possibility to control the flow control, the sender must allow a minimum time interval between the individual packets. This is 10-200 ms. This takes account of any slow network nodes.

Connection-Oriented – Connection Mode Data Transfer (RTS/CTS)

With this protocol, the sender establishes a connection with the receiver. The receiver can control and influence the flow control of the individual data packets. Both pages, receiver and sender, can terminate the connection (e.g. in case of an error).

The Connection Mode Data Transfer protocol is not subject to any time limit. All participants potentially exchange their data with each other at the maximum speed of their capabilities.

Functionality

The following communication diagram shows the schematic sequence of a transmission according to J1939-21. The BAM protocol starts with a TP.CM_BAM message , which contains the parameter group to be transmitted, the length and the number of packets. The packets are then sent, observing the time limits.

While with the BAM protocol only the messages originate from the sender, with RTS/CTS connections the receiver must communicate explicitly to be able to receive packets.

  • The sender starts the connection setup with a TP.CM_RTS message. This contains the parameter group to be transmitted, the length, the number of packets, and the maximum number of packets that the receiver can request per transmission segment.
  • Provided the receiver has sufficient resources available for a connection, it responds with a TP.CM_CTS message. This contains the number of packets the receiver can receive at one time. The value is limited by that in the RTS message.
  • If the parameter group is not yet completely transmitted after this transmission unit, the receiver repeats the process and sends further TP.CM_CTS messages until the transmission is complete.
  • To subsequently confirm correct transmission and close the connection, the receiver acknowledges receipt of the parameter group with a TP.CM_EndOfMsgACK message.

With respect to J1939-22 (CAN FD) there are small differences in the transport protocols. According to the specification, parameter groups up to a length of 60 bytes can be transmitted in a single CAN FD frame. Only if the parameter group exceeds this length, one of the transport protocols must be used. The sequence of both protocols is expanded by a FD.TP.CM_EndOfMsgStatus message. This is transmitted as the last message from the sender and contains control information to verify correct transmission as well as optional Assurance Data (Safety or Security checksums). The receiver therefore only sends the FD.TP.CM_EndOfMsgACK message after the FD.TP.CM_EndOfMsgStatus message.
