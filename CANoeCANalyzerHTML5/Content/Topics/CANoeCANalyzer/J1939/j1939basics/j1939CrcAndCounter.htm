J1939 Functional Safety – Messages With Integrated Checksum and Counter

 

The J1939 Digital Annex specifies checksums and message counters for more than 200 messages (as of March 2021) to detect error states of the ECUs involved. The message counter and the checksum are part of the user data of a PG. Each time a message is transmitted, the message counter is incremented by one. Since the checksum is calculated from the user data, the checksum consequently also changes with each correct transmission, regardless of whether other signals have changed or not. However, if the message counter is not increased during a subsequent transmission, or if the checksum does not match the user data, the receiver can assume an error condition of the sender.

The Digital Annex describes five different calculation rules for the checksums. Two of them are based on a typical CRC (Cyclic Redundancy Counter) calculation with eight-bit length. The other three calculations use simple addition, bit shift and bitmask. In CANoe, the calculation rules are identified by a capital letter (A-E) and stored in the message database. Six possible positions within the message are specified for the message counter. We identify them with lowercase letters a-f. Support in CANalyzer

CANoe supports you both in detecting erroneous checksums and message counters in the Trace Window and in automatically calculating these values when you send messages with the J1939 Modeling Library (J1939_IL). So that CANoe knows according to which procedure the values are to be set, you must first add the message attribute FsJ1939UseCrcAndCounter to the database (if it does not already exist). Then define the calculation formula for checksum and message counter by setting the attribute for the desired messages (format [A-E][a-f], e.g. Bc).

Special CAPL functions are also available. This allows you to calculate the checksum for any message according to one of the five available calculation rules. You can set the checksum and message counter for a message to send the message. Or you can extract the checksum and message counter from a received message, e.g. to check it for correctness.

CANoe has already supported the calculation of checksums in earlier versions. However, the function was only available to PGs with PGN 0, 1024, 2816, 61469 and 61483. You can still use this function. However, if you add the FsJ1939UseCrcAndCounter message attribute to your database, only the new implementation will be used. The following applies: all messages that do not specify a value for the attribute are ignored when the checksum and message counter are checked.
