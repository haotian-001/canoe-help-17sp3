1

RTS/CTS Transport: Successfully receives a valid RTS (content and format) (DUT as Responder) 
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  4. At least one RTS‑>Abort PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Start sending TP.CM_RTS for PGN flagged with RTS‑>CTS and RTS‑>Abort.
  • Await the DUT's answer (TP.CM_CTS or TP.CM_Abort) within Tr (200 ms):
  • Expectation: DUT sends TP.CM_CTS or TP.CM_Abort within Tr (200 ms).
  • For RTS‑>CTS PGN the tool completes the TP. (Errors during this completion lead not to fail the test).
  1. Check the DUT's answer.
  • Check the PGN in the DUT's answer.
  • Expectation: The PGN in the DUT's answer (including DP/EDP) fits the PGN in the TP.CM_RTS
  • If the PGN was configured to be RTS‑>Abort:
  • Expectation: The DUT sent an Abort.
  1. Check the DP/EDP bits.
  • Start sending TP.CM_RTS for PGN flagged with RTS‑>CTS and RTS‑>Abort with manipulated DP/EDP bits.
  • Await the DUT's answer within Tr (200 ms):
  • Check the PGN in the answer.
  • Expectation: DUT answers within Tr (200 ms) and the PGN fits the manipulated PGN.
  • The tool completes the TP. (Errors during this completion lead not to fail the test).
  1. Inconclusive, if no RTS‑>CTS and RTS‑>Abort PGN is configured. (Both are needed).
  2. Fails, if DUT does not respond properly within Tr (200 ms) to the TP.CM_RTS.
  3. Fails, if DUT aborts an RTS‑>CTS PGN, or does not abort an RTS‑>Abort PGN.
  4. Fails, if PGN in the DUT's answer does not fit the PGN in the TP.CM_RTS. (timeout).

2

RTS/CTS Transport: CTS following RTS is valid (content and format) (DUT as Responder)
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Start sending TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the DUT's TP.CM_CTS within Tr (200 ms):
  • Expectation: DUT sends TP.CM_CTS within Tr (200 ms).
  • Expectation: TP.CM_CTS 'Number of packets that can be sent' <= TP.CM_RTS limitations.
  • Expectation: TP.CM_CTS 'Next packet number to be send' starts with 1.
  • Expectation: TP.CM_CTS is not send to global address.
  • Expectation: DUT does not abort.
  • The tool completes the TP.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not respond properly within Tr (200 ms) to the TP.CM_RTS.
  3. Fails, if DUT aborts an RTS‑>CTS PGN.
  4. Fails, if PGN in the DUT's answer does not fit the PGN in the TP.CM_RTS. (timeout).
  5. Fails, for unexpected of wrong parameters in TP.CM_CTS.
  6. Fails, if DUT sends a TP.CM_CTS to the global address (never allowed!).

3

RTS/CTS Transport: Multiple RTS received from same SA for the same PGN within short period (DUT as Responder)
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Start very quickly send two TP.CM_RTS for PGN flagged with RTS‑>CTS of different message sizes.
  • Await the DUT's TP.CM_CTS within Tr (200 ms):
  • Expectation: DUT sends only one TP.CM_CTS within Tr (200 ms) for the most recent TP.CM_RTS.
  • Expectation: DUT does not abort.
  • After the tool completed the TP:
  • Expectation: DUT sends TP.CM_EndOfMsgACK for the most recent RTS.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not respond properly within Tr (200 ms) to the TP.CM_RTS.
  3. Fails, if DUT aborts an RTS‑>CTS PGN.
  4. Inconclusive, if DUT aborts the multiple RTS. (See Testreport for further information).
  5. Fails, if DUT send multiple TP.CM_CTS messages as answer to each of the TP.CM_RTS.

  3. Fails, if DUT accepts the TP.CM_RTS.
  4. Fails, if PGN in the abort does not fit the PGN in th TP.CM_RTS.
  5. Fails, if DUT sends abort to the global address (never allowed).

RTS/CTS Transport: Receiving an invalid TP.CM_RTS (content and format) (DUT as Responder)

  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send TP.CM_RTS for PGN flagged with RTS‑>CTS with invalid number of packets.
  • Await the DUT's TP.CM_Abort within Tr (200 ms):
  • Expectation: DUT aborts the connection by sending TP.CM_Abort within Tr (200 ms).
  • Expectation: The TP.CM_Abort is not send to global address.
  1. Send TP.CM_RTS with message size <9 and await abort within Tr (200 ms).
  • Await the DUT's TP.CM_Abort within Tr (200 ms):
  • Expectation: DUT aborts the connection by sending TP.CM_Abort within Tr (200 ms).
  • Expectation: The TP.CM_Abort is not send to global address.
  1. Send TP.CM_RTS with message size >1785 and await abort within Tr (200 ms).
  • Await the DUT's TP.CM_Abort within Tr (200 ms):
  • Expectation: DUT aborts the connection by sending TP.CM_Abort within Tr (200 ms).
  • Expectation: The TP.CM_Abort is not send to global address.
  1. Send TP.CM_RTS to global address.
  • Wait T3 (1250 ms) + Tr (200 ms):
  • Expectation: DUT does NOT abort an TP.CM_RTS sent to global address.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not abort (reject) within Tr (200 ms).
  3. Fails, if DUT accepts the TP.CM_RTS.
  4. Fails, if DUT sends abort to the global address (never allowed).

RTS/CTS Transport: Does not send CTS outside of connection (DUT as Responder)
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the TP.CM_EndOfMsgACK of the DUT to have the TP to be finalized.
  • Expectation: DUT accepts connection with TP.CM_CTS within Tr (200 ms).
  • Expectation: DUT finalizes the TP with TP.CM_EndOfMsgACK.
  • Expectation: DUT does not send any TP.CM_CTS for PGNs not in the current connection.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT produces a TP error in any way - so TP gets not finalized.
  3. Fails, if DUT sends a TP.CM_CTS to the tool, for a PGN, that is not in current connection.

RTS/CTS Transport: Multiple CTS (same SA, DA, PGN) transmitted in short period (DUT as Responder)

  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the DUT's TP.CM_CTS within Tr (200 ms).
  • Await a (second) TP.CM_CTS within T2 (1250 ms).
  • Expectation: DUT does not send a (second) TP.CM_CTS within T2 (1250 ms).
  • Abort the connection if DUT didn't abort the connection so far.
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the DUT's TP.CM_CTS within Tr (200 ms).
  • Send only the first TP.DT packet.
  • Await a (second) TP.CM_CTS within T1 (750 ms).
  • Expectation: DUT does not reorders the missing TP.DT with a TP.CM_CTS within T1 (750 ms).
  • Abort the connection if DUT didn't abort the connection so far.
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the TP.EndOfMsgACK of the DUT to have the TP to be finalized.
  • Expectation: DUT accepts connection with TP.CM_CTS within Tr (200 ms).
  • Expectation: DUT finalizes the TP with TP.CM_EndOfMsgACK.
  • Expectation: DUT does not send TP.CM_CTS messages that are not suitable for the TP.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not react on RTS within Tr (200 ms).
  3. Fails, if DUT does not abort or reorder packets when they are missing.
  4. Fails on any other TP errors (see testreport for details).
  5. Fails, if DUT sends TP.CM_CTS packets not suitable for the current connection.

RTS/CTS Transport: Timing of first TP.DT for RTS/CTS following CTS (DUT as Responder)

  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the TP.CM_CTS within Tr (200 ms).
  • Wait T2 (1250 ms).
  • Expectation: DUT does not try to reorder TP.CM_CTS within T2 (1250 ms).
  • Await reaction of DUT within Tr (200 ms).
  • Expectation: DUT sends abort.
  • Expectation: DUT reorders (correctly) the missing data, by (re)sending the TP.CM_CTS.
  • Abort the connection if not done by the DUT already.
  • Wait T3 (1250 ms).
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not react on RTS within Tr (200 ms).
  3. Fails, if DUT does neither abort, nor re-order the missing data-block.
  4. Fails, if DUT re-orders, but the CTS' NextPacketNumberToBeSent is not 1.
  5. Fails, if DUT tries to re-order by CTS within the timeout T2 (1250 ms).

15

RTS/CTS Transport: TP.DT Data Transport timing following CTS (DUT as Responder)

  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS with max 3 packets.
  • Await the TP.CM_CTS within Tr (200 ms) for the first 3 packets.
  • Expectation: DUT sends TP.CM_CTS for 3 packets starting with packet #1.
  • Send the first 3 TP.DT packets.
  • Await the TP.CM_CTS within Tr (200 ms) for the next 3 packets.
  • Expectation: DUT sends TP.CM_CTS for 3 packets starting with packet #4.
  • Await 1st reaction of DUT within T2 (1250 ms) when nothing is sent anymore.
  • Expectation: DUT can abort the missing TP.DT packets -> OK.
  • Expectation: DUT can reorder, by sending TP.CM_CTS for 3 packets starting with packet #4.
  • On reorder, await 2nd reaction of DUT within T2 (1250 ms) when nothing is sent anymore.
  • Expectation: DUT can abort the missing TP.DT packets -> OK
  • Expectation: DUT can reorder, by sending TP.CM_CTS for 3 packets starting with packet #4.
  • On reorder, await 3rd reaction of DUT within T2 (1250 ms) when nothing is sent anymore.
  • Expectation: DUT has to abort the missing TP.DT packets for the 3rd consecutive time. -> OK.
  • Abort the connection if not done by the DUT already.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not react on TP.CM_RTS within Tr (200 ms).
  3. Fails, if DUT reorders by TP.CM_CTS too early - within T2 (1250 ms).
  4. Fails, if DUT does not abort, if the data is missing after the 3rd consecutive time.
  5. Succeeds, if DUT aborts instead of re-ordering with TP.CM_CTS.
  6. Fails, if DUT reorders the first data block, but does not start with packet #1.
  7. Fails, if DUT reorders the second data block, but does not start with packet #4.
  8. Fails, if DUT does not order 3 packets with TP.CM_CTS.

RTS/CTS Transport: Time between consecutive Transport Data Packets for RTS/CTS when the CTS allows more than 1 packet (DUT as Responder)
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS with max 4 packets.
  2. Await the TP.CM_CTS within T2 (1250 ms) for the first 4 packets.
  • Expectation: DUT sends TP.CM_CTS for 4 packets starting with packet #1.
  • Expectation: Test is inconclusive if DUT rejects the connection with abort.
  1. Send the first TP.DT packet with sequence #1 within Tr (200 ms) after the TP.CM_CTS.
  2. Await the next TP.CM within T1 (750 ms).
  • Expectation: If DUT sends TP.CM_CTS test fails, due TP.CM_CTS is not allowed here.
  • Expectation: Test is inconclusive if DUT aborts the connection.
  1. Send the second TP.DT packet with sequence #2 within Tr (200 ms) and T1 (750 ms) after the last TP.DT.
  2. Await the next TP.CM within T1 (750 ms).
  • Expectation: If DUT sends TP.CM_CTS test fails, due TP.CM_CTS is not allowed here.
  • Expectation: Test is inconclusive if DUT aborts the connection.
  1. Await the next TP.CM within Tr (200 ms).
  • Expectation: If DUT sends TP.CM_CTS this is a resend #1. The content is checked.
  • Expectation: If DUT aborts, the test is passed.
  1. Await the next TP.CM within T1 (750 ms).
  • Expectation: If DUT sends TP.CM_CTS test fails, due TP.CM_CTS is not allowed here.
  • Expectation: If DUT aborts, the test is passed.
  1. Await the next TP.CM within Tr (200 ms).
  • Expectation: If DUT sends TP.CM_CTS this is a resend #2. The content is checked.
  • Expectation: If DUT aborts, the test is passed.
  1. Await the next TP.CM within T1 (750 ms).
  • Expectation: If DUT sends TP.CM_CTS test fails, due TP.CM_CTS is not allowed here.
  1. Await the next TP.CM within Tr (200 ms).
  • Expectation: DUT aborts the third consecutive time there is no data sent.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not react on TP.CM_RTS within Tr (200 ms).
  3. Fails, if DUT sends a TP.CM_CTS too early.
  4. Fails, if DUT does not abort, if the data is missing after the 3rd consecutive time.
  5. Succeeds, if DUT aborts instead of reordering with TP.CM_CTS.
  6. Fails, if DUT sends a TP.CM_CTS with unexpected starting index or number of packets.
  7. Inconclusive, if DUT rejects the connection after the TP.CM_RTS.
  8. Inconclusive, if DUT aborts the connection before a TP.DT timeout can be detected.

17

RTS/CTS Transport: Sequential order of Transport Data Packets for RTS/CTS (DUT as Responder)
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one RTS‑>CTS PGN.
  1. DUT has to support Transport Protocol RTS/CTS (0x00EB00, 0x00EC00).
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS.
  • Await the TP.CM_CTS within Tr (200 ms).
  • Expectation: DUT reacts to the TP.CM_RTS and TP without sequence errors finalizes correctly.
  1. Send valid TP.CM_RTS for PGN flagged with RTS‑>CTS (a second time).
  • Await the TP.CM_CTS within Tr (200 ms).
  • Expectation: DUT sends TP.CM_CTS within Tr (200 ms).
  • Send TP.DT packets with a wrong order in sequence.
  • Inspect if the DUT aborted or reordered packets with TP.CM_CTS within the TP.DT data block.
  • Expectation: DUT aborts the connection or reorders packets with TP.CM_CTS.
  • Expectation: DUT's reaction may also follow after the last TP.DT.
  • If DUT didn't react within the data block, check the reaction after the last TP.DT within Tr (200 ms).
  • Expectation: DUT can abort within Tr (200 ms). Then the test is passed.
  • Expectation: DUT can reorder with TP.CM_CTS within Tr (200 ms).
  • Expectation: DUT does not fnalize the connection with TP.CM_EndOfMsgAck.
  • If DUT reorders packets with TP.CM_CTS, send the packets and await the TP.CM_EndOfMsgAck.
  • Expectation: DUT finalizes the resent packets with TP.CM_EndOfMsgAck.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Inconclusive, if DUT rejects the connection.
  3. Fails on any TP error (see testreport for details).
  4. Fails, if DUT does neither reorders, nor sends abort on wrong sequence.
  5. Fails, if DUT sends TP.CM_EndOfMsgAck after wrong sequence.
  6. Fails, if DUT does not finalizes the proper resent packets with TP.CM_EndOfMsgAck.
  7. Inconclusive, if DUT aborts a TP that contains no errors in between.

By using the PGNs flagged with RTS‑>CTS...

Within TP, the tool provokes the DUT to produce an abort.

Then it checks the aborts reason, the DA and the PGN.

 

Following provocations are tested (if testable):

  • Reason 1: Already in one or more connections: Not testable.
  • Reason 2: System ressource problem: Not testable.
  • Reason 3: Timeout: Send RTS and after receiving CTS, send first TP.DT ans stop. After DUT aborted with timeout, send the missing TP.DTs and check if DUT does react on them.
  • Reason 4: CTS within connection: Not testable in this TP-direction.
  • Reason 5: Max retransmit: Not testable in this TP-direction.
  • Reason 6: Unexpected Data Transfer packet: Not testable due semantic is not cleared.
  • Reason 7: Bad sequence: Send RTS, await CTS and send a bad TP.DT sequence.
  • Reason 8: Duplicate sequence: Send RTS, await CTS and send a duplicate TP.DT packet.
  • Reason 9: Total msg size > 1785: Send an RTS with a byte size of > 1785 bytes.
  • Reason 10-249: SAE range. Not directly testable, but if DUT uses any of this abort code in the other checks, the test fails with a hint to the reserved SAE range.
  • Reason 250: Not listed. Not testable.
  • Reason 251-255: J1939-71 - Vehicle Application Layer. Not testable.
  1. Inconclusive, if no RTS‑>CTS PGN is configured.
  2. Fails, if DUT does not abort as expected.
  3. Fails, if DUT uses an unexpected abort reason.
  4. Fails, if DUT uses the global address as DA.
  5. Fails, if DUT uses an unexpected PGN in the abort.
  6. Fails, if DUT accepts packets after abort.
  7. Fails, if DUT aborts too early, before timeout exceeds.
