J1939-82 Compliance Test (2015)

 

Multimedia Link

Generating a J1939 Compliance Test Conveniently with CANoe J1939

This online video tutorial will show you how you can generate a J1939-82 (2015) compliance test module (Revision 2015). You will see a step by step sample which is assisted by text explanations for further information to the steps.

The tests serve to cover test cases from the J1939 Compliance Specification (2015).
For further information, please refer to the specification.

Advantages

Configuration

With the Compliance Test (2015) a test unit is created via vTestStudio that was programmed with CAPL. It depicts the content of the Compliance Tests 2015. The J1939.CT_Configurator panel serves to configure the data to be used by the test, for example, which messages support the DUT.

Test Results

The error description in the test reports was extended to simplify the interpretation of the test results. The HTML report has a detailed description of individual test cases and provides links to them.

Test Strategies

Note

  • On failure, especially on inconclusion, there is no need to diagnose a wrong behavior of the DUT too early. Try to repeat the test isolated some (3-4) times and check what lead to the trouble code within the DUT. If the test at any time succeeds, it's a strong indication the DUT works correctly but has another issue that should to be investigated in depth.
    Also there are some randomly generated things implemented, like randomly generated CAN messages for lifting the busload, that lead to different runtimes.
  • Dependant on your DUT implementation a DENY/BUSY ACK may be not trivially to be available at testing time. But the test needs to have a PGN that is produced and responded always with DENY/BUSY. For the test it's impossible to set a DUT into a mode that will configure the DUT to react this way, so it heavily depends on your testing/debugging interfaces, like Testing-Mode-DIP-Switches, etc. It is planned to evolve a solution for this problem e.g a user-code space or similar.

Execution

The J1939.CT_Configurator panel is available for productive execution of your compliance tests (2015).

To create a new productive compliance test project, open the File Ribbon Tab, click on New and select the J1939-82 Compliance Test (2015) template.

J1939.CT_Configurator

This panel serves to configure the data used by the test. The individual panel areas and their functionality are described here.

DUT – Device Under Test

  • Address [hex]
    Hexadecimal J1939 node address of the DUT. It will be used to identify with which node it must communicate.
  • NAME [hex]
    The J1939 device name of the DUT. It is used for, e.g. examining the byte order of the response. Enter the device name as hexadecimal numerical order in mathematical sequence (corresponding to Little Endian or Intel format), for instance, 4B280D0AE8A03039.
  • Commanded Address
    You can select whether the DUT supports a Commanded Address (CA). CAs are used so that the DUT starts a Transport Protocol (TP). The CA has a DLC of more than 8 bytes.
  • Name Management
    DUT supports Name Management (NM).
  • DM1 indicates Addr.Viol.
    If activated, the DUT behaves with a system notifiaction over DM1, when it detects an address conflict (violation).
  • Msg.Tolerance Abs. [ms]
    In Test A04 Row1 the time interval of cyclic messages is measured. Enter the absolute time in milliseconds which should be considered as the tolerance threshold.
    Deviations of the cycle times above the tolerance values Msg.Tolerance Abs. [ms] and Msg.Tolerance Rel. lead to a failure. A faster transmission within the cycle time is accepted.
    Possible values: 3 - 99999 ms
    Good default value: 3 ms.
  • Msg.Tolerance Rel.
    Apart from the absolute time indication of the tolerance (see above), you can also enter a relative indication.
    Deviations of the cycle times above the tolerance values Msg.Tolerance Abs. [ms] and Msg.Tolerance Rel. lead to a failure. A faster transmission within the cycle time is accepted.
    Possible values: 0.05 - 100 %
    Good default value: 0.1%.

PGNs

Here you can configure the PGNs that are supported by DUT.

No.

Number, for which the PGN's sequence is to be stored and reset.

PGN

Hexadecimal indication of the PGN (0x0 - 0x3FFFF).

DLC>8*

PG response has a DLC of more than 8 bytes, so that the the content must be transmitted with the transport protocol.

Cycle*

Cycle time for cyclic messages in [ms]. 0 corresponds to non-cyclic.

Req.*

This PGN is requestable.

Resp.ACK*

Response to a Request of this PGN results in an ACK message.

Resp.DENY*

Response to a Request of this PGN results in a DENY message.

Resp.BUSY*

Response to a Request of this PGN results in a BUSY message.

RTS->CTS**

Reaction to an RTS for this PGN is answered with a CTS by the DUT.

RTS->Abort**

Reaction to an RTS for this PGN is answered with an Abort by the DUT.

*) These fields are describing the DUT's bahavior when sending packets.

**) These fields are describing the DUT's behavior when receiving packets.

PG with known data (DLC>8)

There are tests which evaluate the content (Payload) in conjunction with the transport protocol. This indicates the PGN and the content of the PG (DATA) in hexadecimal.

Without this information the affected tests will be listed with inconclusive.

Note

To run most tests, the following PGNs must be configured:

  1. PGs sent by the DUT

a) PDU1 not cyclic, requestable, DLC=8

b) PDU1 not cyclic, requestable, DLC>8

c) PDU1 cyclic, DLC=8

d) PDU2 not cyclic, requestable, DLC=8

e) PDU2 not cyclic, requestable, DLC>8

f) PDU2 cyclic, DLC=8

  1. PGN that can be requested, but for which the positive ACKM (ACK) is sent as the response rather than the requested PG. This could be e.g. DM11 (PGN=0xFED3) or DM3 (PGN=FECC).

  2. PGN that is not supported by the DUT. If such a PGN is requested, a negative ACKM (NACK) is sent in response.

  3. Multi-frame PGs that are sent with RTS/CTS and received by the DUT. For these, DLC>8 and "RTS->CTS" must be configured.

  4. To test the reception of BAM messages (Table A6), the DUT must support the Commanded Address parameter group.

  5. To test sending with RTS/CTS and BAM (tables A5 and A7), there must be a parameter group with DLC>8 and a known and constant payload. This parameter group must be described in the area "PG with known data (DLC>8)".

  6. To test the reception with RTS/CTS (Table A8), at least one PGN with DLC>8 must be received by the DUT. The flags "DLC>8" and "RTS->CTS" must be set for this PGN.

If fewer PGNs are configured, the test case for which there is no matching PGN is noted as inconclusive.

TOOL

  • Tool Address [hex]

    • Represents the J1939 node address of the testing tool.
    Standard: 0xF9.

DEBUG

You can use these selections to control the outputs in the Write Window during the test procedure. The additional outputs enable a deeper insight into the precise test procedure.

  • CONFIG
    Information on the configuration management of the test (e.g. on the configured PGNs in the panel)
  • HELPER
    System internal function calls
  • TEST
    Information on the test structure (each test step has a test start, a test end and a verdict). An unsupported nesting can thus be uncovered.
  • CHECKS
    Information on the test functions called upon

Since the buffer of the Write Window is limited, you should activate these settings for individual test runs only if necessary.

REPORT

Here you can control the output in "Report". You can add the Preconditions and the Strategy of the test cases to the test.

Use Inconclusive As to specify what will be performed in test cases, which will be marked as Inconclusive. They can be maintained as Inconclusive or alternatively marked as Passed or Failed.

Note

Inconclusive: Tests that could not be completed. You will find the reason in the test report.

TEST

  • Autostart Test
    Controls a system variable which is combined with the test module in order to start the measurement.
    If you have already activated Autostart Test, the test unit starts automatically with the measurement. Otherwise you need to manually start the test unit after the start of measurement.
  • Autostop Measurement
    Stops the measurement, as soon as the test unit runs down.
  • Full Mode
    Particularly for the A06 Row3 test. The PGs that are not supported by DUT will be under examination. Since the number of these PGs is unknown, the a brute force approach will be used to check all possible PGs (if you have chosen this option). If not, only one out of every ten PGs will be examined.
