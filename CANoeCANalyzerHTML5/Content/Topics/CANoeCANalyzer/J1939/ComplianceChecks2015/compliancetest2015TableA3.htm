Precondition DUT

Strategy

Constriction / Limitation / Typical Errors / Resolution Notes

1

Check position and order of PGN in the 29-bit Frame Header
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one requestable PGN.
  1. DUT has to support and answer a J1939 Request (0x00EA00).
  1. Request the configured requestable PGNs.
  2. Await DUT response for the requested PGN within response timeout Tr (200 ms).
  3. Only accept the response, if the Frame Header of the response contains the requested PGN at the expected position and order.
  1. Inconclusive if no requestable PGNs are configured.
  2. Timeout if DUT does not send an expected response to the request within response timeout Tr (200 ms).
  3. Wrong positioned / ordered PGNs within the Frame Header of the response are not accepted by the tester and lead to a response timeout.

2

Use of EDP Bit to identify PGN
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one requestable PGN.
  1. DUT has to support and answer a J1939 Request (0x00EA00).
  1. Request the configured requestable PGNs.
  2. Await DUT response for the requested PGN within response timeout Tr (200 ms).
  3. Only accept the response, if the Frame Header of the response contains the requested PGN EDP bit.
  1. Inconclusive if no requestable PGNs are configured.
  2. Timeout if DUT does not send an expected response to the request within response timeout Tr (200 ms).
  3. Unexpected EDP bit within the Frame Header of the response is not accepted by the tester and lead to a response timeout.

3

Use of DP Bit to identify PGN
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one requestable PGN.
  1. DUT has to support and answer a J1939 Request (0x00EA00).
  1. Request the configured requestable PGNs.
  2. Await DUT response for the requested PGN within response timeout Tr (200 ms).
  3. Only accept the response, if the Frame Header of the response contains the requested PGN's DP bit.
  1. Inconclusive if no requestable PGNs are configured.
  2. Timeout if DUT does not send an expected response to the request within response timeout Tr (200 ms).
  3. Unexpected DP bit within the Frame Header of the response is not accepted by the tester and lead to a response timeout.

4

Data Field Byte Length
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. At least one requestable PGN.
  1. DUT has to support and answer a J1939 Request (0x00EA00).
  1. Request the configured requestable PGNs.
  2. Await DUT response for the requested PGN within 2 times the response timeout Tr (200 ms).
  3. Inspect all Frame DLCs sent by the DUT starting from the request time onwards that they consist of 8 bytes. Request frames (0x00EA00) consist of 3 bytes only and are also accepted.
  1. Inconclusive if no requestable PGNs are configured.
  2. Timeout if DUT does not send an expected response to the request within 2 times the response timeout Tr (200 ms).
  3. Frames with DLCs with less or more as 8 bytes (except Request frames) are ignored.

5

Byte Ordering within Data Field
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  3. Device NAME.
  1. DUT has to support ACL. This implies the ability to handle J1939 Requesting (0x00EA00).
  1. Request the PGN AC (0x00EE00) from the DUT.
  2. Await the ACL of the DUT within response timeout Tr (200 ms).
  3. Accept the ACL, if it was sent from DUT to the global address.
  4. ACL payload contains the Device NAME of the DUT that is inspected for the right byte ordering.
  1. Fails if the payload does not contains the expected bytes consisting of the Device NAME.
  2. Timeout if DUT does not send an expected ACL to the request within response timeout Tr (200 ms).

6

Multipacket capable PGN with less than 9 bytes of data sent as single CAN Data Frame with 8 byte data field
  1. DUT address.
  2. Tool address (defaults to 0xF9).
  1. DUT has to support and answer a J1939 Request (0x00EA00).
  2. DUT has to support multipacket PGNs.
  3. DUT has to support DM11 (clear).
  4. DUT has to support DM1 (trouble codes).
  5. DUT properly fills unused payload with padding bytes (0xFF).
  6. DUT has no serous problems that may flood the trouble code table.
  1. Send request for DM11 to the DUT and expect it clears its trouble codes.
  2. Send request for DM1 to the DUT.
  3. Await the DM1 response. After (2) it has a size of DLC=8 (no errors).
  4. Inspect the last 2 bytes (padding bytes) within the response. They has to be equal to 0xFF.
  1. Fails if the payload is not properly filled with padding bytes.
  2. Timeout if DUT does not send an expected DM1 response to the request within response timeout Tr (200 ms).
  3. DUT tries to use the Transport Protocol for a DM1 response with a payload of DLC=8 bytes. I this case the test also detects this and fails.
  4. Inconclusive if the DUT answers with a DM1 of DLC>8 bytes using the Transport Protocol. Either it didn't clear the trouble codes, or trouble code table may be flooded by rapidly occurring errors.

Note

Even if the DUT fits the preconditions and cleared the trouble codes, in the meanwhile of the request and response time and despite there are trouble codes within the DM1 answer.

The test can't be really be done, due the DUT seems to have another serious problem that has to be found and fixed first.
