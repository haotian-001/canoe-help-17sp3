Testing in a J1939 Network

 

Introduction

Tests for J1939 ECUs are set up according to the same principle as for all other bus systems or protocols:

  1. A bus environment is created in which the Device Under Test (DUT) can operate. For example, some devices rely on other devices to provide them with certain messages. For example, an ABS device needs information about the speeds of all wheels.
  2. The tester stimulates the DUT by placing certain messages on the bus.
  3. Optionally, the tester can also initiate electrical voltages or currents representing sensors directly connected to DUT. For this, however, CANoe must be expanded by a VT System.
  4. In the next step the tester checks the reactions of the DUT. The reaction of the DUT is composed of data that the DUT sends in the form of CAN frames. Optionally, you can also check the voltage and/or current at DUT outputs. For this purpose CANoe must be expanded by a VT System (just like in the previous step).
  5. If the DUT response is as expected, the test step can be marked as successful. Otherwise, the test step is marked as faulty.

Possible Solutions

The following explains how you can implement the points listed in the introduction in concrete terms.

Simulating a J1939 ECU for the Necessary Bus Environment

Devices that are necessary for stimulating the DUT can be simulated by a simulation node in CANoe.

To do this, create a network node in the database for the device. Assign the modeling library J1939_IL.dll to this node and configure its transmission behavior by assigning messages.

Now open the configuration dialog of the simulation node and assign the network node from the database to it.

This already defines the basis - the simulated node sends the configured cyclic messages and you only have to make sure that messages are filled with correct data.

You can do this either by setting the signal values or by defining the payload of a message directly. Setting the signal values can be done with the J1939ILSetSignal function in a CAPL program, a Signal Generator, in the Symbol Explorer or via a self-configured panel. The payload can only be set in a CAPL program with the help of the J1939ILSetMsgRawData function.

To send a message spontaneously to the DUT during a test, you can use the CAPL functions output and J1939ILSetMsgEvent. If you want to influence the parameters of the J1939 transport protocol or the components of the J1939-NAME, you can use the CAPL function J1939ILSetNodeProperty. You can use the functions J1939ILActivateDTC, J1939ILAddDTC and further functions from the DTC Support group to fill diagnostic messages with corresponding DTCs.

For more information on how to include the J1939 IL for a test, see the Interaction Layer (J1939) page.

Stimulating the Physical Inputs of the DUT

To connect physical inputs of the DUT, a VT System must be connected. You only have to set system variables in your test and the connected VT System converts the value of the variables into the voltage or current.

Checking the Behavior of the DUT

There are several options available when checking the response:

The easiest way to control the reaction of the DUT is "manually": Either in the Trace Window, Graphics Window or Data Window. Or you prepare raw data with the help of a simple CAPL program and output results in a panel you compiled. This is fast, but error-prone.

To check the reactions of the DUT automatically, you can create a test module in CANoe or execute a test unit created with it. In the test module or test unit you have, among other things, special CAPL functions at your disposal with which you can wait for a specific event or for a combination of several events. These include both the general functions such as TestWaitForSignalMatch or TestWaitForMessage and the J1939-specific functions such as TestWaitForJ1939PG and TestWaitForJ1939DTC.

When the called TestWaitForJ1939xxx function returns, you can use the TestGetWaitJ1939PGData function to get the data of the parameter group that triggered the wait condition and then analyze it.

In addition to such test functions, CANoe also provides you with checks that, once started, continuously check whether the DUT complies with a certain aspect of the defined transmission model or J1939 protocol. Several checks can check their respective conditions in parallel.

General Checks

The following general checks also provide J1939 support:

  • Cycle Time Checks
  • DLC Check
  • Fallback
  • Message Distance
  • Node Active
  • Occurrence Count
  • Occurrence Distance
  • Occurrence Of Messages

Special Checks

In addition, there are special checks that check J1939-specific regulations (such as transport protocols):

  • J1939 Address Claiming
  • J1939 BAM
  • J1939 Request Check
  • J1939 Request2 Check
  • J1939 RTS/CTS
  • J1939-76 Functional Safety Check

Note

All the above mentioned checks can be used both in CAPL and in the table editor of vTESTstudio).

Test Commands

In addition to checks, there is another type of test, test commands.

Test commands are test modules that combine a stimulation of the DUT and a test of the DUT response to the stimulation.

The main difference between checks and test commands:

While checks run in the background, test commands allow you to provoke a specific behavior and check that behavior at the same time.

The following test commands are available:

Name

Task

J1939 Request

J1939 Request2

Check whether a J1939 request is answered correctly.

J1939 Address Claiming Conflict

Check if address claiming is supported correctly.

J1939 Wait For DTC

Wait until a DM message with defined SPNs is received.

Note

Test commands are only available in the table editor in vTESTstudio.

Note

The J1939 Test Service Library is no longer being developed and should therefore no longer be used.

Manipulation of Transport Protocols

You have the option of intervening in the flow of Transport Protocols. With a series of CAPL funktionen, you can specifically block the transmission of transport protocol messages or delay them as required. With the functions for blocking and delaying individual messages, you can specifically test simple scenarios. However, if the test scenario becomes more complex, these functions are no longer sufficient. CANoe therefore offers you the option of freezing transport protocols and continuing execution in order to inject your own manipulated TP messages in a targeted manner. The functions J1939ILFreezeTp, J1939ILUnfreezeTp and J1939KillTp are available for this purpose.
