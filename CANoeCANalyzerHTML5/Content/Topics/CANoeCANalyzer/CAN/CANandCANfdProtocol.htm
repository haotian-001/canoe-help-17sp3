Protocol Specific Peculiarities (CAN/CAN FD)

Protocol-Specific Properties (CAN and CAN FD)

What is the 'Bus Integration State'?

The CAN controller is in this state in the following situations:

  • Start of communication
  • After a Bus Off Recovery
  • After a Protocol Exception Event

The CAN controller does not participate in communication in this state.

The CAN controller becomes active again in the network only when it has detected the Bus Idle condition.

Where are the Time Stamps Recorded for Frames of CAN or CAN FD?

  • For a received frame, the time stamp is recorded after the sixth EOF bit. For a recipient, a frame is valid when no error has occurred up to this position.
  • For a sent frame, the time stamp is recorded after the seventh EOF bit. For a sender, a frame is valid when no error has occurred up to this position.

The different position for acceptance of a frame should ensure that a receive node can still signal an error in the sixth EOF bit to the sender in the seventh EOF bit. This causes the sender to send the frame again.

How is the Frame Length Calculated?

The frame length is determined from the bit fields of the frame, the Stuff bits and the Intermission field (IFS). The IFS is added to the frame length because no other transmission can be started at this point. The IFS thus counts at the moment in which the network is occupied. (status Bus busy). This also affects the calculation of the busload.

Protocol-Specific Properties (CAN FD)

What is a 'Protocol Exception Event'?

The Protocol Exception Event was introduced with ISO11898-1:2015.

Its purpose is to enable backwards compatibility of existing controllers for future protocol expansions.

In ISO11898-1:2015 the res bit (reserved bit for future expansions) is specified as a bit with dominant level.

If the res bit is sent recessively, a Protocol Exception Event can be sent by the controller instead of an Error Frame. In this case, the controller withdraws from the network, sends only regressively, and transitions to the Bus Integration State.

What is the Purpose of 'Restricted Operation Mode'?

The Restricted Operation Mode is an optional mode of the CAN Controller that enables the receiving of frames and the acknowledgment. It is not possible to send Frames, Error Flags or Overload Flags. If an error or overload occurs, a Protocol Exception is signaled. The error counters are not affected.

Restricted Operation Mode is used with CAN controllers that use a so-called CAN Calibration Unit (PLL with adjustable clock divider) instead of a quartz. The CAN Calibration Unit is used for calibration to the utilized CAN bitrate. During calibration the CAN module is in Restricted Operation Mode so that communication is not disturbed when the controller still sees an incorrect pulse.

Why are 'Transmitting Node Delay (TD)' and 'Secondary Sample Point (SSP)' Needed?

To detect send errors, a CAN controller receives the bits of a frame it sent itself and compares these with the desired values. The Transmitting Node Delay (TD) is the signal propagation delay from Tx FlipFlop to Rx FlipFlop of a controller.

The TD comprises the delay times of the controller and transceiver. At bitrates in the CAN FD data phase of 1 Mbit/s or higher, the TD can be greater than a physical bit time. The time interval from the start of the sent bit to the sampling point of the read-back bit (SSP) is thus SSP = TD + SSP Offset.
