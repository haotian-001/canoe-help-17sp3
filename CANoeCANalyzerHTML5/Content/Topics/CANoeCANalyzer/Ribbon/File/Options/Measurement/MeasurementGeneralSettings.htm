Configuration: General

Ribbon: File Tab|Options|Measurement|General

Channel Usage

In this dialog you can configure the number of channels you wish to work with in your active configuration. Besides affecting the measurement, the channel definition also affects the input options of the various configuration dialogs. Only defined channels are offered to you for selection.

In case you do not need more than one channel per bus system in the current configuration, please set the respective channel count values to 1.

  • In online mode the defined channels correspond to the channels of the current measurement (hardware or simulation).
  • In offline mode the defined channels correspond to the channels on which the replayed messages were logged in the logging file.
  • In simulation mode (online mode, simulated bus) each channel corresponds to a simulated bus.

Note

The number of channels is configuration-specific. It is saved in the configuration file and is restored when the configuration is loaded.

User Defined Application Name

If CANoe is started with a user defined application name not equal to CANoe, the name of this group includes the user defined application name.

CAPL blocks represent a special case. With these program blocks the check to determine whether all referenced channels are valid is not made until the compiling process. A warning is output if invalid channels are detected. It is advisable to compile all nodes after each redefinition of channels.

Perform Consistency Check

You can decide whether or not a consistency check should be performed after the configuration process. The consistency check relates to database associations and the configuration of all function blocks with the exception of CAPL blocks. It detects whether invalid channels are referenced. An inconsistency is reported whenever this is the case.

Example

Consider a configuration that has already been created and has two defined channels, since it operates with two buses.

A Generator is inserted in the Measurement Setup, and when it receives a message with ID 100 on channel 1 it sends the same message out on channel 2.

Now only one bus is to be used. Therefore, the number of channels is reduced to 1 in the Channel Definition Dialog. Channel 2 is thereby no longer defined.

If a consistency check is performed now, a warning is output indicating that the invalid channel 2 is referenced in the Generator.

Display results in the Write Window

The consistency check reports can be output to the Write Window if desired.

Simulation

Working mode

The list with the following working modes are only available in the Home ribbon tab, when you have selected the Online mode:

  • Real bus
    In this mode CANoe can communicate with real buses via a network interface. When a real control unit (ECU) is tested, the communication of the other ECUs is reproduced in a remaining bus simulation.
  • Simulated bus
    In this mode you can operate CANoe without hardware and simulate all buses completely.
  • As fast as possible
    Alternatively you can run the simulation with this option as fast as possible. In this case the simulation does not run with constant speed. The speed depends on the performance of the computer and the load that is provoked by simulation and analysis. The following circumstances slow down the simulation:
  • calculation of complex models
  • high simulated busload
  • many analysis windows
  • complex analyzes in the Measurement Setup
  • Animated with factor
    With this option you can specify an animation factor. The measurement is slowed-down by this factor. For simulation factors between zero and one the simulation is accelerated accordingly. For example, if you enter the value 0.1 the measurement is accelerated by a factor of 10.
  • Slave Mode
    The slave mode is a special simulation mode in which the time response of the measurement (time basis) is controlled by an external program.

Using the supplemental icons and that appear on the Home ribbon tab you can resume the simulation for a defined period of time or stop a running simulation at any desired point in time. The length of the time step is defined in the edit control to the right of the icons; entries are made in milliseconds.

If you are operating CANoe in simulation mode, i.e. without a physical bus, then the functionality of the CAN chip on the interface card is emulated by the program. In this mode, the only system parameter you define in the card's configuration dialog is the baudrate of the bus. The values of other card parameters do not have any effect on the program run in this case.

Via the Virtual MOST ring check box you connect all active MOST channels with the simulation.

In simulation mode CANoe usually reads out the simulation time directly of the hardware clock of the processor. As an alternative you can use the Use Windows timer check box to set that the time should be updated regularly by a windows timer. This means a significant inaccuracy of the simulation time but nevertheless may be helpful if you e.g. develop modeling libraries (DLL files) and want to pause the whole simulation for a while during debugging. I you have activated the As fast as possible option, however there is no correlation between simulation time and realtime.

Values on Measurement Start

Next measurement index

Here you specify which number is used for the filename of the next logging or test report file.
The number is used with the Start of Measurement field function.

Value of signals

You can choose whether the values of signals are to be reset to their initial values of the database at the beginning of a new measurement.

If you activate the option Reset to initial database values then all signal values are reset to their initial values at the beginning of a new measurement.
To manually reset signals to the initial values in the database before the start of a new measurement, press the [Reset Now] button.

Value of system variables

You can choose whether the values of system variables are to be reset to their initial values at the beginning of a new measurement.

All system variable values are preserved even after the end of measurement, and they are assumed as start values if you have selected the option Do not change. With this option you can also assign specific start values to system variables even before the start of measurement. To do this, set the appropriate controls in the panels which are loaded in CANoe, and then start the measurement.

If you activate the option Reset to initial values all system variables values are reset to their initial values at the beginning of a new measurement. System variables do no need a initial value. If a system variable has no defined initial value, their value remains unchanged at the start of measurement.

To manually reset system variables to the initial values in the database before the start of a new measurement, press the [Reset Now] button.

Note

These settings are specific settings of the currently loaded configuration.

With the Start Values Window you can set the start value of signals and system variables individually.

Drift/Jitter

The global settings for Drift and Jitter affect the timer behavior of any network nodes used in the Simulation Setup that do not have a local Drift / Jitter setting defined.

  • The Drift setting ensures that the processing times of all timers are lengthened or shortened by a specified percentage.
  • With Jitter the timer processing times are modified randomly. The setting specifies the interval within which the fluctuation should lie.
    The fluctuation is evenly distributed.

The entries in the corresponding fields are percent values. The maximum bandwidth is from –100.00% to +100.00%.

Note

  • A global Drift/Jitter setting should be specified if there are multiple timer-controlled events that can occur at the same time. This is particularly the case when several timers are triggered simultaneously. As a result, there can be a delay in the processing of individual events. Thus, the interval between two cyclically transmitted events may be subject to considerable deviations.

The global Drift / Jitter setting can help regulate these intervals.

  • Furthermore, please note the limitations for the use of the CANoe demo version.
  • These settings are specific settings of the currently loaded configuration.

Compatibility

Bus Validation

If a node outputs a message to a bus, a check is performed to determine whether the node actually has a link to this bus. The behavior of this function can be adjusted here:

  • No validation (ignore)
    No check will be performed.
  • Warn
    If an incorrect channel is being used, a warning will appear in the Write Window, but the message will still be sent.
    The format of the message is as follows:
    Node XXX: Output to bus system YYY channel ZZZ not allowed!
  • Stop measurement
    The message will not be sent and the measurement will be stopped.
    A message appears in the Write Window, as was the case with the Warn setting:
    Node XXX: Output to bus system YYY channel ZZZ not allowed!

Runtime Environment for Node Layer Modules

The runtime environment of the CANoe simulation has been improved in terms of using node layer modules and now allows for easier and at the same time more flexible configuration.
Up through Version 3.2 of CANoe, the (CAN) channel to be used had to be explicitly specified during configuration of modules. This generally involved editing INI files, databases or both.
Beginning with version CANoe 3.2 SP2, this channel configuration is superfluous since the channel to be used is implicitly specified by the configuration of the Simulation Setup. This also makes it possible to use node layer modules simultaneously on multiple buses. This allows you to run the OSEK NM on 2 buses at the same time, for example. Of course multiple use like this is also possible within a CAPL block (control device), which simplifies modeling of gateways significantly.
Not all currently available node layers are suitable for the new runtime environment in the same way, however. For this reason, a compatibility mode has been introduced corresponding exactly to the previous runtime environment, thus ensuring the same functionality of these node layers – including the previous restrictions.

  • Compatible (version 3.2 and earlier)
    This is the mode that is set by default for CANoe configurations that were generated with CANoe 3.2 or older.
    The runtime environment runs as it did previously in this mode. The correct functionality of node layer modules that were written for CANoe versions 3.2 or earlier is guaranteed with no restrictions. Whether or not it is possible to use modules on multiple buses simultaneously depends on specific configuration properties of the node layer in question. Generally it is not possible. Running a module on multiple buses simultaneously within a CAPL block (Gateway control device) is definitely not possible.
  • Standard
    This is the mode set by default for new configurations.
    This mode should be used for node layers that were created for CANoe 3.2 SP2 or later.

Note

These settings are configuration-independent settings of CANoe.

You'll find further information on the compatibility between configurations on the CANoe/CANalyzer Compatibility page.

String Encoding

Here you can set the coding in system variables of String type.

If you use Unicode (UTF-8) (the default setting for new configurations), the values of system variables of String type can contain any characters, especially also characters from other languages.
However, you can only use these values in CAPL if your CAPL program was also stored with UTF-8 coding.
You also cannot pass the values directly to any CAPL function or write the values to ASCII logging files. If you use the individual bytes in CAPL, note that the special characters are coded using UTF-8.

Alternatively, you can use the current character set of the computer for the coding same as in up to and including CANoe version 8.2 SP4. You must then limit the characters to those of the current language setting in the computer.
To use your configuration internationally, you should limit yourself to the ASCII character set.

Execution Environment

Here you can select in which variant (32-bit or 64-bit) the RT kernel should be used as execution environment.

If you change variant of the RT kernel, the current execution environment will be terminated and the selected one will be started. At the same time all modules will be loaded in the selected execution environment.

Please note that the 32-bit variant is still used for the execution in real-time mode (distributed mode or standalone mode) and this setting has no effect on it.

If the EnforceRtkTargetArchitecture switch in [SYSTEM] section in the CAN.ini is set to 1 (for 32-bit variant) or 2 (for 64-bit variant), the selection in this dialog is deactivated.

You can find further information on the Notes on 64-Bit/32-Bit help page.
