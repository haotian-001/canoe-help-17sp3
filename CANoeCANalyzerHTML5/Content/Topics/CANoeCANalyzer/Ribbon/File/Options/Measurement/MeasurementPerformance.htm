Configuration: Performance

Ribbon: File Tab|Options|Measurement|Performance

Statistics

  • Calculate bus statistics
    Here you can activate/deactivate the calculation of all bus statistics values.
  • Enable performance monitor
    Here you can activate/deactivate the performance monitoring of critical system parameters.

Timer accuracy check interval

Specifies the interval with which a cyclic timer is executed in the Real Bus working mode. When the interval expires, the Simulation::TimerAccuracy system variable is updated with the difference between realtime and simulated time.

If you have specific requirements for the accuracy of a timer, you can monitor it to some extent with this system variable. To do this, set the interval to the expiration time of the timer in question.

Note that a discrepancy between realtime and simulated time is quite normal and is not a problem in most cases. However, if you have realtime requirements and the simulation is held up, for example, by external events or by a slow simulation program, you can usually detect this by looking at the system variable.

Also note that especially at low busload the simulation is normally calculated only every 1 ms, so in this situation deviations up to 1 ms are normal.

You can meet higher realtime requirements by using an RT Setup or ERT.

Throughput

Here you can optimize the throughput by filtering events in the real-time area (RT server) of the application. This means that the data will not be transmitted to the analysis area(user computer).

  • Early filtering of events

With activated option a Channel Filter and a logging file will be included in the Measurement Setup before the Online/Offline switch.
The filtered data will not be displayed nor could they be logged in the Measurement Setup.

With the Channel Filter you can filter messages and events of all available bus systems.

If you want to increase the throughput without filtering entire bus systems, you can add different filters with the Hot Spot before the Online/Offline switch.
As in the case of the channel filter, the filtered data are not received in the analysis area. However, all other data can be analyzed and recorded in the Measurement Setup.
The filtered data can be recorded before the Online/Offline switch using the logging file.

Note

If you save the configuration in a CANoe version up to and including 8.2 ( all Service Packs inclusive) the filters before the Online/Offline switch will be deleted.
  • Filtering of the output (logging file) of the CAPL functions WriteToLog and WriteToLogEx.

The filtered data will not be displayed nor could they be logged in the Measurement Setup.

Extended Real Time (ERT)

Optimizes latency and deterministic characteristics of the network node simulation.

  • Activate ERT network nodes
    Activate this check box, to enable the creation of ERT Network Nodes in the CANoe Simulation Setup. ERT network nodes are executed in the ERT area of the configured Vector Hardware and thus benefit from the advantages of the ERT (better latency, better determinism).
  • Selective synchronization of ERT variable updates
    By default the selective synchronization of ERT variable updates is disabled and all variable updates from ERT Network Nodes are synchronized. This might have a negative impact on the realtime performance of ERT. If this option is enabled, the synchronization is limited to all user-defined system variables and system-defined system variables with the Sync ERT Update flag. The Sync ERT Update flag can be specified in the System Variable Configuration dialog

CAN Error Frames

If no other ECU confirms a transmitted message by setting the acknowledge bit, a Not Acknowledge Error Frame is reported and the message will be repeated. If the acknowledge keeps missing this may lead to a burst of not acknowledge messages that impacts the system performance.

  • Filter CAN NACK Error Frames (if transmit error count ≥ 128)
    If the CAN NACK Error Frame filter is active, messages are blocked when the transmit error count of the CAN controller is 128 or larger.
    Remark: Filtered NACK Error Frames will appear neither in the Simulation Setup nor in the Measurement Setup.
  • Flush Tx queue if queue overflows
    If transmitting continues despite missing NACK, this may lead to an overflow of the TX queue. This will be shown in the Write Window. All following send requests will be lost. Activating this option flushes the queue automatically when an overflow occurs.
    Remark: The transmissions that have been placed in the Tx queue will be lost.

VN8970 Throughput Optimization

Here you can configure mechanisms for throughput optimization when using CAN or FlexRay on a VN8900 device with a VN8970 or VN8972 slot. All optimization mechanisms are active as default.

  • Activate Tx buffering for CAN
    Multiple transmit calls of CAN messages will be accumulated and send with one function call to the driver.
  • Activate Tx buffering for FlexRay
    Multiple transmit calls of FlexRay frames will be accumulated and send with one function call to the driver.
  • Activate IRQ reduction for CAN and FlexRay
    The number of the interrupts generated by the interface will be reduced.

Under Accumulation time you can configure the maximum duration during which sent off transmit calls (with active send buffering) and incoming events (with active IRQ reduction)will be buffered. Thus this parameter enables limiting the impact of throughput optimization on the realtime behavior.

Threading in Measurement Setup

Here you can control the parallel processing of logging branches.

Maximum number of threads controls how many additional parallel threads CANoe will use at maximum. By default, it is calculated from the number of processor cores of your computer. How many threads CANoe actually uses, depends on how many thread-safe logging branches exist in your configuration.

There are the following options to configure:

  • When signal values are updated either from a logging branch or from messages on a real or simulated bus, the time when this change would be visible in other branches depends on how far the processing of each thread has advanced in time. To have predictable results, any branch that reads or writes signals, cannot be parallelized.
    The option Parallelize signal access allows to parallelize branches that read signal values (for example using the $ syntax in CAPL programs). Signals are processed in each branch at the correct point in time. Signals on messages that are filtered out before the logging branches (for example with a channel filter) are not visible in the branches. In CANoe versions prior to 8.2, signals were always visible, even if the message carrying it was filtered out or modified.
  • When the simulation is stopped by a trigger in a logging branch, other branches will process some additional messages, before the measurement is stopped. That means that in log files there will be messages with timestamps later than the timestamp of the measurement stop. Disabling the option Parallelize triggers with simulation stop prevents this by preventing branches from being parallelized, when there is a trigger that could stop the measurement.
  • Parallelize CAPL file access controls, whether branches with CAPL nodes that write to files should be parallelized. When multiple CAPL nodes write to the same file, this option ensures that the accessing commands are processed in the order of their corresponding simulation time. This is only relevant when multiple CAPL programs write to the same file.

The options can be configured manually. Alternatively you can choose from one of the following presets:

  • Off (no parallelization of logging branches)
  • Compatible (use threading only when the resulting behavior is identical to CANoe 8.1)
  • Max. performance (use the setting that yield the highest performance)

When the option Output threading information at the start of measurement is active, a summary of how many threads will be used is printed to the Write Window.
