Configuration: Time Source & Event Sorting

Ribbon: File Tab|Options|Measurement|Time Source & Event Sorting

Configure here the simulation time source and the event sorting for hardware and analysis.

Simulation Time Source

If you are using several network interfaces simultaneously (such as a VN1530 and a VN1630), you can configure the simulation behavior here:

Every hardware device usually has its own integrated clock. This clock is used to produce a time stamp for messages, frames, and other data received from the network. In order for these time stamps to match, you should use time synchronization.

In addition, there is an internal simulation time in the program that you can query, for example, with the timeNow function in CAPL. This time should be established by exactly one of the hardware clocks to ensure that it increases evenly and monotonically.
With Reference time is delivered by you select the channel whose hardware clock sets the simulation time.

If you have activated hardware event sorting, this channel is simultaneously the reference channel for sorting. Therefore, the events received on this channel are regularly processed with a time delay according to the Reference delay setting.

If you have deactivated hardware event sorting, only the simulation time for events of the selected hardware is guaranteed to match the time stamp of the event (timeNow() == this.time).

In most cases, you can leave the setting at Auto and this will automatically result in selection of a good time source.

  • Select a different channel, for example, if you want to guarantee that the simulation time for its events matches the time stamp of the event or if very few events are received from the automatically selected channel in your configuration.
  • Select All suitable channels in order to produce the behavior of versions older than Version 7.1. Then, if event sorting is deactivated, the simulation time of every good hardware clock is affected. Select this setting only if required explicitly by your configuration, for example, because your CAPL programs rely on the consistency of timeNow() and this.time.

Hardware Event Sorting

No sorting

Events are not sorted. That means that a received message is immediately passed on without further delay to the nodes in the Simulation Setup from CANoe. If a message with a lower time stamp is received after a message with a higher time stamp, it is also processed after the message with the higher time stamp.

Note

  • Depending on the settings in the analysis event sorting, events in the analysis (e.g. in the Trace Window) can be displayed in a different order than specified with the hardware event sorting.
  • The activated hardware event sorting always takes place before an activated analysis event sorting.

Automatic sorting

CANoe selects a delay time for the sorting depending on the number of configured channels and the hardware. If the configuration uses only one device and one network type, the delay is 0 (not sorting required). You can read the delay times selected in the disabled input fields for manual setting.

Manual sorting

You can enter your own settings here. Only use this option if automatic sorting does not work well enough for your specific hardware and bus configuration.

  • Maximum delay
    Here you set the maximum number of microseconds a received event can be delayed before its processing begins.
  • Reference delay
    Enter here how many microseconds the reference channel events are delayed. Each reference channel event is delayed by this time before it is processed.

Output statistics at measurement stop

When this option is enabled, a statistic is output at measurement stop in the Write Window from which you can read, what share of the events have actually been processed after sorting in the correct order. Since some program-internal events are never sorted, the share is normally less than 100%.

Note

These settings are specific settings of the currently loaded configuration.

Note

  • Incorrect settings can lead to unexpected program behavior.
  • The settings only apply to the concrete configuration and hardware. If you use the configuration with other hardware, CANoe checks the plausibility of the setting and defaults to automatic settings if necessary (if the selected reference channel is no longer assigned to any suitable hardware).
  • The difference between reference delay and maximum delay is important, if you anticipate that a driver delivers events too early compared to the reference channel. Such events are delayed longer than the reference events.
  • The system time is increased in the 1 millisecond cycle only under certain circumstances. The real delay can thereby increase accordingly.
  • Selecting the simulation time source is limited with older drivers (before version 5.4.46).

Analysis Event Sorting

Note

These settings are configuration-independent settings of CANoe.

You can also sort the events for analysis (e.g. in the Trace Window).

The analysis event sorting always tries to sort the events in the order of their timestamps.

Note

Depending on the settings, events can be displayed in the analysis in a different order than specified with the hardware event sorting.

Activate sorting

Here you activate/deactivate the event sorting for the analysis. The set Maximum delay and Reference delay are used for the sorting.

  • Maximum delay
    Here you set the maximum number of milliseconds a received event can be delayed before its processing begins.
  • Reference delay
    Here you set by how many milliseconds all events are to be delayed

    		•

    Example

    • Events that arrive in the correct order according to their timestamp are delayed by the reference delay before they are processed.
    • Events that arrive too late can be pushed into the correct order by applying the reference delay before they are processed.
      Events that arrive much too late can no longer be pushed into the correct order and are not delayed.
    • Events that arrive too early can still be pushed into the correct order by applying the maximum delay.
      Events that arrive much too early are only delayed by the reference delay and are not pushed into the correct order.

Output statistics at measurement stop

When this option is enabled, a statistic is output at measurement stop in the Write Window from which you can read, what share of the events have actually been processed after sorting in the correct order.
