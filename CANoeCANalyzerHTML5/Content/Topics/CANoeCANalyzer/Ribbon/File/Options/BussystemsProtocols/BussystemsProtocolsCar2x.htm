Configuration: Car2x

Ribbon: File Tab|Options|Bus Systems/Protocols|Car2x

The following options are available.

Physical Layer

The choice of the physical layer of the frame determines which technology is used in the construction of the frame and in the wireless transmission:

  • IEEE 802.11p DSRC
  • C-V2X PC5

This setting is valid for all CANoe configurations until it is changed again. Select C-V2X PC5 when using a connected C-V2X device when measuring with CANoe on the real bus.

Station Manager

Assign frames to stations based on

  • Signer HashedID8 and Source MAC Address: Attempt to assign a message to a station primarily by means of the HashedID8. Only if no HashedID8 is available, the tool will attempt to assign the message to a station by means of the MAC address.
  • Source MAC Address:Attempt to assign a message to a station primarily by means of the MAC address.
  • Signer HashedID8: Attempt to assign a message to a station primarily by means of the HashedID8.

Note

There must always be at least one option activated, therefore it is not possible to deactivate all options.

Packet assignment based on station movement

If you activate this check box, an algorithm is used supporting the automatic recognition of a vehicle based on its position data if a certificate change occurs. This helps e.g. to maintain a continuous display in the Map Window as well as to analyze signals in the Graphics and Data Window.

Deactivate the check box to deactivate the automatic recognition. In this case, if a certificate change occurs , the Graphics and the Data Window cannot assign the signal automatically any longer. This means that the "old" vehicle is no longer displayed and a new one is created. However, you still have the possibility to assign the packets manually (see procedures).

Assign HashedID/ MAC addresses

One HashedID can contain various MAC addresses.
Either the MAC address is already existent in the database or a new ITS station brings a new MAC address.
One or various MAC addresses may be changed or deleted, HashedID8s are unalterable.

Troubleshooting

With the Station Manager you may change assignments that were generated based on the vehicle movement. You can find more information about this in the chapter Procedures.

Trace Window

Trace Window display mode

  • Station based: Selecting this option sums up stations with various known certificates to one row in the Trace Window.
  • Packet based: Selecting this option shows stations with various known certificates individually (one row for each certificate) in the Trace Window.

DCC Settings

Overwrite previous unsent messages

Due to DCC rules it’s not possible to send messages at every time. If a message shall be sent, but DCC prevents immediate sending, queuing is necessary.

Overwrite enabled:

Use this setting to only send the latest message of a specific application message (e.g. always only the latest CAM/DENM/BSM, older messages of the same type are overwritten)

Overwrite disabled:

CANoe tries to keep the history of the messages until the limit of the queue. If the queue is full, messages with oldest time stamp and lowest priority are overwritten.

Note

Each ITS station has its own queue and is considered as a separate ITS Station within the DCC. If messages in the queues are deleted, a warning appears in the Write-Window.

Certificate Manager

A public key method is used for secure Car2x communication. This area manages the certificates needed for this, which contain, among other things, the public keys, and the corresponding private keys.

The private keys can be stored in the configuration file either unencrypted or encrypted. To save the keys securely, you activate the check box Use password to secure private keys. This opens the password dialog, in which you must enter the password twice to avoid typing errors. If the keys need to be saved in the configuration file again in plain text, you deactivate this option. Then the password dialog opens to enter the assigned password.

With [Change password...] you open the password dialog, in which you can assign a new password. This requires entering the old password once and the new password twice.

If you have assigned a password, you must enter it whenever you load the configuration, so that the private keys can be used. If you lose the password, the keys cannot be restored. You can then load the configuration without a password, but all private keys are removed from the configuration. When saving this configuration, the private keys are also deleted from the configuration file.

Important Note on Security!

  • The private keys are only protected by the password in the configuration file.

    At runtime, they exist in the main memory as plain text.

    The keys are unprotected in a memory map that is created in a crash, for example.

  • In the distributed mode no password can be entered on the RT Server, therefore the private keys cannot be transferred there encrypted, rather only masked.

    That is why the transmission path must be protected, e.g. by a VPN.

  • In standalone mode no password can be entered on the RT device.

    Private keys are not saved encrypted there, rather only masked.

    Therefore, special measures must be taken for secure access to the RT device.

The table shows all certificates saved in the configuration. Information displayed includes an optionally assigned plain text name, the digest (HashedId8) of the certificate and whether the certificate can be classified as trustworthy. Certificates that do not match the set version of the standard are inactive and are not taken into consideration during the measurement.

With [Add…] you add a certificate to the configuration. This opens the Certificate dialog where you can configure the new certificate.

With [Edit…] or double-clicking on a certificate you open the Certificate dialog to edit the selected certificate.

With [Remove] you can remove a selected certificate from the configuration.

With [Generate EU...] you can create a new ETSI TS 103 097 certificate. In the Generate Certificates dialog you can set the properties of the new certificate.

With [Generate US...] you can create a new IEEE 1609.2 certificate. In the Generate Certificates dialog you can set the properties of the new certificate.

With [Generate CN...] you can create a new China 2020 certificate. In the Generate Certificates dialog you can set the properties of the new certificate.

With [Import...] and [Export...], certificates can be transferred between configurations. When the button is pressed, the Import/Export dialog opens. Here, you can limit the certificates and keys that are to be loaded or saved. A simple XML format is used as the file format.

Certificate Dialog

You can assign a name to the certificate in the Certificate dialog. The certificate is then identified by this name in CAPL or in the Trace Window.

When you activate the Is Trusted Certificate check box, this certificate and all certificates or messages properly signed with it are considered trustworthy. In general, this applies to Root certificates.

The Protocol check box is used to specify whether the certificate involves a certificate according to ETSI TS 103 097, IEEE 1609.2 or China 2020. The version of the standard is taken from the Security Layer area.

The certificate is inserted in the dialog as ETSI TS 103 097, IEEE 1609.2 or China 2020 coded, hexadecimal character string.

If you have a certificate in binary format, you can add it with [Import binary certificate]. The certificate is then also entered as a hexadecimal string. Make sure that the set protocol version matches the version of the binary certificate, otherwise an error message will be displayed.

If messages should be signed with the certificate, you need to enter the private signature key for it. You can enter the key as a hexadecimal character string. When the Certificate dialog is opened later, it is shown in hexadecimal format.

If you have a private signature key in binary format, you can add it with [Import binary key]. The key will then also be entered as a hexadecimal string.

If the certificate also contains a public encryption key, you can specify an associated private encryption key. However, the key is not used.

If you have added a private key in the Certificate dialog and have not assigned a password for protected saving, the password dialog is opened for entering a password when you close the Certificate dialog.

Generate Certificates Dialog

For the generation of certificates, you must first specify various certificate properties.

Property

Description

Subject Type

The type subject specifies the use purpose of the certificate.

Root certificates can always be generated. All other certificate types can only be generated when a parent certificate with matching private key is available.

Subject Name

The subject name identifies the certificate owner in the case of CA certificates. The name remains empty for Pseudonym certificates.

Signer

The signer specifies which certificate is used for signing the new certificate.

For Root CA certificates, this field is not present because these are self-signed. For all other certificate types, only a certificate with known private key can be selected as signer.

Signature Key

The signature key pair is generated for every new certificate.

EU ETSI TS 103 097 V1.3.1 and V1.4.1 only: You can choose between the algorithms NIST-256, Brainpool-256 and Brainpool-384.

US IEEE 1609.2-2016 only: For root CA certificates, you can choose between the algorithms NIST-256 and NIST-384.

Encryption Key

Optionally, an encryption key pair can be generated for a certificate.

EU ETSI TS 103 097 V1.3.1 and V1.4.1 only: You can choose between the algorithms NIST-256, Brainpool-256 and Brainpool-384.

Assurance/Confidence

(only EU: ETSI TS 103 097 V1.1.1 and V1.2.1)

The assurance value must be between 0 and 7, the confidence value must be between 0 and 3.

Application IDs/SSPs

(only EU: ETSI TS 103 097 V1.1.1 and V1.2.1)

The Application IDs specify which message types may be signed with the certificate. In addition, the service-specific permissions (SSPs) can be specified as a hexadecimal character string.

Data PSIDs/SSPs

Application PSIDs/SSPs

Application AIDs/SSPs

(only US: IEEE 1609.2, EU: ETSI TS 103 097 V1.3.1 and V1.4.1, CN: 2020)

The PSIDs specify the data message types for which the certificate can be used. In addition, the service-specific permissions (SSPs) can be specified as a hexadecimal character string.

The Application IDs specify which message types may be signed with the certificate. In addition, the service-specific permissions (SSPs) can be specified as a hexadecimal character string.

Cert Issue PSIDs/SSPs

Cert Issue AIDs/SSPs

(only US: IEEE 1609.2-2016, EU: ETSI TS 103 097 V1.3.1 and V1.4.1, CN: 2020)

The PSIDs specify which rights may be awarded in issued certificates. In addition, the certificate type and the length of the certificate chain can be specified.

The Application IDs specify which message types may be signed with the certificate. In addition, the service-specific permissions (SSPs) can be specified as a hexadecimal character string.

Note

If you want to create an intermediate certificate, you must specify a Cert Issue AID / SSP. An intermediate certificate differs from a pseudonym certificate only by the presence of at least one Cert Issue AID/PSID. If none is set, then only a pseudonym certificate is generated as a result.

WSA PSIDs/SSPs

(only US: IEEE 1609.2-2013 and IEEE 1609.2/D9.3 2011-09)

The PSIDs specify the WSA message types for which the certificate can be used. In addition, the service-specific permissions (SSPs) can be specified as a hexadecimal character string.

The Application IDs specify which message types may be signed with the certificate. In addition, the service-specific permissions (SSPs) can be specified as a hexadecimal character string.

Valid From/Valid To

The validity period of the certificate is limited by the validity start and validity end.

Region

(only EU: ETSI TS 103 097 V1.1.1 and V1.2.1)

The region for which the certificate is to be valid can be specified using the numeric country code according to ISO 3166-1.

Valid Regions

(only US: IEEE 1609.2-2016, EU: ETSI TS 103 097 V1.3.1 and V1.4.1, CN: 2020)

The validity of the certificate can be restricted to selected countries.

In addition to the certificate properties, you can also specify CANoe properties. Unlike the certificate properties, which cannot be changed after the certificate is generated, the CANoe properties can be changed in the Certificate Manager using [Edit...].

  • You can assign a name to the certificate. It is used to identify the certificate in the Trace Window and in CAPL.
  • If you mark the certificate as trusted, this certificate as well as all certificates or messages correctly signed with it are then trusted.

You can generate multiple certificates at a time with Number of certificates. In this case, an index within brackets is added to the certificate subject name and the CANoe name, if present.

Import/Export Dialog

In the Certificate Type area, you can define the type of certificates to be loaded or saved: Root CA certificates, Pseudonym CA certificates, Pseudonym certificates, or other certificates.

In the Private Key area, you can define the handling of the private keys belonging to the certificates, if these are present.

  • For the export, it is possible not to save private keys, to save them unprotected, or to save them password-encrypted. The password specified for this applies only to the exported file and does not change the password that protects the private keys saved in the configuration.
  • For the import, you can specify whether private keys are ignored or whether they are present in the file unprotected or protected. In this case, you must specify the password used to protect the keys. During the import, existing certificates are overwritten. If private keys are imported and a password for the configuration has not yet been assigned, the password dialog opens after the import.

Note

The certificates that are known at the start of the measurement are configured in the Certificate Manager. If you would like to analyze known certificates during runtime, use the Car2x Certificate Explorer.

Security Verification

The Enable Security Verification option activates or deactivates the signature verification of the Car2x frames. The result of the signature verification is displayed separately for each Car2x frame in the detail view of the Trace Window and, in the case of an error, the background color of the frame in the Trace Window will be changed (configurable under CANoe Options|Windows/Blocks|Trace Window|Event Highlighting|WLAN|Errors|Security Error).

The Signature verification is automatically deactivated when performance optimization under CANoe Options|Performance|Optimization is activated.
