Strictness - AFDX protocol violations

Sets the strictness of error messages in the AFDX messages analysis.

The strictness can range over three levels:

  • Low: base checks
  • Medium: additional checks for address errors
  • High: additional checks for MAC address details

The strictness has an effect on the performance of the CANoe system. If the CANoe system is running at full load, it is advisable to reduce the strictness. This reduces the level of information detail accordingly.

AFDX VL Statistics

With the check box Activate you activate or deactivate the calculation of AFDX VL Statistic values.

The statistic values are recalculated every 100 ms based on the so called "rolling average". The epoch duration defines the integration period for average values and standard deviation. The update is carried out independently every second.

The default value for the epoch duration is 1000 ms.

Note

The calculation of AFDX VL Statistic values depends on the state of the Calculate statistics check box on page Measurement | Performance under Bus Statistic.

If the check box is active the state of check box Activate and the epoch duration can be changed. If the check box is inactive the check box Activate and the input box for the epoch duration are inactive too.

AFDX Interactive Generator

The initial standard IP address is used as the preset IP address of an Interactive Generator (IG) that has just been created. Afterward, the IP address can be individually changed and saved in the IG.

To send out AFDX packets, the default IP address must be input in the format 10.x.x.x. The default IP address is 10.0.0.0.

This standard IP address is mandatory if there is no valid source address (0.0.0.0) stored in the database. Alternatively, it can always be forced onto messages generated by the IG.

Logging

Dependent on the configuration, it is possible to generate either Ethernet or AFDX packets from PCAP or PCAPNG files. In contrast to PCAP files, PCAPNG files can contain packets of various channels (Interfaces).

Although the PCAP format can be selected as a log-format for writing, this is not supported for AFDX bus system. PCAPNG should be used instead, because it stores meta info about the channel assignment.

However, reading from PCAP format files is supported for AFDX.

It is possible to replay PCAPNG files for AFDX and Ethernet simultaneously independent of this configuration setting, if the corresponding meta information is available.

However this is different for PCAP-format files: If a change is made to the configuration, either Ethernet or AFDX packets can be generated from the PCAP files only after a CANoe restart. It is not possible to replay PCAP files with different event generation for AFDX or Ethernet simultaneously.

Usually statistic values but no values of statistic system variables are written to different logging files in CANoe. With the Log statistic variables check box the logging of these statistic variables can be explicitly enabled and is then valid for bus statistic and VL statistic values. These are then saved like the user-defined system variables.

This allows a simple offline analysis of the statistic values, e.g. by converting them in a CSV format with the converting tool. Afterwards, they can be processed as similarly described for signal values.

AFDX FDS mode

With the Activate check box in the AFDX FDS mode area, you activate or deactivate the FDS mode (or PDU mode).

The FDS mode is based not on messages but on the contained Functional Data Sets.

The activation of the FDS mode has various effects. For one, the display changes to some extent in the various analysis windows. For another, you change from a bus-oriented view (conventional frame mode) to an application-oriented view (FDS mode).

You can use the same databases (DBCs) for both modes, but all databases must be reloaded following a changeover of this mode. Note that signal names may change on such a changeover.

Activate signal evaluation for Functional State

Each Functional Data Set (FDS) is assigned a so-called Functional Status that can assume four different values.

The Functional Status does not automatically define the validity of the signal contents, i.e., whether or not corresponding signal values are to be decoded and applied.

You can define the validity of the signal contents specifically by activating or deactivating the individual check boxes corresponding to the four valid values of the Functional Status in the area.Activate signal evaluation for Functional State.

  • NO - Normal Operation
  • FT - Functional Test
  • NCD - No Computed Data
  • ND - No Data

You can thus define which FDS is to be ignored in FDS mode due to its functional status.

If you are working with the conventional frame mode, you can also define which signals are not to be displayed for the frame mode by activating or deactivating the check boxes.

These signals are ignored in the Graphics Window, Data Window, etc. In the Trace Window, however, they are always explicitly displayed with a color background depending on the Functional Status.

AFDX Payload Filter (Performance Filter)

With the drop unchanged payload check box in the AFDX Payload Filter area, you activate or deactivate an automatic filtering in which messages or functional data sets with identical signal values are filtered out and their further processing is suppressed. This enables a significant performance increase, especially when signal contents change only rarely.

Statistic calculation is not influenced by this. Events are only displayed in the Trace Window, Graphics Window, and Data Window if the content of the message (frame mode) or the FDS (FDS mode) has changed.

Note

The filter for AFDX payload data relates only to Rx messages.

The following are not filtered:

  • Sent messages
  • Faulty messages

You can also combine the AFDX Payload Filter with the so-called EarlyFilter. In the internal order, the throughput filter precedes the AFDX Payload Filter. The combination of both filters enables you to record all frames but to display and actively analyze online only the frames whose payload has changed. In this way, a subsequent offline analysis on the complete data set is possible at any time.
