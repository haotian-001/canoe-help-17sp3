Configuration: .NET

Ribbon: File Tab|Options|Programming|.NET

.NET Target Framework

  • .NET 4.7 (recommended)
  • .NET 4.0 (version compatible to earlier CANoe versions)
  • .NET 2.0 (version compatible to very old .NET DLLs)

Newly created CANoe configurations use the .NET 4.7 Runtime and the corresponding CANoe API. Existing configurations (created before CANoe version 13.0) use the .NET Target Framework 4.0 with the .NET 4.7 Runtime for compatibility reasons. Even older configurations (created before CANoe version 8.2 SP3) use the .NET 2 Framework and the .NET 2 Runtime.

Information about conversion of configurations to the .NET 4 Runtime can be found here: Migrating from .NET 2 API to .NET 4 API

Type Libraries Creation Path

Here you find the following three options:

  • Create Type Libraries in the database folder
  • Create Type Libraries in a folder for use with the current configuration only
  • Create Type Libraries in a folder shareable with multiple configurations

Classes are generated automatically for access to symbols in .NET. The type libraries are normally stored with the corresponding configuration or database (option 1). However, you can also select a different directory here (option 2 and option 3).

In case the target directory is used with this configuration only, you should select option 2. With this selection the path is stored as relative path to the configuration.

In case you want to share the target directory between multiple configurations, you should select option 3. With this selection the path is stored as absolute path. Furthermore, the Type Libraries are created within subdirectories to allow same database filenames within the different configurations.

Note

Note that when .NET files are compiled, the type libraries are also copied automatically into the output directory of the .NET assemblies (usually the directory in which the .NET file is located).

Type Libraries Selection

Normally type libraries are generated for all symbols. However, you can also select which libraries are to be generated, and thus disable libraries that are not required in particular. This is useful, for example, if there are very large databases in the configuration for which generation takes a long time. Sometimes databases are also internally inconsistent or faulty, so that no type library can be generated for them.

The type libraries are divided according to the source of the symbols. There is one type library per database, another per system variable file, a common library for the system-defined system variables and the system variables stored in the configuration, and a type library for the objects of the application layer.

Qualification Settings for .NET Namespace Generation

If you are using .NET to program test modules or network nodes, classes will be generated from the databases automatically for access to signal values, for example.

To avoid ambiguities, these classes sometimes have to be divided across namespaces. If you have two Status signals in your database, for example, two classes with the name Status will be generated for them. Without further division by namespace, these classes would result in a conflict.

You can use the following namespaces to resolve ambiguities:

  • Network: Name of the network in the Simulation Setup.
    You have to select this namespace, for example, if you are using a database in more than one network.
  • Database: Name of the database.
    You have to select this namespace, for example, if you are using two apparently similar databases in one network. Please note that in this case you will need to change your code if you change the name of the database.
  • Node (for signals only): Name of the send node for a signal.
    You can select this namespace instead of the frame name in order to abstract from the concrete assignment of signals to frames.
  • Frame (for signals only): Name of the frame in which the signal is located.

All four namespaces are located within the uppermost namespace NetworkDB, which is used for all classes generated from databases. If you select more than one namespace, these will be nested in the above sequence. If you do not select any namespaces, signal classes will not only be generated without a namespace but also in the namespace associated with the send node.

Example

If you send two signals called Status from the Engine and Gateway nodes and assign the database in which these signals are defined to both the Comfort and the PowerTrain networks, you should select namespaces called Network and Node. The following four classes will then be generated:

  • NetworkDB.Comfort.Engine.Status
  • NetworkDB.Comfort.Gateway.Status
  • NetworkDB.PowerTrain.Engine.Status
  • NetworkDB.PowerTrain.Gateway.Status

Note

Select namespaces once, prior to writing your programs (this is so that you do not need to make further changes subsequently).

Create Type Libraries Now

With this button you'll get automatically generated classes without any .NET program in the configuration. If you get an error notification due to double defined types you have to use more namespaces.
