Diagnostic Tester

 

To access the diagnostic service of a – real or simulated – control unit with CANoe, please take the following steps:

Basic Steps

Add diagnostic description file (CDD, ODX/PDX, MDX)

  1. Use Configuration|Diagnostics/ISO TP Configuration...|Network to add a diagnostic description file (CDD/ODX) to your configuration. Select the page which corresponds to the Network you wish to send diagnostic requests on.
  2. CANoe uses the diagnostic description file to automatically populate the settings in the pages of the following dialog. If these are not appropriate for your control unit – e.g. because you want the diagnostic communication to use different CAN IDs than those specified in the diagnostic description file – you can adapt the values manually.

Note

If you want to use CANoe to run diagnostic on a network node and you also want to simulate its diagnostic interface, you may still assign only one diagnostic description file to your CANoe configuration for this network node. The settings for both functions are saved in the same diagnostic configuration (a list entry in the Assigned Diagnostic Descriptions field).

Please refer to the following descriptions for a detailed explanation of each individual entry field:

  • Diagnostic Description
    with CDD, ODX/PDX, MDX
  • Basic Diagnostic Description
    with UDS, KWP
  • Standard Diagnostic Description
    with reduced CDD
  • Protocol Settings
  • Diagnostic Layer

Note

The values in the diagnostic description file itself remain unchanged regardless of any manual modifications made in this dialog and its pages. You can restore the default values stored in the diagnostic description file to the dialog at any time by clicking the [Get Defaults] button.

Specify identifiers for the diagnostic configuration (Assignments page)

  • To access the diagnostic services via the Diagnostic Console or the Fault Memory Window:
    The name of the Diagnostic Console or Fault Memory Window is determined by the identifier specified in the ECU(CDD) field.
    You can open these windows at any time via View|<descriptor> Diagnostic Console or View|<descriptor> Fault Memory Window.
  • To access the diagnostic services from within CAPL/XML test nodes or test modules:
  • CAPL
    The DiagSetTarget CAPL function in a test node, test module or test library uses the identifier specified in the ECU(CDD) field to send the diagnostic requests contained in this diagnostic description to the control unit (see also Basic CAPL Procedure for a Tester Implementation).
  • XML
    Several XML test patterns are available. The Diagnostics Service command (ecu="target control unit identifier" parameter) in an XML test module uses the identifier specified in the ECU(CDD) field to send the diagnostic requests contained in this diagnostic description to the control unit. The fault memory test patterns automatically determine the diagnostic services that can be used to access the fault memory.

Select network node or bus to be diagnosed (Assignments page)

  • To diagnose a real control unit on a bus:
    In the Function/Node field select the setting (Send-only).
    This ensures that the diagnostic requests and responses are automatically sent and received on the correct bus, i.e., the bus configured for this network.
  • To diagnose a simulated control unit
    Use Function/Node to select the Simulation Setup node that simulates the diagnostic services via CAPL event procedures on diagRequest (see also Basic CAPL Procedure for an ECU Implementation). As previously, the diagnostic requests and responses are sent and received in the network to which the - in this case simulated - node belongs.

Once you have confirmed these steps by clicking [OK], you can send diagnostic requests to a real CAN control unit via the Diagnostic Console.

Example

You can find sample configurations for using CANoe as a diagnostic tester under CAN Sample Configurations: Diagnostics.

Please see Application Note: CANoe/CANalyzer as a Diagnostic Tool for a detailed introduction to using CANoe for diagnostics.

Next steps for CAPL diagnostic testers and XML test modules

In addition to sending individual diagnostic requests and evaluating the responses interactively via the Diagnostic Console and the Fault Memory Window, you can also create application-specific diagnostic testers with CAPL or an XML test module in order, for example, to

  • send diagnostic requests via keyboard entries or display panels,
  • to process the diagnostic responses or display them on panels in a program-controlled manner, or
  • to check diagnostic services via CAPL or XML test modules or Test Feature Set test case libraries automatically (see also using diagnostic functions in test cases).

Take the following additional steps to create an application-specific diagnostic tester:

Define/create the connection between the diagnostic tester and the communication layer

If you do not require your diagnostic tester to further influence the communication with the control unit on the transport protocol (TP) layer, you do not need to implement a CAPL callback interface. If this is the case, the diagnostic tester is connected to the communication layer using the settings specified for the Diagnostic Console and Fault Memory Window, as described in the previous steps.

Notes on functional addressing

To send functional requests from a diagnostic tester to ECUs in a network you must not implement a CAPL callback interface. Currently the CAPL callback interface does not allow processing of functional requests. Nevertheless you can use both physical and functional addressing in the same tester. To do so you have to assign both a concrete diagnostic description file for each ECU and a diagnostic description file for functional group requests. Inside a tester you can switch to the respective diagnostic description file using the DiagSetTarget command. Doing so selects the channel (physical/functional) on which the subsequent diagnostic requests are sent.

You will need to implement a CAPL callback interface to connect the diagnostic tester to the communication layer in the following cases:

  • the control unit is also to be checked on the level of the transport protocol layer (e.g. injecting and checking TP protocol disturbances via the CanTpFI_-functions of the TP DLL; see also Fault Injection Support in OSEK TP)
  • other application-specific behaviors are desired, e.g. additional transport layer messages in the Write Window
  • To do this you must first assign the corresponding transport protocol functionality to the test node or test module as a modeling library (DLL file) (Test module: Configuration|Module), and then add the callback function to the CAPL code in the node or module. 
  • Please see Connecting the Communication Layer / Reference Implementations and Application Note: CANoe/CANalyzer as a Diagnostic Tool for further information on implementing a CAPL callback interface in order to connect a diagnostic tester to the communication layer.

Create the application or test program in CAPL:

The last step consists of implementing the actual test routines and sequences in CAPL. You can specify the control unit that is to be the target of the diagnosis (DiagSetTarget), send diagnostic requests stored in the diagnostic description to the control unit, and evaluate the responses from the control unit.

Please see Basic CAPL Procedure for a Diagnostic Tester Implementation for detailed notes on how to set up diagnostic testers using CAPL.

If your tester needs to access security locked diagnostic functions of an ECU, you can use a Seed & Key DLL to calculate access keys.
