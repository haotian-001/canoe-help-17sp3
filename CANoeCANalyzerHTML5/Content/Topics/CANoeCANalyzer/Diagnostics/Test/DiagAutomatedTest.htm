Automated Testing of ECU Diagnostics

 

Control unit diagnostic functions can be tested with CANoe in the following ways:

  • Functional testing of diagnostics using CAPL test cases (see Using diagnostic functions in test cases):
    You can use this to insert diagnostic commands into complex test runs and to add diagnostic commands to existing test cases or test modules. Doing so is particularly useful if testing a diagnostic function involves several complex steps.
    See UDS Diagnostic System Configuration.
    Example: Stimulate a physical input on the control unit; start a diagnostic routine with a request; end and read the results of a diagnostic routine with another request; evaluate the response dependent on an input signal.
  • Testing diagnostics using the Diagnostics Service XML test function:
    If the expected response parameters for a diagnostic service are known and fixed (e.g., "Maximum response time for a request < 50ms"), you can use these to test a control unit diagnostic interface without having to program a test control. Numeric and symbolic parameters can be copied directly from the test specification to the test pattern (see also XML test modules structure).
    See CentralLockingSystem sample configuration.
  • Testing fault memory using dedicated XML test patterns for using information from the fault memory for application tests:
    There are three XML test patterns (faultmem_clear, faultmem_status and faultmem_dtcs) available with which the fault memory can be tested. These patterns provide an easy-to-use way for accessing fault memory.
