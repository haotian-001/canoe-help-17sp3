Diagnostics and Testing

 

For test nodes and test modules, there are different methods for diagnostic communications with an ECU, depending on the bus system.

CANoe Diagnostic Channel

If you are using diagnostic description files (CDD/ODX) or Basic Diagnostics, you can send data from a diagnostic tester to the ECU to be diagnosed via a diagnostic channel built directly into CANoe. The transport protocol parameters (e.g. CAN IDs, IP addresses of the tester and the ECU) and diagnostic layer parameters (e.g. P2/P2ext timings) needed for the configuration are either read from the diagnostic description file or use predefined values in the case of default interfaces. These parameters can be overwritten with custom values in the CANoe diagnostic configuration. In the case of diagnostics with FlexRay, Diagnostics over Internet Protocol (DoIP), High-Speed-Fahrzeugzugang (HSFZ) and diagnostics with LIN, such configuration data is drawn from network-specific databases (FIBEX, LDF) or from an INI file.

Using a Transport Protocol DLL in the Tester

One or more transport protocols are available for the CAN, LIN, FlexRay and DoIP/HSFZ bus systems in the form of modeling libraries (DLL files). These can be integrated into test nodes manually (see Components page). The diagnostic layer of the tester is connected to the transport protocol layer via a CAPL Callback Interface (CCI). Sample implementations of these CCIs for diagnostics are currently available for CAN, LIN, FlexRay (FlexRay TP AUTOSAR and FlexRay TP ISO) and DoIP/HSFZ. Because it is possible to make wide-ranging modifications in CAPL code, it can also be used for test cases that require the transport protocol itself to be influenced at test runtime.

Simulated Diagnostic Gateway

The tester can communicate with a target ECU or ECU network via a diagnostic gateway that is simulated in CANoe, i.e. a virtual diagnostic gateway. This gateway is populated with the transport protocols available for all bus systems. It receives the tester's diagnostic requests on one network - e.g. on the CAN bus using the osek_tp.DLL - and forwards them to the target ECU on another network, e.g. the MOST bus using the MOST High Protocol (MOST_MHP.DLL).

This is especially useful in the following cases:

  • A gateway ECU used in the final setup is not available during early development stages or is not available to all diagnostic users e.g. for cost reasons. In this case, CANoe can be used to implement a virtual gateway, either for a tester that is also implemented in CANoe or for an external diagnostic tester.
  • A diagnostic description file cannot be natively assigned to a network or diagnostic connection (e.g. MOST, K-Line, legacy connections). In this case, the diagnostic description is assigned to the network/diagnostic connection "on one side / diagnostic bus" of the gateway (typically CAN or DoIP), and the gateway routes the diagnostic data transparently between the diagnostic and target buses. A diagnostic tester can in this case also work symbolically with diagnostic objects defined in the description file; diagnostic objects in the CANoe windows are interpreted according to the description file.

For more information on simulating gateways in CANoe see the Application Note Diagnostics via CANoe Gateways (PDF).

Further general notes on setting up the CANoe configuration for diagnostics are available in Using CANoe as a Diagnostic Tester and Simulating ECU Diagnostic Services with CANoe.
