Simulating ECU Diagnostic Services

 

Please take the following steps to simulate the diagnostic services of a control unit in CANoe:

Basic Steps

Procedure if your simulation doesn't contain a network node yet:

Procedure to add a Diagnostic Description file (CDD, ODX/PDX, MDX):

  1. Use Configuration|Diagnostic/ISO TP Configuration...|Network to add a diagnostic description file to your CANoe configuration. Select the page which corresponds to the Network the simulated network node belongs.
  2. CANoe uses the diagnostic description file to automatically populate the settings in the pages of the following dialog. If these are not appropriate for your control unit - e.g. because you want the diagnostic communication to use different CAN IDs than those specified in the diagnostic description file - you can adapt the values manually.

Please refer to the following descriptions for a detailed explanation of each entry field.

  • Diagnostic Description
    with CDD, ODX/PDX, MDX
  • Basic Diagnostic Description
    with UDS, KWP
  • Standard Diagnostic Description
    with reduced CDD
  • Protocol Settings
  • Diagnostic Layer

Important Note about changed Features

In a diagnostic description multiple services with the same signature may be used, which differ only in their identifier table. Since the version 8.0 SP4 CANoe always assigns the appropriate service to the diagnostic message. In CAPL simulations, this may result in the fact that other on diagRequest handlers are called than before. CAPL programs that expect that CANoe selects a particular service (this was not assured), must therefore be corrected.

Note

The values in the diagnostic description file itself remain unchanged regardless of any manual modifications made in this dialog and its pages.
You can restore the default values stored in the diagnostic description file to the dialog at any time by clicking the [Reset to defaults] button.

Specify identifiers for the diagnostic configuration (Assignments):

The identifier specified in the ECU qualifier field is used by the Diagnostic Consoles and Fault Memory Windows, as well as by the CAPL/XML test nodes/test modules to select a diagnostic target — and by doing so, to indirectly select the diagnostic interface of the simulated network node that has been saved here.

You can use a diagnostic description for the following use cases:

  1. Add diagnostic interpretation to the trace.
  2. Communicate with an ECU, based on this diagnostic description.
  3. Simulate the diagnostic behavior of an ECU.

All these use cases you can configure in the Diagnostic/ISO TP Configuration dialog in the Usage of the Diagnostic Description area.

Example

You can find sample configurations for using CANoe to simulate control unit diagnostic services under CAN Sample Configurations: Diagnostics.

Connecting to the Communication Layer

You will only have to implement a CAPL callback interface to connect your simulation to the communication layer:

  • if you have to be backward compatible to older CANoe versions up to and including 9.0. SP6
  • if you want to implement fault injection on the transport layer or
  • if you need to implement complex simulations like gateway ECUs:

To do this you need to first assign the corresponding transport protocol functionality to the simulation node as a modeling library (DLL file) (Simulation Setup: Node Configuration|Module page), and then add the callback functions to the node's CAPL code.

For simple ECU simulations, reference CAPL Callback Interface (CCI) implementations are available in the following directory
“%PUBLIC%\Documents\Vector\CANoe\<version>\Reusable\CAPL_Includes\Diagnostics”
(replace <version> in the path by your CANoe version).

Please see Connecting the Communication Layer / Reference Implementations and application notes CANoe/CANalyzer as a Diagnostic Tool and CAPL Callback interface for Diagnostics for further information on implementing a CAPL callback interface in order to connect an ECU simulation to the communication layer.
