Basic Diagnostic Editor: Edit Window

 

ECU

If you have selected an ECU node in the tree view, you can add or delete the Negative Response Codes (NRC) and edit them in the Edit window using Edit. Enter a hexadecimal value (1 byte) for the NRC and a symbolic text for this purpose. This text is then displayed in the Diagnostic Console or in the Trace Window if a negative response is received.

Protocol Service Template

If you have selected a protocol service template in the tree view, all services defined based on this template are displayed. You can add or delete new services using the context menu. When you add a service, the new service is automatically displayed in the Edit window.

Service

Once you have selected a service, you can concretely populate the service by setting the subfunction/local ID and editing the parameters of the request/response.

  • Service Name
    Enter the name of the service in this input box. The name must be unique within the ECU and may only contain the characters [0 to 9, a to Z, _] and no spaces.
  • Request PDU / Response PDU
    Hexadecimal display of the service PDUs.
  • Synchronize subfunction/local ID
    This check box is used to keep the value of the subfunction between request and response synchronous. If you activate the check box, the value of the subfunction under Response is derived from the request and cannot be changed under Response.

Request / Response

On these pages, you can add/edit/delete the request/response parameters for the currently selected service.

After being committed to the Diagnostic Console, the request parameters can be set to concrete values and the service can be sent to the ECU.

The response parameters are used for symbolic interpretation in the Diagnostic Console and Trace Window when receiving a response.

ID Parameters

Via this table you can set the names and the values for subfunction/local ID depending on the protocol service.

Note

The combination of protocol service ID and concrete values for subfunction/local ID clearly identifies a service and may only occur once within the same basic diagnostic ECU.

The possible values for subfunctions/local IDs and their length depend on the respective protocol service.

Example

For the protocol service $10 "DiagnosticSessionControl", only values 0x00 - 0xFF are permissible for the "diagnosticSessionType" subfunction.

With certain protocol services, default values are defined for a subfunction/local ID in the diagnostic protocol. You can select these values from a value table opened with in the Value column.

Parameters

Byte Order
Via this list you can specifies the byte order of all parameters contained in the request/response. This setting is used for all parameters in the currently selected service (Note: Protocol ID and subfunctions/local IDs are always transmitted in MOTOROLA format).

Via the context menu of the table you can add new parameters, delete, cut, copy or paste parameters:

  • You can copy already defined parameters using the parameter context menu and insert them in a different place, e.g. in another service. Multiple parameters can be selected with <Shift> + mouse click or <Ctrl> + mouse click and then copied, inserted or deleted at the same time. When inserted, parameters are always added to the end of the current parameter list.
  • The Align Parameters context menu command compensates for overlaps or gaps in the PDU byte order. Start bits are adjusted if necessary.

The parameters are displayed in the table with the following columns and can be edited there:

  • Start Bit
    Here, you specify the start bit of the parameter within the service. Parameters may not overlap with other parameters, protocol service ID or subfunction/local ID. Adapt the start bit and length [bit] of the parameter if necessary.

Note

Parameters may only be located on byte limits (start bit) and occupy entire bytes (length [bit]).

The last parameter of a parameter list determines the effective length of the transmitted service.

If you allow gaps in the parameter list, they are automatically displayed as gap parameters in the Diagnostic Console.

Note about the settings for the start bit position

The start bit position refers to the start of the request/response PDU.

When you add new parameters to the parameter list using the context menu, the parameters are added to the end of the list and the start bit is adapted automatically. In the case of subsequent changes to the length of a parameter, you must adapt the start bit of the following parameters manually. The INTEL/MOTOROLA byte order must also be taken into account here:

INTEL: ascending counting

Example: Service ID (1Byte / Bit 0..7) Subfunction (1Byte / Bit 8..15)

Parameter1 (1 byte / Bit 16..23 / Start bit 16)

Parameter2 (2 Byte / Bit 24..39 / Start bit 24)

...

MOTOROLA: "Sawtooth" counting

Example: Service ID (1Byte / Bit 0..7) Subfunction (1Byte / Bit 8..15)

Parameter1 (1 byte / Bit 16..23 / Start bit 23)

Parameter2 (2 Byte / Bit 24..39 / Start bit 31)

...

For this reason, we recommend setting a parameter's length in bits after it is inserted and before the next parameter is inserted.
  • Name
    Here, you specify the name of the parameter. The same parameter name may only occur once within the request/response population of a service.
  • Default Value (Request only)
    Here, you specify the default value of the parameter. This default value is sent along with the request from the diagnostic console if it is not overwritten with another value before sending occurs. Parameters that do not have a default value are initialized with 0x00 based on their length.
  • Value Type
    Here, you specify the value type of the parameter. With certain value types, the length in bits is fixed (e.g. float: 32 bits).
  • UINT, INT, FLOAT, DOUBLE – simple numerical data types
  • ARRAY – raw byte array
  • ASCII – ASCII string
  • Conversion
    Here, you specify the conversion for the parameter. Depending on the value type, some conversions may not be available.

With the formula conversion type, a linear conversion of the raw value can be specified.
With the value table conversion type, a table can be stored with symbolic values.

  • Length [Bit]
    Here, you specify the length of the parameter in bits. Parameters may not overlap with other parameters, protocol service ID or subfunction/local ID. Adapt the start bit and length [bit] of the parameter if necessary.
