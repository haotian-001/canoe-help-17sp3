Diagnostic Objects in CAPL

 

CAPL programs that communicate with ECUs via diagnostic services will use DiagRequest and DiagResponse objects typically. The following operations can be performed on these objects:

  • The value of parameters in the objects can be read and changed. This is possible in different modes. Details: DiagGetParameter, DiagSetParameter, etc.
  • Objects can be sent, where a tester may only send request objects, and an ECU simulation may only send response objects. Details: DiagSendRequest, DiagSendResponse
  • CAPL programs can react on the reception of a diagnostic object by implementing a matching handler function. Details: On diagRequest, On diagResponse
  • CAPL programs can react on the send confirmation for a diagnostic request by implementing a matching callback function. Details: On diagRequestSent
  • Tester programs can access a DiagResponse object via a DiagRequest object, if the former was received using the latter. Details: DiagGetRespParameter, etc.
  • DiagRequest and DiagResponse objects can be used as arguments for CAPL functions. Details: Data Types for Function Parameters

Declaration and Initialization 

Diagnostic objects are declared and initialized as follows:

Syntax with ECU qualifier in DiagRequest/Response objects:

DiagRequest EcuQualifier.ServiceQualifier request;

DiagResponse EcuQualifier.ServiceQualifier response;

Obsolete syntax:

DiagRequest ServiceQualifier request;

DiagResponse ServiceQualifier response;

This declares the CAPL variables request and response, and initializes them with the request or the response primitive of the given service. For the obsolete Syntax, the initialization is performed every time DiagSetTarget is called in a tester program, and once during measurement start in ECU simulations.
Hints regarding overloaded response primitives in ODX can be found at Diagnostic Descriptions with Overloaded Responses.

If an asterisk * is given as ServiceQualifier, the object is initialized as raw without using the diagnostic description. The object's bytes must be set before sending it.

Example for the Hand Over of Diagnostic Objects to Functions

variables
{
  byte StatusByte=0x0;
  mstimer RepeatTimer;
  diagRequest Door.FaultMemory_ReadAllIdentified req; // "FaultMemory_ReadAllIdentified" is a service defined for the ECU "Door"
}

on diagResponse Door.FaultMemory_ReadAllIdentified
{
  EvaluateResponse(this);
}

void ReadAllIdentifiedByStatusByte (diagRequest * req, byte StatusByte)
{
  if (DiagSetParameter(req, "DtcStatusbyte", StatusByte) < 0) write("*** Error calling DiagSetParameter!");
  if (DiagSendRequest(req) < 0) write("*** Error calling DiagSendRequest!");
}

void EvaluateResponse(diagResponse * resp)
{
  // Do some evaluation here using the response, e.g. xy=DiagGetPrimitiveByte (resp, 2);
  // ...
}

on timer RepeatTimer
{
  // Send a Diagnostic Request for each possible value of "StatusByte"
  if (0xFF==StatusByte) cancelTimer(RepeatTimer);
  ReadAllIdentifiedByStatusByte (req, StatusByte);
  StatusByte++;
}

on key 'a'
{
  // Initialize the timer to send a Diagnostic Request at each timer event
  SetTimerCyclic(RepeatTimer, 0, 200);
}

Read & Write Complex Diagnostic Parameters

The CAPL notation for accessing structured diagnostic parameters, especially for bit fields, slightly differs between CANdela files (CDD) and ODX/MDX files:

  • If a request/response is generated using a CANdela file, elements of a structured parameter are accessed using a "." operator.
  • If a request/response is generated using an ODX/MDX file, elements of a structured parameter are accessed using a "/" operator.

Example

Fault memory entries

// CDD Notation uses "." operator
...
DiagGetComplexParameter(faultMemResponse,"LIST_OF_DTC_AND_STATUS", i, "DTCStatusByte.TestFailed");
...
DiagGetParameter(deviceControlResponse,"Door_contact.Door_contact_rear_left")
...
// ODX notation uses "/" operator
...
DiagGetComplexParameter(faultMemResponseODX,"LIST_OF_DTC_AND_STATUS",i,"DTCStatusByte_STRUCTURE/TestFailed");
...
DiagGetParameter(deviceControlResponseODX,"Door_contact_STRUCTURE/Door_contact_rear_left");
...

Note

These examples can also be found in the UDS Diagnostic System Configuration.

Support in the CAPL Browser

The CAPL Browser offers support for entering qualifiers of diagnostic services or other diagnostic symbols. When using the syntax with ECU qualifier in DiagRequest/Response objects, you will be able to use the Autocomplete Input Assistance:

Alternatively, you may

  • Select in the Symbol Explorer the view Diagnostic Classes or Diagnostic Services and Drag and drop diagnostic symbols from the Symbol Explorer.
  • Press the right mouse button in the input box and then select one of the sub menu commands of Insert Diagnostic Symbol.
  • Use one of the key combinations indicated in the same context menu. The respective symbols defined in the loaded diagnostic descriptions are displayed and can be inserted in the source code with a double click.
