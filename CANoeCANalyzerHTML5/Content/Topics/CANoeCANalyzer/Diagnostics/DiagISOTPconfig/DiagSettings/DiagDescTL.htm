Parameters from diagnostic description for transport and diagnostic layer

Override manually

If the diagnostic description has an interface matching to the network, the default parameters from the interface of the diagnostic description are used. These parameters (see below) you can adjust manually, after activating the Override manually check box.

For basic diagnostic descriptions the parameters can be overwritten any time.

Note

Note that you cannot change this setting in the following cases:

  • An interface using a transport protocol other than ISO TP is selected.
  • The description is loaded on a non-CAN network.

Reset to Defaults

Via the [Reset to defaults] button, you can set back your manual changes (see above) to the parameters of the used diagnostic description.

Addressing

The provided parameters are listed in a table. You can change the CAN ID and activate or deactivate the 29 bit identifier via the check box.

For the communication between testers and control units (ECUs), several "transport channels" are used:

  • Physical requests
    The tester sends these to exactly one ECU.
  • Physical responses
    The response to a physical request.
  • Functional requests
    The tester sends these to a group of control units, whereby the target address specifies this group (e.g. "all"). Typically control units respond either physically or not at all.
  • Multiple responses (OEM specific)
    Some manufacturers allow ECUs to send additional "UUDT" responses (unacknowledged unsegmented data transport) to a request, e.g. at intervals of a second.

Depending on the manufacturer and class, various address modes are used for these transport channels. CANoe now permits the changing of the parameters of these modes.

Additional ISO TP protocol parameters

In addition, protocol parameters of the ISO TP can be set here. This is relevant for CAPL programs and the access to the ECU via the consoles.

  • STmin
    This specifies the minimum interval (in ms) between consecutive frames (CAN messages of the transmitter).
  • Block size
    After so many consecutive frames, the transmitter awaits a flow control frame from the receiver.
    0 switches this off.
  • FC delay
    The tester will delay the sending of a flow control frame for this many ms.
    Delaying the frame may be necessary to ensure that the ECU is ready to receive it.
  • Max. length
    Transmissions shall have this maximum length in bytes. With ISO 15765-2 (2015) the maximum can be increased to 4 GiB, while CANoe supports up to 16 MiB at the moment. A value > 4095 has to be given here to activate the new header format. When a transmission with larger length is received, it is aborted with an overflow error.
  • The following two functions are only available if you have activated the CAN FD mode via the Hardware ribbon tab|Network Hardware:
  • DLC
    With this list you decide, if the transport protocol for diagnosis will use CAN 2.0 messages only (CAN 2.0) or CAN FD messages with the corresponding maximum length (FD <= 8 up to FD = 64). As soon as you select a CAN FD entry the Enable BRS check box appears.
  • Enable BRS
    With this check box you activate the CAN FD Bit Rate Switch, i.e. the transport protocol uses the higher bitrate during the data phase of the CAN FD message.

Mixing of CAN 2.0 and FD frames

For certain scenarios the build-in diagnostic tester shall be able to handle CAN 2.0 and FD frames with the same communication channel, i.e. instead of configuring 2 diagnostic descriptions for CAN 2.0 and CAN FD, only one description shall suffice. The tester might have to adapt to the frame type received by the ECU:

  • ignore
    Default. The tester ignores CAN frames that do not match the configured type.
  • accept
    The tester will accept and process frames of the other type, but continue to send with the configured type.
  • adapt
    The tester will not only accept and process frames of the other type, but also continue to send using this type.
