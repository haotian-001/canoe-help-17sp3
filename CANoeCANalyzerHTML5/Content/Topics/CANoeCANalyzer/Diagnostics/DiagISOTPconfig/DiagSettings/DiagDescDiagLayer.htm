Diagnostic Layer

Ribbon: Diagnostics & XCP Tab|Diagnostics/ISO TP Configuration|Administrate Diagnostic Descriptions|Assignments|Diagnostic Layer

Here you can set the behavior of the consoles and specify parameters for CAPL programs. These parameters can be reset during measurement in CAPL.

Parameters from diagnostic description for transport and diagnostic layer

Override manually

If the diagnostic description has an interface matching to the network, the default parameters from the interface of the diagnostic description are used. These parameters (see below) you can adjust manually, after activating the Override manually check box.

Note

Note that you cannot change this setting in the following cases:

  • An interface using a transport protocol other than ISO TP is selected.
  • The description is loaded on a non-CAN network.

Reset to Defaults

Via the [Reset to defaults] button, you can set back your manual changes (see above) to the parameters of the used diagnostic description.

'Tester Present' request

Many ECUs must receive such a request in order to remain in the (expanded) diagnostic session. Using this dialog settings can be changed, which control the sending of a Tester Present Service for the Diagnostic Console, the Fault Memory Window and such diagnostic testers, which do not use a CAPL callback interface for diagnostics.

Note

If within a tester a CAPL callback interface for diagnostics is used, definition of the request content and cyclically sending it (e.g. by use of a timer) must be taken care of in the tester as well. In this case the dialog's settings will not be used.
  • Send tester present
    Here you decide whether a tester present should be sent cyclically to the ECU at all. If selected, a Tester Present is only sent after the measurement has been started and once a diagnostic session has been established with the ECU by sending one diagnostic request. To stop sending cyclical Tester Present requests either
  • Toggle the (Tester Present on/off) button in the Diagnostic Console or in the Fault Memory Window, or
  • In a CAPL tester call the DiagStopTesterPresent() function.

Note

  • Tester present is always off at the start of measurement.
  • If Send tester present is active (Diagnostics Configuration|Diagnostic Layer), it will be switched on as soon as a request is being sent.
  • When starting a COM client, tester present will neither be switched on nor switched off (prior state remains).
  • When a COM client ends, tester present is switched off. After that, tester present remains switched off even if Send tester present is active and a request is being sent.
  • S3 client time [ms]
    Time the diagnostic client (=tester) should wait before sending a tester present request.
  • S3 server time [ms]
    Timeout in the diagnostic server for leaving a non-default session.

Note

This time must be larger than S3 Client Time. If a value lower or equal is entered, 1.25 * S3 Client Time is used.

Note

For networks that were configured for DoIP via the Enable diagnostics via common DoIP gateway for this network option, you find as parameter for timing See Gateway:. In this case the used values are listed under
Gateway to <Network Name>|Node-Specific Settings.
  • From description
    From the Tester Present services defined in the diagnostic description file, select the one to be sent. If no Tester Present service should be defined in the diagnostic description file, always a default request may be selected.
  • Manually defined
    The Tester Present service can be defined by manually entering a byte stream (e.g. 0x3e 0x03 0x04).

Timing

These parameters determine the timing conditions for diagnostic responses, differentiated by tester (= client) and ECU (= server):

  • Client Timings:
  • P2 Client
    After sending the request, the ECU must react within this time (in ms), that is send either a positive or negative message insofar as the tester expects a response for the request.
  • P2 Client Extended
    If the ECU sends a negative response with the error code 0x78 "Response Pending" then the tester is waiting once again for a response, whereby this time (in ms) is used.

If diagnostics over IP (DoIP) is used, P6 instead of P2 parameters are used:

  • P6 Client
    After sending the request the response from the vehicle must be received within this time.
  • P6 Client Extended
    If the vehicle sends a negative response with the error code 0x78 "Response Pending", the tester waits this time for another response.

Note

On CAN the behavior of CANoe after reception of a negative response with response code 0x21 busy - repeat request can be configured: if active, CANoe as a tester will repeat the request after the configured time, but not longer as for the complete within time.
  • Server Timings:
  • P2 Server
    Determines the maximum amount of time (in ms) during which the ECU has to start with the diagnostic response after the reception of a diagnostic request.
  • P2 Server Eextended
    Determines the maximum amount of time (in ms) during which the ECU has to start with the diagnostic response after the ECU has sent a negative diagnostic response with negative response code 0x78 (enhanced response timing).

Security Access

Seed & Key DLL

If you are using diagnostic services for whose execution the ECU is to be unlocked, you must specify a DLL here:

Link to Seed & Key DLL File

  1. Diagnostics & XCP ribbon tab: Diagnostics/ISO-TP.
  2. In the Dialog select the network and the ECU.
  3. Click on Diagnostic Layer.
  4. Via the Button with the three points link to the location of the Seed & Key DLL.
  5. Commit with OK.

Further information you find on page Seed & Key DLL / Security Access.

Diagnostic Console

  • Prevent constant parameters from being overwritten unintentionally
    If checked, the Diagnostic Console will not allow changing the value of request parameters that are defined constant.
    If such a parameter must be changed there, deactivate the check box.

With [Reset to defaults] you reset all settings back to the values from the diagnostic description.
