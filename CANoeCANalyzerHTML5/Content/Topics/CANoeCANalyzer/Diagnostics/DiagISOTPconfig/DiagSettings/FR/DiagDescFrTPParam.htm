FrTP Parameter

Ribbon: Diagnostics & XCP Tab|Diagnostics/ISO TP Configuration|Administrate Diagnostic Descriptions|Assignments|FrTP Parameter

Here you can configure the Transport Protocol that is used in order to send the diagnostics requests.

Note

Per default, those settings will be read from the assigned CDD file and its interface definition (see DiagnosticsDescription Settings: Assignments | Interface), but the settings from the CDD file can be overwritten with this dialog page.

Parameters from Diagnostic Description for Transport and Diagnostic Layer

Override manually

If the diagnostic description has an interface matching to the network, the default parameters from the interface of the diagnostic description are used. These parameters (see below) you can adjust manually, after activating the Override manually check box.

FlexRay Transport Protocol

Here you can select the protocol that is to be used and its parameters. Normally these settings are taken over from the interface definition of the assigned CDD file (see Diagnostic Description: Assignment|Interface). These settings can be overwritten here.

FlexRay-ISO-TP-CAPL-Functions or FlexRay-CAPL-Functions

Protocol type

Currently the following transport protocols for FlexRay are supported:

  • FlexRay ISO 10681-2 TP
  • AUTOSAR FlexRay TP 2.x / 3.x
  • BMW FlexRay TP

Note

  • For using the AUTOSAR FlexRay TP the AUTOSAR FlexRay Support Package must be installed.
  • For using the BMW FlexRay TP the BMW FlexRay TP Support Package must be installed.

For using these transport protocols in CANalyzer, please contact the Vector support.

According to the selected protocol type different settings are required:

ISO 10681-2 TP

This protocol can work on FlexRay frames or on AUTOSAR PDUs.

Parameter

Description

ECU Address 2

Sets optionally 2nd ECU address that is used for incoming or outgoing connection data.

This allows e.g. in the context of the Tester to receive responses from the ECU from 2 different (remote) TP addresses. In the context of the ECU (simulation) this allows to accept/receive requests that are directed to 2 different (local) TP addresses.

The usage of this 2nd address must explicitly be enabled by the check box behind the input field.

Valid value range: [0x0000 – 0xFFFF] 16 Bit

Connection Mode

With this setting the acknowledging and retry operation modes of the protocol can be activated.

Sep. Cycles (SC)

This is a bandwidth control (BC) parameter and specifies the minimum number of cycles the sender has to wait for the next transmission of a C-PDU.

In case that no bandwidth control is required this parameter may be set to 0. (Sent as fast as possible)

Valid value range: (0, 1, 3, 7, 15, 31, 63, 127)

Num. Frames (MNPC)

This is a bandwidth control (BC) parameter and specifies the maximum number of C-PDUs the sender is allowed to send within one cycle.

In case that no bandwidth control is required set parameter SC to 0.

Valid value range: [0, 31]

Buffer Size (BfS)

This parameter indicates to the sender the maximum number of data bytes that can be received without an intermediate FC frame from the receiving network entity.

The BfS parameter value zero (0) shall be used to indicate that no more FC frames shall be sent during the transmission of the segmented message.

Valid value range: [0, 232 - 1]

 

AUTOSAR FlexRay TP 2.x / 3.x

This protocol can work on FlexRay frames or on AUTOSAR PDUs.

Parameter

Description

Connection Mode

With this setting the acknowledging and retry operation modes of the protocol can be activated.

Message Length

This specifies the maximum length a transport protocol's communication unit (C-PDU) can have. Implicitly this specifies the maximum amount of data bytes that can be transferred with one segmented message transfer:

  • ISO (< 4096 Byte)
  • ISO6 (< 4096 Byte)
    Additionally the maximum amount of user bytes per one C-PDU is limited to 6 Bytes
  • L16M (< 224 Byte)
  • L4G (< 232 Byte)

Format

Here you can select one byte or two byte addressing.

STmin

This is a flow control (FC) parameter and specifies the minimum time [ms] the sender has to wait for the next transmission of a C-PDU.

In case that no flow control is required this parameter may be set to 0. (Sent as fast as possible)

Block Size (BS)

This parameter indicates to the sender the maximum number of C-PDUs that can be received without an intermediate FC frame from the receiving network entity.

The BS parameter value zero (0) shall be used to indicate that no FC frames shall be sent during the transmission of the segmented message. (Sent as fast as possible)

Valid value range: [0, 255]

 

BMW FlexRay TP

This protocol can work only on FlexRay frames.

Parameter

Description

Receiver Type

With this setting the Single Receiver (physical/unicast network requests) or Group Receiver (functional/multicast network requests) mode is activated.

Bandwidth Ctrl. (BC)

Either a frame-oriented or a cycle-oriented bandwidth control (BC) can be selected.

Sep. Cycles (MNSC)

This parameter is only valid if a cycle-oriented BC is chosen.

The parameter specifies the minimum number of cycles the sender has to wait for the next transmission of a C-PDU.

In case that no bandwidth control is required this parameter may be set to 0. (Sent as fast as possible)

Valid value range: [0, 9].

Num. Frames (MNFC)

This parameter is only valid if a frame-oriented BC is chosen.

This parameter specifies the maximum number of C-PDUs the sender is allowed to send within one cycle.

Valid value range: [2, 127].

Block Size (BS)

This parameter indicates to the sender the maximum number of C-PDUs that can be received without an intermediate FC frame from the receiving network entity.

The BS parameter value zero (0) shall be used to indicate that no FC frames shall be sent during the transmission of the segmented message.

Valid value range: [0, 255]

Padding byte

The remaining space inside of an L-PDU (FlexRay Frame or AUTOSAR PDU) that is not used by the definition of the maximum size of a C-PDU of the transport protocol will be filled with the appropriate byte, if the padding is enabled.

When the transport protocol is using AUTOSAR PDUs as message containers, then the maximum C-PDU size is normally equal to the size of the AUTOSAR PDU. Thus, filling/padding will not take place.

When the transport protocol is directly using FlexRay frames, then all remaining bytes after the transport protocol's service data unit (C-PDU) will be filled. Even if padding is disabled and the size of the C-PDU is odd, the last byte of the FlexRay frame (which has always an even payload size) will be filled.

Addressing

Tester

This specifies the address in the protocol header of the transport protocol that is used of CANoe/CANalyzer as diagnostic tester. With this diagnostic tester CANoe/CANalyzer is sending the requests or receiving the responses.

ECU

This specifies the address in the protocol header of the transport protocol for the target of the diagnostic requests (and the source of the diagnostic responses).

Functional

This specifies the address for the functional channel.

FlexRay Slot/PDU Assignments

Currently the diagnostic implementation uses for sending requests and for receiving responses exactly one frame/PDU per direction. In the FlexRay Slot/PDU assignments section you can select appropriate frames/PDUs from the FIBEX database via the displayed check boxes.

Note

Note that you can only select those frames/PDUs, which are sent/received to/from the selected node that is the target for the diagnostic communication (see Diagnostic Description Settings: Assignments | Function/Node).

Only those frames/PDUs will be listed for selection that have the following service type:

  • TPL
  • DIAG-REQUEST
  • DIAG-RESPONSE
  • DIAG-STATE

Tester node

Show only selected

Timing

On this page you define timing parameter that handle the distance between two C-PDUs for transmission (time) or reception (timeout):

  • Timeout As
    Maximum tolerated time [ms] between the request to send a FlexRay frame and its transmission (Tx acknowledge from bus) on the sender side.
    Default: 125 ms
  • Timeout Ar
    Maximum tolerated time [ms] between the request to send a FlexRay frame and its transmission (Tx acknowledge from bus) on the receiver side.
    Default: 125 ms
  • Timeout Bs
    Timeout [ms] for the reception of a flow control frame on the sender side.
    Default: 150 ms
  • Time Br
    Time [ms] between receiving a frame from the sender and sending out a flow control frame by the receiver, if necessary.
    Default: 25 ms
  • Time Cs
    Time [ms] between the sending of two frames by the sender (25 ms).
  • Timeout Cr
    Timeout [ms] waiting for a frame from the sender after receiving a consecutive frame or sending a flow control frame.
    Default: 150 ms

Note

Please note that the ISO TP parameters have to follow certain restrictions to allow correct behavior of the transport protocol!

For example, the following has to hold:

TimeBR + TimeoutAR ≤ TimeoutBS
TimeCS + TimeoutAS ≤ TimeoutCR
