Standard Diagnostic Description with reduced CDD

Ribbon: Diagnostics & XCP Tab|Diagnostics/ISO TP Configuration|Administrate Diagnostic Descriptions|Standard Diagnostic Description with reduced CDD

If you click in the Diagnostics/ISO TP Configuration dialog in the tree view in level 3 on a standard diagnostic description you see this sub dialog:

Cross Reference

Default Functions to Configure Diagnostic Description

For the standard diagnostic description is in CDD format, you only need to add one diagnostic description file, as these contain all communication parameters.

For standard diagnostic descriptions the Common variant is only available.

Further descriptions for the functions in this dialog you find here: Diagnostic Description with CDD, ODX/PDX, MDX.

Diagnostic Description

ECU Qualifier

Here the ECU qualifier is displayed which will be used by CANoe to refer to the current diagnostic description. The qualifiers for each diagnostic description configured in a CANoe configuration must be unique.

When a description file is added, as default the ECU qualifier as defined in the description will be used. Yet it is also possible to manually overwrite this qualifier here. The qualifier name may only contain letters A…Z and a…z, numbers and underscores _.

Furthermore it is ensured that unique qualifiers are assigned to all ECUs.

Note

In particular this has effects on CAPL programs.

When you configure the target ECU (DiagSetTarget or DiagInitECUSimulation) you have to specify the correct qualifier!

It is also the name used in the Diagnostic Console, Diagnostic Session Control the Fault Memory Window and the Symbol Explorer.

File

Path of the currently assigned diagnostic description. Via the button with the three points you can change the diagnostic description.

Use Cases

Standard diagnostic descriptions are especially useful when:

  • the diagnostic protocol (KWP, UDS) as well as the essential diagnostic communication parameters (such as CAN identifier and timeout times) of the ECU are known
  • there is no specific diagnostic description for the relevant ECU

Possible CDD Types

Depending on the network used and the diagnostic implementation of the ECU, you can choose between:

  • GenericKWP (for Keyword Protocol 2000-based implementations)
  • GenericUDS (for Unified Diagnostic Services-based implementations)
  • GenericUDSLIN (for Unified Diagnostic Services-based implementations for LIN networks)

Check Settings

The default settings, e.g., for CAN IDs or timeouts, in the template represent only values chosen by way of example and typically do not agree with those of the ECU used. Because of that please check the main settings after adding the standard CDD for ECUs in CAN networks:

  • Protocol Settings
  • Diagnostic Layer (Seed & Key DLL)

Available Options

Interface

Interfaces in diagnostic description files define the communication parameters for accessing an ECU. Typically not all access types are defined in a definition file e.g. for some bus systems they might be missing.

To allow the assignment of description files even in such cases, CANoe starting with version 7.5 offers default interfaces which can be selected in the configuration dialog (with the prefix {generated}).

Hence it is possible to use a diagnostic description file on all supported bus system and diagnostic access types (e.g. DoIP/HSFZ, K-Line).

Default interfaces for supported bus systems are also offered in case of basic diagnostic ECUs.

These default interfaces are available for the different Addressing Schemes.

Note

Please mind that older versions of CANoe cannot make use of default interfaces in configurations. Upon saving a configuration in a predecessor format, the default interfaces will be removed.

Note on Diagnostics Using the Vector CANoe VAG Package

If you have installed Vector CANoe VAG package version 2.0 (or higher), you can select the VWTP 2.0 (CANoe) interface in addition to the interfaces defined in the diagnostic description. In this case you can configure the settings for the transport protocol layer on the Diagnosics Description Settings: Transport Layer VW TP page.

For using this package in CANalyzer, please contact the Vector support.

Variant

If a description contains more than one variant, you will need to specify a default variant before measurement start. The variant determines which services and parameters are available for diagnostics and communication.

Language

A diagnostic description file can contain several language versions of which one has to be selected. Note, that some diagnostic descriptions only support one single language.

Target group

You can define so called "target groups" (e.g. "Developer", "Supplier", "Workshop"). Each instance and service of a diagnostic description can have one or more target groups assigned. By doing so you can provide tailored information.

  • When you select one of these target groups the Diagnostic Console will only show the services that have been activated for the specific target group.
  • If no target group is selected, all services will be shown.

Note

These settings do not affect CAPL programs.

Usage of the Diagnostic Description

Activate the option which fits to your use case(s), i.e. specify what you want to do with your diagnostic description:

  1. Add diagnostic interpretation to the trace.
  2. Communicate with an ECU, based on this diagnostic description.
  3. Simulate the diagnostic behavior of an ECU.

For the first use case, i.e. to use the diagnostic observer feature of CANoe, it is suffcient to simply add a diagnostic configuration in the Diagnostics/ISO TP Configuration dialog without activating any additional check boxes there. No diagnostics windows (Diagnostic Console, Fault Memory or Session Control) will be provided. (This use case was formerly provided with the interpretation only mode).

For the second use case, i.e. configuring the tester side of the built-in diagnostic channel, you need to activate the Diagnostic tester check box and select the desired addressing type for it. This will provide a Diagnostic Console, Fault Memory and Session Control Window in CANoe. In CANoe, this diagnostic description will be available for Test Modules or Test Units as well. The following addressing types are available:

  • Physical Requests:
    This is the typical way of diagnostic communication to an ECU
    In the case of a real ECU on a bus, the diagnostic requests from this description are transmitted and received on the currently selected network.
  • Functional Group Requests
    Only Diagnostic Console and Fault Memory Window available. Sending functional requests to the group.
    The diagnostic description is used to send functional requests to all ECUs on the network. The functional IDs defined in the diagnostic description (unless manually overridden) are used for this purpose.

Note

Depending on whether or not the diagnostic description contains information about fault memory model and session model, the Fault Memory and Session Control Window might not be available.

For the third use case, i.e. configuring the simulation side of the built-in diagnostic channel, you need to activate the Simulation by check box and select the desired simulation node for it. This will activate the simplified diagnostic simulation in CANoe, i.e. CANoe will respond to a diagnostic request to the simulated ECU with a positive response, with each parameter value in the response set to 0 (zero). You can modify the response behavior of the simulated ECU by adding on diagRequest handlers/event procedures in CANoe.

This is explained in more detail on the Basic CAPL Procedure for an ECU Implementation page.

Note

If a simulation node with the name (ECU Qualifier) of the ECU of the diagnostic description already exists when the diagnostic description file is loaded this node is automatically assigned to the diagnostic description.
This setting can be changed manually.
