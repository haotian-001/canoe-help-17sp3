DoIP/HSFZ Settings (Node)

Ribbon: Diagnostics & XCP Tab|Diagnostics/ISO TP Configuration|Diagnostics over IP/High-Speed-Fahrzeugzugang/Vehicle Ethernet|Administrate Diagnostic Descriptions|Assignments|Settings (Node)

Here you set the node specific parameters for communication, that are used for sending and receiving diagnostic requests and responses over a DoIP/HSFZ Gateway.

Parameters from diagnostic description for transport and diagnostic layer

Override manually

If the diagnostic description has an interface matching to the network, the default parameters from the interface of the diagnostic description are used. These parameters (see below) you can adjust manually, after activating the Override manually check box.

Reset to Defaults

Via the [Reset to defaults] button, you can set back your manual changes (see above) to the parameters of the used diagnostic description.

Node Specific Settings

If you activate the Enable diagnostics via common DoIP/HSFZ gateway for this network check box the DoIP/HSFZ settings for every single ECU behind the gateways can be adjusted here.

Interface

Here you can select the DoIP/HSFZ interface which is to be used for diagnostics of the ECUs behind the gateway, from the diagnostic description of the ECU.

By default the interface {generated} DoIP/HSFZ is used, whose parameters you can change manually. This default interface is also available if no other valid DoIP/HSFZ interface should be defined in any of the CDD/ODX diagnostic description files assigned to the network or for an ECU.

Log. address (phys / func)

This is the logical address of the ECU for physical (phys) or functional (func) diagnostic requests, in conformance with the DoIP/HSFZ protocol.

'Tester Present' Request

  • S3 client time [ms]
    Time the diagnostic client (=tester) should wait before sending a tester present request.
  • S3 server time [ms]
    Timeout in the diagnostic server for leaving a non-default session.

Further details you find here: Settings Diagnostic Layer

Timing

These parameters determine when the tester takes on a timeout of the ECU:

  • P6 timeout
    After sending the request, the ECU must react within this time (in ms), that is send either a positive or negative message insofar as the tester expects a response for the request.
  • P6 extended
    If the ECU sends a negative response with the error code 0x78 "Response Pending" then the tester is waiting once again for a response, whereby this time (in ms) is used.

Further details you find here: Settings Diagnostic Layer
