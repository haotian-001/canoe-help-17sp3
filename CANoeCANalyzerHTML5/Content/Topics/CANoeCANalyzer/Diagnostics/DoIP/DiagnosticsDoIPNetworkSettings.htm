DoIP/HSFZ Settings

Ribbon: Diagnostics & XCP Tab|Diagnostics/ISO TP Configuration|Diagnostics over IP/High-Speed-Fahrzeugzugang/Vehicle Ethernet|Administrate Diagnostic Descriptions|Assignments|DoIP/HSFZ Settings

Here you set the basic parameters for communication, that are used for sending and receiving diagnostic requests and responses over IP or HSFZ.

Note

For both new and existing configurations, the fields in this dialog page are populated with values from both the DoIP.ini file and the interface definitions in the corresponding diagnostic description (see Settings Interface). When you save the configuration, settings made in this dialog page are added to the configuration.

Parameters from Diagnostic Description regarding DoIP/HSFZ

Override manually

If the diagnostic description has an interface matching to the network, the default parameters from the interface of the diagnostic description are used. These parameters (see below) you can adjust manually, after activating the Override manually check box.

Reset to Defaults

Via the [Reset to defaults] button, you can set back your manual changes (see above) to the parameters of the used diagnostic description.

Tester

Log. address

This is the logical address of the tester, in conformance with the DoIP/HSFZ protocol.

Routing activation type

ISO 13400-2 defines a routing activation request activation type sent by the tester when establishing a connection to a vehicle. This is the initial value used by CANoe, where 0 is the default. Other values may be configured at the selected communication interface in the diagnostic description. Further it is possible to change the value from CAPL.

It is also possible to configure several communication channels to allow sending requests from diagnostic consoles with different activation types. This is not necessary if the type can be changed from CAPL.

Please refer to the following CAPL functions for details:

  • DoIP_SetRoutingActivationType
  • _DoIP_RoutingActivationRequest (ECU simulation callback)

Adapter

Here you can specify the local network interface and IP address via which CANoe (in its role as tester) or a simulated tester (e.g. in CAPL) shall communicate. If a valid address is selected, the tester will bind to this address and send Vehicle Identification Requests to the broadcast address derived from it. You can select an existing IPv4 or IPv6 address (with or without VLAN tag).

A tester node implementation using the file DoIP.dll can retrieve the IP address and/or adapter selected here using the communication parameters DoIP.TESTER_IPAddress and DoIP.TESTER_Adapter in diagGetCommParameter (without or with FlexRay).

Note

It is recommended to choose a dedicated IP address since the setting Automatic may lead to unexpected results. Especially in case of multiple network adapters/addresses are available on your computer, it depends on the selected TCP/IP stack in CANoe and the IP address settings of each adapter, which adapter is chosen for communication in case Automatic is selected.

The algorithm used in case of Automatic is chosen is as follows:

  1. The local endpoint is connected to the any address, i.e. listening is activated on all fitting adapters on Port 13400. If a request is received on an adapter, the communication is continued via this adapter.
  2. If a broadcast is sent, this is directed to port 13400 with the broadcast address which fits to the protocol of a random available adapter. If this adapter supports IPv4, an IPv4-Broadcast is sent. If this adapter supports IPv6 only, the link-local all nodes multicast address with the FF02::1 label is used.

Security profile

This profile is only visible if a security profile was activated for the selected tester adapter. You may change the security profile in the Security Configuration dialog. It contains all information necessary for a TLS connection, such as certificates, allowed cipher suites etc.

Note

When configuring the security profile in the Security Configuration dialog, you need to make sure to configure it at the stack of the corresponding simulation node (Stacks page). A configuration for a network will have no effect. If you do not use a dedicated simulation node with an Individual Stack for the tester, we recommend to activate the Use Security check box for the tester at the Shared CANoe Stack, which is typically used for all diagnostic windows (e.g. Diagnostic Console, Fault Memory Window, …).

TLS mode

This check box is only visible if a security profile was activated for the selected tester adapter. You may change the security profile in the Security Configuration dialog. Use this check box to define the behavior of the diagnostic tester with regard to the connection setup for encrypted communication via TLS. The individual settings have the following meaning:

Setting

Description

Disable TLS

The tester does not use TLS even if a security profile for TLS is configured at the TCP/IP stack of the tester. No connection is established, if the ECU is responding with routing activation response code (RARC) 0x07:

Routing activation denied because the specified activation type requires a secure TLS TCP_DATA socket

Enable TLS by routing activation (NACK 0x07)

The tester at first tries to set up an unencrypted TCP connection. If the ECU responds with RARC 0x10 (Routing successfully activated), the tester will continue with an unencrypted TCP connection. If the ECU responds with RARC 0x07, the tester tries to connect via TLS.

Enable TLS, allow secure connection only

The tester at first tries to set up an unencrypted TCP connection. If the ECU responds with RARC 0x10, the tester stops communication. If the ECU responds with RARC 0x07, the tester tries to connect via TLS.

Enable TLS, direct connection to TLS port for DoIP

The tester does not try to set up an unencrypted TCP connection. Instead, the tester directly tries to connect to the ECU via TLS. This can speed up the connection setup but needs to be supported by the ECU.

ECU

Log. address (phys / func)

This is the logical address of the ECU for physical (phys) or functional (func) diagnostic requests, in conformance with the DoIP/HSFZ protocol.

Note

For networks that were configured for DoIP/HSFZ via the Enable diagnostics via common DoIP/HSFZ gateway for this network option, you have to enter the logical addresses (physical or functional) of the corresponding ECU for the node:
via Gateway to <Network Name>|Node-Specific Settings.

Adapter

Here you can specify the local Windows network interface and IP address via which a simulated ECU shall communicate, unless the ECU is located at an Ethernet network. In the latter case the TCP/IP stack is configured directly at the simulation node. You can select an existing IPv4 or IPv6 address (with or without VLAN tag).

The simulation program can retrieve the IP address and/or adapter selected here using the communication parameters DoIP.VEHICLE_IPAddress and/or DoIP.VEHICLE_Adapter in diagGetCommParameter (without or with FlexRay). The value returned can be used in DoIP_SetLocalIPaddress.

Note

It is recommended to choose a dedicated IP address since the setting Automatic may lead to unexpected results. Especially in case of multiple network adapters/addresses are available on your computer, it depends on the selected TCP/IP stack in CANoe and the IP address settings of each adapter, which adapter is chosen for communication in case Automatic is selected.

The algorithm used in case of Automatic is chosen is as follows:

  1. The local endpoint is connected to the any address, i.e. listening is activated on all fitting adapters on Port 13400. If a request is received on an adapter, the communication is continued via this adapter.
  2. If a broadcast is sent, this is directed to port 13400 with the broadcast address which fits to the protocol of a random available adapter. If this adapter supports IPv4, an IPv4-Broadcast is sent. If this adapter supports IPv6 only, the link-local all nodes multicast address with the FF02::1 label is used.

Security profile

This profile is only visible if a security profile was activated for the selected ECUadapter. You may change the security profile in the Security Configuration dialog. It contains all information necessary for a TLS connection, such as certificates, allowed cipher suites etc.

Note

When configuring the security profile in the Security Configuration dialog, you need to make sure to configure it at the stack of the corresponding simulation node (Stacks page). A configuration for a network will have no effect.

Enable TLS

This check box is only visible if a security profile was activated for the selected ECUadapter. You may change the security profile in the Security Configuration dialog. If this check box is activated, the built-in diagnostics channel of CANoe will try to establish an encrypted communication with the ECU using the TLS protocol. This profile is only visible if a security profile was activated for the selected Adapter. You may change the security profile in the Security Configuration dialog.

Certificate

This input box is only visible if a security profile was activated for the selected ECU adapter. Here you can specify the name of the certificate which should be used by the ECU simulation. Note: this certificate needs to be available in the security profile and which was activated at the TCP/IP stack which is selected for the corresponding adapter.

Note

Please make sure not to add leading or trailing blanks by mistake when entering the certificate name. Otherwise, the certificate can not be found and therefore, no TLS connection can be established with the simulated ECU.

Gateway Settings

Log. address

This is the logical address of the gateway for the corresponding ECU.

In case of configurations with more than one gateway in a vehicle it is necessary to specify the DoIP Gateway Logical Address (GLA) of an ECU. This address is needed by a tester in order to decide with which gateway to connect when trying to reach a specific ECU.

Vehicle Identification Request Parameter

Address

The vehicle address together with the logical address provide unique identification for an ECU.

Because concrete vehicle data may only be partially available during ECU development, a range of vehicle address formats are supported. The address format is continuously checked when an address is entered, with the recognized address format displayed to the right of the entry field. If the text you enter does not correspond to a valid format, a question mark is shown.

The list below shows supported vehicle address formats:

  • If you leave the entry field empty (Format: Any), a connection is made to any ECU with a corresponding logical address.
  • VIN (Vehicle Identification Number according to ISO 3779)
    A 17-character string consisting of upper-case letters A-Z and numbers 0-9, whereby the letters I, O and Q must not be used due to their similarity to the numbers 0 and 1. In addition, the last four characters can only be numbers.
    Example: WP0ZZZ99ZSS331234
  • IPv4
    An internet protocol address conforming to IP Version 4
    Example: 127.0.0.1
  • IPv6
    An internet protocol address conforming to IP Version 6
    Example: 1002:db6:0:8d3::8b2e:70:6374
  • MAC
    MAC or hardware address of a network adapter
    Example: AB-CD-EF-12-34-56
    Or: AB:CD:EF:12:34:56
  • Hostname
    A unique name for a computer in a network
    Example: wikipedia.org

Note

If by chance a hostname is recognized as being in a VIN format, it is possible to force its interpretation as a hostname by using lowercase letters or appending a period to the name.

Request As

Here you can specify how the Vehicle Identification Request is sent to the network.

Vehicle Simulation Parameter

The list below shows supported vehicle address formats for simulation:

  • VIN (Vehicle Identification Number according to ISO 3779)
    A 17-character string consisting of upper-case letters A-Z and numbers 0-9, whereby the letters I, O and Q must not be used due to their similarity to the numbers 0 and 1. In addition, the last four characters can only be numbers.
    Example: WP0ZZZ99ZSS331234
  • EID (Entity Identification according to ISO 13400-2)
    Is typically identical to the MAC address of the corresponding network adapter.
    Example: AB-CD-EF-12-34-56
  • GID (Group Identification according to ISO 13400-2)
    Unique identification of DoIP devices within the same vehicle in the case that a VIN is not configured for this vehicle.
    Example: AB-CD-EF-12-34-56

Send VIN/GID sync. status

Only if this check box is activated, the optional parameter VIN/GID sync. status in a vehicle identification response or a vehicle announcement message will be sent.

Behavior when ECU closes connection

These parameters define how the tester communication channel reacts when the ECU closes the TCP connection.

Wait before reconnect attempt

The channel will wait this many ms before trying to open a new TCP connection to the vehicle. If this value is set to 0, the channel will not try to reestablish a connection.

Interval between attempts

When this many ms have passed and the vehicle has not confirmed the TCP connection, the channel will close the connection just initiated. (This procedure avoids communication errors from a possible response packet that the vehicle has already sent and that has not reached the tester yet). After that the channel tries to open a new connection.

Maximum number of attempts

The channel will only perform this many retries. A value of 0 here indicates that only 1 attempt after the time configured under Wait before reconnect attempt is performed (if that value is >0).

Notes

  • These parameters can be queried with the CAPL function DiagGetCommParameter.
  • When using the CAPL Callback Interface in a tester node (i.e. not the built-in diagnostic channel of a tester), the reconnect behavior can be configured using the CAPL functions DoIP_SetReconnectDelay and DoIP_SetReconnectInterval.
