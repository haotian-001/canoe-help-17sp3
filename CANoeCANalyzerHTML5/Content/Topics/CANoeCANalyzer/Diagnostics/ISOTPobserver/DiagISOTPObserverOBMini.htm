Configuration

The ISO TP observer uses the settings in OSEK_OBM.INI the configuration file to identify and interpret messages of the Transport Layer. The observer searches for the OSEK_OBM.INI file first in the directory where the configuration file is and second in the application data directory.

As long as you use only the address modes NormalFixed and/or Mixed the INI file doesn't need to contain any information – actually it doesn't need exist.

As long as you use only the address modes Normal and/or Extended you have to set the parameters standard address mode, base address and RxMask (see below).

Notes

  • The predefined default values can be replaced by attribute values from the database that is associated with the configuration if a node has been specified there and the transmitted data can be assigned to it.
  • The parameter names in the INI file are not case sensitive; e.g. ADRMODE and AdrMode are both accepted as the same.
    The attribute names in the database are case sensitive.

Example of a configuration file OSEK_OBM.INI

[OSEK TP OBSERVER]

// Address mode Normal=0/Extended=1
ADRMODE=1

// Global Default Values
TpBaseAddress=0x600
TpRxMask=0xff

// Ignore connections between nodes that are not specified in the database?
OnlyKnownNodes=0

Parameters

  • ADRMODE

The ADRMODE parameter must be set to the actually used mode (Normal or Extended). If it isn't set correctly and the attribute TpAddressingMode cannot be found for the node, the messages of the transport layer cannot be identified correctly.

  • TpBaseAddress / TpRxMask

The parameters TpBaseAddress and TpRxMask determine the range for the identifiers used for the messages of the transport layer. (Address modes Normal and Extended.)

  • OnlyKnownNodes

The OnlyKnownNodes parameter can be set to "1" if messages without specified node have to be ignored. This is useful if several ECU's communicate and only the communication of the ones that are defined in the database has to be analyzed.

Note

If you set the value to "1" without specifying the node(s) correctly in the database, the observer will not output anything.

The configuration file OSEK_OBM.INI may optionally contain the following described information.

Flow Control Messages

Additionally, the general behavior of the observer can be configured. These parameters will influence what the observer does with information in the control messages and also which values it expects to find.

Example

// Take block size and STmin from Flow Control messages?
EvalFC=1

// If this and EvalFC is not 0, all Flow Control messages are evaluated
EvalAllFC=0

// Which sequence number is expected in the first CF?
FirstSN=1

// Maximum tolerated number of Wait-FC messages
WFTMax=5

Timeouts

The ISO/DIS 15765-2 transport protocol defines various timeouts that can be monitored by the Observer. This is done by inserting the relevant parameters in the [TIMEOUTS] section of the configuration file OSEK_OBM.INI. Names of the timeouts reference the document ISO/DIS 15765-2.

Example

TIMEOUTS]

// Timeouts described in ISO/TC22/SC3/WG1/TF2 N124
// time in ms

Tar=1000
Tbr=900
Tcr=1000
Tdr=1000
Ter=1000

Zero Padding

The ISO TP Observer can be configured to check messages for correct Zero Padding (the filling of all CAN messages to DLC-value bytes, even if fewer data is actually used by the protocol). This requires inputting the parameters ZeroPadding and ZeroPaddingValue in the [OSEK TP OBSERVER] section of the configuration file OSEK_OBM.ini.

Example

[OSEK TP OBSERVER]

ADRMODE=1
TpRxMask=0x600
TpRxMask=0xff

// ZeroPadding on=1/off=0
ZeroPadding=1

// {PaddingValue}
ZeroPaddingValue=0x36

Output Control

In the section [OUTPUT], you can specify in detail which type of information the observer will report in the Trace Window. It is possible to specify e.g.

  • the messages
  • timer actions
  • warnings that it should display
  • it is also possible to limit the amount of data that is stored for every transmission (e. g. to support long running measurements without memory shortage).

Please note that the observer can be deactivated completely in the Configuration|Diagnostics/ISO TP Configuration...|ISO TP Observer dialog.

Deactivated warnings and information cannot be displayed later, i. e. they are not generated at all and not only hidden.

The main parameter Message can be used to deactivate the interpretation of messages completely. If its value is not "0", the display of message types can be activated and deactivated individually:

[OUTPUT]

// Configure the output of the observer into the trace window
// REM Global de-/activation can be done in menu CANoe/Configure/Options.../Global settings/Hook DLL

// which messages to interpret - first value is dominant
// 0 = do not print any messages
Message=1

// If message<>0, these values control if messages are printed (0 = no)
SingleFrame=1
FirstFrame=1
ContiFrame=1
FlowControl=1

// Report messages of unknown type
UnrecognizedType=1

The display of the setup, canceling and expiration of timers can be activated/deactivated by several timer parameters:

// report timer? Timer == 0 for none, otherwise actions can be selected.
Timer=1
TimerSetup=0
TimerCancel=0
TimerExpire=1

The observer will report possible errors and problems of a data transmission only if the Warning parameters are activated, i. e. not "0".

// Warning <> 0 means print warnings
Warning=1

// Report when a transmission is broken by a Single or First Frame.
Break=1

// Report when connections interleave, i.e. during a running transmission another
// connection is set up.
Interleaving=1

// Report messages that to not have the maximum DLC of 8
NotFilled=0

// Report when the receiver rejects a transmission due to buffer overflow.
// (FC.OVFLW message)
Overflow=1

// Report when the sequence numbers are incorrect
SeqNr=1

// Report when first CF is send without keeping STmin after reception of FC
STminFirstFC=1

// Report messages that should not arrive/be sent in the current state of a node.
UnexpectedMessage=1

// Report unexpected or missing FlowControl messages?
WatchFC=1

// Check if a message was sent by the right sender?
WrongSender=1

Finally, the number of bytes that are stored for a data transmission may be specified. This is useful for long time measurements a system with a low amount of available memory.

// Print transported data

// <0 ignore completely

// ==0 only indicate

// >0 store up to number bytes (4095 is maximum)
MaxDataLen=20

Recommendation

Since the output control values are re-read on every measurement start, it is convenient to open the INI file in a text editor to change parameter values between measurements.

Note

An additional parameter – WriteLevel – may be given in the [OSEK TP OBSERVER] section of the INI file. If its value is >0, the observer will output additional information to the Write Window (marked with OBS: at the beginning of the line), e. g. about the initialization and its state.
