LIN Settings

Ribbon: Diagnostics & XCP Tab|Diagnostics/ISO TP Configuration|Diagnostic Descriptions|Assignments|LIN Settings

The LDF file contains LIN diagnostic data. This data is imported and used for the diagnostics. The values in the CDD file are ignored.

The majority of diagnostic values in the LDF file are optional. If a value is not present in the LDF file, a default value from the LIN 2.1 specification is used.

Parameters from LDF File for LIN Settings and Diagnostic Layer

Override manually

If the diagnostic description has an interface matching to the network, the default parameters from the interface of the diagnostic description are used. These parameters (see below) you can adjust manually, after activating the Override manually check box.

Reset to Defaults

Via the [Reset to defaults] button, you can set back your manual changes (see above) to the parameters of the used diagnostic description.

Transport Protocol

  • STMin
    Minimum separation time (in ms) between two frames that the sender of a segmented message shall keep.
  • N_As_timeout
    Time the sender of a frame waits for the send confirmation.
  • N_Cr_timeout
    Time the receiver waits for a frame that is due from the sender.
  • NAD
    LIN Node address, typically specified in the LDF file and assigned automatically with the selection of a network node.

Scheduling Settings

LIN diagnostics use two predefined LIN frames, Master Request and Slave Response. So, the currently active schedule has to contain them.

The LIN 2.1 specification assumes that a separate table containing precisely these frames exists, i.e. one table each for the Master Request and the Slave Response frames. The system must therefore switch between these two tables in between the diagnostic request and the diagnostic response. The following settings relate to this issue.

Selected Scheduling

The currently running schedule table is used. It is important to make sure that this table contains both Master Request and Slave Response frames.

Diagnostics only

The LDF file is searched to establish whether separate schedule tables for the Master Request and Slave Response frames exist. If this is the case, the system will automatically switch between these two tables after a diagnostic request and a diagnostic response, otherwise this option is disabled. After the diagnostic communication has finished the schedule table running before the diagnostic request was started is run again.

Note

If no further communication is desired after finishing diagnostics, the table with the Master Request needs to be listed first in the LDF.

Interleaved

Like for the Diagnostics only mode separate schedule tables for the Master Request and Slave Response frames must exist. If a diagnostic request is transmitted, between each segment of the request the schedule table that was running before the request was started is run once. The same behavior is true for a diagnostic response. After the diagnostic communication has finished the schedule table running before the diagnostic communication was started is run again.

Direct sending

When transmitting directly, the Master Request Header is generated automatically, independent of the scheduling. If a Master Request and a Slave Response table exist, the system automatically switches to the Slave Response table for the diagnostic response. If this is not the case, a table containing a Slave Response frame must be set up in the configuration. It is important to make sure that in this mode the specified table does not contain a Master Request frame, as this would cause the Master Request header to be transmitted too often. The frame can be deactivated via the Interactive Master if required.
