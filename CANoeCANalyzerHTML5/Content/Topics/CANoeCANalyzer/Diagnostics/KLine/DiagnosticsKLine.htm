K-Line

 

General

For diagnostics of ECUs on K-Line CANoe offers a built-in diagnostic channel according to ISO 14230-2 (Keyword 2000).

You can use one or more K-Line capable VN network interfaces to access the ECU.

Alternatively you can use a serial port built-in to your computer and a Ser2K level converter from Vector attached to that serial port as hardware access the ECU. However, Vector recommends instead of the serial port to use the Vector interfaces listed above due to the higher timing accuracy. Some K-Line features are supported only with these interfaces.

Notes on hardware using serial ports

In order to meet timing requirements as good as possible – especially when performing diagnostics on FastInitialization ECUs – either a built-in computer serial port or a serial port connected to the computer via PCI/PCIe/PCMCIA/expressBus should be used.

Common "off the shelf" USB/RS232 converters typically cannot provide the same performance and should not be used.

Yet even when built-in serial ports are used, some restrictions may apply:

  • Jitter in wake-up pattern timing / ECU does not react on wake-up pattern
    Adapt the K-Line parameters in the configuration of the diagnostic channel to meet the requirements of your ECU (IniL time, Wup time).
  • No communication on the K-Line
    Some serial interfaces and their drivers do not provide the baudrates necessary for K-Line communication (5baud, 10400baud). In this case use to a different serial hardware.
  • Termination of measurement takes overly long (when using 5 baud initialization)
    If the ECU is not available or does not respond, in some cases longer than usual waiting times are to be expected when terminating the measurement and closing the serial port.

Note about compatibility

Starting with CANoe version 8.0 the previous solution (CAN-K-Line gateway and KLineCAPL.DLL) is no longer supported.
Adapt your existing diagnostic configuration to use the built-in K-Line diagnostic channel.
In case of a need of the previous interface please contact Vector Support.

Diagnostic Channel Features

The K-Line diagnostic channel supports the following features:

  • Physical addressing
  • Functional addressing
    This feature is only available in CAPL. If a diagnostic request is sent with DiagSendFunctional also a functional channel to the ECU will be established.
  • Fast & 5 baud initialization
  • Automatic baudrate switching
    In case a service $10 ("startDiagnosticSession") request with respective parameterization is sent to the ECU, the baudrate on the K-Line will be automatically adapted according to those parameters.
  • Configurable usage of 4 byte headers irrespective of actual frame length
  • Interpretation of K-Line protocol in the Trace Window (K-Line bytes, K-Line messages, K-Line initialization phases)

Note on ECUs, which rely on a diagnostic channel being closed before being opened anew:

Currently StopCommmunication is not automatically sent to the ECU on termination of the measurement.

In case your ECUs needs this StopCommunication for its correct internal workings, you can insert this service in a CAPL node though by calling the function diagCloseChannel(<ECU_Qualifier>). Here ECU_Qualifier is the name/qualifier of the ECU as inserted in the diagnostic configuration dialog.

K-Line Tester Simulation (only available with Vector hardware)

Using the K-Line CAPL-API following features are supported for a K-Line tester:

  • Reception and transmission callbacks on K-Line message and byte level
  • Access to byte values and byte time stamps
  • Pre-send callbacks to manipulate requests
  • Sending of raw byte streams
  • Controlling system baudrate
  • Controlling timing parameters of frames and init-patterns
  • Controlling K-Line header format and data

K-Line ECU Simulation (only available with Vector hardware)

Using the K-Line CAPL-API following features are supported for a ECU simulation:

  • Functions for initializing a ECU simulation
  • Reception and transmission callbacks on K-Line frame and byte level
  • Access to byte values and byte-end time stamps
  • Receive callback for the fast-init pattern
  • Pre-send callback to manipulate responses
  • Sending of raw byte streams
  • Sending of K-Line diagnostic responses
  • Changing the inter-byte times of a response
  • Changing the baudrate of the reception

K-Line Monitoring (only available with Vector hardware)

  • Analyzing K-Line communication on the level of messages and bytes.
  • Reporting fast init pattern with Tinil and TWup times.
  • Logging communication in ASC format and BLF format.

Note: Not supported are:

  • five baud pattern detection
  • detection of baudrate changes

Note

The analysis-only mode for K-Line is only supported with Vector hardware!
