Glossary: Â»Diagnostics

Diagnostics describes the functional area in CANoe for:

  • Analyzing diagnostic communication between a diagnostic tester and ECUs.
  • Testing diagnostic functionality of ECUs interactively, semi-automated and automated.
  • Simulating diagnostic functionality of ECUs (Test the tester).

General Information

The Diagnostic Feature Set contains various features that are needed for developing, testing, and calibrating control units with or by means of diagnostics.

The integrated diagnostic features allow both analysis as well as simulation of diagnostic services of a control unit. In addition to the use of specific diagnostic commands within CANoe (CAPL), external applications can easily access diagnostic objects and diagnostic parameters via the COM server.

Besides CANoe, the Diagnostic Feature Set is also contained in Vector products CANape MC+D and CANdito. As a result, the same support is provided throughout the development process.

Use of CANoe in Development Phases

CANoe can be used in all phases of development and use of diagnostics in control units:

  • Design of the diagnostic functionality (system simulation)
  • Implementation of the diagnostic functionality in a control unit (remaining bus simulation)
  • Diagnosing of ECUs with the test functionality integrated in CANoe
  • Specification/integration/regression tests
  • Analysis of the communication of real control units
  • Troubleshooting
  • Automatic creation of diagnostic test modules in connection with CANoe Option DiVa
  • Description of tests in XML

Supported Protocols for Diagnostic Communication

The following protocols are supported for diagnostic communication of CANoe:

  • Physical layer: CAN, LIN, MOST, FlexRay and K-Line
  • Diagnostic protocols: KWP2000, UDS, OBD, J1939 (according to SAE J1939-73)
  • Transport protocols: OSEK-TP,VW-TP as well as UDP and TCP for DoIP/HSFZ
  • Diagnostic via IP (DoIP)
  • High-Speed-Fahrzeugzugang (HSFZ)

Off-Board Diagnostics

For error detection with off-board diagnostics, one or more control units are connected to an external diagnostic tester that executes the error detection functions (e.g., CANoe). The diagnostic communication occurs via a 1:1 connection between the tester and the control unit using standardized diagnostic protocols.

Diagnostics with CDD, ODX/PDX, MDX Diagnostic Description

During active diagnostics, the tester and the control unit typically exchange large quantities of complex data. The interpretation of these data depends not only on the (vehicle) manufacturer but also on the model of the control unit, its year of manufacture, or even on the individual control unit itself.

Information regarding the layout and semantics of the data are managed in diagnostic descriptions (CDD, ODX/PDX, MDX).

Diagnostic descriptions can be used universally in CANoe. This enables you to work with predefined symbolic diagnostic requests for creation of diagnostic testers as well as control unit simulations. Consequently, this procedure is a significantly time-saving alternative to manual/byte-by-byte definition of diagnostic requests.

Based on the diagnostic description files (CDD, ODX/PDX, MDX), the Diagnostic Console contained in the Diagnostic Feature Set provides interactive access to all diagnostic requests. These can be selected, parameterized, and displayed with their response.

The Fault Memory Window enables you to read out the fault memory of a control unit and to delete entries (all entries or individual entries) from the fault memory (DTC - Diagnostic Trouble Code).

You can use the Diagnostic Session Control to change interactively from one status of a control unit to another (session change, security access) and to control the communication management settings of a control unit.

The diagnostic feature Variant Coding is used to read, write and compare variant coding data of an ECU. Security aspects from possibly configured security sources are taken into account. See Variant Coding Window.

Diagnostics with UDS/KWP Basic Diagnostic Description

For simple use cases, Basic Diagnostics is an alternative to the process-oriented approach using diagnostic descriptions. You can exchange diagnostic information with the control unit without diagnostic description files (CDD, ODX/PDX, MDX).
You can use the Basic Diagnostic Editor to code the diagnostic requests in accordance with specifications and to transfer the services coded in this way to the Diagnostic Console and send requests from there.

Diagnostics with Standard Diagnostic Description (reduced CDD)

Use the Standard Diagnostics if no other of the above described diagnostic types are possible. A Standard Diagnostic Description is a reduced CDD diagnostic description, that contains all necessary information to run a simple diagnostic communication.

On-Board Diagnostics

On-Board Diagnostics (OBD) refers to diagnostics that are integrated in modern motor vehicles. During driving operation, all systems that affect emissions are monitored. Errors that occur are displayed to the driver via a warning light and are stored permanently in the respective control unit. Error messages can then be queried subsequently by a specialized garage via standardized interfaces.

With the OBD-II Window, it is possible to perform On-Board Diagnostics. The control units, their associated requests, and the information are determined and provided to the user in the window.

Display of Diagnostic Data

As described previously in the section on off-board diagnostics, diagnostic requests and their responses are shown in the Diagnostic Console; the fault memory is output in the Fault Memory Window.

Diagnostic communication can also be displayed in some CANoe analysis windows, such as Graphics Window, Data Window, and Trace Window.

Sending of Diagnostic Requests

There are two different methods of sending diagnostic requests: interactive and programmatic.

To send data interactively, you have the Diagnostic Console, the Fault Memory Window, the Session Control, the Variant Coding Window and the OBD-II Window available.
CAPL and .NET programming are available for sending data with a program.
In contrast to interactive sending, programmatic sending is used when it is necessary to comply with time conditions of sequential requests and these time conditions cannot be met with manual operation. This can also be implemented with the Test Feature Set supported by CANoe.

Addressing Types

CANoe enables multiple control units in a network to be diagnosed at once. An entire network can be assigned multiple diagnostic descriptions in CDD, ODX/PDX, MDX format for any of the following:

  • Typical 1:1 connection between the tester and the control unit or
  • Functional group requests (functional addressing).

For functional group requests, a request that multiple control units answer to (functional ID) is sent for all control units in the network. These are typically requests for session handling, e.g., switch all control units in the network to normal/default diagnostic mode.
