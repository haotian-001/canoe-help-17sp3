Discrete vs. Continuous Conditions

 

Each user-defined condition allows specifying

  • a required runtime event
  • its timing (optional) and
  • its required data (if applicable).

We distinguish between discrete and continuous conditions according to the process of their evaluation.

Note, that user-defined condition passes evaluation on every incoming runtime event.

Discrete condition

Discrete condition is reset before each new evaluation and becomes satisfied if the runtime event is the required runtime event with the required data (if applicable) and with the required timing (if configured).

The following types of conditions are discrete:

Condition

Note
  • Symbolic Message
  • Predefined Bus Event
  • Raw Message

As long as there are no settings for Delay… or Timeout after…
(see set of user-defined conditions: Timing)
  • Symbolic Signal
  • Environment Variable
  • Bus Statistics Signal

As long as there are no settings for Delay… or Timeout after…
(see set of user-defined conditions: Timing)

Mode of the required value
(see set of user-defined conditions: Mode of the required value)

  • Absolute with Discrete flag
  • Consecutive Delta
  • Change from … to …

Continuous condition

A Continuous condition is not reset before a new evaluation; nevertheless it becomes satisfied the same way as a Discrete condition.
It is reset if the runtime event is the required runtime event and one of the following is true:

  • the required value is not satisfied,
  • "Absolute" timing is configured and it is not satisfied,
  • the required value is satisfied, Delay… or Timeout after… is configured.
    In that case the timer of the condition is restarted.

So, only conditions that specify a required value or have timers to measure timing configuration may be Continuous.

The following types of conditions are continuous:

Condition

Note
  • Symbolic Message
  • Predefined Bus Event
  • Raw Message

The settings for Delay… or Timeout after… are required
(see set of user-defined conditions: Timing)
  • Symbolic Signal
  • Environment Variable
  • Bus Statistics Signal

The settings for Delay… or Timeout after… are required
(see set of user-defined conditions: Timing)

Mode of the required value
(see set of user-defined conditions: Mode of the required value)

  • Absolute without Discrete flag

Example for discrete and continuous conditions

Lets suppose we have two user-defined conditions configured

  • Discrete condition C1:

    • The required runtime event is a symbolic message M

  • Continuous condition C2:

    • The required runtime event is a symbolic signal S, which is mapped to the symbolic message M.

The required data is the value <=6.
The mode of the required value is Absolute without Discrete flag.

The following chart shows incoming runtime events over time axis and the evaluation of the two above conditions on each event.

Description

  1. Initially both conditions are not satisfied.
  2. On getting message M with a value of signal S equal to 5:
    the evaluation leads to satisfaction of both C1 and C2.
  3. On getting message M1 with a value of signal S1 equal to 0:
    C1 becomes not satisfied, C2 remains satisfied because the message M1 is not a required runtime event.
  4. On getting message M with a value of signal S equal to 5:
    C1 becomes satisfied, C2 doesn’t change its state because it wasn’t yet reset.
  5. On getting message M with a value of signal S equal to 6:
    the result of the evaluation is the same as before.
  6. On getting message M with a value of signal S equal to 7:
    C1 is satisfied, C2 is reset because message M is the required runtime event and the value 7 is not the required value.
