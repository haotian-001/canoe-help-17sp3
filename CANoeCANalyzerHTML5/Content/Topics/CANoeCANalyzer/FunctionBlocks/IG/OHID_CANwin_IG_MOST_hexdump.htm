MOST IG: Hexadecimal Representation of the MOST Messages

 

The hexadecimal representation of the MOST messages (hex dump) is located in the configuration dialog of the MOST IG between the transmit list and the parameter tree.

If a message is marked in the transmit list, the hexadecimal representation of this message is shown in the hex dump. Most values in the hex dump fields can be modified directly.

Most fields match the fields described in the transmit list.

Setup of the hex dump:

Application Messages

  • Send
    …via this button the message can be sent. Corresponds with 'Send' in the transmit list.
  • Dest.
    …physical address of the recipient. Corresponds with 'Destination' in the transmit list.
  • Rtype
    …this field represents the transmission type of the message.
    Normal messages have the transmission type 0x00, system messages a transmission type between 0x01 and 0x05 according to the type of the system message.
  • FBlock
    …Function block. Corresponds with 'FBlock' in the transmit list
  • Instance
    …corresponds with 'Instance' in the transmit list
  • Function
    …Function within a given function block. Corresponds with 'Function' in the transmit list
  • OpType
    …Operation within a given function. Corresponds with 'OpType' in the transmit list
  • TelID
    …this field is only editable with messages that were not sent via the AMS. The TelID designates the respective protocol and the message type within the protocol.
    The following values apply:
    0: normal CMS message
    1: first AMS telegram
    2: second..(n-1) AMS telegram
    3: last AMS telegram
    8: MOST High user data telegram
    9: MOST High control data telegram
    If the message is sent via the AMS, the service automatically enters the corresponding TelIDs (1, 2 and 3) in the respective telegrams.
  • TelLen
    …indicates the number of bytes contained in the utilizable data of a message. This value is automatically adapted to the data entered. If the message is not sent via the AMS it is possible to change the value manually after the data has been entered completely. The maximum value for the TelLen is 12 bytes, or 11 bytes when Broadcast/Groupcast addressing is used.
  • Data
    …hexadecimal representation of the parameter list

    • If an invalid value is entered in a field, a corresponding message appears in the status line. If a value is valid, but not defined in the associated database ('FBlock', 'Function' and 'OpType' fields), the relevant fields of the message are highlighted in color in the transmit list, and the values they contain are represented in hexadecimal.

    Parameters for which no values have yet been entered in the parameter tree are represented by underscores ('__') in the 'Data' hex dump field. If a message that is incomplete in this manner is sent, the data bytes will only be sent until the first unspecified value.

    Post-editing of messages

    Parameter values can be entered in the hex dump field at any time. The hex dump is suitable for post-editing of messages as well as manual entry of errors, if required.

  • TelID
    The parameter tree is only available for TelIDs 0 and 1. If a higher value is entered, the parameter tree will be hidden automatically. By means of the corresponding check box in the parameter area it is possible to re-activate the display of the parameter tree at any time. This automatically resets the TelID back to 0.
  • Data
    Changing the values in the Data field will cause the parameter tree to be hidden. It is then possible to enter data that is not supported by the parameter structure. By means of the corresponding check box in the Parameter area it is possible to re-activate the display of the parameter tree at any time. In this case the data entries are mapped to the parameters as well as possible. Intentional errors may, however, be lost in the process.

Raw / System Messages

Every single byte of a control frame can be set individually. The RType of a message is set in the message list.

  • Normal

  • Send:
    Via this button the message can be sent. Corresponds with 'Send' in the transmit list.
  • Dest.:
    Physical address of the recipient. Corresponds with 'Destination' in the transmit list.
  • RType:
    This field represents the transmission type of the message.
  • 0..16:
    Single bytes of the control frame

For system messages additionally the specific terms of the several bytes are displayed:

  • RemoteRead

  • MAP:
    Address to read from
  • D0..D7:
    Values of the registers starting at the address MAP
  • D0 contains the value of register MAP, D1 of MAP+1, etc.
  • RemoteWrite

  • MAP:
    Address to write to
  • Length:
    Number of data bytes to be written (max. 8)
  • D0..D7:
    Values to be written to the registers starting at the address MAP
  • D0 contains the value of the register MAP, D1 of MAP+1, etc.
  • Allocate

  • REQ:
    Number of channels to be allocated (max. 8)
  • AW1:
    Allocation status of the TimingMaster
  • 0x01 (ALLOC_GRANT):
    Channels successfully allocated
  • 0x02 (ALLOC_BUSY):
    TimingMaster is busy working on an allocation / deallocation process
  • 0x03 (ALLOC_DENY):
    Not enough channels available
  • 0x04 (ALLOC_WRONG):
    The value of REQ is set to 0 or greater than 8
  • 0x05 (WRONG_TARGET):
    The Allocate Message was sent to a TimingSlave
  • AW2:
    Number of free channels (if Answer1 = 0x01..0x04) or 0x00
  • P0..P7:
    MOST Routing Table addresses of the allocated channels
  • P0 matches the connection label (CL), which is used for deallocation (s. Deallocate)
  • Deallocate

  • Label:
    Connection label, which was assigned during allocation
  • AW:
    Deallocation status of the TimingMasters
  • 0x01 (DEALLOC_GRANT):
    Channels successfully deallocated
  • 0x02 (DEALLOC_BUSY):
    TimingMaster is busy working on an allocation / deallocation process
  • 0x04 (DEALLOC_WRONG):
    Connection label has a value greater than 0x7F
  • 0x05 (WRONG_TARGET):
    The Deallocate Message was sent to a TimingSlave
  • GetSource

  • Label:
    Connection label (CL) or physical channel (address in the MOST Routing Table)
  • NP:
    Node position of the node, which streams data on the channels specified by the CL
  • GA:
    Group address of the node, which streams data on the channels specified by the CL
  • NAH:
    High byte of the node address of the node, which streams data on the channels specified by the CL
  • NAL:
    Low byte of the node address of the node, which streams data on the channels specified by the CL

Raw Packets

  • Send
    …via this button the message can be sent. Corresponds with 'Send' in the transmit list.
  • Dest.
    …physical address of the recipient. Corresponds with 'Destination' in the transmit list.
  • Length
    ...Number of data bytes in the packet
  • User Data
