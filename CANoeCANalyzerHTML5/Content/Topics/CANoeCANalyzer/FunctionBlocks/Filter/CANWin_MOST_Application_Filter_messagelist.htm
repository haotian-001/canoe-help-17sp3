MOST Application Filter: Message List

 

The message list is located in the upper half of the MOST Application Filter configuration dialog. For each message to be filtered, an entry must be added to the message list. The description of a message is contained in one line.

The different buttons can be used to add and delete messages and to resort the messages in the list.

Note

In case the MOST Application Filter is configured as Pass Filter any message or packet type (application message/packet, system message, raw or MAMAC packet) that is not inserted into the message list will be blocked (e.g. in case system messages should always pass the filter add a new system message with all parameters set to "*" (wildcard)).

For a block filter the respective message or packet types are blocked in case they are inserted into the message list.

Structure of the Message List:

Active

  • In this column you can activate or deactivate single messages
  • Through the short cut menu you can activate or deactivate all messages in the message list

Address

Here you can define the source and destination address of the respective message

  • SA - source address
  • DA - destination address

Message / Packet

  • Application Messages/Packets

The [New] button can be used to add a new application message / packet via the XML selection dialog. Application packets are those packets, which are sent on the asynchronous channel via the MOST High Protocol (MHP). This is only possible after an XML function catalog has been added to the configuration. Otherwise, the drop-down list on the [New] button can be used to add a new application message to the message list. In this case, all fields are filled with a "*" (wildcard). In case the configuration contains an XML function catalog the input of FBlock, Function and OpType can be done via a choice list.

  • FBlockID

... function block. Corresponds to 'Fblock' in the message list

  • InstID

... corresponds to 'InstId' in the message list

  • FunctionID

... function within a specified function block. Corresponds to 'Function' in the message list

  • OpType

... operation within a specified function. Corresponds to 'OpType' in the message list

  • TelID
  • ... this field only affects individual frames within the messages. TelID describes the protocol as well as the message type within the protocol to which the message belongs. The following applies:
  • 0: Normal CMS message
  • 1: First AMS telegram
  • 2: Second .. (n-1) AMS telegram
  • 3: Last AMS telegram
  • 8: MOST High user data telegram
  • 9: MOST High control data telegram
  • CMS (0..3)
  • MHP (8..9)
  • CMS & MHP
  • any other (4, 5, 6, 7 A, B, C, D, E, F)
  • TelLen

... TelLen describes the number of payload data bytes used. It only affects the individual frames or telegrams       within a message. It is automatically adapted to the data that is entered, insofar as the maximum number of payload data bytes for the corresponding telegram (see TelID) is not exceeded. The following applies:

    • A maximum of 12 payload data bytes for AMS telegrams; 11 payload data bytes maximum for broadcast/groupcast addressing
    • A maximum of 11 payload data bytes for MHP telegrams on the control channel
    • A maximum of 1006 payload data bytes for MHP telegrams on the asynchronous channel

If the maximum number of payload data bytes is exceeded, the individual telegrams are:

    • Blocked by an 'allow filter'
    • Allowed by a 'block filter'

Changes made to this field affect the "Data" field, unless a wildcard ("*") has been entered there:

    • If fewer bytes are entered there, the remaining bytes are hidden
    • If more bytes are entered there, the superfluous bytes are deleted

  • Raw/System Messages

The drop-down list on the [New] button can be used to add a line for a raw/system message to the message list. In this case, all fields are filled with a "*" (wildcard). For a specific system message (Remote Read, Remote Write, Allocation, Deallocation, or Get Source) please select the corresponding entry from the list.

Combiner (Comb.)

Along with the individual telegrams of an application message, the combined messages of the Application Message Service (AMS) can also be filtered.

The message type can be activated/deactivated by clicking in the relevant cell:

  • activated means that this message type is also included in the filtering process
  • deactivated means that this message type is not included in the filtering process

Note

Since the MOST High Protocol can be used for transmission on the control as well as on the asynchronous channel, all combined events (MHPBlock, MHPPacket and MHPError) on both channels will be considered for highlighting.

Context Menu of the Message List

The context menu of message list is displayed if you execute a right mouse click in on of the cells of the message list. The following menu commands are available:

Read configuration from file...
(Opens a file dialog to select the file (.afm) from which the configuration (message list) should be read)

Write configuration from file...
(Opens a file dialog to select the file (.afm) to which the configuration (message list) should be written)

Activate selected message(s)
(Activates the selected message(s) in the message list)

Deactivate selected message(s)
(Deactivates the selected message(s) in the message list)

Activate all messages
(Activates all messages in the message list)

Deactivate all messages
(Deactivates all messages in the message list)

General

  • Not only specific values, but also "wildcards" or masking patterns can be used for the individual parameters of a message. Wildcards are specified by means of an asterisk "*" and indicate that the given parameter may have any value. A masking pattern is specified by means of an underscore "_". Masking is carried out at nibble level, which means you have to specify a "_" for each nibble.

Example:

Masking of the FBlock: "0_"  (include all function blocks from 0x00..0x0F)

Masking of an address: "3__" (include all group addresses)

  • Clicking on the last, gray line in the message list adds a new application message to the list. In this case, all fields are filled with a "*" (wildcard). The fields of the new message can be filled even without an embedded function catalog.

Note

Only AMS atoms can be written into logging file. In case you want to write MOST messages into a logging file, make sure that in each MOST Application filter before a Logging Block the several AMS atoms are let through.
