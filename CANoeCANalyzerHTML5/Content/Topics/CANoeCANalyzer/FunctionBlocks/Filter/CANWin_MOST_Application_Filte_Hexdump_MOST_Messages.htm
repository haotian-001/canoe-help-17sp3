MOST Application Filter: Hexadecimal Representation of the MOST Messages

 

The hexadecimal representation (hex dump) of the MOST messages is located below the message list.

When you select a message in the message list, the hexadecimal representation of this message appears in the hex dump. You can alter most of the values in the hex dump fields directly.

Most of the fields correspond to the fields shown in the message list.

Hex Dump Structure

Application Messages

  • FBlockID
    …function block; corresponds to Fblock in the message list
  • InstID
    …corresponds to InstId in the message list
  • FunctionID
    …function within a specified function block; corresponds to Function in the message list
  • OpType
    …operation within a specified function; corresponds to OpType in the message list
  • TelID
    …this field only affects individual frames within the messages.
    It describes the protocol as well as the message type within the protocol to which the message belongs.
    The following applies:
  • 0: Normal CMS message
  • 1: First AMS telegram
  • 2: Second…(n-1) AMS telegram
  • 3: Last AMS telegram
  • 8: MOST High user data telegram
  • 9: MOST High control data telegram
  • TelLen
    …describes the number of payload data bytes used.
    It only affects the individual frames or telegrams within a message.
    It is automatically adapted to the data that is entered, insofar as the maximum number of payload data bytes for the corresponding telegram (see TelID) is not exceeded.
  • The following applies:
  • A maximum of 12 payload data bytes for AMS telegrams; 11 payload data bytes maximum for broadcast/groupcast addressing
  • A maximum of 11 payload data bytes for MHP telegrams on the control channel
  • A maximum of 1006 payload data bytes for MHP telegrams on the asynchronous channel
  • If the maximum number of payload data bytes is exceeded, the individual telegrams are:
  • Blocked by an "allow filter"
  • Allowed by a "block filter"
  • Changes made to this field affect the Data field, unless a wildcard ("*") has been entered there:
  • If fewer bytes are entered there, the remaining bytes are hidden
  • If more bytes are entered there, the superfluous bytes are deleted
  • User Data
    …hexadecimal representation of the user data

Raw/System Messages

Every single byte of a control frame can be set individually. The RType of a message is set in the message list. 

For system messages additionally the specific terms of the several bytes are displayed:

  • RemoteRead

  • MAP: Address to read from
  • D0…D7: Values of the registers starting at the address MAP
  • D0 contains the value of register MAP, D1 of MAP+1, etc.
  • RemoteWrite

  • MAP: Address to write to
  • Length: Number of data bytes to be written (max. 8)
  • D0…D7: Values to be written to the registers starting at the address MAP
  • D0 contains the value of the register MAP, D1 of MAP+1, etc.
  • Allocate

  • REQ: Number of channels to be allocated (max. 8)
  • AW1: Allocation status of the TimingMaster
  • 0x01 (ALLOC_GRANT): Channels successfully allocated
  • 0x02 (ALLOC_BUSY): TimingMaster is busy working on an allocation / deallocation process
  • 0x03 (ALLOC_DENY): Not enough channels available
  • 0x04 (ALLOC_WRONG): The value of REQ is set to 0 or greater than 8
  • 0x05 (WRONG_TARGET): The Allocate Message was sent to a TimingSlave
  • AW2: Number of free channels (if Answer1 = 0x01..0x04) or 0x00
  • P0…P7: MOST Routing Table addresses of the allocated channels
  • P0 matches the connection label (CL), which is used for deallocation (s. Deallocate)
  • Deallocate

  • Label: Connection label, which was assigned during allocation
  • AW: Deallocation status of the TimingMasters
  • 0x01 (DEALLOC_GRANT): Channels successfully deallocated
  • 0x02 (DEALLOC_BUSY): TimingMaster is busy working on an allocation / deallocation process
  • 0x04 (DEALLOC_WRONG): Connection label has a value greater than 0x7F
  • 0x05 (WRONG_TARGET): The Deallocate Message was sent to a TimingSlave
  • GetSource

  • Label: Connection label (CL) or physical channel (address in the MOST Routing Table)
  • NP: Node position of the node, which streams data on the channels specified by the CL
  • GA: Group address of the node, which streams data on the channels specified by the CL
  • NAH: High byte of the node address of the node, which streams data on the channels specified by the CL
  • NAL: Low byte of the node address of the node, which streams data on the channels specified by the CL

Raw Packets

  • Length
    …Number of data bytes in the packet
  • User Data
    …hexadecimal representation of the user data

MAMAC Packets

  • TelID
    Telegram ID of the MAMAC packet
  • 0x0A
    unsegmented MAMAC packet
  • 0x0A
    unsegmented MAMAC packet
  • TelLen
    …Number of data bytes of the MAMAC packet
  • User Data
    …hexadecimal representation of the MAMAC packet user data

If you enter an invalid value into a field, a corresponding message appears in the status bar. If a value is valid, but has not been defined in the database that is in use (FBlock, Function and OpType fields), the affected message fields are marked with a colored background in the message list, and the individual values shown in hexadecimal form.
