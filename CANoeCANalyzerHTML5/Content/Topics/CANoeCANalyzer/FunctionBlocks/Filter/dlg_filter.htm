Filter Block: Configuration

 

This Filter dialog is used to configure a filter.
Depending on the options you have installed, you can have access to multiple/different databases, e.g. CAN, LIN, MOST.

The table in the upper half of the dialog shows the active configuration.
In the box below this, all configured objects are listed briefly. Furthermore, the Filter Type can be toggled between Pass Filter and Stop Filter (blocking filter) by the relevant selection buttons.

Note

Please select the directions as "seen" from CANoe, e.g. so that CANoe is the sender or receiver.

Buttons and Options

[Events… |V]

Here you can select messages from the database or standard predefined messages of the bus system.

  • If the button is clicked within the area of the right arrow symbol you can choose from a context menu whether the message should be taken from a DBC database or from a XML function catalog (only MOST messages). An appropriate selection dialog is opened.
  • If the button is activated outside of the right arrow symbol the selection dialog for DBC databases is always opened.

[Raw data…]

Here you can enter either a single ID or a range you want to filter.

[Node…]

[Node…] allows you to select network nodes from the database and thereby filter all Messages that this node transmits and/or receives.
In the Filter Configuration dialog you can select certain Message Attributes (Tx, Rx, TXRQ) to be filtered.

[Remove]

Individually selected lines can be deleted by [Remove].

  • Filter Type – Pass Filter
    All listed Events/Ranges will pass the filter.
    Exception: seeFilter unconfigured Status- and Error Events option.
  • Filter Type – Stop Filter
    All listed events will be stopped/blocked.
    Exception: see Filter unconfigured Status- and Error Events option.

Filter unconfigured Status- and Error Events.

This control field can be used to filter status and error events.

To display status and error events in the analysis windows the check box must be cleared in case of a block filter; the check box must be selected in case of a Pass Filter.

However, if rules are defined for one or more of these status and error events in the list, these take precedence over the Filter unconfigured Status- and Error Events option for the relevant event.

Typical Procedures for Filtering

  • Messages

To create a new filter condition for individual messages, the specific identifier can be input directly by double clicking in the ID box. If the configuration is associated to a database, the message can also be selected by double clicking in the Name box or by pressing the [Add Message] button.

By pressing the [Raw data…] button you get a dialog in which you can enter start- and stop value (ID) of a message range and the according channel. If you enter just one value (ID) you select a single message for filtering.

If a database is assigned to this configuration, you can select messages with a double click in the Name field or by pressing the [Events…] button.

  • Message Range

Entire ranges of messages and the according channel can be created by entering start and end values in the ID box (e.g. 200-2ff). or by pressing [Raw data…].

  • Network Nodes

[Add Node] allows you to select network nodes from the database and thereby filter all messages that this node transmits and/or receives.

  • Error Frames

The [Events…] button open a dialog in which you can select CAN Error Frames in the Predefined Messages|CAN|Messages (top entry) area.

  • Rx, Tx, TXRQ, Data, Remote

Select the desired message attributes by check boxes.

  • Channels

Here you can select the CAN controller for the message.

If you set the entry to All this option is ignored.

Let all MOST messages and events pass

Note

  • The MOST Application Filter, available since version 6.0, allows much more flexible configuration of MOST messages.

  • The filter described here is still supported to ensure compatibility with older configurations.

    The 'Pass all MOST messages and events' check box can be used to run a series of filters and MOST Application Filters, whereby the multibus Filter has no influence on the MOST messages.

    This makes it possible to filter different bus systems separately.

When this check box is activated, all MOST messages and events passes this filter. As long as this check box is activated no MOST messages can be inserted. In case there are already MOST messages configured in the filter, a dialog appears, in which you can decide whether:

  • all MOST messages and events shall be filtered anyway

=> The check box will be deactivated again

  • all MOST messages and events shall pass the filter

=> all MOST messages and events will pass the filter, but the MOST message remain in the configuration list

  • all MOST message and events shall pass the filter and the configured MOST messages should be removed
