Scheduling Concepts

 

For an AFDX compliant stimulation of an end system, besides the message contents, timing aspects in particular are also relevant. It must be possible to send the messages periodically or sporadically, depending on their type, and they may also need to be fragmented. In addition, it must be assured that the frames fulfill BAG requirements.

Definition: BAG

Bandwidth Allocation Gap

This is the time in milliseconds that must elapse at a minimum between the transmission of two AFDX messages on the same VL.

To this end, there are two different schedulers in CANoe which work on different levels: the Message Scheduler and the VL Scheduler.

Message-Scheduler

The Message Scheduler accepts message information (payload and control information) from all CANoe components (IG, Replay, CAPL, Signal Server) intended for this purpose.

Periodic transmission

If a message is defined in the database, the AfdxMsgTxRate > 0 attribute is set, and the control information periodic sending, the Message Scheduler makes sure that this message is sent to the VL Scheduler at the given period without requiring further user actions. Message updates by the clients (2, 4, 6) are then only used to modify the payload without affecting the send rhythm, unless the control information requires that sending be stopped.

Sporadic transmission

For non-periodic messages or for <Single Shot> control (e.g. in replaying logging files), the messages are transferred directly to the VL Scheduler via the IP software stack.

Fragmenting

In relaying messages from the Message Scheduler to the VL Scheduler, the messages pass through the IP software stack and are saved as frames in various queues, depending on whether VL or SubVL. The messages are also fragmented here, if this is required according to the database (AfdxVLMaxFrame) and control information.

VL-Scheduler

The VL Scheduler is essentially the scheduler described in the AFDX specification. It periodically checks (period is found in attribute AfdxVLBag) to see whether a frame is available in the VL or SubVL queues. If yes, it removes this frame by a round-robin method and routes it to the hardware via the AFDX SW stack. The AFDX SW stack takes care of the sequence number and redundancy per AFDX specification based on control information.

Routing to the hardware also depends on the hardware type that is used:

  • Vector hardware that supports time-triggered transmission (e.g. VN5610): In this case, a target time stamp is calculated which is offset a few milliseconds into the future (default: 5 ms), and it defines when the hardware should send the frame. This time stamp is buffered in the hardware together with the frame. This minimizes jitter effects due to task switching and hardware access times (e.g. via USB). Here, the Scheduler also considers the hardware queue size, which can differ according to hardware type.
  • Vector hardware that does not support time-triggered sending (in network-based mode): In this case, scheduling is not done by the hardware, but software-based time-triggered sending with limited accuracy is used.
  • Other hardware such as regular network cards (NICs): In this case, the frames are immediately routed to the hardware and from there they are sent out by a "best effort" method.

Notes

Messages which should be sent fragmented and/or periodically must be described in the database or the send parameters must be described using the CAPL control API (A664VLConfig; A664MSgConfig).

Replay of logging files always occurs in what is referred to as the transparent mode (the logging files contain frames and not necessarily messages). Here, the frames are routed while minimizing hardware effects as much as possible, and the send time point is based on the time difference recorded in the log file. Therefore, no information is needed from the database to replay log files. In time-triggered hardware, there is a future time offset for these frames as well.

In principle, the Message Scheduler accepts Tx requests from all clients (IG, Replay, CAPL, Panels etc). When the Message Scheduler is simultaneous accessed by multiple clients, the user is responsible for avoiding meaningless sequences.
