AFDX Protocol Checks

 

Integrity Check

Each AFDX frame contains a sequence number, which is counted from 1 to 255 by the VL and then starts with 1 again. After a restart or reset the sequence number starts with 0.

Based on IP length and Ethernet length the integrity check analyzes if a sequence number is available and if its value is 1 or 2 higher than the sequence number received before, in consideration of the skip from 255 to 1. The value 0 is allowed always. Packets with wrong sequence number are marked as erroneous.

Redundancy Check

If redundant channels are used, only the first valid packet should be evaluated for redundant packets and the slower packet should be discarded (referred to AFDX). Therefore both packets have to be valid, have to pass the integrity check and the second packet must be received in a specified time limit (SkewMax) which is defined in the database.

If a second packet is received but violates the criteria, it is not discarded but detected as redundancy error for the statistics as well as a missing redundant packet.

Value Range Checks

The AFDX specification ARINC-664p7 defines several requirements referring to value ranges of protocol relevant parameters, e.g. IP addresses, UDP ports etc. With a strict check a violation of this requirements within a packet should be handled as error and consequently the payload (signals) should not be decoded.

Within test environments the requirements are not fulfilled intentionally (e.g. Interface-ID 0 or 3, IP addresses not conform to ICD etc.), however the packet should be completely decoded.

For this use case you can select between strict and non-strict check in the options dialog of CANoe. With the strict check erroneous packets are not decoded, with the non-strict check only the following error types are relevant:

  • InvalidFrames
  • Protocol errors in IP or UDP protocol

Checks and error message output are processed and do not block the decoding.
