Integration Time and Transfer Cycle Time

 

The VT System modules function as follows when making measurement signals available:

  1. The measurement electronics scan the signal with great frequency (on the VT1004, for instance, with 250 kSamples/s). These measurement values are raw values that cannot be transmitted to CANoe in their original form.
  2. The VT System module pre-processes these raw values; for example, it can calculate an average over a given time span. This time span, which can be adjusted, is called the integration time in CANoe. This is a gliding time span. An average with an integration time of 10 ms, for instance, is recalculated every 1 ms using a time span of the past 10 ms. The 10 ms window thus glides over the measurement values every 1 ms (this ratio of 10:1 applies to all values calculated).
  3. CANoe cyclically collects the values thus calculated on the system and uses them to refresh the contents of the corresponding system variables. The transfer cycle time, which can be adjusted, is independent of the frequency with which the values are recalculated (see 2.). This collection cycle time is called the transfer cycle time.
    Please note that measurement values not be updated more often than calculated by the hardware. An average value with an integration time of 1 s will only be updated every 100 ms even if a shorter cycle time is set.

For example, you can derive the average of the past 10 ms on the module every 1 ms (the average is therefore derived from 2500 values, since the VT1004 samples the raw data at 250 kSamples/s). By specifying a transfer cycle time of 10 ms, you can have CANoe update the VTS::Kanal::Avg system variable every 10 ms. The point in time at which the average is derived therefore lies no more than 1 ms in the past (plus a small imprecision due to processing and data transmission).

When values are transferred from the VT System to CANoe, the exact time instant of the value determination is also transferred (time instant of value determination = measurement of a value, or measurement of the last value when a calculation is made based on several measured values). Therefore, the transfer time significantly affects the number of measuring points available and the maximum latency until a value is processed in CANoe. Time uncertainties for transmission and processing also affect only the latency (jitter) and not the time stamp of the values.

Note

  • You can set the transfer cycle time for measurement values in CANoe only for very low values to a short period of time (typically 4 to 10 values to a transfer cycle time of 1 ms, depending on load and processor).

  • To prevent overloading CANoe, you should initially set all measurement values to a longer transfer cycle time (typically 100 ms).

  • If you need a lower value for a specific test case, you should set the transfer cycle time to this lower value at the start of the test case and revert to the longer cycle time at the end of the test case.

In addition, there are special separate system variables for values provided to CANoe with very high resolution (e.g. Avg128us in VT1004). For these values, a certain number of values are stored in a fixed time interval in the system variable (for example, 10 values at an interval of 100 Âµs for Avg128us). Therefore there is only one reasonable transfer cycle (1 ms in this example). The block transmission achieves a high temporal resolution without CANoe becoming overloaded by too many system variable updates. In this case, the latency is relatively high compared to the time resolution (in the example, the resolution of the values is 100 Âµs, but the latency is 1 ms, because the system variable only has to be reset every 1 ms). For these system variables additional slower transfer cycles are only useful to reduce the system load temporarily as described above. Due to the relative high amount of data this option is quite important.
