Timestamps

 

For the bigger part of VT System measuring values, timestamps are made available, that originate directly from the VT System hardware. For this reason the delay of the transmission from the VT System hardware to CANoe has no negative impact on the accuracy of the timestamps.

You have to use the CAPL function sysGetVariableTimeNS to have access to the timestamp that has been provided by the VT System hardware. The function returns a timestamp that uses the same time base as the CANoe simulation time. It is necessary to use this CAPL function since the VT System timestamps are not identical to the current CANoe simulation time, which you can query e.g. with timeNowInt64. Basically these synchronized timestamps that origin from the VT hardware are used when logging VT System measuring values or for the Graphics Window display. The page VT System: Trigger shows an example of how to use this command.

In rare cases it can be reasonable to access the unsynchronized timestamps of the VT System hardware directly. This is the case if solely VT timestamps are compared among each other. In such a case you avoid minimal inaccuracies that result from synchronizing with the CANoe simulation time and hence with the connected hardware devices. For that you can use the CAPL function SysGetOrigTimeNS. A comparison with timestamps from other sources â e.g. CAN interfaces â is not possible.

In order that VT System timestamps use the same time base as the CANoe simulation time, this means that they can be compared with timestamps from other sources, a synchronization mechanism is needed. To reach an optimum of accuracy, Hardware-based Time Synchronization should be used. Therefore a VT6104-, VT6204-, or VT7001 module is necessary. See the help page Time Synchronization for further information.
