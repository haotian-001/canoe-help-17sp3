Remote Debugging

 

Overview

When working with CANoe it may be advisable, for performance reasons, to execute the realtime portions on a separate, dedicated computer. You can do this using a standard computer or also a special realtime computer (e.g. VN8900 or VT6000). If a CANoe configuration that is loaded in this way contains test nodes implemented in a .NET language, you can analyze their execution using the Microsoft Remote Debugging Monitor.

This page describes how to establish and use such a remote debugging connection between a computer and a Vector realtime module (VT6000 family).

When your system is set up as described below you can use CANoe's integrated remote debug support.

For Remote Debugging you need Visual Studio 2003 or higher as a full version (not Express version!).

Concept

As described above, remote debugging of .NET code involves monitoring two computers, the Host PC and the Remote PC.

  • The host PC
    The host computer runs the GUI portions of CANoe. It serves to load the configuration and control the measurement sequence.
    This requires installing Microsoft Visual Studio (Version 2003 or higher) on the host computer to enable remote debugging.
  • The remote PC (VT realtime module)
    The remote computer runs the realtime portions of CANoe. Thus the .NET assemblies are also executed on this computer. In order to debug the .NET assemblies, you need Microsoft Visual Studio Remote Debugging Monitor (MSVSMON.EXE) on the remote computer. The Remote Debugging Monitor communicates with the Visual Studio installed on the host computer to enable debugging via the network interface.

Procedure

Host PC Configuration

The remote debugging connection for .NET code requires a user account on the host computer and the remote computer with the same name and password. There is already a user account configured on the VT realtime module for this purpose:

  • Name rtdebug
  • Password: Debug@VT6000

Proceed as follows to set up the corresponding account on the host computer:

  1. Start the user account administration with command lusrmgr.msc.
  2. Open the Users folder.

  3. Add a new user account by choosing Action|New User…

    Enter rtdebug as the user name and Debug@VT6000 as the password for the new user.
    Deactivate the User must change password at next logon option.
    Activate the Password never expires option (see image).

  4. Right click on the new user account and choose Properties.
  5. Change to the Member Of page.
  6. Click [Add...].

  7. Enter the name of the administrator group (e.g. Administrators; see image)

  8. Confirm the open dialogs with [OK].

VT6000 Configuration

Step 1: Connection between the computer and the VT6000

  1. Connect the VT6000 module with a dedicated network card on the computer using a crossover cable.

    When the upper connection port of the VT6000 is used, it is assigned the static IP address 192.168.100.1.
    When the lower connection port of the VT6000 is used, it is assigned the static IP address 192.168.100.2.

    Upon successful connection, the new network is displayed in the Windows panel.

  2. Now assign a static IP address and the subnet mask 255.255.255.0 of the VT6000 to the computer (see image).

  3. Use the Ping command in the command line to check whether the computer and the VT6000 are connected (e.g.: ping 192.168.100.1).

Step 2: Installation of the Remote Debugging Monitor on the VT6000

After connecting the computer with the VT6000 you can copy the Remote Debugging Monitor to the VT6000.

  1. In Windows Explorer, enter the path to the released Tmp directory on the VT6000.
    Depending on the connection port/IP address used (see above) the required path is \\192.168.100.1\Tmp or \\192.168.100.2\Tmp.

  2. Now copy the Remote Debugging Monitor to the Tmp directory on theVT6000, either

    • from the Visual Studio installation directory (e.g. \Microsoft Visual Studio 10.0\Common7\IDE\Remote Debugger) or
    • from the Visual Studio installation DVD (\vs\Remote Debugger).

    To connect successfully you need to use corresponding versions of the Remote Debugging Monitor and Visual Studio.

Step 3: Starting the Remote Desktop Connection

To configure the Remote Debugging Monitor on the VT6000 you need a remote desktop connection.

  1. In CANoe, open the menu path Home ribbon tab|Group Measurement|Vector Tool Platform|Connection|General|Remote Desktop|Connect.
  2. Enter Administrator as the user name and leave the password field empty.
  3. Confirm your entries with [OK] to start the remote desktop connection.

Step 4: Starting the Remote Debugging Monitor

  1. To start Windows Explorer on the VT6000, enter explorer in the command console of RuntimeServer.EXE.
    The console may be hidden behind another window, in which case you can move the other window to the side.
  2. Navigate to the D:\Tmp directory into which you previously copied the Remote Debugging Monitor.
  3. Right click on the msvsmon.EXE file and choose the command Run as….
    Select rtdebug as the user and enter the password (Debug@VT6000).

Step 5: Configuration of the Remote Debugging Monitor

  1. Choose Tools|Options… to open the Remote Debugging Monitor's options dialog.
  2. Select the Windows Authentication option.
  3. Make a note of the server name (format: Domain\User@ComputerName), as you will need to enter this name in Visual Studio later on.

Checking the Connection

Before debugging the .NET assemblies of a CANoe configuration, you should check whether you can actually connect between Visual Studio (host computer) and the Remote Debugging Monitor (remote computer).
You can use the following steps to check the connection:

  1. Begin by starting the Remote Debugging Monitor (MSVSMON.EXE) on the remote computer as described above.
  2. Make sure the authentication mode in the Remote Debugging Monitor is set to Windows Authentication.
  3. Make a note of the server name (format: Domain\User@ComputerName), as you will need to enter this name in Visual Studio later on.
  4. Start Visual Studio on the host computer (likewise via the Run as… command with user rtdebug).
  5. Open the menu path Debug|Attach to process… from the Visual Studio menu to open the connection dialog.
    Establishing the connection may take several seconds.
  6. Set the Transport field to Default to enable debugging of managed code.
  7. As qualifier enter the server name from the Remote Debugging Monitor's options dialog (Domain\User@ComputerName).

A complete list of processes on the remote computer is displayed.

Error Handling

If Visual Studio and the Remote Debugging Monitor don't connect, please check the following points:

  • The Remote Debugging Monitor has to be started in Windows Authentication mode.
    The /noauth mode may be used for connection checks but does not allow debugging of managed code.
  • In the Visual Studio Attach to process dialog, the transport mode has to be set to Default, as managed code can only be debugged in this mode.
  • The server name of the Remote Debugging Monitor has to be entered correctly in Visual Studio.
  • Both programs need to be able to access the respective other computer.
    This requires that the shared user account rtdebug is set up correctly on both computers (same user name, same password, member of the administrator group).
  • In the Permissions menu of the Remote Debugging Monitor you can explicitly grant user rights to allow access by the host computer.
  • Visual Studio and the Remote Debugging Monitor have to be the same version.
  • Visual Studio and the Remote Debugging Monitor have to be run by the same user rtdebug.
  • On the remote computer you need to configure the network access security in the operating system so that access is enabled based on user name and password authentication, not Guest mode.
    You can configure these setting under Control Panel|Administration|Local Security Policy|Local Policies|Security Options|Network access: Sharing and security model for local accounts.

For further information on error handling, visit Microsoft MSDN pages (https://learn.microsoft.com/en-us/previous-versions/windows/embedded/bb521559(v=winembedded.5)).

Remote Debugging a .NET Assembly

  1. Start CANoe on the host computer and load a configuration.
  2. On the remote computer, start RT kernel (RuntimeKernel.EXE in the Exec32 directory of the CANoe installation).
  3. Configure CANoe on the host computer so that it runs the RT portions on the remote computer, see Vector Tool Platform dialog (Home ribbon tab|Group Measurement|Vector Tool Platform).
  4. If the two computers are connected you can start a measurement in CANoe to check.
    If this test measurement is initialized and executed correctly you can proceed to the next step.

  5. Compile the .NET assemblies in CANoe by choosing the Environment ribbon tab|Compile All Nodes.

    Since this process does not generate any debug information (.pdb file), you need to open the Visual Studio solution of the .NET assembly.
    If CANoe is configured accordingly (see help) you can do this by clicking the [Edit] button on the .NET test node

    .

  6. Execute a rebuild of the assembly in Visual Studio.

    The assembly is then recompiled and the debug information is generated.
    This should generate two files:

    • AssemblyName.DLL
    • AssemblyName.pdb
  7. Now copy the debug information generated in the previous step (AssemblyName.pdb) to the C:\CANrtk\Exec32\SynchronizedFiles directory on the remote computer. The assembly corresponding to the debug information is automatically stored in this directory.

  8. Now start the Microsoft Remote Debugging Monitor (MSVSMON.EXE) on the remote computer.
    Also open the Visual Studio solution of the.NET assembly on the host computer (see above).

    Note that both the Remote Debugging Monitor and Visual Studio need to be started by user rtdebug (e.g. using the RunAs command).

  9. Now start the measurement in CANoe.

    You can debug the .NET assembly in Visual Studio while the measurement is running.

  10. To do so, choose Debug|Attach to Process… in Visual Studio.

    As described above, you can then connect to the Remote Debugging Monitor on the remote computer.

  11. From the list of processes choose the RuntimeKernel.EXE process for debugging.

Error Handling

  • Visual Studio cannot connect with the Remote Debugging Monitor.
    For information on connection problems, see section Checking the Connection.
  • Breakpoints are not activated or there are no debug symbols available.
    This happens when the debug symbol file (.pdb) has not been copied to the remote computer (see above).
    If there are still no debug symbols available even after copying the file you can load the debug symbols file manually.
    To do so, open the Modules window in Visual Studio, and then right click the DLL to be debugged to search for and load the corresponding PDB file.
  • Despite correct loading of the debug symbols the debugger does not stop at the breakpoints.
    In this case you should check whether the .NET code is actually linked with the Simulation Setup in CANoe; it could be that it is linked with the test setup instead. Debugging of .NET assemblies used in the test setup is currently not supported.
