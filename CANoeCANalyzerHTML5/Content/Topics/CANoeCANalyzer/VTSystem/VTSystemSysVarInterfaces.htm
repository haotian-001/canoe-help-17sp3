General

There are groups of system variables that belong together logically and occur more than once as a group. Examples can be found, for example, in the modules of the VT System. Each system variable namespace that represents a VT1004 channel contains the same system variables that can be used to change settings and query measured values. This grouping is represented using system variable interfaces. In addition to accessing system variables, system variable interfaces can also be used to call member functions, such as SetLoadMode for VT1004 channels.

System variable interfaces define a data type that can be used for defining auxiliary functions in CAPL, similar to structures. For namespaces to which a system variable interface is assigned, it is displayed as a data type in the Symbol Explorer and in the System Variable Configuration dialog. Such a data type makes it possible to use an auxiliary function for all similar channels, e. g. VT1004 channels. Otherwise, this is only possible with the VT System, for example, if the path of a channel is passed as a string. Compared to string transfer, system variable interfaces have the advantage that many errors can be detected earlier. This way, typos in names of system variables are detected when compiling the CAPL program instead of only at runtime.

Examples of the use of interfaces are:

  • Modules of VT System
  • Channels of VT System

List of all system variable interfaces of VT System

A list of all available system variable interfaces for the VT System can be found in the System Variables Configuration dialog. Select the System-Defined page and drag the definition area with the splitter to the right. Below the root VTS you will find all available interfaces. In the Data type column, you can see which other interface a type inherits from. Expand the interfaces to see the available system variables. Note that measured value system variables must be activated and may not be available.

System variable interfaces as parameter types

System variable interfaces can be used as parameter types of user-defined functions. The syntax for this is sysvarIF::<Path> <ParameterName>. Neither sysvarIF nor parts of the path can be omitted.

Example

void MyFunction(sysvarIF::VTS::IVT1004Channel myChannel)
{
  ...
}

Namespace transfer

A function that adopts an interface parameter as the data type can accept any namespace that has been assigned the system variable interface (or an interface extending it) as the type. The namespace is identified by sysvar::<Path>.

Example

  MyFunction(sysvar::VTS::M1_Ch1);
  MyFunction(sysvar::VTS::M1_Ch2);

Parameter passing

The concrete parameter of the function can be passed to other functions where the corresponding parameter has the same interface type.

Example

  void MyFunction(sysvarIF::VTS::VT_1004 myChannel) { â¦ }
  void AnotherFunction(sysvarIF::VTS::VT_1004 someChannel)
  {
    MyFunction(someChannel);
  }

Interface check

The parameter of the function can be passed to other functions if a type conversion into the interface type of the respective function is possible.

The special member function IsOfType allows users to check at runtime whether the concrete parameter has a specific interface type. If possible, there is a Cast.

Example

void MySetImpedanceMode(sysvarIF::VTS::IVT1004AChannel someChannel)
{
  someChannel.SetImpedanceMode(eVTSImpedanceModeLow);
}
void MyFunction(sysvarIF::VTS::IVT1004Channel someChannel)
{
  if (someChannel.IsOfType(sysvarIF::VTS::IVT1004AChannel))
  {
    MySetImpedanceMode((sysvarIF::VTS::IVT1004AChannel)someChannel);
  }
}

Access to namespace

Within the function, variables can be accessed in the namespace passed. The compiler knows the name and type of the variables based on the prototype definition in the system variable interface. If one type expands another type, access to variables of the base interface is also possible.

Example

void MyFunction(sysvarIF::VTS::IVT1004AChannel myChannel)
{
  @myChannel::IntLoadResistor = 100.0;
  write("Current voltage: %f", @myChannel::Cur);
}
void SampleFunction()
{
  MyFunction(sysvar::VTS::M1_Ch1);
  MyFunction(sysvar::VTS::M1_Ch2);
}

Name and path

For all system variable interfaces the name and the path can be queried to determine the calling namespace.

Example

void ShowNameAndPath(sysvarIF::VTS::IVT2004Channel myChannel)
{
  write("Channel Name: %s", myChannel.Name);
  write("Channel Path: %s", myChannel.Path);
}

Error codes

The prototype definition of the system variable interfaces contains all system variables that can be contained in the data type. In many cases, system variables can be configured and are only created if they are activated. With the VT System, this applies to all non-writable system variables which in most cases represent measured values. If a variable is not activated in the concrete parameter and therefore does not exist at runtime, the internal function call returns an error code. Function calls like sysGetVariableFloat return this error code directly. For value accesses via the @ syntax, the value 0 is returned or no value is set if the system variable does not exist.

Further functions

Apart from the value access with @, the variables can also be passed to the sysGet/SetVariableType functions and other SV functions (sysGetOrigTimeNS, sysGetVariableDescriptionForValue, sysFilterAddVariable,...). Furthermore, system variables from system variable interfaces can also be passed to all CAPL functions whose parameters have the correct type (sysvarInt *, sysvarFloat *, etc.).

In addition to accessing system variables, in many cases suitable member functions can also be called upon system variable interfaces, such as the VT2004 channel shown here:

Example

void OutputPWMSignal(sysvarIF::VTS::IVT2004Channel channel, double dutyCycle, double frequency)
{
  channel.SetCurveType(eVTSCurveTypePWM);   // PWM output
  ...
}
