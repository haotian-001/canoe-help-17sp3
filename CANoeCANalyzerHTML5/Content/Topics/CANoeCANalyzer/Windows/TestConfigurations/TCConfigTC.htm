Configuring a Test Configuration

Toolbar of the Test Configuration Window: Configuration...

You use this dialog to configure a test configuration.

Start/Stop Event

Here you can define when the test configuration is to be started/stopped:

  • Immediately after measurement start.
  • When the specified environment variable changes to a value other than 0 during the measurement.
  • When the specified system variable changes to a value other than 0 during the measurement.
  • When the specified key is pressed during the measurement.

Alternatively, the test configuration can also be started/stopped manually in the test configuration dialog. A test configuration can be started as often as desired during a measurement. Therefore, multiple start/stop conditions can also be specified and used. While a test configuration is being executed, however, it cannot be started a second time.

Note

  • A test configuration can be started again only a short time (1 ms) after completion.
  • Start events are ignored until processing of the test report has finished.

Execution Options

Multiple Execution

Here you can specify whether a test configuration is to be executed multiple times in succession. It is possible to define the number or duration of the test runs. If you limit the duration to a particular time (days, hours, minutes), the current test run will still be completed once this time elapses.

In spite of multiple execution, only one Test Report is created in which the individual test runs are marked, however.

If a test configuration for which multiple execution is configured is terminated, the verdict remains unchanged if a complete test run is successfully completed. That is, starting from the second test run, the test unit can be manually terminated without this causing the test unit to receive a Failed verdict.

Execution in Random Order

Using this option test sequences / test cases / test case lists / test sequence lists can be executed in random order instead of sequential order. The structure given by test units / test groups / test case lists / test sequence lists is preserved in the test configuration. This means that the mixture of test sequences and test cases just happens within a single test unit / test group / test case list / test sequence list. Thereby the preparation and completion sections remain valid for the same part of the test unit. Furthermore all test cases and test sequences that have been activated in the execution dialog will be executed exactly once.

The Test Report shows all test cases in their actual execution order – numbered according to their appearance in the test report.

For multiple execution in random order of a test configuration you can additionally chose whether to keep the same random order for all test runs or to have a new random order for each test run.

Verdict Impact of Entire Execution Tree

Here you specify whether the test execution of this test configuration shall be aborted if one of the verdicts fail, error in test system or inconclusive occurs.
If the compatibility mode is active, only fail will be shown.

Whether the failed test case or the failed test sequence itself will be aborted immediately is specified during the test creation step in vTESTstudio. This setting can be overwritten with Ignore [Break on Fail] presetting from vTESTstudio.

On error in test system the currently executed test case or test sequence will always be aborted immediately. This abort behavior can be suppressed with the setting Do not cancel execution of elements on verdict "Error in test system".

Debug Mode

Here you define if and when the test run shall automatically be paused.

After pausing the further execution of the test run has to be resumed manually by pressing [Start].

Offline Mode

Here you define whether the test configuration should be available in offline mode or not.

Offline tests enable the observation/check of certain conditions (e.g. cycle times) based on a recorded logging file. Stimulations are not applicable in offline mode, they will be ignored during test execution.

Test Report

Test Report

  • Create joint test report for all test units
    Via this check box you can specify whether an overall test report is to be created for all test units within the test configuration or whether an individual test report is to be created for each test unit.
    If a test report is not to be created at all, deselect this option as well as the corresponding option in the configuration dialog of the test unit.
  • Next test run
    Specifies the next test run value to be used for the {TestRun} field code. The test run will be incremented automatically with each test start.

Output Options

This options are only relevant if the Create jont test report for all test units check box is activated. See above under Report.

Test Report Path

Specify the filename of the test report file in the Test report path input box:

  • use your keyboard to type in a file path in the field
  • use the Field Codes button to create the test report file path
  • use the […] button to open a file selection dialog

An existing file will be overwritten when the test configuration is executed.

Test Report in CANoe Test Report Viewer Format

If you selected the recommended Test Report Viewer Format in the global CANoe options, User Data is available under Test Report Filter. Hereby you can decide whether user-specific data will be written to the test report, e.g. if tester-related information must not be revealed to others due to data privacy reasons. Using the filter feature of CANoe Test Report Viewer you can use further filter criteria. Therefore you don't have to define these criteria in advance.

Test Report in XML/HTML Format

If you selected the XML/HTML test report format in the global CANoe options, you can activate an additional test report in HTML format by selecting Create HTML test report using the following XSLT stylesheet. For that purpose the XML test report file is transformed into an HTML test report file after the test has been performed. This HTML test report has the same filename as the XML test report, except that the extension is HTML instead of XML.

If you don't want to use the standard transformation script, you can select a different XSLT transformation stylesheet by clicking the [...] button.

If you want to used a different XSLT stylesheet once, you can select the XSLT file in the execution dialog in the Test Report menu and used it for an immediate transformation of the last generated XML file.

Test Report Filter

On this page you select the information that is to be included in the XML test report. In the default setting, all generated information will be included in the test report.

Test Cases

Here you can select whether the test report details shall contain all test cases or only those with specific verdicts.
In either case the overview table lists all test cases.

Test Steps and Commands

Here you can select whether the test report details shall contain all test steps and commands or only those with specific verdicts.

For test steps that are implemented in CAPL or .NET, you can also select the test step level at which the test steps are to be reported.

  • If you specify Max. Level: 2, the test steps will be reported with detailing 0 (no information), 1, and 2. The test step level can be specified via the API for the documentation of a test step in the test report.
  • If a test step level less than 2 is selected, test steps not inserted by the user but originating from CANoe itself will also be omitted.
  • In test step level 1, test steps that are purely informative in nature are removed. Error messages, warnings, and other important information are retained.
  • In test step level 0, only test steps that provide information about errors are displayed.

Options

Here you can select the type of information that will be written in the test report:

  • Test unit information
    Here, general information on test units will be documented in the test report; e.g. Description, Misc Info, Extended Info and test unit file.
  • System Under Test, test setup
    Here, the properties of the System Under Test and the test setup are documented in the test report.
  • Tester, test designer
    Here, information on tester and test designer are documented in the test report.
  • Information from test structure elements
    Here, general information from test groups and test cases is documented in the test report.
  • Comments and details
    If this property is selected, comments and additional information, such as the stimulation of signal values, will be documented in the test report.
  • Variable definition, Configuration
    If this property is selected, variable definition (of tests described in tabular form) and configuration information will be documented in the test report.
  • Triggering of simple timeout wait conditions
    Simple timeout wait conditions are waits that depend solely on time and are thus independent of other conditions (such as signal conditions). Such simple wait conditions are used in CAPL, for example, by means of TestWaitForTimeout() or with the Test Table Editor in a [State Change] or [Wait] command.
  • Triggering of complex wait conditions
    Complex wait conditions depend on multiple criteria, usually a timeout plus events or the occurrence of statuses. The reason for triggering the wait condition is always documented in the test report (condition met or wait time elapsed).
  • Report only the first background check violation
    If this property is selected, only the first violation of a check object assigned as a condition or constraint within each test case will be documented in detail. Additional violations are included only in the statistics. If each violation in each test case is to be documented, this property must be deselected.
  • Report result and statistics of background checks
    Check objects keep statistics during the test period that will be output for each test case (while the check is active). Depending on the test task, these may include maximum and minimum information, frequencies, or even distributions.
  • Show hardware information
    If this property is selected, hardware information on the configured VT System modules and on connected network interfaces will be included in the test report.
  • Start and end of background check
    If this property is selected, the start and end of a background check with time stamp will be documented in the test run details of the test report.

Additional Report Information

Here you can enter additional information on the test setup and the system under test. They are transferred to the test report along with CANoe's automatically generated information. In both categories you can add properties consisting of name and value, e.g. hardware and software versions.
