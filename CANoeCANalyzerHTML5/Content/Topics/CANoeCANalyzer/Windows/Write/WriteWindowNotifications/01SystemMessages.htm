01-0001

Assertion <Error number> in <Filename> <Condition>

At the current level of technology, it is not possible to develop completely error-free programs that are non-trivial. Unfortunately, this also applies to CANoe. To better localize and correct system errors that occur infrequently, CANoe has diagnostic mechanisms for reporting these errors. Please be sure to write down the exact text of the error message. With this information our telephone Customer Service can then help you to solve the problem more quickly.

01-0007

can board: Rx queue is full

The receive buffer could not record the received messages any longer.

There are several possible remedies:

  • Disconnect unnecessary branches in the data flow diagram. In an extreme case the statistics block, data block and trace block can be disconnected while the measurement is being recorded with the Logging Block. Afterwards, the measurement is evaluated offline.
  • With Basic-CAN controllers the data stream can be reduced by acceptance filtering. In particular, the second controller could be disconnected entirely by this method, except in special applications.
  • With Full-CAN controllers a data reduction can be achieved by eliminating messages in the message setup in conjunction with Filters.
  • Deactivation of the Statistics Report option or other unnecessary options.

01-0009

can board: Tx queue is full

The transmit buffer is still full. The new transmit request cannot be processed.

There are several possible reasons for this:

  • The baudrate has been set incorrectly; no messages can be transmitted.
  • CANoe is transmitting data faster than the firmware can receive them and pass them to the CAN controller.
    This may occur, for example, if higher priority messages are being transmitted on the CAN bus.

  • The number of messages transmitted directly after one another in a CAPL program is greater than the size of the transmit buffer.

    Above all, this problem occurs when CAPL programs transmit in a loop:

    for (i=0;i<50;i=i+1) output(Msg);

    Remedy: Quick transmission by setting msTimers and reaction to the timer event.

  • The addressed CAN controller is in the BUSOFF state and therefore cannot accept any more transmit tasks.
    This can be recognized in the Bus Statistics Window.

01-0016

If you are playing back replay files that you previously recorded with the logging function in symbolic mode, you should make sure that the same database is associated for the playback.

01-0022

In current mode windows timer is used

If you are using CANoe in simulation mode without a expansion card, you must indicate a substitute time base in the project file CAN.INI. You can use either the computer's realtime clock (RTC) or the less accurate but always available timer of the Windows operating system.

This message appears if you have entered the line WindowsTimer = 1 in section [System] of the project file CAN.INI.

You will also receive this message in the Demo version of the program. This version uses the timer of the Windows operating system as the time base.

01-0023

This message will appear if a timer is not available. The measurement is then terminated.

You should close some applications that use Windows timers before restarting the measurement in Animation mode.

01-0024

Load transition

The ring buffer between the realtime library and the main program has its own fill level indicator.

If the buffer is full or overflowing due to high system loading, an appropriate message will appear (NORMAL->QUEUE HIGH resp. NORMAL->DATA LOST) in the Write Window.

If the loading drops again (possibly after a message burst), you are also informed of this by the message QUEUE HIGH -> NORMAL.

01-0025

Measurement stopped due to recursive lock on system variables update

User-programmability of CANoe with CAPL programs is also associated with the risk of difficult-to-detect programming errors, due to the multitude of possible individual applications. Critical areas include CAPL-programmable recursives. In the simplest case an system variable X might be modified e.g. by the press of a key. This involves calling the CAPL event procedure on sysvar X {}. A recursion occurs when the value of the system variable X is changed in this procedure. More complicated and difficult to detect recursions analogous to this simple example are also conceivable.

To prevent serious system errors due to user-programmed recursions, CANoe has a mechanism for detecting such errors. The measurement is stopped with the error message noted above if the value of an system variable is modified often enough at the same time.

However, in such a case you will receive other instructions in the Write Window in addition to the message shown above.

01-0026

Data history is full: Measurement stopped.

In Options dialog (Measurement|Data History) the behavior is set to stop the measurement as soon as the swap file is full. Now this state is reached. If you don't want this behavior you can set it back to Delete older data gradually.

As an alternative you can enlarge the swap file for the data history.

01-0027

No NODE LAYER (e.g. NWM) available; DUMMY DLL LOADED!!

Some CANoe configurations require supplemental expansions for the CAPL nodes (e.g. network management).
These so-called node layers exist as runtime libraries (DLLs).

You will receive this error message if CANoe does not find the necessary DLL in the Windows path. Make sure that the DLL is located in the CANoe system directory CANWIN\EXEC32 or in the working directory of the CANoe program icon.

You will find additional instructions regarding installation in the description of the runtime library.

01-0028

NWM call not allowed in this node

This error message reports that a simulation of the body electronics bus (comfort bus) for older Mercedes vehicles should be used.

Until version 3.2 SP2 the Network Management was part of the CANoe main program, in later versions this functionality has been sourced out into an add-on.

To have the simulation work properly with CANoe 4.0 or later, please contact Vector to receive the DCznwm (DaimlerChrysler Central Network Management) add-on. It also includes a detailed description of how to port the simulation.

01-0029

Critical memory shortage: Measurement stopped.

See page Critical Memory Shortage - Measurement Stopped for further information.

01-0030

Not enough memory configured for swapping the data history to a file: data history will not be swapped out.

When the data history is stored in a file, a portion of the data (approx. 40 MB) is stored in the main memory for each window with history (especially Trace and Graphics windows) to improve the speed. If you have set a small user-defined use of the main memory in the data history settings, this is not sufficient, depending on the number of windows. In this case, CANoe does not automatically use a swap, but saves the data in the main memory (with the set size). The data history is then correspondingly shorter. If you want a long data history, set a higher main memory usage.

01-0034

01-0035

Replay: Loading was delayed

Large replay files might not fit in Windows main memory under certain conditions. In such cases, CANoe reloads parts of the replay file during the measurement. This warning message appears if the data cannot be reloaded quickly enough. You must assume that the messages are being output on the bus with a time delay.

This situation can occur when CANoe is in a high-load situation or other processes are reducing the overall system performance. To rectify the delayed output of the replay file, make sure that processor-intensive and hard disk-access-intensive processes are not active while playing the logging files.

01-0037

You are using a timer in a CAPL program that was set by the setTimer function, but before the previously set timer has elapsed.

In a CAPL program with multiple timers you can localize a timer by opening the CAPL program with a text editor and counting through the on timer event procedures â starting with 0 â from top to bottom.

The timer number indicated in the warning will match the procedure number of the event procedure counted by this method.

01-0041

In CANoe, environment variables are managed by internal numbers, so-called Handles. When modifications are made to the database, these numbers may â under some circumstances â be modified, even if the environment variable names do not change.

This warning informs you that CANoe has detected an invalid environment variable handle in a program node. This problem can generally be solved by recompiling all CAPL programs.

01-0042

Warning: Bus transmission time resulting from preset baudrateâ¦

In simulation mode CANoe conducts the bus simulation with a time resolution of one millisecond. Bus transmission times below this time resolution cannot be simulated exactly. Therefore, CANoe calculates the bus transmission time of short messages for the preset baudrate. If this is less than the time resolution in simulation mode, you will receive the above warning. Furthermore, CANoe informs you of the smallest animation factor that permits an exact simulation.

01-0043

01-0044

WRITE WINDOW QUEUE LOST MESSAGES

If the system load is too high during a measurement low-priority output of the Write Window are not immediately being displayed but stored in a queue.

  • Some Write Window output might get lost due to the limited size of the queue.
  • Users have no access to lost output.
  • Alternatively you can use the WriteToLog function.
    The output will no longer be displayed in the Write Window but stored in a logging file.

01-0045

! Queue: HighWaterMark/ ! Queue: LowWaterMark

The messages provide information about an internal precautionary measure. If the event queue level in CANoe exceeds the HighWaterMark level, the updating of the windows in the analysis section will stop, i.e., the display and the bus events will no longer be synchronized. This will free up more resources for processing events and the queue level should drop. As soon as the level undershoots the LowWaterMark, the updating of the windows will be reactivated. Marker values are set in the CAN.INI file; typical values are 50 and 5000.

01-0046

Realtime event processing was interrupted at simulation time <t>. Events may be lost!

Temporarily the connected network hardware delivered more events to CANoe than the program could process. Therefore the acceptance of further events from the driver has been stopped temporarily. During such a break some events may get lost if a buffer overflow occurs in the driver. Depending on the bus system this will be shown by further messages.

Use the following remedies to resolve the overload situation:

  • reduce the transfer rate on the bus
  • reduce the complexity of the event processing (simplify the model)
  • enhance the calculating capacity

01-0047

Measurement stopped due to realtime event processing overrun at simulation time <t>

The number of events that can be processed by CANoe was permanently exceeded. Therefore the event processing has been stopped.

Use the following remedies to resolve the overload situation:

  • reduce the transfer rate on the bus
  • reduce the complexity of the event processing (simplify the model)
  • enhance the calculating capacity

01-0048

Measurement stopped due to insufficient event buffer size

In rare cases the event sorting buffer may overflow, e.g. when event sorting is active and many events occur on a channel that is not the selected reference channel, while on the reference channel itself only a few events occur.

Check the settings in Options dialog (Measurement|Time Source & Event Sorting); particularly the selected reference channel and the maximum delay.

01-0050

The Simulation Setup will be adjusted automatically during loading an older configuration with databases that have been assigned to all channels. Please check this database assignment.

01-0060

In CANoe run and CANoe pex you cannot start a measurement, if a CAPL or .NET program has been changed since the last time the configuration has been saved with a full version of CANoe. The variants run and pex do not allow changes in simulation or analysis programs. You have to either reverse the changes or ask for an updated configuration file including the changed programs.

01-0083

SetSignal was called for signal <Signal Name> but there is no Signal Driver available.

CAN

CAN signals can only be sent when the send node is contained in the Simulation Setup and there is an interaction layer configured for it. If there is no interaction layer configured, you can switch on the default interaction layer in Options diaolg (Bus Systems/Protocols|Bus Systems). The default interaction layer sends messages based on event triggering and is used for all nodes that do not have an interaction layer configured. The default interaction layer ignores NM, TP and diagnostic messages.

FlexRay

FlexRay signals can only be sent, if the send node exists in the Simulation Setup. Further reasons for this message can be: no database is assigned for the simulation node, the message is deactivated in the Tx buffer list or the sending relation is not unique (multiple Tx nodes). In case of using a PDU database (FIBEX >= 3.x or AUTOSAR System Description) also the disabling of the property PDU buffer in the Tx Buffer List, the entry of the according raw frame in a Frame Panel or the explicit configuration of the raw frame buffer in CAPL by using FrSetSendFrame(â¦) might lead to this error message on updating a signal of this raw frame.

01-0110

Too many used Handles

CANoe was unable to open a window, because too many handles (Windows USER objects) are already in use. Please close CANoe windows (e.g. panels) that use many graphical elements such as images and try again.

Per default, Windows limits the number of "User Objects" to 10000 per process. You can change this limit in the Windows registry. To avoid depletion of all handles, CANoe refuses to open new windows above a configurable threshold of used handles. You can configure this threshold in the file CAN.INI with the key LimitUserObjects in section [Gui].

01-0111

Duplicated CAN identifier in databases detected.

The databases that are assigned to a CAN channel contain messages with the same CAN identifier. Therefore a message can no longer be assigned unambiguously to the symbols of the databases. As a result some signals of this message cannot be displayed any more.

01-0120

Value of system variable could not be set because conversion from codepage a to codepage b failed.

A CAPL program tried to write a string value that contains unsupported characters to a system variable.

This may happen for example if you execute a CAPL program that has been written in China, on a computer on which the Chinese codepage (the first codepage listed in the error message) isn't installed.

To avoid the problem you can convert the CAPL code to UTF-8, e.g. by using an editor like Notepad++. Alternatively you can install the missing codepage. If the problem occurs on a VN8900 or VT6010, please contact Vector Support.

01-0159

No response to the method call was received within the predefined time and resources for receiving the response are no longer held. If a response still arrives, it will no longer trigger handlers in application models. See the Asynchronous Call section on the Function Calls help page in the CANoe communication concept.
