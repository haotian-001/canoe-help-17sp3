45-0037

Configuration Warning: Not all available GNSS channels are configured. Therefore the time quality cannot be monitored for all channels.

When several VN4610 are used, the time precision and the time synchronization depends on the GNSS-capability of each individual device. The time accuracy decreases if no fix is available, but this can be detected only, if the corresponding GNSS-channel is configured.

For a single VN4610 configuration the complete timing depends typically on the availability of the GNSS-channel, otherwise no precise timing can be guaranteed.

45-0038

Configuration Warning: No GNSS capable time reference channel is configured.

CANoe allows to configure the time source in the Options dialog on the Time Source & Event Sorting page. It is highly recommended to configure an W11p-channel there, because this provides the most precise time synchronization.

If another channel is selected however, this message is shown to indicate a potential measurement risk (timing unprecise).

45-0039

Timing Warning: UTC time reference is based on computer clock only.

The reference clock channel in the Option dialog is set to 'computer clock' or ‘all appropriate’ channels, which forces the system to take the UTC base time from the host computer.

45-0040

Timing Warning: Found UTC time difference > 100 ms for Ch#<channel>: <delta time> ms.

If several GNSS channels are used, usually they should be synchronized over air by GNSS. If some channels don’t have a valid fix or have lost their fix, the local clock on the device is no longer synchronized and may drift away. If the time difference becomes significant, this warning is shown to indicate insufficient timing information.

45-0041

Timing Warning: Time difference between computer time and GPS time is > 10 s: <delta time> s.

At startup the time difference between the host computer and the assigned GNSS-reference channel is checked to have get an additional indication for potential time precision issues. If the difference is too huge, this warning is shown.

45-0042

Timing Warning: Lost fix since more than 5 min for Ch#<channel>; timing may be inaccurate.

For a specific GNSS channel a fix was availbale already, but has been lost since a significant time (5min), which may cause potential time inaccuracies, because the device clock is no longer synchronized by GNSS.

45-0043

WLAN Transmission Skipped: [VN4610 <channel number>] Time <Measurement time> <DCC rule violation>

Following Write Window notifications will be displayed when a frame transmit request is affected by one of the DCC rules and restrictions. As a consequence, this transmit request at the specified measurement time will not be sent at all or not sent immediately as a radio frame over the VN4610 device with the European region setting enabled.

<DCC rule violation>

This part of the Write Window notification explains which one of the DCC restrictions was violated by the individual frame transmit request created at the specified measurement time:

  • DCC does not allow to send this frame immediately due to DCC T_off time limit (CBR=<Percent>%, limit=<N>msec)
    A new transmit request was initiated before the T_off time limit was elapsed since the last successful transmission (among other conditions, T_off is also dependent from the length of the frame in bytes and the current channel busy ratio (CBR) value, here expressed in percent). The new transmit request will be postponed.

  • DCC replaced message in TX queue with(<SrcMACAddress>, <FrameType>, QoS priority=<N> - CBR=<Percent>%, DCC-gap limit=<N>msec)
    In case the DCC Overwrite previous unsent messages setting is enabled in CANoe Options Dialog|Bus Systems/Protocols |Car2x|DCC Settings and a new transmit request has the same origin and event identity (belongs to the same ITS station, have the same frame type (DENM, MAP, …)), the old (outdated) transmit request will be overwritten and never sent as a real radio frame.

  • DCC has surpressed a message transmission with priority <N> and BTP dest. port <N> (CBR=<Percent>%, DCC-gap limit=<N>msec); too many messages with higher priority are waiting
    In case the multiple number of transmit requests are present and due to DCC restrictions not all of them are allowed to be sent immediately, the frames with higher priority will be sent first. For example, older DENM frames will be sent before a (lower prioritized) CAM message.

  • DCC has removed a message with lower priority from transmission queue (priority <N> and BTP dest. port <N>, CBR=<Percent>%, DCC-gap limit=<N>msec) allowing a message with priority <N> and BTP dest. port <N>
    In case the internal transmit queue is completely full of frame transmit requests waiting to be sent as a radio frame, the pending transmit requests with higher priority (DENM) will remain in the transmit queue, lower prioritized transmit requests (e.g. CAM) will be removed and not sent at all. This DCC mechanism ensures that safety relevant frames (e.g. DENM with EEBL flag) will be still sent also in transmit request overload or traffic congestion (high CBR) situation.

45-0044

Coexistence mode disabled: <TIME>: Tx power is no longer limited.

When the geographical position of the sending Car2x device (VN4610) is not in the range of any tolling areas, the 10dBm transmit power restriction for VN4610 will be deactivated. This allows to send Car2x frames with transmit power determined by the Network Hardware Configuration dialog in CANoe and C2xConfigureChannel or C2xSetSendParameters Car2x Interaction Layer CAPL functions.

Coexistence mode enabled: <TIME>: Tx power is limited to 10 dBm.

At least one reason for enabling the tolling zone coexistence mode is present (no valid GPS fix, geographical proximity to a stationary tolling zone (known from tolling zones database or from received CAM frames)).

Coexistence mode enabled: <TIME>: GPS fix lost and timed out.

When the geographical localization of the transmitting VN4610 device is not available due to missing valid GPS signal, the transmit power will be limited to 10dBm until valid GPS position is available again.

Coexistence mode enabled: <TIME>: Approaching stationary tolling zone <ZONE_ID> at <LATITUDE°;LONGITUDE°>

The transmit power will be limited to 10dBm when the transmitting VN4610 device approaches closer than 250m to the geographical location of one of known permanent installed tolling station.

Coexistence mode enabled: <TIME>: Approaching the stationary tolling zone <ZONE_ID> defined in received CAM frame at <LATITUDE°;LONGITUDE°>

Coexistence mode enabled: <TIME>: Approaching the mobile/temporary tolling zone <ZONE_ID> defined in received CAM frame at <LATITUDE°;LONGITUDE°>

The transmit power will be limited to 10dBm when the VN4610 device receives a CAM frame containing the definition and position of one or more tolling zones and the VN4610 device is located less than 250 m from one of the zones.

See also transmit power restrictions applied in tolling areas.

45-0045

Configured Physical Layer: C-V2X PC5

This notification will be displayed each time during the measurement start, when the C-V2X PC5 frame physical layer is activated.

The choice of the physical layer of the frame determines which technology is used in the construction of the frame and in the wireless transmission:

  • IEEE 802.11p DSRC
  • C-V2X PC5

The default frame physical layer is IEEE 802.11p (DSRC). The C-V2X PC5 physical layer can be activated in the Options dialog under Bus Systems/Protocols|Car2x|Physical Layer|Technology.

See also Connect C-V2X Device.

45-0046

See also LPD/EPD protocol type and VN4610 country code. The protocol layout can be changed in the Network Hardware Configuration dialog under Ath|Setup|Protocol Type.
