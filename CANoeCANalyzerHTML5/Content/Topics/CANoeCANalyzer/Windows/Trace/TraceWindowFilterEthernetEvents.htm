Filter Condition | Ethernet Events

Context Menu Path: Add Condition|Ethernet Filter.../AVB Filter.../IP Filter

In order to create filters that are as effective as possible, a distinction is made between Ethernet filter, AVB filter, and IP filter, each of which enables configuration of protocol-specific filter conditions.

You can create these filters not only using the context menu but also with drag and drop from the Trace Output Window.

If a filter is created using drag and drop, the specific values of the event are applied as filter conditions.
If a filter is created using the context menu, the configuration dialog opens immediately where you can set the filter conditions.

Filter Conditions

Attributes Ethernet Filter, AVB Filter, IP Filter

General attributes of the events can be configured for all three filters:

  • With Rx and Tx you can specify the direction into which the filter is to act. At least one direction must be selected.
  • In case of channel-based access: With the Channel attribute you can limit the effect of the filter to a particular channel. If the filter is to act on events of all channels, you can activate this by selecting Eth all.
  • In case of network-based access: With the attributes Network and Ports you can limit the effect of the filter to particular ports of a network. If the filter is to act on events of all ports of the selected network, you can activate this by selecting All.

Note

In case of network-based access the Rx/Tx direction is based on the network interface perspective. If an Ethernet packet is transmitted by the simulation (e.g. by a simulated network node), then this Ethernet packet has direction Rx at the simulation port.

Ethernet Filter

  • The Source MAC address specifies the source of the Ethernet event.
    This sub-condition of the filter applies to events with the specified source MAC address.
    Through specification of a mask as postfix notation, the sub-condition can be softened.
    If 02:A4:00:00:00:00/16 is input, the sub-condition of the filter applies to all events for which the first 16 bits of the source MAC address matches the specified address, or in other words events beginning with 02:A4.
  • The Destination MAC address specifies the destination of the Ethernet event.
    This sub-condition of the filter applies to events with the specified destination MAC address. The filter condition for the destination MAC address can be softened through masks similarly to the filter condition for the source MAC address.
  • With VLAN you can specify the virtual LAN on which a filter is to act.
    If a particular VLAN ID is specified here, the sub-condition applies to all events with the same VLAN ID. For events with multiple VLAN tags, the innermost VLAN tag is considered for this. To adapt the filter for events without VLAN, you can select the Not tagged option. In this case the sub-condition of the filter applies to all events that do not belong to a VLAN.
  • The sub-condition Ethernet type specifies the type of Ethernet events on which the filter is to act. You can select this from the selection list or specify it by name (e.g., ARP) or by entry of its numerical value (e.g., 0x0806).

AVB Filter

  • The Source MAC address specifies the source of the event.
    This sub-condition of the filter applies to events with the specified source MAC address.
    Through specification of a mask as postfix notation, the sub-condition can be softened.
    If 02:A4:00:00:00:00/16 is input, the sub-condition of the filter applies to all events for which the first 16 bits of the source MAC address matches the specified address, or in other words events beginning with 02:A4.
  • The Destination MAC address specifies the destination of the event.
    This sub-condition of the filter applies to events with the specified destination MAC address. The filter condition for the destination MAC address can be softened through masks similarly to the filter condition for the source MAC address.
  • In addition, the AVB Stream ID can be specified as a filter criterion.
    The sub-condition of the filter applies to all AVB packets with the specified Stream ID. If the filter is to act on AVB packets without valid Stream ID, you can select the Invalid ID option.

IP Filter

  • With Source you can specify the source node or the source IP address.
    This sub-condition of the filter applies to events with the specified source node or source IP address.
    Through specification of a mask as postfix notation, the sub-condition can be softened.
    If 192.168.0.0/16 is input, the sub-condition of the filter applies to all events for which the first 16 bits of the source IP address matches the specified address, or in other words events beginning with 192.168.
  • With Destination you can specify the destination node or the destination IP address.
    This sub-condition of the filter applies to events with the specified destination node or destination IP address.
    The filter condition for the destination IP address can be softened through masks similarly to the filter condition for the source IP address.
  • With Protocol you can set that the filter acts on events of a particular protocol.
    The sub-condition applies to all events that contain the specified protocol.
    If UDP is specified, for example, the condition applies to all UDP events as well as all events that use UDP such as SOME/IP via UDP.
  • If you have selected UDP or TCP as the protocol, you can further limit the filter in the TCP/UDP Ports area with Source and Destination.
    In this case, the two sub-conditions apply to all events that contain not only the specified ports but also the specified protocol.
    For all other selected protocols, no ports can be specified. The corresponding fields are therefore deactivated.
    For protocols of a higher layer than TCP and UDP, these are obtained automatically from the Options for Bus Systems/Protocols.
    For protocols of a lower layer than TCP and UDP, ports likewise cannot be specified here because these protocols do not use ports.
  • If you have activated the Conversation option, the filter acts not only on events with the selected Source and/or Destination, but also on the opposite direction.

Wildcards

In order to further adapt an analysis filter, it is possible to ignore certain sub-conditions of the Ethernet protocol by deactivating the desired condition.

Using the destination MAC address as an example, this means that the sub-condition applies to all Ethernet events regardless of their destination MAC address, which corresponds to a wildcard.

Examples

Example 1

The filter acts on all events that are received (Rx) in network Vehicle_Ethernet at port AMP and have Ethernet type 0x88F7 (ptp). In case AMP is a simulation port, this would mean that the filter acts on all PTP packets which the simulation node AMP transmits on Network Vehicle_Ethernet.

Example 2

The filter acts on all events that are received (Rx) in network Vehicle_Ethernet at port AMP and have Ethernet type 0x88F7 (ptp) and the destination MAC address 01:80:C2:00:00:0E and whose source MAC address begins with 02.

Example 3

The filter acts on all events that are received (Rx) in network Vehicle_Ethernet at port AMP transmitted (Tx) on channel Eth 1 and have Ethernet type 0x88F7 (ptp) and the destination MAC address 01:80:C2:00:00:0E and whose source MAC address begins with 02 and that belongs to the VLAN with ID 5.
