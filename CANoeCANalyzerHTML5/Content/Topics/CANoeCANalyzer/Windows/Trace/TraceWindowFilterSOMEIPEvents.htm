Filter Condition | SOME/IP Events

To allow you to specify filters more exactly, a distinction is made between a SOME/IP Filter and a SOME/IP – Service Discovery Filter, each of which enables configuration of protocol-specific filter conditions.

You can create these filters drag and drop from the Trace Output Window.

The Filter Condition dialog is opened

  • with context menu command Edit Condition
  • with a double click on the filter

If a filter is created using drag and drop, the specific values of the event are applied as filter conditions.

Note

Because a SOME/IP-SD event can contain multiple SD entries, a drag and drop of the event creates a filter for each entry. If, on the other hand, a filter is to be created for a particular SD entry of the event, this can be achieved by a drag and drop of the desired SD entry.

Filter Conditions

Attributes

General attributes of the events can be configured for both filters:

  • With Rx and Tx you can specify the direction into which the filter is to act. At least one direction must be selected.
  • In case of channel-based access: With the Channel attribute you can limit the effect of the filter to a particular channel. If the filter is to act on events of all channels, you can activate this by selecting Eth all.
  • In case of network-based access: With the attributes Network and Ports you can limit the effect of the filter to particular ports of a network. If the filter is to act on events of all ports of the selected network, you can activate this by selecting All.

Note

In case of network-based access the Rx/Tx direction is based on the network interface perspective. If an Ethernet packet is transmitted by the simulation (e.g. by a simulated network node), then this Ethernet packet has direction Rx at the simulation port.

SOME/IP

  • Provider specifies the node from which the respective SOME/IP service of the filter is made available.
    The sub-condition of the filter applies to SOME/IP events whose providers match that of the filter.
  • Consumer specifies who makes use of the offered SOME/IP service of the filter.
    The sub-condition of the filter applies to SOME/IP events whose consumers match that of the filter.
  • Different service instances can be differentiated from each other using the Instance ID.
    This sub-condition applies to all SOME/IP events whose Instance IDs match that of the filter.
  • The Service ID specifies the service on which the SOME/IP analysis filter acts.
    This sub-condition applies to all SOME/IP events whose Service IDs match that of the filter.
  • With Method ID you can specify the method of a service on which the filter acts.
    The sub-condition applies to all SOME/IP events in the case of which both the Service IDs as well as the Method IDs match the corresponding values of the filter.
  • The Message Type specifies the types of SOME/IP events on which the analysis filter acts.

SOME/IP-SD

  • Provider specifies the node from which the respective SOME/IP service of the filter is made available.
  • Consumer specifies who makes use of the offered SOME/IP service of the filter.
  • The SOME/IP-SD filter can be further limited by selection of the Service.
  • The Major Version indicates the highest version of the service interface.
  • Different service instances can be differentiated from each other using the Instance ID.
  • The filter can be further refined by selection of the Event Group.
  • The SOME/IP-SD analysis filter can be configured even further, in that it is possible to select SOME/IP-SD types on which the filter is to act, such as Find Service, Offer Service, or Subscribe Eventgroup.

Note

The SOME/IP-SD analysis filter acts on all SOME/IP-SD events that contain at least one SOME/IP-SD entry that meets the filter criteria.

Wildcards

In order to further adapt an analysis filter, it is possible to ignore certain sub-conditions of the filter by deactivating the desired condition.

Using the Instance ID as an example, this means that the sub-condition applies to all events regardless of their instance ID, which corresponds to a wildcard.

Note

The Method ID of the SOME/IP filter is unique only in combination with the Service ID. For this reason the configuration of a wildcard for the Service ID also operates as a wildcard for the Method ID. This situation is also correspondingly indicated in the configuration dialog in that the Method ID can neither be activated nor deactivated.

Examples

Example 1

The filter acts on all SOME/IP events that are received (Rx) in network Vehicle_Ethernet at port AMP. In case AMP is a simulation port, this would mean that the filter acts on all SOME/IP events which the simulation node AMP transmits on Network Vehicle_Ethernet.

Example 2

The filter acts on all SOME/IP notifications that are transmitted in network Vehicle_Ethernet at port MeasurementPort1 from the providerAMP to the consumer HU and whose method PeakLevel of the service CDPlayer matches.

Example 3

The filter acts on all SOME/IP events which occur at one of the configured ports in network Vehicle_Ethernet whose message type of a request matches.

Example 4

The filter acts on all SOME/IP-SD events that are received (Rx) at one of the configured ports in network Vehicle_Ethernet and that have a SOME/IP-SD Offer Service or SOME/IP-SD Subscribe Eventgroup Acknowledgment entry with the Provider CAMF and the service CameraObjectDetection.
