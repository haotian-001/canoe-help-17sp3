MOST Stress Window: Busload on the Control Channel

 

Control

The settings for busload on the control channel are organized into these different areas:

Busload

Here you set how high the busload should be (messages per second) and how many messages to send (total number of messages).

  • The busload can be set to a value between one message per second and a maximum.
    This maximum, shown to the right of the input box for reference, will depend on the bus frequency (typically 44.1 or 48 kHz).
  • If you specify a total number of messages, exactly this number of messages is generated at the preset rate, and afterwards the stress mode is ended.
  • The total number of messages may lie between 1 and 2147483647.
  • To the right of the input box is the Unlimited option.
    If this option is activated the busload is generated continuously.

Message

Here you configure the message to be used to generate the busload.

  • You can configure the message's destination address, message type (normal, remote read, remote write, allocate, deallocate, get source), priority and contents (Hexdump).
  • The [Defaults] button resets the message contents (Hexdump) to the default value for the configured message type.
    This action does not reset the destination address, message type or priority.
  • The Hexdump has the format specified for the configured MOST message type.
  • In the Counter column you can add a passing counter to the generated control messages. The width of the counter can be set from 1 to 4 bytes. The counter's position in the message can be selected in that way that devices receiving these messages will not be influenced by the counter value.

Message: Differences to MOST150

  • You can configure the message's destination address, priority and contents (Hexdump) - there aren’t different message types anymore.
  • The Hexdump has a minimum length of 6 bytes (12 hexadecimal characters).
  • The counter's position has to be set after the sixth byte.
  • The bytes 4 to 6 contain the TelLen. This number indicates the actual length of the user data bytes. After entering the Hexdump the TelLen will be calculated and adjusted automatically.
  • All messages have priority 1.

Low Level Retry Parameter (MOST25)

If a message is not accepted by the receiver, the MOST chip attempts to resend the message after a brief wait time (retry delay).
This is subject to a maximum number of send attempts.

  • Retries may occur if the destination address does not exist in the MOST ring, or if the receiver's Rx buffer is still full and therefore it cannot accept the message.
  • If sending to addresses that do not exist on the ring, or if the addressed device cannot accept the stress mode messages quickly enough, retries will further increase the busload. For this reason it is advisable to set the number of retries (send attempts) to 1. This means that only one send attempt is made per message. This is a way to ensure precise maintenance of the settings made under Busload.

Rx Buffer Disable

As described under Low-Level Retry Parameter retries may occur if a device's Rx buffer is full when the next message arrives. To intentionally induce this state, you can disable the Rx buffer to prevent it from being automatically emptied. The [Disable] button disables the Rx buffer. Now exactly one message, with MOST150 up to four messages, can be received. Afterwards, the MOST Chip rejects all additional messages sent to this device (the Not Acknowledge bit is set in the message).

  • When the disable is activated the [Start disable] button becomes the [Stop disable] button used to deactivate the Rx buffer disable.
  • The [Empty] button is used to clear the Rx buffer once when the disable is activated, to receive the next arriving message.
    Afterwards all further messages are rejected.

Note

When the disable is deactivated the Rx buffer is automatically emptied, so that the node can operate normally again.

MOST150: When the disable is deactivated, up to four messages that were received during the disable can be reported to CANoe. The time stamp of these messages then corresponds to approximately the time at which the disable was canceled, and not the time of receipt.
