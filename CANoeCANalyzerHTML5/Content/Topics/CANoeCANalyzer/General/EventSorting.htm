Event Sorting

 

Since version 5.2, CANoe can sort events chronologically that it receives from different networks before they are processed. The time stamps of the events are used for sorting. An event in this context is, for example, a CAN message or an Error Frame.

It is important to sort events if the different networks are connected via well-synchronized hardware, but some hardware can only pass the events on to CANoe after a certain delay (this is the case, for example, for hardware connected via USB interface). Without sorting, an event with a higher time stamp would sometimes be processed before an event (from another network) with a lower time stamp, which can lead to problems in gateway nodes.

CANoe determines whether sorting is necessary based on the hardware and network configuration automatically. However, you can also configure or turn off the sorting yourself in the Options dialog. The latency time between the receipt of an event and its processing is somewhat lower without sorting.

The Sorting Function

To sort the results, one of the channels is first selected as a reference channel. The reference channel must precisely generate and guarantee time stamps that events are delivered in short time intervals, since it determines the program-internal time.

Reference channel events are then delayed a certain period before processing. If an event from another channel arrives with a lower time stamp in this time, it can be processed before the event from the reference channel. The delay depends on the "slowest" hardware that does not belong to the reference channel.

If an event arrives too early, i.e. it has a time stamp that is greater than that of the last received reference event, it should be delayed somewhat longer, so that it can be correctly sorted. This delay is then normally the delay of the reference events plus the time that the event arrived to early. However, a maximum delay is defined that should never be exceeded.

Unsorted events

If an event arrives so late that another event with a higher time stamp has already been processed, or if it arrives so early that the maximum delay must be exceeded, it can no longer be sorted. In this case, it is processed immediately upon arrival (unsorted). The internal time that is, for example, queried in CAPL with the timeNow function, can then differ considerably from the event's time stamp.

Notes

The chronologically incorrect processing sequence relates only to the processing of the events in the Simulation Setup.

Normally, the unsorted, processed events appear in the analysis windows in the correct sequence – for example in the Trace Window – because the events are sorted once again in the analysis portion of the program. The same applies for CAPL programs in the Measurement Setup.

You can use the function GetEventSortingStatus in CAPL to query whether an event was processed sorted or unsorted in the Simulation Setup.
When event sorting is activated in the Trace Window, the unsorted processed events are displayed with a red exclamation point .
