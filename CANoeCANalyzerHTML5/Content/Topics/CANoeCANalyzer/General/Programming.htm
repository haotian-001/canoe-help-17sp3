Programming

 

CANoe Programming Options

Modeling / Testing / Analysis

  • CAPL

    Using the CAPL Browser, you can write CAPL programs (Communication Access Programming Language) for stimulating, simulating, testing and diagnostics.

    Thus the problem of remaining bus subscribers not yet being available for tests arises when developing control units, for example. To map the system environment, the data traffic of all the other network subscribers can be simulated, for example, using CAPL.

    You can also use CAPL to write programs for problem-oriented analysis of the data traffic or program a gateway to exchange data between different buses.

  • .NET

    You can use .NET programming languages in different areas of CANoe:

    • programming of simulated network nodes

    • programming of test modules, test cases and test libraries

    • programming application models

    • programming of snippets

  • Python

    In CANoe you can use Python scripts for programming application models.

  • C API

    The CANoe C library feature (CCL for short) enables you to write application code in the C programming language and execute it as part of a CANoe measurement.

    A simple C API for accessing services of CANoe is available in your application code. This API enables the sending and receiving of CAN and LIN frames, provides timer services and enables access to the system variables of CANoe.

Communication With External Programs

  • COM

    With the help of the COM Interface you can control CANoe from external programs.

    Besides applications, scripts also come into consideration as external programs.

  • FDX

    CANoe FDX (Fast Data eXchange) is a protocol that can be used for easy, fast, and realtime data exchange between CANoe and other systems via an Ethernet connection.

    The protocol gives other systems read/write access to system variables, environment variables, and bus signals of CANoe.

    In addition, it is possible to send control commands (e.g., for starting and stopping the measurement) to CANoe or to receive status information via FDX.

Realtime Execution With Lowest Latencies

  • CAPL-on-Board

    The network nodes of a CANoe remaining bus simulation can be executed directly on a standard Vector network interface.

    This has the advantage of lower latency times, especially with USB devices. It thus provides a simple way to execute code under realtime conditions.

What Programming Language Should Be Used?

  • CAPL

    CAPL is suitable for all use cases in CANoe. The strengths of the programming language are in particular the realtime capability and the many predefined types and functions with which one can directly solve domain-specific tasks. However, for tasks outside the domain -â e.g. parsing XML files or statistical calculations â it is easier to use the libraries of other programming languages.

    Therefore, program in CAPL in particular if

    • there are realtime requirements for the configuration

    • the programs are to access frames or PDUs of bus systems used in the vehicle, or are to influence the bus directly

    • the functions of the Test Service Library or for vehicle diagnostics are to be used

    • the programs are to influence the simulation of ECUs by using the functions of the corresponding modeling libraries

  • C# / .NET

    C# / .NET is useful for tests and application models if

    • there are no special realtime requirements for the configuration (note that a .NET program and especially the loading of .NET assemblies can also have an impact on the rest of the simulation)

    • the programs mainly work on the application or signal level

    • execution on a Linux platform is not required

    • the developers already have experience with the programming language

    • .NET libraries are to be used or code is to be reused by means of classes and assemblies

  • Python

    Python is useful for application models if

    • there are no special realtime requirements for the configuration (note that the execution of a Python program may also affect the rest of the simulation)

    • the programs work exclusively on the application layer

    • the developers already have experience with the programming language

    • Python libraries are to be used

  • C / C++

    C or C++ is useful for simulation models if

    • the developers have experience with the programming language

    • the programs mainly work on the application or signal level

    • external C libraries or generated C code are to be connected

In many cases it can also be useful to combine CAPL with other programming languages. For example, you can call .NET functions or functions of a DLL developed with C/C++ from CAPL; in tests, you can also call CAPL functions from .NET in reverse.

Programs in different programming languages can also communicate using system variables or internal members of Distributed Objects.

vTESTstudio offers even more possibilities to combine different programming languages when developing tests.
