Configure Sending Behavior

Car2x » Procedures » Configure Sending Behavior

 

General

With the Car2x Network Explorer, network nodes can be configured in such a way that enables cyclic sending of messages.

Attributes View

The Attributes view is located in the bottom left corner of the Car2x Network Explorer. A distinction is made between node attributes, general attributes and stack attributes. Node attributes influence the transmission behavior for the complete node, general attributes influence the send behavior and stack attributes influence the message structure.

Adding Attributes

The Attributes view is empty for databases created with CANoe Version 8.5 SP4 or earlier. No node attributes are available for databases created with CANoe Version 12.0 SP4 or lower. To add attributes to the database, the Add Attributes button must be clicked once.

Configuring Attributes

If a message with attributes is added to a network node, these attributes apply to this network node, effectively immediately.

Node Attributes

Stack Values

On: Fields are automatically set correctly by the stack (depending on the application message and event). Stack attributes in Car2x Network Explorer for manual configuration are deactivated.

Off: Fields are not set automatically. Stack attributes in Car2x Network Explorer for manual configuration are enabled.

Stack Timing

On: The timing of the application message is controlled by the stack (depending on application message and event). The general attributes Send Type and Cycle Time are deactivated.

Off: The timing of the application message is controlled by the general attributes Send Type and Cycle Time.

Use GNSS Sysvar

On: The system-defined system variable GNSS::ATH1 is used to set Latitude, Longitude, Heading, Speed and Elevation in the CAM, DENM, US BSM and CN BSM application messages. GNSS::ATH1 is filled with real position data when using a VN4610 with GNSS receiver, but can also be filled via CAPL. If a scenario is available, it will be ported to the real GNSS position. Can only be activated for one node.

Off: Latitude, Longitude, Heading, Speed and Elevation in the application messages CAM, DENM, US BSM and CN BSM are filled via the data of the existing scenario. If no scenario exists, the values remain empty.

Message Sending

Active: The node sends messages.

Inactive: The node sends no messages. But it is active, which means that the CAPL code is running and any callbacks and tests are working.

Start Node On

Measurement Start: The node is immediately active at the start of measurement.

Scenario Start: The node starts itself on scenario start.

C2xStartNode Command: The node is disabled at the start of measurement and will not send any messages. The node can be activated by executing the CAPL function C2xStartNode.

Station ID

Sets the field for the station ID of the vehicle in the application messages.

Vehicle Length

Sets the field for the length of the vehicle in the application message.

Vehicle Width

Sets the field for the width of the vehicle in the application message.

Station Type

Sets the station type field (e.g. Passenger Car) in the extended header and in the application message.

General Attributes

Send Type
  • NoMsgSendType: No send behavior is defined for the network node.
  • Cyclic: Cyclic send behavior is defined for the network node.

  • CyclicIfActive: A cyclic transmission behavior is defined for the network node if it has been activated. An activation can be done e.g. via the Scenario Editor timeline or via the CAPL function C2xEnableMsg

Cycle Time

Cycle time in milliseconds for sending messages.

Certificate Name

If the messages are to have a security header, the name of a certificate that was previously added to the configuration must be entered here.

Certificate Signer Type
  • Unsecured: A security header is inserted into the message, but the payload is not signed.
  • Digest: The security header contains only the digest of the certificate.
  • Certificate: The security header contains the full certificate.
  • CertificateChain: The security header contains the full chain of certificates.

Certificate Sendtime [ms]

Specifies the time period, in milliseconds, after which the certificate is included in a message. If the Certificate Name and the Certificate Matching fields are empty, this has no effect.

Certificate Matching

Specifies which certificates are used at a change of certificate. For example, if certificates are available named

  • ITS_Cert_1,
  • ITS_Cert_2,
  • ITS_Cert_3, and
  • V2x_Cert_1,

ITS can be entered in this field. Only the certificates beginning with ITS are then considered at a change of certificate.

Certificate Change Time [minute]

Specifies the time period, in minutes, after which a certificate will be changed. If the Certificate Name and the Certificate Matching fields are empty, this has no effect.

Initial Content

If the application message is to be pre-filled with a specific payload (BTP), this can be specified here by selecting the associated, binary-coded file (.per). The message will then be initialized with the corresponding content.

Note: For better portability of the configuration, the binary-coded file (.per) should be stored within the configuration folder (e.g. in a subfolder).

Vehicle Class

(CN stack only)

Specifies the value of the bsmFrame::vehicleClass::classification field of the BasicSafetyMessage message. The value of this attribute is used only if the Stack Values attribute is enabled.

Priority AID (Vehicle Events)

(CN stack only)

If the BasicSafetyMessage contains at least one set Vehicle Event flag (in the bsmFrame.safetyExt.events field), this Application Identifier (AID) value is used in DSMP and Security Header. If this condition is not met, AID values specified in the CN Stack Attributes table for the BasicSafetyMessage are used. The value of this attribute is used only if the Stack Values attribute is enabled.

Stack Attributes US

wssv3 – Provider Service ID

Specifies the Provider Service ID. Must match the Provider Service ID in the WSMP Header.

wssv3 – Generation Time

If this parameter is present, the optional field Generation Time is set to a correct value automatically when sending a message. If this attribute is not present, the Generation Time is not set when sending.

wsmp – Version

Sets the Version field in the WSMP Header.

wsmp – Provider Service ID

Specifies the Provider Service ID. Must match the Provider Service ID in the wssv3 Header.

wsmp – WAVE Element ID

Sets the WAVE Element ID field in the WSMP Header.

Stack Attributes EU

geo_bh – Version

Sets the Version field in the Basic Header.

geo_ch – Next Header

Sets the Next Header field in the Common Header (e.g. BTP-A).

geo_ch – Header Type

Sets the Header Type field in the Common Header (e.g. Beacon).

geo_ch – Header Subtype

Sets the Header Subtype field in the Common Header (e.g.Circular Area).

geo_eh – Country Code

Sets the Country Code field in the Extended Header (e.g. 310 for United States of America).

Stack Attributes CN

al – Protocol Type

Defines the Protocol Type in the Adaption Layer.

dsmp – Version

Sets the version field in DSMP.

dsmp – AID

Defines the Application Identifier in DSMP.

cnsh20 – AID

Defines the application identifier in the security header.

Database Template

The database templates are located next to the demo directroy in Templates\Database\Car2xTemplate.xml. Values that are changed or added are taken into account when Add Attribute is clicked in the future.

CAPL

If other values of the message are to be cyclically changed, a network node must be inserted for this. The C2xSetSignal and C2xRegisterCallback functions can be used to change signals before sending a message.

Supported protocols

The signal sending concept currently supports messages according to the following protocols:

  • IEEE 1609.2 / IEEE 1609.3 with application messages according to SAE J2735.
  • ETSI EN 302 636-4-1 / ETSI TS 103 097 with the application messages ETSI EN 302 637-2 (CAM) and ETSI EN 302 637-3 (DENM).
  • GB/T 31024.3 with the application messages according to C-SAE 0053.

Node Attributes – How to do it

This section shows an overview of the functionality and possible combinations of some node attributes. The examples are always seen from the perspective of a "Device Under Test (DuT)" network node. In addition, a pre-created CANoe.Car2x configuration with one scenario (DuT and SimVehicle) is assumed. The SimVehicle brakes relatively strongly before the DuT and triggers an Emergency Electronic Brake Light Warning.

Example 1: Simulation of a scenario

Configuration:
  • Stack Values: ON (for EU databases)
  • Timing: ON (for EU databases)
  • Use GNSS Sysvars: Off
  • Message Sending: Active
  • Start Node On: Measurement Start or Scenario Start

Result:

When the measurement is started, the configured scenario is simulated.

The position data for the application messages are taken from the scenario.

  • For EU databases, the remaining values of the messages are filled automatically and the messages are sent with correct timing.
  • For US and CN databases, this must be done manually (e.g. in a CAPL script in the preTx function).

Example 2: Stimulation of a V2x control unit in HiL

Configuration:
  • Use GNSS Sysvars: On
  • Message Sending: Inactive
  • Start Node On: Measurement Start or Scenario Start

Result:

When the measurement starts, the configured scenario is started in real, ported to the position received via the system variable GNSS::ATH1.

The position can be generated by a GNSS generator and set to the system variable in CAPL.

The DuT node itself does not send any messages anymore (no longer needed since there is a real V2x controller in HiL).

Nevertheless, evaluations and tests can be performed for this node.

Example 3: Stimulation of a real V2x vehicle (e.g. on a test track)

Configuration:
  • Use GNSS Sysvars: On
  • Message Sending: Inactive
  • Start Node On: Scenario Start or C2xStartNode Command.

Result:

The configured scenario is started real at scenario start or via CAPL function, ports to the position received via the GNSS::ATH1 system variable; this is filled from the GNSS receiver of the VN4610.

The DuT-node itself does not send any more messages (not needed anymore because there is a real vehicle with V2x control unit).

Nevertheless, evaluations and tests can be performed for this node.

Example 4: Use CANoe.Car2x and the VN4610 to turn a normal vehicle into a V2x vehicle

Configuration:
  • Stack Values: ON (for EU databases)
  • Timing: ON (for EU databases)
  • Use GNSS Sysvars: ON
  • Message Sending: Active
  • Start Node On: Measurement Start

Result:

The node is started in real mode at the start of measurement.

The position data for the application messages are filled via the system variable GNSS::ATH1; this is filled from the GNSS receiver of the VN4610.

  • For EU databases, the remaining values of the messages are filled automatically and the messages are sent with correct timing.
  • For US and CN databases, this must be done manually (e.g. in a CAPL script in the preTx function).
