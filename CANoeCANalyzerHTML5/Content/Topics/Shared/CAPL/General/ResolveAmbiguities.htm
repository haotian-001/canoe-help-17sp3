Resolution of Ambiguities

 

With the use of multiple databases ambiguities are possible when symbolic names are used, and these must be resolved by the program. On the one hand, messages arriving from the bus which are registered by the program over one of the two CAN controllers can have different symbolic names in two databases. On the other hand, in certain cases the user may wish to define symbolic messages which have the same name in different databases.

Ambiguities of the first type are resolved by the order of databases on the channel on which the message was received. You can change the order in CANoe in the Simulation Setup.

To resolve name conflicts when configuring measurement windows and function blocks the search sequence of the database list is also utilized. However, you also have the option of resolving ambiguities of this type by qualifying the symbolic name.

Explicit Resolution by Exact Type Specification in CAPL

With this procedure the search for priorities is not necessary. The procedure is available since CANoe 10.0 SP2.

The following type qualifier can be used as prefix to set a specific type for a part of a name:

  • message: dbMsg::<msg-Name>
  • node: dbNode::<node-Name>
  • database: dbNetwork::<database-Name>
  • frFrame: dbFrFrame::<frFrame-Name>
  • frPDU: dbFrPDU::<frPDU-Name>
  • pdu: dbPDU::<PDU-Name>
  • hdr-id: dbHdrId::<ID>

See also: Signal Qualification for Signal Ambiguity

Examples

Example for 2 qualifiers (search via priorities)

message MyNode::message myMessageVar
// Here all 3 priorities are evaluated for CAN messages to search for the specific message.

 

message dbNode::MyNode::message myMessageVar
// Here only the third priority is used for CAN messages.

The use of type qualifiers which are not defined according to the above mentioned rules will lead to an error:

Examples for errors

message dbNode::MyNode::MyMessage::MyXYZ myMessageVar
// The message is already completely specified with "MyMessage"

message dbNode::MyNode::dbNetwork::MyDatabase::MyMessage myMessageVar
// The database name must be set before the node name

message dbNode::MyNode::dbFrFrame::MyMessage myMessageVar
// Even if "MyMessage" is of "message" type the correct type qualifier must be set here thus "dbMsg" instead of "dbFrFrame"
