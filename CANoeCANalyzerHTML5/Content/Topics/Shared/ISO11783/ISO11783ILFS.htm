File Server Interaction Layer

 

A File Server (FS) for ISOBUS is described in specification ISO11783-13. This device can be used by tractors and implements to save and call up data.

CANoe Option ISO11783 provides a File Server interaction layer (FS IL) with which you can expand a simulation node of an ISOBUS network into a File Server.

File Server (FS) Interaction Layer

  • must be supplemented by a CAPL program
  • offers extensive possibilities for fault injection
  • has a programming interface (CAPL)
  • is controlled by a CAPL program
  • can be controlled from the test unit

Functions of the File Server Interaction Layer (FS IL)

  • Configuration of the root directory and other File Server properties.
  • Communication with multiple File Server clients.
  • Control and configuration of the FS IL, including in test nodes.

Setting Up FS IL [▲ back]

The file server interaction layer is configured using the database. Perform the steps below so that a node can use the FS IL:

  1. Define a node in the database (see Defining a J1939 Node).
  2. Set all NmJ1939xxx attributes for this node according to the desired device name.
  3. In the NodeLayerModules node attribute , enter the value ISO11783_FS_IL.dll.
  4. If required, add the diagnostic messages DM1, DM2, DM3, ECUID, DP, PII, IIC, SOFT or DT in the Tx list, which are then sent by the FS IL upon request.

The FSIL_SetNodeProperty function is used to specify the properties of the simulated File Server. In Simulation Setup, you can do this in on preStart or in on start. In a test unit, it can be done in preStart. To change the properties during a test at a later point in time (i.e. during the measurement), the FS IL must first be stopped with FSIL_ControlStop and then restarted after the change is made with FSIL_ControlStart.

With the rootDirectory property you specify the directory where the fixed volumes are located. Each subdirectory is considered as a volume. Which of these subdirectories is to be used as the primary volume can be defined with the property primaryVolume. If no primary volume is defined, the first Removable Volume (i.e. an inserted USB stick or a directory defined with FSIL_ConnectRemovableDevice) is used as primary volume.

Activating FS IL Functions [▲ back]

With the default setting, the File Server interaction layer (FS IL for short) uses network management (Address Claiming).

The FS IL does not respond to a J1939 request automatically (exception: request for address claimed), even with a negative acknowledgment, if a message is not supported. Automatic responding can be activated by assigning various messages in the Tx list of the node, however.

The following messages are filled automatically upon a J1939 request from the FS IL if they are contained in the Tx list of the node:

  • PII (product identification, PGN 0xFC8D)
  • DP (ECU diagnostic protocol, PGN 0xFD32)
  • ICC (ISOBUS certification, PGN 0xFD42)
  • ECUID (ECU identification information, PGN 0xFDC5)
  • DM1 (active diagnostic trouble codes, PGN 0xFECA)
  • DM2 (previously active diagnostic trouble codes, PGN 0xFECB)
  • DM3 (diagnostic data delete/reset previously active DTCs, 0xFECC)
  • SOFT (ECU software identification, PGN 0xFEDA)
  • TD (time date, PGN 0xFEE6)

In the FSIL_OnTxPrepare callback, you can adjust the contents of messages to be sent directly before sending.

Network Management

The FS IL automatically carries out address claiming upon start-up. For this purpose the IL uses the device name that is defined in the database with the NmJ1939xxx node attributes. More information about network management and behavior in the event of address conflicts can be found on the following page: J1939/ISO11783 Interaction Layer Network Management.

Acknowledgment

In order for the FS IL to send the Acknowledgment message (PGN E800h), it must be assigned as a Tx message to the node. If the FS IL finds this message in the send list of the node, but the requested message is not entered in the send list of the node, a request is answered with a negative acknowledgment.

FS IL States [▲ back]

Depending on the state of the FS IL, various functions are activated:

State

Description

Initialized

The FS IL is in this state after the measurement start. It changes automatically to the Claiming state if this is not suppressed with FSIL_ControlInit.

Claiming

The FS IL is in this state during the Address Claiming phase. No message is sent, and only messages of the Address Claiming type are responded to.

Once Address Claiming has been carried out successfully, the FS IL switches to the Active status. If a conflict occurs during address claiming, the FS IL switches to the Stopped state.

Active

The FS IL is activated. The File Server Status message is sent cyclically.

Stopped

The FS IL is deactivated. Messages are no longer sent.

The FS IL changes to this state if, for example, a conflict occurs during address claiming, the advanced address claiming support is not activated and the J1939 IL loses address arbitration.

The state of the FS IL can be queried using the FSIL_GetState function. When the state changes, the FSIL_OnChangedState callback function is called.

The state and the address of the FS IL can also be queried and influenced using system variables, which behave as described for the J1939 Interaction Layer (only the Suspended state is not supported).

The following CAPL functions are available to control the state of the FS IL:

  • FSIL_ControlInit
  • FSIL_ControlStart
  • FSIL_ControlStop

Fault Injection [▲ back]

You can influence the behavior of the FS IL from test units or test modules in order to inject faults. For example, you can generate communication errors during a remaining bus simulation to which the SUT (System Under Test) has to respond.

You can find a description of the associated CAPL functions in the CAPL function overview FS - Fault Injection.

Manipulation of Transport Protocols

You have the option of intervening in the flow of Transport Protocols. With a series of CAPL functions, you can specifically block the transmission of transport protocol messages or delay them as required. With the functions for blocking and delaying individual messages, you can specifically test simple scenarios. However, if the test scenario becomes more complex, these functions are no longer sufficient. CANoe therefore offers you the option of freezing transport protocols and continuing execution in order to inject your own manipulated TP messages in a targeted manner. The functions FSIL_FreezeTp, FSIL_UnfreezeTp and FSIL_KillTp are available for this purpose.

ISO11783 Diagnostics [▲ back]

In order for the FS IL to send diagnostic messages automatically, first activate ISO11783 diagnostic support. Via CAPL you can activate or deactivate errors which are then communicated via the ISO11783 diagnostic messages (DM1 and DM2).

The description of the corresponding CAPL functions you find in the FSIL CAPL Function Overview - DTC Support.

See also Testing Transport Protocols (J1939/ISO11783).
