Task Controller Interaction Layer

 

The CANoe Option ISO11783 contains an Interaction Layer DLL for the Task Controller (TC). This Task Controller Interaction Layer (TC IL) represents an alternative control option for the existing Interactive Task Controller. The TC IL is handled by means of CAPL commands, some of which are also available in a Test Unit.

With the help of the TC IL you can expand a simulation node in an ISOBUS network into a Task Controller without a visible user interface. You assign the ISO11783_TC_IL.dll file to a node in the Simulation Setup for this. The TC IL carries out communication with clients and can be controlled with CAPL commands.

Functions of the Task Controller Interaction Layer (TC IL)

  • Varied configuration of the Task Controller:
    TC version, number of booms/sections/control channels, boot time, etc.
  • Handling of the Task Controller using CAPL commands:
    For example, Starting/stopping a task, sending/requesting the value of a process variable
  • Defining current process variable values and device properties.
    (can also be used in passive mode in parallel with a real Task Controller)
  • Connecting process variables to system variables.
  • Manipulation of response messages of the Task Controller (fault injection).
  • Manipulation of the TC Status message (fault injection).

TC IL CAPL Functions

Setting Up TC IL [▲ back]

The Task Controller Interaction Layer is configured using the database.Perform the steps below so that a node can use the TC IL:
  1. Define a node in the database (see Defining a J1939 Node).
  2. Set all NmJ1939xxx attributes for this node according to the desired device name.
  3. In node attribute NodeLayerModules, enter the value ISO11783_TC_IL.dll.
  4. Add the PD, and ACKM messages to the Tx list.
  5. If required, add the diagnostic messages DM1, DM2, ECUID, DP, LC, SOFT or DT, which are then sent by the TC IL at a request.

The TCIL_SetNodeProperty function is used to specify the properties of the simulated Task Controller. To change the properties during a test, the TC IL must first be stopped with TCIL_ControlStop and then restarted after the change is made with TCIL_ControlStart.

Activating TC IL Functions [▲ back]

In the default setting, the TC IL uses no network management (Address Claiming), sends no PD message, and does not respond to a request, not even with a negative acknowledgment, if a message is not supported. However, this can be activated by assigning various messages.

Network Management

The TC IL automatically carries out address claiming upon start-up. For this purpose the IL uses the device name that is defined in the database with the NmJ1939xxx node attributes. More information about network management and behavior in the event of address conflicts can be found on the following page: J1939/ISO11783 Interaction Layer Network Management.

PD Message

So that the TC IL sends the PD message (PGN CB00h) and thus can actually communicate with the TC clients, this message must be assigned as a Tx message to the node.

Acknowledgment

So that the TC IL sends the acknowledgment message (PGN E800h), it must be assigned as a Tx message to the node. If the TC IL finds the acknowledgment message in the send list of the node but the requested message is not entered in the send list of the node, a request is answered with a negative acknowledgment.

Connecting Objects to System Variables [▲ back]

The TC IL can link objects such as process variables from a transferred object pool (DDOP) to system variables. In addition, the state of an individual section can be linked to a system variable. If the value of an object or the state of section is changed, the system variable is also changed.

The object or state of a section is connected to a system variable with the TCIL_ConnectSysVarWithDataEntity or TCIL_ConnectSysVarWithSection function.

The system variables must be defined by the user. A change of the system variables does not affect the Task Controller.

TC IL States [▲ back]

Depending on the state of the TC IL, various functions are activated:

State

Description

Initialized

The TC IL is in this state after the measurement start. It changes automatically to Claiming state if this is not suppressed with TCIL_ControlInit.

Claiming

The TC IL is in this state during the Address Claiming phase. No message is sent and only Address Claimed messages are responded to.

Once Address Claiming has been successfully completed, the TC IL changes to Active state. If a conflict occurs during Address Claiming, the TC IL changes to the Stopped state.

Active

The TC IL is activated. The TC Status message is sent cyclically.

Stopped

The TC IL is deactivated. Messages are no longer sent. When the TC IL changes to this state, a Cannot Claim Address message is sent.

The TC IL changes to this state if, for example, a conflict occurs during address claiming, the advanced address claiming support is not activated and the J1939 IL loses address arbitration.

The state of the TC IL can be queried using the TCIL_GetState function. At a change of state, the TCIL_OnChangedState callback function is called.

The state and the address of the TC IL can also be queried and influenced using system variables, which behave as described for the J1939 Interaction Layer (only the Suspended state is not supported).

The following CAPL functions are available to control the state of the TC IL:

  • TCIL_ControlInit
  • TCIL_ControlStart
  • TCIL_ControlStop

Fault Injection [▲ back]

You can influence the behavior of the TC IL from Test Units or test modules in order to inject faults. For example, you can generate communication errors during a residual bus simulation to which the SUT (System Under Test) has to respond. You will find a description of the associated CAPL functions in the CAPL function overview TC – Fault Injection.

Manipulation of Transport Protocols

You have the option of intervening in the flow of Transport Protocols. With a series of CAPL functions, you can specifically block the transmission of transport protocol messages or delay them as required. With the functions for blocking and delaying individual messages, you can specifically test simple scenarios. However, if the test scenario becomes more complex, these functions are no longer sufficient. CANoe therefore offers you the option of freezing transport protocols and continuing execution in order to inject your own manipulated TP messages in a targeted manner. The functions TCIL_FreezeTp, TCIL_UnfreezeTp and TCIL_KillTp are available for this purpose.

Controlling a TC Client [▲ back]

You can use the Task Controller Interaction Layer (TC IL) from Test Units or test modules in order to control one or more Task Controller clients. It is possible to change process variables, set trigger conditions, and start or stop tasks. Other functions can be used to check the values transferred to the Task Controller.

The TC IL has no graphical user interface. If you want to visually track the operating steps, you can insert an additional Interactive Task Controller in the configuration in which you then activate passive mode. In this way you can also visually check the operating steps.

You will find a description of the CAPL functions in the following CAPL function overviews:

Making Object Pools Available to the Task Controller [▲ back]

If you do not want to have to transfer a Device Description Object Pool (DDOP) to the TC IL, you can use the TCIL_MakeDdopAvailable function to make a DDOP file (*.xml) known to the TC IL.

If a Task Controller client now uses the Request Structure Label message to check whether the device description is already present and there is a matching structure label, it receives a positive response and the DDOP is not transferred.

Exporting Object Pools [▲ back]

You can use the TCIL_ExportDdop function to export each of the currently existing Device Description Object Pools (DDOP) with its current data.

Peer Control [▲ back]

You can create Peer Control assignment with function TCIL_CreatePeerControlAssignment and you can remove them with function TCIL_RemovePeerControlAssignment again.

You can also create Peer Control assignments manually using the functions TCIL_AssignReceiver and TCIL_AssignTransmitter. To remove the assignments you can use TCIL_UnassignReceiver and TCIL_UnassignTransmitter.

ISO11783 Diagnostics [▲ back]

In order for the TC IL to send diagnostic messages automatically, first activate ISO11783 diagnostic support. Via CAPL you can activate or deactivate errors which are then communicated via the ISO11783 diagnostic messages (DM1 and DM2).

The description of the corresponding CAPL functions you find in the TCIL CAPL Function Overview - DTC Support.
