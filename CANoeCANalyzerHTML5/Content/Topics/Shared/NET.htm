.NET Programming

 

General

You can use .NET programming languages in different areas of CANoe.

  • For the programming of simulated network nodes / application models
  • For the programming of test modules, test cases and test libraries
  • And for the programming of snippets

CANoe provides a special .NET API with specific language extensions (i.e. it creates Embedded Domain Specific Languages). The following programming languages are supported directly:

  • C# (programming language recommended by Vector)
  • Visual Basic .NET

Note on .NET Framework Compatibility

CANoe supports .NET 4.7 language features for .NET-based network nodes / application models. To guarantee compatibility for all environments including VN/VT-Systems higher versions are not supported. For compatibility with existing .NET libraries, you can also choose to build against the .NET Framework 4.0 or .NET Framework 2.0 APIs.

Due to Microsoft's decision to deliver .NET Framework minor version updates as "in-place updates", actually replacing the previous files results in CANoe not being able to build your CANoe network nodes / application models for .NET 4.7 (or 4.0) if the ".NET 4.0 Framework Reference Assemblies" are not installed.

You will typically not encounter problems when a warning appears, if you do not use "extension methods" or .NET features >4.7/4.0. If you however encounter a "TypeLoadException" when running the simulation on another machine than the build machine, you can assure the compatibility on build time, by providing the Reference Assemblies. To do so, please

  • install the .NET 4.7/4.0 reference assemblies by installing
  • Microsoft Visual Studio 2017 (2010 for 4.0) or higher, or
  • the NuGet package "Microsoft.NETFramework.ReferenceAssemblies" or
  • .NET Framework 4.7.2 Developer Pack or
  • Windows SDK 7.1 for version 4.0 (only "Intellisense and Reference Assemblies" component is needed)
    Note: If the installation of Windows SDK 7.1 fails of unclear origin, you may need to uninstall the "Microsoft Visual C++ 2010 x86 Redistributable" package before. (It will automatically be re-installed by the Windows SDK setup.)

Note on .NET Frameworks

CANoe compiles the .NET programs against the .NET Framework 4.7, or alternatively in compatibility mode against the .NET Framework 4.0 or 2.0. Other versions are not supported. Therefore, in Visual Studio, you must specify one of these version numbers for compilation as the Target Framework. The compiled programs can then be run with .NET Framework 4.7, for example.

Starting with Windows 8 you must activate .NET 3.5 Framework manually in Windows in order to use .NET 2 API. Otherwise you may receive the following error message: Compiler executable file csc.exe cannot be found. Additional information on installation of .NET 3.5 Framework can be found in this document: CANoeCANalyzer_Activation_Installation_EN.pdf

Using .NET

Using Visual Studio / Visual Studio Code as your Programming Editor

You can comfortably edit .NET applications by using, for example, Visual Studio or Visual Studio Code as your development environment. Visual Studio Code is available free of charge, as is the Community Edition of Visual Studio under certain circumstances. Express Editions of older Visual Studio versions (up to 2015) are also supported. In order to use Visual Studio for the development of .NET applications in CANoe, open the External Programs page in the Options dialog, click on .NET file editor and select Edit_NET_Source_With_VS_2019.vbs (or another, e.g. 2017), which is located in the Exec64\Scripts folder of your CANoe installation. For Visual Studio Code use Edit_NET_Source_With_VS_Code.vbs.

Instead of the source file you may also configure Visual Studio's solution file (SLN) or the built assembly (DLL) in a network node, test node or for an application model in the Communication Setup. When you select a solution file, CANoe automatically uses an installed Visual Studio to compile the sources. Therefore the solution must have the same name as the assembly generated by CANoe and the assembly must be stored in the path of the solution. If you specify an assembly in CANoe, you have to rebuild it if e.g. the databases have changed.

Note on Creation

It is recommended not to create the solution manually, but to enter the name of the not yet existing solution directly in the configuration dialog. After that you can start the automatic generation of the solution with [Edit]. Thus all settings are automatically correct.

Note on Versions

  • For .NET 4.7 Visual Studio since version 2017 is needed.
  • For .NET 4.0 Visual Studio since version 2010 is needed.

Note

To build Visual Studio Solutions in CANoe or in Visual Studio Code, the msbuild program is required. It is included in a Visual Studio setup or can be installed separately using the Visual Studio Build Tools. Without msbuild only single C# files can be compiled from CANoe.

Access to Signals, Environment Variables and System Variables

For every signal and every environment variable or system variable .NET provides a class which you can use to access the values in the CANoe runtime environment. Example:

double value = EnvSpeedEntry.Value;

Each of these classes has a static property Value. You cannot create instances of these classes, since the value only occurs once in the runtime environment.

Please note that setting the value is just a request to CANoe to change the value. The changed value is not available until this has happened, in general after a certain delay (e.g. after the system variable has been transmitted and received again on the bus).

Access to CAN Frames

For each CAN frame defined in the database an own class is available in .NET. You may create instances of these classes, set signal values and put the frames on the bus with the Send method. Furthermore you can react on received CAN frames with the [OnCANFrame] attribute.

If you don't use a database you may directly use the general class CANFrame or derive own classes from this class. If so, use the [Signal] attribute to define signals.

The API Reference contains further information.

Access to Diagnostics

If one or more diagnostic descriptions are configured, test modules and snippets may send diagnostic requests and receive diagnostic responses using a .NET library. It is possible to set parameters in requests, read them from responses and use any .NET language.

The library Vector.Diagnostics, often referred to as Vector Diagnostic Scripting Library (VDS Library), is supported by several other Vector applications, i.e. diagnostic procedures can be reused easily in CANoe, CANape, CANdito and Indigo.

More information is available in the API reference Vector.Diagnostics Namespace and the VDS Library Quickstart Guide VDS_Library_QuickStart.pdf.

Generating .NET Namespaces

To have access to symbols in .NET, classes will be generated automatically to provide access to objects from databases and other symbol files. Resultant is a restriction when choosing an identifier in these files: they have to follow rules for C# identifiers. In particular C# keywords (e.g. class, public, interface, return..) must not be used. Identifiers that begin with get_ or with set_ can also cause problems.

Note

The validation depends on the exact spelling: "Class" is allowed, "class" is not allowed.
Please see C# language definition for details.

Among others the following files are concerned:

  • databases (signals, messages, environment variables…)
  • system variables
  • CAPL functions that can be called in .NET

Event Procedures

If you want to react to events in CANoe you can add special attributes to methods. The methods are then called when the event occurs (just like in CAPL).
The following attributes are available in CANoe 7.0:

  • OnChange(Type t)
    This method is called whenever the value of the runtime object specified by Type t changes.

Example

[OnChange(typeof(EnvMotorSwitch))]
public void MotorSwitchChanged()
  • OnKey(char ch)
    This method is called whenever a key is pressed.

Example

[OnKey('a')]
public void KeyPressed(char ch)
  • OnTimer(long interval)
    This method is called regularly every interval milliseconds.

Example

[OnTimer(2000)]
public void TimeElapsed()  //  called every 2 seconds
  • OnCANFrame()
    This method is called, if a certain CAN frame is received. The frame can either be identified by parameters of the attribute or by the parameter type of the method.

Example

[OnCANFrame]
public void MotorStateReceived(NetworkDB.Frames.MotorState frame)
  • OnAnyCANFrame()
    This method is called on any received CAN frame. You can use a parameter of the attribute to restrict the call to a certain channel.

Example

[OnAnyCANFrame(1)]
public void CANFrameReceived(CANFrame frame)

For further information on the attributes please refer to the API Reference.

The event procedures in your main class are automatically activated and deactivated by CANoe. You can dynamically activate and deactivate event procedures in other classes. The classes Vector.CANoe.Runtime.Runtime and Vector.CANoe.Runtime.DynamicHandlers can be used for this purpose.

Programming Network Nodes / Application Models

For the programming of network nodes / application models you can simply specify a .NET source file instead of a CAPL file. Alternatively you can also select an assembly (i.e. a .NET DLL) you have created - this is particularly useful if you want to make the configuration available without source code. The assembly must contain exactly one class, derived from Vector.CANoe.Runtime.Script. The class has to have a public default constructor.

This class is instantiated by CANoe at runtime and its event procedures are automatically registered.

If you want to react to the measurement start or stop event, for example to execute initializations or cleanup work, you should derive your main class from Vector.CANoe.Runtime.MeasurementScript. This class contains overwritable methods that are called by the CANoe runtime environment at the start and end of the measurement.

The types from the Vector.CANoe.Threading namespace cannot be used in network nodes / application models.

Programming Snippets

Snippets are small procedures that are used, for example, for the stimulation of tests. To program snippets, configure the .NET source file in the macro configuration dialog. In the resulting assembly all methods are treated as snippets that meet the following requirements:

  • They are public
  • They have no parameters and no return value
  • They are not static
  • Their class is public and has a public default constructor

These snippets can then be carried out in the macro configuration dialog or via the macro toolbar.

Event procedures in a snippet's class are automatically registered while the snippet is running. Before and after the snippet is carried out, they are not registered.

A snippet can be edited and recompiled while the measurement is running (but not while the snippet itself is being carried out).

The types for the Vector.CANoe.Threading namespace can be used in snippets to control the timing.

Migrating from .NET 2 API to .NET 4 API

For the binary compatibility with existing DLLs, the .NET 2.0 version of the CANoe API continues to be supported. In order to benefit from the further API developments, we recommend conversion to .NET 4.7 API, since the .NET 2.0 API supports only the scope of functions of CANoe 8.2 SP2 and will not be further developed. For this you must have access to the source code of your test modules.

Migrating the CANoe Configuration

The .NET 2.0 assemblies are not binary compatible with the .NET API. These must therefore be recompiled.

You cannot migrate to .NET if you are using compiled assemblies of an older CANoe version and have no access to the source code. An error message to that effect is output in the CANoe output window in this case.

You can set the .NET version for each CANoe configuration in the Options dialog (Programming|.NET|.NET Runtime).

The version .NET 4.7 is the default setting for new configurations.

Migrating .NET Test Modules and Test Libraries

The .NET API uses the same files as the .NET 2.0 API. The files of the two APIs are in separate directories, however. The .NET files are in directory <CANoe Installation>/Exec64/NETDev/v40. Already existing test libraries refer to the assemblies in directory <CANoe installation>/Exec64/NETDev/.

For test modules, you generally refer to a source code file. If you have selected Visual Studio as the editor, CANoe automatically creates a project file and a solution file for Visual Studio. You can safely delete the SLN and CSPROJ files generated by CANoe. When test modules are edited, CANoe generates these files again with the set .NET Runtime version and the suitable CANoe API library.

If you have changed the project file during development and want to keep these changes, you can also make the changes manually.

For test libraries, you can create a new CANoe test library project that uses the .NET API.

The exact procedure is described in the Using CANoe .NET API document, Migrating a .NET Module section.
