MOST Tester - Desktops

 

The sample configuration contains a total of six desktops. The last four provide different test scenarios in which test modules can be executed by using fault injection under different conditions:

Setup [▲ back]

This desktop contains the Simulation Setup, the Measurement Setup, a test modul window, the MOST State Window and the Write Window.

Trace [▲ back]

This desktop contains the ADP Control panel, the MOST State Window, two Trace Windows Trace Node Only and Trace Spy Only and the Write Window.

Function Tests 1 [▲ back]

The CAPL test module Tester shows the use of fundamental wait functions of the test feature set: The test case TC_GetMediaInfo works with TestWaitForMostAMSMessage and TestSendMostAMSMessage to test the behavior of the function AudioDiskPlayer.MediaInfo. The test case TC_TrackPosition uses TestWaitForMostAMSReport to analyze the response messages of AudioDiskPlayer.TrackPosition. In addition, it shows how to access the data of a message captured by a wait function. TC_Notification uses TestJoinMostAMSMessageEvent and TestWaitForAllJoinedEvents to test the arrival of the notification messages of the functions TimePosition and TrackPosition. TC_ChannelAlloc uses TestWaitForMostAMSResponse to test whether sending Allocate.StartResult generates a response by AudioDiskPlayer.

The XML test module RequestResponse uses the test patterns <RequestResponse> and <Initialize> to formulate corresponding test processes in XML. Changing the response times and the response behavior with the help of the fault injection panel Notification Error Injection, ADP Error Injection &ndash; TrackPosition, and ADP Error Injection &ndash; MediaInfo allows for creating different test results.

Function Tests 2 [▲ back]

The CAPL test module Extended supplements the uses of wait functions demonstrated in the Tester test module. The test case TC_GatherNotifications uses the function TestWaitForMostAMSReport to collect all notification messages arriving within the specified time frame and lists their function IDs in the report. This allows for influencing the list by means of the notification error injection panel. The test case TC_MethodResponse tests whether sending an Allocate.StartResult message results in the occurrence of a result message and, in the process, also counts any occurring processing messages. The fault injection panel for allocate can be used here to obtain different test results.

The XML test module FunctionsInFBlock checks for the existence of certain functions in a function block by using the test pattern <FunctionsInFBlock>. In its original state, the test module is finished with failed to better demonstrate the operating principle of the test pattern it uses. Other test results can only be created by editing the XML specification of the test module.

FBlock Tests [▲ back]

It contains three XML test modules which are demonstrating the test patterns <FBlocksExistence>, <NetBlockComplete>, and <GenericFBlockTest> that are each operating on complete function blocks. The test modules NetBlockComplete and GenericFBockTest feature a longer execution time and can be disturbed using the system error injection panel since corresponding light&lock or NetState checks are set up in these modules.

The test result of GenericFBlockTest can be influenced by the application, e.g. by using the fault injection panel to prevent the DeckStatus function from responding.

Checks [▲ back]

The test module PropertyProtocol uses the PropertyProtocolError check to monitor the response time of the AudioDiskPlayer.TrackPosition function. It can be set using the corresponding fault injection panel. After its start, the test module runs for approximately 20 seconds and continuously sends request messages to the function to generate status messages. The check is continuously queried about its collected information. The determined values for the number of request messages, the number of unresolved request messages and the average valid response time are displayed in the Graphics Window together with the specified response time. It is recommended to change the response time during the runtime of the test module to be able to observe interesting results in the Graphics Window.
