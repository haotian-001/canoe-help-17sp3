MATLAB Integration: Multitasking

 

This sample configuration is only available after you have installed the MATLAB Integration Package: Open Setup Folder

See further: MATLAB Integration and MATLAB Integration Setup Introduction

Click on the blue link to load the sample configuration in CANoe.

Multitasking – MATLAB MultitaskingDemo.cfg
Shows the effects of the multitasking mode. (MATLAB Integration Package installation required).

Introduction

After opening the sample configuration in CANoe, start the measurement.

The Simulink Model

The model consists of one continuous task (sine wave blocks) and 4 discrete tasks with different sample times, as depicted in the following figure. The first task (red) has a sample time of 0.5 s and outputs a sine wave to CANoe, while the other tasks have sample times of 1 s (green), 2 s (dark blue), and 2.5 s (light blue) and route two system variables through the model.

To simulate a bigger model where computation time matters, additional S-Function blocks are added. These S-Functions read their input value and approximately consume that amount of CPU time in seconds:

  • Task 0 takes about 0.1 s
  • Task 1 takes 0.2 s
  • Task 2 takes 0.4 s
  • Task 3 has a variable execution time. After 3 samples of 0.5 s, one sample takes 1.5 s.

The CANoe Configuration

Desktops

This configuration contains the following desktops:

Multitasking

Parameterization

 

Multitasking [▲ back]

On this desktop the input system variables of the model can be set in Panel1. The corresponding output values can be observed in the Data Window. Furthermore the Data Window shows the currently active task and the task information of all tasks. The Graphics Window also shows the currently active task.

Parameterization [▲ back]

On this desktop the amplitude of the block Sine Wave can be changed in the Panel. The Graphics Window shows the signal before and after the rate transition block.

After starting the simulation, modify the values of the input system variables and observe the output.

  • Increasing In will increase Out in two steps.
  • First, Out is increased as soon as Task 1 is executed.
  • Then, the change has to pass two rate transitions (to Task 3 and back), after that the value is added to the result again.
  • Increasing In2 has the same effect, but the first change is delayed until Task 2 is executed.
  • In3 acts as an amplitude of a sine wave.
  • As soon as this value is unequal 0, the sine value is subtracted from Out2.
  • The value is updated every 2.5 s.
  • The changes are also propagated to Task 1, where it influences Out.

In the Graphics Window, one can observe the scheduling of the tasks (see the figure below):

  • At t = 0.5
  • all tasks are due
  • scheduler first schedules Task 0, which takes 0.1 s
  • at t = 0.6, it schedules Task 1, which takes 0.2 s
  • at t = 0.8, it schedulers Task 2, which takes 0.4 s
  • At t = 1.0
  • task 0 is due again, but Task 2 is not yet finished (red circle in the figure)
  • task 2 is preempted, Task 0 is executed and Task 2 is continued at t = 1.1
  • At t = 1.3
  • task 3 is scheduled
  • during its execution, it is preempted two times (green circles in the figure)
  • At t = 2.2
  • all tasks are completed, the emulated processor goes idle until t = 2.5

Due to the input to the S-Function block, the fourth execution of Task 3 takes longer than its previous executions. It is due at t = 8.0 (first time due at t = 0.5, then each 2.5 s), and during its execution, it is preempted several times. Therefore, at t = 10.5, Task 3 is due again, but its previous execution is not yet finished. Instead of passing the task to the scheduler, the overrun counter is increased now.

On the Parameterization desktop, one can observe the output signal of the Sine Wave block before Sine_Continuous and after Sine_Task3 the rate transposition block. These signals are executed at the beginning of the execution of the corresponding tasks. Because the continuous blocks are computed in Task 0, Sine_Continuous is updated every 0.5 seconds with only a small jitter. Sine_Task3 should be updated every 2.5 seconds. However, due to the fact that Task 3 is often delayed by higher prior task, this update shows a large jitter.
