J1939-76 Basic Functional Safety

 

Click on the blue link to load the sample configuration in CANoe.

  • J1939-76 Basic Functional Safety – J1939-76BasicFunctionalSafety.cfg
    Demonstrates the communication of two safety-critical applications using the SAE J1939-76 Functional Safety Communication Protocol.

In this sample configuration, a safety-critical application (Producer) sends messages that are received and checked by another application (Consumer). Communication takes place using SAE J1939-76 Functional Safety Communication Protocol.

With the Producer panel you can falsify the messages, which is then displayed as an error in the Consumer panel.

After the starting of the measurement, both applications start sending an Address-Claimed message. Then the Producer starts sending the Safety Data Group (consisting of messages SHM and SafetyMessage1) cyclically. The Consumer checks the received messages and displays the positive result by means of green lamps.

Desktops

This configuration contains the following desktops:

Setup

The Simulation Setup contains the network nodes Producer and Consumer, which both use the J1939 Interaction Layer. In the J1939 database, the network node Producer is assigned the messages SafetyMessage1 and SafetyMessage2 as Tx messages in addition to the address-claimed message. Since the database attribute FsJ1939NeedsSHM is set to Yes for both messages, they are sent by the J1939 Interaction Layer with the SAE J1939-76 Functional Safety Communication Protocol.

The Measurement Setup contains a J1939 Scanner (which can be opened to analyze the devices involved), a Trace Window and a Graphics Window.

A warning appears in the Write Window at the start of measurement because the delay between SHM and SafetyMessage2 set by the message attribute FsJ1939Delay leads to a violation of the Safety-Relevant Validation Time (SRVTMaximum).

Measurement

Producer Safety Application

In this panel, you can activate or deactivate the sending of two messages via the Send Safety Data group (SDB) check box. The messages are then sent in the form of a Safety Data Group (SDG). An SDG consists of a Safety Header Message (SHM) and the subsequent Safety Data Message (SDM), which represents the actual message.

While SafetyMessage1 complies with the time constraints for the SDG, SafetyMessage2 violates the maximum Safety-Relevant Validation Time (SRVTMaximum).

In the Fault Injection area of the first message you can block or falsify the associated SHM to force a faulty transmission. The actual falsification or blocking is realized by CAPL functions in the Producer.can file.

Graphics

The Graphics Window displays the sequence numbers of the two messages.

Consumer Safety Application

This panel shows the evaluation of the received SafetyMessage1 message. While a green lamp indicates correct transmission, a red lamp indicates one of the following faults:

  • The SHM was not received.

  • The sequence number in the SHM was not continuously incremented.

  • The CRC transmitted with the SHM does not match the data of the subsequent SafetyMessage2.

The concrete error notification is displayed in the output window.

Trace

The Trace Window performs an independent check of the communication of J1939-76 applications. If it detects an error, it marks the corresponding message with an error icon and outputs a description of the error in the J1939 Violation column.

In addition, the Trace Window displays the contents of the SHM in the J1939 Interpretation column.

The Trace Window also offers a context search, with which you can search for a corresponding SDM or SHM.
