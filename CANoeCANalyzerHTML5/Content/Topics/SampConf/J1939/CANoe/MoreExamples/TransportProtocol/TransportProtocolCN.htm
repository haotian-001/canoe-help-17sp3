Transport Protocol (J1939)

 

Click on the blue link to load the sample configuration in CANoe.

  • Transport Protocol (J1939) – J1939TransportProtocol.cfg
    Demonstration of the use of the J1939 transport protocols.

This sample demonstrates the use of the J1939 CMDT and BAM transport protocols. These transport protocols are specified in SAE J1939-21.

Sending and Receiving of Transport Protocol Messages

You can use the Node1 and Node2 panels to send messages with the transport protocols. Use the input fields to set the parameters of the respective transport protocol. Among other things, you can enter the length of the message to be sent or the destination address. The buttons initiate the transport.

The other node in each case receives the message and displays the number of received data bytes.

The transport can be viewed in the Trace Window. Multiple TPCM and TPDT messages are displayed. CANoe interprets the transport protocols and displays the complete message with a DLC greater than 8.

Sending with J1939 IL

The J1939 Interaction Layer (J1939_IL.dll) provides functions for sending messages with CAPL.

The program nodes show an example of the procedure.

Via the function J1939ILSetNodeProperty you can set the parameters from the Panel for the node.

Then you can append the user data with J1939ILSetMsgRawData and transfer the message to the interaction layer via J1939ILSetMsgEvent.

The interaction layer then sends the message over the bus with or without transport protocol, depending on its length.

Code Example

on sysvar Node1::sysvarSendBAMClicked
{
  byte payload[1785];
  int index;

  if(@this == 1)
  {
    // disable send button during transmission
    enableControl( gNodeName, "SendBAM", 0 );
    //Reset the button variable
    @Node1::sysvarSendBAMClicked = 0;

    J1939ILSetNodeProperty("BAM_DT_Interval", @Node1::sysvarTxBAMInterval);

    //Create some payload
    for(index = 0; index < 1785 && index < @Node1::sysvarTxBAMDLC; index++)
    {
      payload[index] = index;
    }

    //ProprietaryB_Node1 is a PDU2 (Broadcast) message.
    J1939 ILSetMsgRawData(Node1::ProprietaryB_Node1, @Node1::sysvarTxBAMDLC, payload);
    J1939ILSetMsgEvent(Node1::ProprietaryB_Node1);
  }
}

Receiving with J1939 IL

If messages that are sent with a transport protocol are to be received in CAPL, the data bytes can be read out, same as for a normal CAN frame, using the selectors this.Byte(index), this.Word(index), and this.DWord(index). The signal access is also possible.

on pg BigMessage
{
if (this.DLC > 20)
{
write( "Value %d", this.Byte(19) );
}
// access signal, which has offset is greater than 8
write( "Signal value %f", this.SignalName.phys );
}

It is enough to react to the event of the message to access the message in full length.

The interaction layer takes over the transport logic and assembles the message completely.

It is therefore not necessary to manually send Clear To Send (CTS) messages and an End of Message Acknowledgment (EoMA) message in CAPL.

To determine the maximum number of packets per sequence that can be received, the property must first be set using J1939ILSetNodeProperty (Packets_Per_CTS, ctsPacketsCount).

If a node that can also act as a receiver of the CMDT protocol is to be simulated with CAPL, the CTS and EoMA messages must be sent. The sending of the CTS and EoMA messages must be programmed in CAPL.

Desktops

This configuration contains the following desktops:

Setup

The Setup desktop contains the Simulation Setup and Measurement Setup. The Simulation Setup contains 2 simulated nodes, which use the interaction layer (J1939_IL.dll) for the protocol logic.

Measurement

This desktop contains the Trace Window and the panels of the simulated nodes.

The panels for controlling the nodes are structurally divided into two sections.

In the upper half are the input and output elements for the protocol Broadcast Announce Message (BAM).

DLC specifies the length of the message to be sent and Packet interval time specifies the delay between successive packets. The Send BAM button triggers the transmission. Afterwards the length of the received data is displayed in the Received data length field at the other node. This corresponds to the DLC at the sending node in case of an error-free transmission.

The lower half shows the input and output elements for the protocol Connection Management Data Transfer (CMDT).

You can set the following properties:

DLC
Length of the message to be sent.

Packet delay
Delay of successive TP.DT packets containing the user data.

Packets per RTS
Maximum possible number of packets to be sent within a sequence.

Destination address
The destination address to which the data is sent.

Packets per CTS
Maximum possible number of packets to be received within a sequence.

Send CMDT
This button triggers the transfer.

Reveived data length
This output field of the other node shows the length of the received data.
