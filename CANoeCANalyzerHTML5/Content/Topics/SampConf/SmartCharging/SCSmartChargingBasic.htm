Smart Charging Basic Simulation

 

Click on the blue link to load the sample configuration in CANoe.

  • Smart Charging Basic Simulation - SmartChargingBasic.cfg
    Demonstrates the communication between an electric vehicle and a charge point according to DIN70121 and ISO 15118‑2, including possibilities for testing and fault injection

Introduction

The Smart Charging Basic Simulation configuration presents the most important features of the Option SmartCharging by combining a vehicle simulation to a charge point simulation.

The Simulation Setup includes the following simulation nodes:

  • EV: the vehicle simulation.
  • EVSE: the charge point simulation.
  • Crosstalk_EVSE: an additional charge point which only participates during the SLAC phase. It is intended to show the mechanics of the SLAC procedure.
  • Monitor: an SCC Monitor node.

Desktops

This configuration contains the following desktops:

Simulation Monitoring V2G Test SLAC Test Configuration

Simulation

On the Simulation desktop, you can observe the communication of an individual vehicle with the charge point. The behavior of the participants can be influenced using the respective panels. To start the communication, simply click on the Connect button of the vehicle. The simulation will run until the charge loop, at which point you can increase the charge point’s output values to transfer energy to the vehicle.

In the Trace Window, the SmartCharging messages (SLAC, SDP, V2G) are shown in decoded form with full data interpretation.

Note

When the battery is fully charged, the vehicle will automatically stop the session. If the connection is interrupted more than three times, the vehicle stops trying to connect. You can reset this by restarting the measurement.

Vehicle Panel

This panel serves as the control and display element for the vehicle simulation module. The following controls are available:

Control Description

Connector button

Connects the vehicle to the charge point or disconnects it again. As soon as the button is clicked, the sequence of the SCC protocol begins.

Charging Mode

Switches between AC and DC mode. The charging mode is adopted as soon as the ChargeParameterDiscovery message is sent. It cannot be changed thereafter for the protocol session.

Battery

Indicates the current charging state of the battery. The [Clear] button can be used to reset the battery state.

Vehicle state

The current state within the (simplified) SCC state machine.

Stop / Play / Pause

The two buttons on the right can be used to pause and resume the vehicle simulation. The [Stop] button can be used to reset the simulation. The state of the simulation module is displayed by the state LED inside the picture.

Schema

Use this switch to select the protocol version to be used for the connection. The protocol version is adopted as soon as the SupportedAppProtocol message is sent.

PnC Active

When the ISO 15118 schema is selected, you can also switch between PnC and EIM mode here. Set the Active check box for PnC mode.

Certificate Installation

Initiates a Certificate Installation procedure at the next protocol start (PnC mode only).

Certificate Update

Initiates a Certificate Update procedure at the next protocol start (PnC mode only).

Stop Charging

Performs a regular shutdown of the protocol once in the charge loop.

ReNegotiation

Initiates a re-negotiation procedure while the protocol is inside the charge loop.

Message Tx LED

Flashes each time a message is sent by the vehicle.

DC tab

Access to values relevant only to DC charging: the charging mode (core or extended, the DC status code, the Ready flag and the voltage and current values of the requests.

Limits tab

Electrical limit values used in various messages.

Misc. tab

Controls for changing the charge loop interval, which defines the interval of successive requests when inside the charge loop.

EVSE Panel

The charge point panel serves as the control and display element for the charge point simulation module. The following controls are available:

Control Description

Sessions

Displays the Session IDs of all active connections.

Message Tx LED

Flashes each time a message is sent by the charge point.

Return FAILED

Instructs the charge point to label the next response message as an error.

Abort Conn.

Aborts the connection immediately, without notifying the vehicle.

Processing

The text Processing… is displayed while the flag EVSEProcessing is set to 1. Turn on EVSE Processing Time under Simulations Settings to enable the use of EVSEProcessing.

Stop / Play / Pause

The two buttons on the right can be used to pause and resume the charge point simulation. The Stop button can be used to reset the simulation. The state of the module is displayed by the state LED inside the picture.

Payment Options

Choses the available payment option(s) sent in the ServiceDiscoveryRes.

ReNegotiation

Initiates a renegotiation procedure when inside the charge loop.

Generate NMK

Generates a new pair of NMK and NID for the SLAC process and sends a message to an assumed HomePlug Green PHY chip. This pair is consumed with the next SLAC session.

Stop Charging

Sends a stop request. The exact mode of operation depends on the chosen schema.

DC status

Indicates the internal EVSEStatusCode (in DC charging mode only).

Power, Limits… tabs

Influence the power, voltage and current limits of the charge point which are sent within several messages.

Note

When in a DC charging session, the charge point will automatically increase the voltage during the PreCharge loop up to the requested value, simulating the behavior of a real charge point in the PreCharge loop.

Protocol Panel

In the protocol panel you can track the sequence of the DIN70121/ISO15118 protocol and configure the general behavior. If messages are received, the current protocol status is marked in the sequence view, and the return value of the charge point is displayed. For simplification purposes, not every displayed status corresponds to an individual message.

The following controls are available:

Control Description

Sequence View

Initialization LEDs

These LEDs display the progress of the SLAC protocol.

Protocol bar

Indicates the current state of the protocol and the response codes sent by the EVSE.

Signature verification icons

When a signed message is received (only when ISO 15118 with PnC mode is configured), the status of the verification is shown as green or red icon overlaid on the protocol bar.
Simulation Settings

Message delay

The linear regulator controls the delay for the sending of the SCC messages. The greater the value is, the larger the interval is between individual messages.

Log message data to Write Window

Deactivate to reduce the Write Window output of the simulation by removing the output of Rx message data.

Processing Time

If checked, the charge point will simulate a processing time for each message pair that contains the flag EVSEProcessing, causing a message loop. If not checked, the simulation will proceed directly to the next message pair.

TLS

Activates TLS usage the next time a connection is established. Configuration is done automatically using the provided security profile.

Verbose Write output

Check to activate additional Write output of the simulation modules.

Note

The message delay regulator does apply to all message pairs. However, the charge loop additionally has its own timing behavior to meet the specifications. Thus the charge loop interval usually doesn’t match the configured message delay.

Monitoring

The desktop Monitoring presents the functionality of the SCC Monitor module. While the simulation is the same as on the Simulation desktop, all data shown is provided by the Monitor node as a passive listener, instead of the simulations modules themselves.

A small subset of the monitoring variables is displayed in Data Windows, and AAG measurements from the SLAC protocol are shown in the Graphics window. Note that these AAG measurements are a union of two different charge point nodes: EVSE and Crosstalk_EVSE. Deactivate the latter node to have a single AAG profile displayed.

V2G Test

This desktop is a demonstration of the simulation modules’ possibility to send configurable SCC messages at arbitrary points in time. One or both of the participants can be run in passive mode for the V2G communication. Its V2G protocol flow can then be manually controlled by sending arbitrary SCC messages from the panel. This is intended to resemble a test scenario, where the active entity would be the System under Test (SUT) and the passive entity would be the Tester. The Tester can check if the SUT replies by evaluating the callback functions.

Protocol Panel

In the protocol panel you can configure and start the testing scenario.

Control Description

Controlled Entity

This sets the entity to be controlled by the user. E.g. if the vehicle is under control, it will go to passive mode directly, while the charge point is running its state machine, and reply to any request sent by the user.

Note

You can still send both requests and responses on demand, no matter which participant is under control. This shows how the free sending feature can be used in both active and passive mode.

Schema

Chosen schema for messages to be sent.

Charging Mode

In case the EVSE is controlled, the EV must decide which charging mode to send in the ChargeParameterDiscoveryReq message. This controls that decision.

Apply

Similar to the Connector button, this connects both participants, but additionally applies the passive mode as configured.

TLS

If checked, the connection will use TLS encryption.

Add optional parameters

Enables or disables the sending of optional message parameters.

Disturb header

Switches bits in the V2G header to induce errors.

Switch roles

If active, requests will be sent by the EVSE instead of the EV, and vice versa.

SendFunctions Panel

This panel is used to send SCC messages. All available messages are shown on the panel and they can be sent at any time. The only limitation is that for all messages except the SECC Discovery, a TCP or TLS connection must exist. Thus, the first message sent must always be the SECC Discovery Request or Response (depending on the controlled entity). Also, it is recommended to send the Session Setup messages to create a valid session ID. Else, the remote station will not accept the messages sent.

Note

The parameters for the messages are all set by CAPL and/or by the test configuration within the XML configuration files. Have a look at the CAPL code to see how messages can be parameterized.

SLAC Test

Similar to the V2G Test desktop, but showcases the passive mode in regard to the SLAC protocol. SLAC messages can be sent freely using the panel.

Configuration

This desktop contains the basic configuration windows, as well as access to the XML configuration files of the two simulation nodes. In these XML files, some basic parameters as well as necessary structures for the V2G messages are stored.
