CCS Basic Simulation (Communication Setup)

 

Click on the blue link to load the sample configuration in CANoe.

  • CCS Basic Simulation (Communication Setup) - CCSBasicDO.cfg
    Demonstrates the communication between an electric vehicle (EV) and a charging station (EVSE) according to the ISO15118-20.

Note

Some functions are not supported yet. You can find more information here.

Introduction

The configuration presents the most important features of the Option SmartCharging by connecting a vehicle simulation to a charge point simulation.

In the Simulation Setup, you will find the following application models:

  • EVSEsimDO.can: Provides data for the charging station simulation

  • EVsimDO.can: Provides data for vehicle simulation

  • Monitor.can: Helps to display the data

Desktops

This configuration contains the following desktops:

Simulation CP State Monitor V2G Test SLAC Monitor SLAC Test Configuration

 

Simulation

On the Simulation desktop, you can observe the communication of an individual vehicle with the charge point. The participants’ behavior can be influenced using the respective panels. To start the communication, simply click the Connect button of the vehicle. The simulation runs automatically until the end of the communication.

In the Trace Window, the SmartCharging messages (SLAC, SDP, V2G) are shown in decoded form with full data interpretation.

Note

At full battery state, the vehicle automatically stops the session.

EV Panel

This panel serves as an operator control and display element for the vehicle simulation module. The following operator controls are available:

Operator Control Description

Connector button

Connects the vehicle to the charge point, or disconnects it again. As soon as the button is pressed, the sequence of the SCC protocol begins.

Charging Mode

Switches between AC, DC and the corresponding BPT modes. The charging mode is adopted as soon as the ChargeParameterDiscovery message is sent. It cannot be changed afterwards for the protocol session.

Battery

Indicates the current charging state of the battery. The [Clear] button can be used to reset the battery state.

Vehicle state

The current state within the (simplified) SCC state machine.

Reset/Pause

The two buttons can be used to pause and reset the vehicle simulation. A restart is triggered by (double) clicking the [Connector] button.

Schema

Using this switch you can select the protocol version to be used for the connection. The protocol version is adopted as soon as the SupportedAppProtocol message is sent.

Stop Charging

Ends the protocol in a regular way, from the charging loop.

Charge Loop Interval

Controls for changing the loading loop interval, which defines the interval of successive requests within the loading loop.

DC tab

Access to values relevant only to DC charging: the charging mode (extended) and the voltage values of the requests.

AC tab

Access to values relevant only to AC charging: the performance values of the requests.

Limits tab

Electrical limit values used in various messages.

EVSE Panel

This panel serves as an operator control and display element for the charge point simulation module. The following operator controls are available:

Operator Control Description

SessionID

Displays the session ID of the active connection.

Return FAILED

Instructs the charge point to label the next response message as an error.

Abort Conn.

Aborts the connection immediately, without notifying the vehicle.

Reset/Pause

The two buttons on the right can be used to pause and resume the charge point simulation. A restart is triggered by (double) clicking the [Connector] button.

Payment Options

Choses the available payment option(s) sent in the ServiceDiscoveryRes.

Stop Charging

Sends a stop request; the exact mode of operation depends on the chosen schema.

EVSE Notification

Displays the last EVSENotification sent.

AC,DC, Limits… tabs

These influence the power, voltage and current limits of the charge point which are sent within several messages.

Protocol Panel

In this panel you can track the sequence of the ISO15118-20 protocol and configure the general behavior. When messages are received, the current protocol status is highlighted in the sequence view and the return value of the charging station is displayed. For simplicity, not every status displayed corresponds to a individual message.

The following controls are available:

Control Description

Sequence View

Initialization LEDs

These LEDs display the progress of the SLAC protocol.

Protocol bar

Indicates the current state of the protocol and the response codes sent by the EVSE.
Simulation Settings

Message delay

The linear regulator controls the delay for the sending of the SCC messages. The greater the value is, the larger the interval is between individual messages.

Processing Time

If checked, the charge point will simulate a processing time for each message pair that contains the flag EVSEProcessing, causing a message loop. If not checked, the simulation will proceed directly to the next message pair.

TLS

Activates TLS usage the next time a connection is established. Configuration is done automatically using the provided security profile.

Note

The message delay regulator does apply to all message pairs. However, the charge loop additionally has its own timing behavior to meet the specifications. Thus the charge loop interval usually doesn’t match the configured message delay.

CP State Monitor

Using this desktop, the measured voltage and the measured, as well as expected, CP state can be tracked on the EV and EVSE side respectively.

V2G Test

This desktop is a demonstration of the simulation modules' ability to send configurable SCC messages at arbitrary times. For V2G communication, one or both participants can be operated in passive mode. The V2G protocol flow can then be manually controlled by sending arbitrary SCC messages from the panel. This is intended to resemble a test scenario where the active unit is the System Under Test (SUT) and the passive unit is the Tester. The tester can verify that the SUT is responding by evaluating the callback functions.

Protocol Panel

In the Log panel you can configure and start the test scenario.

Control Description
Controlled Entity

This sets the entity to be controlled by the user. For example, when the vehicle is under control, it goes directly into passive mode while the charging station operates its state machine, and responds to any request sent by the user.

Note

You can still send both requests and responses on demand, regardless of which participant is under control. This shows how the free send feature can be used in both active and passive modes.
Schema Selected scheme for messages to be sent.
Charging Mode If the EVSE is controlled, the EV must decide which charging mode to send in the ChargeParameterDiscoveryReq message. This controls this decision.
Apply Similar to the Connector button, this connects both participants, but additionally applies passive mode as configured.
TLS If this option is checked, the connection is made using TLS encryption.

PnC Active

If selected, the AuthorizationRequest and -Response will be populated accordingly.

CertificateInstallation

If the EVSE is controlled, this button is used to decide whether the EV sends a CertificateInstallation message.

SendFunctions Panel

This panel is used for sending SCC messages. All available messages are displayed on the panel and can be sent at any time. The only limitation is that for all messages except SECC Discovery there must be a TCP or TLS connection. Therefore, the first message sent must always be the SECC Discovery Request or Response (depending on the controlled entity). Additionally, the SupportedAppProtocol should be sent, as the schema used is read directly from this message pair. It is also recommended to send the Session Setup messages to create a valid session ID. Otherwise the remote station will not accept the sent messages.

Note

The parameters for the messages are all set by CAPL. Take a look at the CAPL code to see how messages can be parameterized.

SLAC Monitor

Monitor Attenuation

This panel serves as a display element of all attenuation values sent during simulation or with hardware.

Trace Window

See Simulation desktop.

SLAC Test

Similar to the V2G Test desktop, but shows the passive mode with respect to the SLAC protocol. SLAC messages can be sent freely via the panel.
