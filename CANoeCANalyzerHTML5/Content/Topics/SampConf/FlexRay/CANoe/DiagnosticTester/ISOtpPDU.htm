Diagnostic Tester ISO TP PDU (FlexRay)

 

Click on the blue link to load the sample configuration in CANoe.

  • Diagnostic Tester ISO TP PDU (FlexRay) – UDSsimExt_Fr_ISOTP_PDU_based.cfg
    Shows the interactive Diagnostic Features in the environment of FlexRay Diagnostics (Diagnostic Console, fault memory and variant coding) as well as the analysis functions for diagnostics, based on the ISO transport protocol.

Introduction

This sample configuration shows the following use cases and functions:

  • Usage of the Diagnostic Windows (Fault Memory Window, Diagnostic Session Control, Diagnostic Parameters Window) and Diagnostic Features (Diagnostic Console, Variant Coding and ECU Control).
  • Usage of the analysis windows (Trace Window) to display the values of the diagnostic parameters.
  • Usage of panels for triggering diagnostic requests and displaying the values of diagnostic parameters.
  • Usage of CDD diagnostic descriptions and basic diagnostic descriptions.
  • Configuration of the FrTP parameters in the Diagnostics/ISO TP Configuration dialog.
  • Security Access by usage of a Seed & Key DLL via the ECU controller.

The sample configuration contains a simulated ECU that implements complex diagnostic functions (ECM program node in Simulation Setup) and a program node that implements test functions provided interactively by the panels (Tester program node in Simulation Setup).

Diagnostic Console [▲ back]

ECU State and Data

This panel contains several areas.

ECU Status

This area shows the three different diagnostic sessions:

  • A vehicle symbol with closed hood represents the Default Session (ECU always locked).
  • The open hood stands for the Extended Session (unlocking of the ECU possible).
  • A computer symbol represents the Programming Session.


The ECU can be blocked, i.e. certain services may not be executed and the ECU denies safety access. If a request is received for which positive responses are suppressed (new UDS function), the Suppressed Response LED lights up briefly.

If the ECU sends a negative response, the Negative Response LED lights up.

Note that if no requests are received for the S3 time (can be found in the Diagnostics/ ISO TP Configuration dialog), the ECU will fall back to the locked default session. If you manually deactivate sending TesterPresent, you must manually activate it again to keep the ECU in a non-default session.

Serial Number

Here you can view the current serial number of the ECU.

Expected Security Key

To allow you to manually unlock the ECU from the Diagnostic Console, the panel displays the key value that the ECU is expecting at the moment. (Do not expect to find such a display on a real ECU).

Variant Coding Data

These values can be changed and queried via the Diagnostic Console or the variant coding control.

Device Control

With these controls you can change the states of the door contacts.

ECM Diagnostic Console

Diagnostic ActiveX Control [▲ back]

This desktop contains, among others, the following windows and panels already described above:
ECU State and Data, Trace Diagnostics, Trace All and Write.

The ECU simulation for ECM includes the state transitions between the three different sessions (Default, Extended and Programming) as defined in the diagnostic description (CDD file), as well as different levels of security access lock. Some service requests require specific sessions or lock states.

You can view the state transition definitions in the CDD file. (The CDD files can be found in the Diagnostics/ ISO TP configuration dialog).

The following is implemented in the simulation ECU:

  • Access to variant coding data is only possible in the unlocked Programming session.
  • The Programming session can only be activated if the ECU is already unlocked.
  • Unlocking the ECU is not possible in the Default session.
  • Changing a session always locks the ECU.
  • After changing the variant, a reset must be performed so that the ECU can reconfigure itself.
  • During the reset, the ECU does not respond to requests.

Therefore, the algorithm for setting variant coding is as follows:

  1. Activate Expanded session.
  2. Unlock ECU (request seed and send key).
  3. Activate Programming session.
  4. Unlock ECU (request seed and send key).
  5. Setting the data for variant coding.
  6. Reset ECU.
  7. Wait some time for the ECU to reconfigure itself.
  8. Activate Default session.

This algorithm is implemented in the VarCodingPanel.VariantCodingControl panel.

VarCodingPanel.VariantCodingControl

This is an example of a VisualBasic 6.0 ActiveX control, the source code can be found in the directory of this configuration (Panel\VariantCodingPanel). Changing the ECU variant requires a sequence of requests, which is programmed in this element to demonstrate how such complex algorithms can be implemented.

ECU Fault Memory
