Ethernet Diagnostics (HSFZ / DoIP)

 

Click on the blue link to load the sample configuration in CANoe.

  • Ethernet Diagnostics (HSFZ / DoIP) – HSFZSystem.cfg
    Shows the use of Diagnostics over High-Speed-Fahrzeugzugang (HSFZ) in CANoe and can be switched to the DoIP protocol. A simple gateway between HSFZ or DoIP and CAN is implemented.

This sample configuration shows the use of diagnostics via High-Speed-Fahrzeugzugang (HSFZ) in CANoe and can be switched to the DoIP protocol. A simple gateway between HSFZ or DoIP and CAN is implemented, which routes the diagnostic communication between tester (to Ethernet) and ECU (to CAN). The tester communicates exclusively via Ethernet.

Note

The HSFZ/CAN gateway itself can only be diagnosed in this sample configuration using HSFZ (or DoIP).

Note

  • You can easily switch the configuration to DoIP. To switch over, via the Diagnostics tab open the Diagnostics/ISO TP Configuration. Under Additional Diagnostic Settings you will find the item DoIP/HSFZ Main Settings. In the Protocol lists, you can switch from HSFZ (High-Speed-Fahrzeugzugang) to one of the DoIP protocol versions.
  • The logical addresses for HSFZ and DoIP differ (HSFZ: 1 byte, DoIP: 2 bytes). If you have switched to DoIP, it may be useful to customize the logical addresses of the tester and ECUs to the value ranges defined in ISO13400 to avoid warnings in the Diagnostic/ISO TP Configuration dialog.

The sample configuration shows the following functionalities:

  • Diagnostic tester for HSFZ (or DoIP) in CANoe (CAPL test module, Diagnostic Console) against real or simulated ECUs.

  • Example of CAPL tester modules Flashdownload in 64 K or 4 K data blocks via diagnostic services.

  • Simulation of an ECU with diagnostic server on a standard network (in the example: CAN).

  • Simulated HSFZ (or DoIP) to CAN gateway for routing diagnostic communication from a HSFZ/DoIP tester to the ECU on the standard network.

  • Switching an entire standard network to diagnostics for HSFZ (or DoIP-) via the CANoe diagnostic configuration. The diagnostic description of the ECU behind the simulated gateway hangs on the CAN network, nevertheless the communication to this ECU is done via the protocol for HSFZ (resp. DoIP) and the gateway simulation node routes the diagnostic communication between tester (to Ethernet) and ECU (to CAN).

Diagnostics & XCP ribbon tab: Diagnostics/ISO-TP|CAN

If you deactivate the Enable diagnostics via common DoIP/HSFZ gateway for this network check box, you switch the diagnostic access from HSFZ (or DoIP) over Ethernet to CAN for the entire network. This also applies when implementing diagnostics in CAPL, e.g. when using DiagSetTarget(<DiagDescriptionQualifier>).

The HSFZ settings for the gateway and the ECUs downstream of it are specified on the Ethernet Networks|Eth1|Gateway to CAN page. Here are the most important settings:

Gateway to CAN

  • Vehicle Identification Request Parameter: Because the example contains only one gateway node (thus only one vehicle), the correct one is also selected by the tester if no specification is made here. In this case, the tester uses a broadcast to query the existing vehicles and is then connected to the single vehicle that signals back.
    If, however, there are multiple ECUs with the same logical address (e.g., the same ECU in multiple vehicles), you must qualify the target vehicle by making a setting in this field (see also DoIP/HSFZ Settings|Vehicle|Address).
  • Tester:
    Adapter: Here, you can select an IP address that was specified for this network in the TCP/IP Stack dialog.
    Log. Address This is the logical address of the tester.
  • ECU:
    The settings for the ECUs are made in the node-specific settings. That is, in the Gateway to CAN\ECU node in this sample configuration.

Gateway to CAN\ECU

  • Log. Address (phys): The logical HSFZ address of the ECU downstream of the HSFZ gateway.
  • Log. Address (funct): Not used in this example.

GW\DoIP/HSFZ Settings

You set the HSFZ settings of the physical HSFZ gateway here.

  • Vehicle Identification Request Parameter:
    Because the CAN ECU ECU is diagnosed via this exact gateway, the Vehicle Identification Request Parameter is specified the same as specified under Gateway to CAN.
  • Tester:
    Adapter and Log. Address must be the same as specified under Gateway to CAN because the same tester is involved.
  • ECU:
  • Log. Address (phys): The logical HSFZ address of the HSFZ gateway.
  • Log. Address (funct): Not used in this example.

Desktops

This configuration contains the following desktops:

Diagnostic

On the Diagnostic desktop, you can send requests using the Diagnostic Console in accordance with the assigned diagnostic description and monitor the symbolic interpretation of the diagnostic objects in the Trace Window. In addition, you can use the UDS Variant Coding to adjust the variant-specific parameters of the ECU via diagnostic services.

In as-delivered condition, diagnostic communication with the ECU is set to occur via HSFZ for the entire CAN network (see Note above).

If you deactivate the Enable diagnostics via common DoIP/HSFZ gateway for this network check box, the ECU Diagnostic Console or the UDSVariantCodingPanel Variant Coding Window, for example, is able to send its requests over the set ISO TP channel on CAN to the ECU without any other changes.

Download

On the Download desktop, the use of an HSFZ diagnostic channel in CAPL test modules is demonstrated for a Flash download procedure. The test modules download data to each network node in test case TCDownload(). The block sizes are adjusted according to the bus system:

  • DownloadGW test module: Flashing of the HSFZ gateway (GW network node) in 64 K data blocks. Since HSFZ also allows large data blocks like this, large amounts of data can also be downloaded to the ECU with comparatively small protocol overhead.
  • DownloadECU test module: Flashing of the ECU ECU (ECU network node) in 4 K data blocks. The data block size is considerably smaller here, because the ISO TP used in the ECU ECU can only process segmented messages up to a total of 4096 bytes on CAN 2.0.

Note

If you deselect the use of the HSFZ channel in the Diagnostics/ISO TP Configuration (Enable diagnostics via common DoIP/HSFZ gateway for this network check box), the DownloadECU test module can still be operated, because the test module automatically uses the CAN ISO TP channel set for this diagnostic description.

System

The ECU network node (Simulation Setup: ECU network node) simulates an ECU with diagnostic functionality on the CAN network.

The gateway (Simulation Setup: network node GW) receives the HSFZ requests from the tester nodes and implements them on CAN/ISO TP for the ECU. The DoIP.dll is used in the GW node for this.
