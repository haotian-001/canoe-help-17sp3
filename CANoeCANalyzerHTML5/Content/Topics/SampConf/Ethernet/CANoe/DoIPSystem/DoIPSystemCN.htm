System Configuration (DoIP)

 

Click on the blue link to load the sample configuration in CANoe.

  • System Configuration (DoIP) – DoIPSystem.cfg
    Shows the setup of a Diagnostics over IP (DoIP) test system consisting of DoIP Tester and DoIP ECU with dynamic IP address assignment by a simulated DHCP server. The usage of TLS encryption is also demonstrated.

Introduction

This sample configuration shows the following use cases and functions using the Diagnostics over IP (DoIP) protocol:

  • Interactive DoIP test using the Diagnostic Console, Fault Memory Window and various panels.
  • Automated DoIP test using a test unit created with vTESTstudio and a CAPL test module.
  • Simulation of an ECU using the CAPL Callback Interface (CCI) reference implementation for DoIP.
  • Simulation of an ECU using the built-in diagnostics channel for DoIP in combination with encrypted communication using TLS.
  • Tester authentication based on UDS Service 0x29.

The sample configuration consists of the following elements:

  • Simulated DoIP ECU implementing complex diagnostic functions (DoIP_ECU node).
  • Simulated DoIP ECU (DoIP_TLS_ECU node) using an encrypted TLS connection to the diagnostic tester.
  • Simulation node that implements test functions that can be called up interactively via panels (DoIP_Tester).
  • CAPL test module (DoIP_Testmodule), which performs a simple diagnostic test and a simulated software download. The CAPL test module can be found in the ribbon under Test | Test Modules.
  • Simple simulated DHCP server that dynamically assigns IP addresses to the nodes involved.
  • Test unit in the test configuration DoIP_TestConfiguration, which performs a more comprehensive test.

The ECU simulations for DoIP_ECU and DoIP_TLS_ECU comprise the state transitions between different sessions (Default Session, Extended Session and Programming Session) as defined in the diagnostic description (CDD file) as well as different levels of security access blocking. Some service requests require tester authentication, certain sessions or lock states. The following behavior is also implemented:

  • Access to the variant coding data is only possible in the extended session. Additionally, the tester needs to authenticate itself at the ECU.
  • The ECU cannot be unlocked in the default session.
  • The ECU is blocked in the following cases so that no further valid requests can be received:
  • session is changed
  • negative response is sent
  • timeout occurs
  • During a hardware reset, any existing TCP connection is terminated, the link to the tester is disconnected and re-established and when the simulated ECU is available again, vehicle announcements are sent.
  • A hardware reset of the ECU takes longer than a software reset. In case of a hardware reset, DoIP_ECU responds with a negative response code 0x78 (request correctly received - response pending), before the positive response is sent after successfully re-establishing the TCP connection and subsequent routing activation.

Note

The programming code for the SeednKey.dll used in this sample configuration is located in the SecurityAccess folder of the sample configuration UDSSystem (Open Folder). For more information about Security Access, see Seed & Key DLL / Security Access.

Panels and Windows

The following panels and windows appear on several desktops of this sample configuration.

DHCP_Server

You control and configure the simulated DHCP server via the DHCP_Server panel. The Enabled LED indicates whether the DHCP server is active. You deactivate the server by deactivating the simulation node DhcpServer in the Simulation Setup. In this case, please make sure that valid IP addresses are configured for the nodes configured in the Simulation Setup and for the Test Unit in the Simulation | TCP/IP Stack dialog. The AddressRangeStart and AddressRangeEnd input fields allow you to specify a range of IP addresses from which the DHCP server can select.

SubnetMask
In this entry field you set the subnet mask for the assigned IP addresses.

DefaultGateway
Here you set the default gateway address propagated by the DHCP server.

LeaseDuration [sec]
Here you set the validity of the assigned IP address in seconds. After half of the validity period the clients start to renew their IP address configuration.

The assigned IP addresses are displayed in the DoIP_ECU_Application and DoIP_Tester_Application panels.

DoIP_Tester_Application

In this panel you can initiate simple diagnostic request sequences and send defined diagnostic requests to the ECU.

Communication Tester – ECU

This area shows the status of the communication in detail. By clicking the Toggle button you switch the Activation Line on and off.

Window Pos.

In this area you can set the window position to open (0%), half open (50%) or closed (100%).

Read cyclically

By clicking the Internals button you can request cyclically sent data from the ECU from the simulated ECU (analogous to Unacknowledged Unsegmented Data Transfer (UUDT) for CAN). Via the buttons Position or Voltage the tester node cyclically queries the current window position as well as the device-internal voltage values of the ECU. The parameter values are then displayed in the Graphics Window.

Use the Read contacts + Serial button to query the DoIP_ECU serial number and the status of the door contacts. Please note that the value of the door contacts depends not only on the status of the (simulated) hardware door contacts, but also on the device-internal status of the ECU (e.g. if the door contact values have been overwritten by using a diagnostic service).

DoIP_ECU_Application

This panel displays the internal states of the ECU DoIP_ECU that controls the front left door. In addition, it is possible to close any open TCP connection to the tester by clicking a button.

Status

This area displays some device-internal data of the ECU, including device-internal states such as Session and Lock Level. Please note that the ECU returns to the locked default session if no requests are received via the P3 time or if a negative response occurs. If you deactivate the sending of Tester Present manually, you must also reactivate the function manually in order to maintain a non-default session for the ECU. The ECU can be locked. This means that certain services may not be running and the ECU may refuse security access.

Rewritable data

This area displays the serial number of DoIP_ECU and the corresponding variant coding data. If DoIP_ECU is configured in such a way that no window regulator is supported via variant coding (for example, by using the VariantCoding panel), the services that read or write the window position respond with negative responses.

Internal values

In this area the values of the simulated device-internal data of the ECU (measured voltage and window position) are displayed.

Communication

This area displays the IP address currently assigned to the ECU. By clicking the Close TCP connection button, you can close the TCP connection from the ECU. Note that the built-in diagnostic channel of the CANoe Diagnostic Tester automatically attempts to re-open the TCP channel after the time set in the Diagnostic/ISO-TP Configuration dialog.

DoIP_TLS_ECU_Application

This panel displays the internal states of the ECU DoIP_TLS_ECU. It has the same functionality as the DoIP_ECU_Application panel, except that there is no Close TCP connection button.

DoIP_ECU_FaultMemory

This panel displays the current contents of the ECU fault memory. You can also add up to six DTCs to the simulated ECU fault memory by setting the corresponding status byte for each DTC individually. When saving a DTC, the current mileage and input voltage are saved as environmental data and as a snapshot with the respective DTC.

ExternalSignals

In this panel, you can manually change external signals to stimulate the ECU.

You can change the current odometer value, the input voltage and the (simulated) status of the hardware door contact of all four doors.

VariantCoding

Similar to the DoIP_Tester_Application panel, you can initiate a diagnostic request sequence in this panel. In this sample configuration, this diagnostic request sequence also includes a tester authentication and a security access sequence.

Note

The tester authentication is done according to the security profile which is configured for network DiagAccess (Eth 1). This security profile uses UDS Service 0x29 for authentication.

By clicking the Read Variant Coding or Write Variant Coding button, you can write selected codes to the simulated ECU memory. The result of the coding process is displayed (OK/not OK). Please note that the variant coding fails if the selected Speed to lock door is above 50 km/h. Just like a real ECU, the simulated ECU randomly generates a seed. Thus, the key required to unlock the ECU changes for each coding process.

DoIP_Trace

Only Ethernet packets that are relevant for a DoIP trace analysis are displayed in this Trace Window. In addition, the DoIP PDUs determined from this and, if applicable, the resulting diagnostic interpretation of the DoIP PDUs are displayed.

Console [▲ back]

This desktop contains the Diagnostic Window Diagnostic Window DoIP_ECU with the Diagnostic Features Diagnostic Console, and ECU Control for the DoIP ECU , the DoIP_ECU_Application, DHCP_Server and DoIP_Tester_Application panels as well as the Trace Window DoIP_Trace.

Diagnostic Window DoIP_ECU

In the Diagnostic Console, all diagnostic services defined in the respective diagnostic description (CDD and PDX file) are displayed and can be selected, parameterized and transmitted. The responses of the simulated ECU are displayed in the Diagnostic Trace within the same Diagnostic Window.

In the ECU Control, the state of the ECU in question is displayed from the tester's point of view and can be changed by sending the request in question. You can interactively unlock the simulated ECU DoIP_ECU via the ECU Control and perform an authentication sequence. Please note that depending on the definition of the respective diagnostic description, some buttons may be deactivated or a different number of sessions, security access levels and ECU reset types may be displayed. Since a Seed & Key DLL is configured for the simulated ECU DoIP_ECU, this DLL is used to automatically execute the Seed / Generate Key / Send Key request sequence after security sequence changes are made in this window. The configured safety profile in the safety configuration (see UDS Diagnostic System Configuration) determines the properties displayed in the ECU Control.

Fault Memory [▲ back]

This desktop contains the DoIP_ECU - Fault Memory Window, the panels DoIP_ECU_Application, DoIP_ECU_FaultMemory, ExternalSignals and Help as well as the Trace Window DoIP_Trace.

DoIP_ECU - Fault Memory

If you have saved one or more DTCs via the DoIP_ECU_FaultMemory panel, the corresponding DTCs appear in the Fault Memory Window after confirming the Update Fault Memory List button. In addition, the snapshot data stored for each DTC is displayed if the Show DTC Details button is enabled.

Automated Test [▲ back]

This desktop contains the panel DoIP_ECU_Application, the test configuration dialog DoIP_TestConfiguration as well as the Trace Window DoIP_Trace.

DoIP_TestConfiguration

An automated diagnostic test is performed when the Test Unit is started.

Manual Test [▲ back]

This desktop contains the Graphics Window, the Panels DoIP_ECU_Application, ExternalSignals, VariantCoding and DoIP_Tester_Application as well as the Trace Window DoIP_Trace.

Graphics Window

Several diagnostic parameters are displayed in this window. Please note that these parameters are only updated after receiving a response with the corresponding parameter. To query the parameters in the DoIP_Tester_Application panel, click on the Internals, Position and Voltage buttons in the Read cyclically group.

DoIP via TLS [▲ back]

This desktop contains the Diagnostic Window Diagnostic Window DoIP TLS ECU with the Diagnostic Features Diagnostic Console and ECU Control for the ECU DoIP_TLS_ECU, the DoIP_TLS_ECU_Application panel, the DoIP_Testmodule test module as well as the Trace Window DoIP_Trace.

Diagnostic Window DoIP_TLS_ECU

When sending diagnostic requests from the Diagnostic Console and the ECU Control in this window, the simulated ECU DoIP_TLS_ECU is addressed and the whole diagnostic communication is encrypted using TLS. Nevertheless, CANoe can decrypt this encrypted communication. Therefore, in the Trace Window, decrypted DoIP PDUs are visible, as well as diagnostic interpretation for DoIP Diagnostic Messages (Payload type 0x8001).

Setup [▲ back]

This desktop contains the Simulation Setup, the Measurement Setup, the Basic Diagnostic Editor and the Help panel.

Basic Diagnostics

If no diagnostic description is available for an ECU, you can define simply structured diagnostic services with the Basic Diagnostic Editor and send the services interactively to the ECU via the Diagnostic Console. In the sample configuration the basic diagnostic description defined with this editor has been added as Additional Description to the DoIP_ECU diagnostic description and completed by the diagnostic services WindowLiftFinePositionRead and WindowLiftFinePositionWrite and by the service UnimplementedServiceData_Read for which DoIP_ECU does not send a response.

You can send diagnostic requests defined in this basic diagnostic description via the Diagnostic Console DoIP_ECU.Additional.

After the measurement is interrupted, you can also define new services in the Basic Diagnostic Editor, apply the changed services to the configuration, and run the services after the measurement is started.
