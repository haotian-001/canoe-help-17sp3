Proprietary IP Protocol

 

Click on the blue link to load the sample configuration in CANoe.

  • Proprietary IP Protocol – EthernetBasicProprietaryProtocol.cfg
    CAPL is used to evaluate a proprietary IP protocol and store the data in system variables. The system variables can be processed further in CANoe.

CANoe supports the interpretation of many Ethernet and IP-based protocols. Standardized protocols such as SOME/IP, AUTOSAR PDU, DoIP, and XCP, are evaluated up to the signal level and can be displayed directly in analysis windows such as the Trace Window and Graphics Window. This example shows how CANoe can be expanded in order to evaluate vendor-specific protocols up to the signal level.

In this example, a simple proprietary protocol is used that is transferred with UDP packets. One UDP packet can contain multiple proprietary messages one after the other. The proprietary message consists of a header with the following structure:

  • 32 bit message length
  • 32 bit message ID
  • 32 bit checksum

This header is defined as a CAPL struct in ProprietaryProtocol.cin.

The payload data of the proprietary message is defined as a system variable struct and can be viewed in the system variable dialog. The layout of the system variable struct corresponds exactly to the layout of the signals that are contained in the payload data in the UDP packet.

The Measurement Setup contains a CAPL program ProprietaryObserver. This evaluates the Ethernet packets. If the UDP matches the proprietary protocol, the data is copied to a system variable struct. The layout of the struct corresponds to the layout of the signals of the proprietary protocol. The fields of the system variable structs can be configured in the analysis windows and analyzed.

In the Trace Window, the system variables are represented as their own events. The time stamps of the system variables correspond to the time stamp of the UDP packet with which they were transferred. In the continuous mode of the Trace Window, system variables of proprietary messages can be seen directly after the Ethernet packet in which they were transferred.

Introduction

At the start of a measurement, the simulated node ProprietarySensor starts to send its data to the broadcast address with UDP packets.

Desktops

This configuration contains the following desktops:

Setup

Analysis

Trace

 

Setup [▲ back]

On this desktop, the Simulation Setup is configured with the simulated node. The simulated CAPL node opens a UDP socket and writes the data of its proprietary protocol to the socket accordingly.

In the Signal Generator, the system variables of the ProprietarySensor node are added and a sine curve is configured. The system variables are read out in the ProprietarySensor node and written to the UDP packet that is to be sent.

Analysis [▲ back]

Analysis windows that represent the system variables of the ProprietaryObserver are configured here.

Trace [▲ back]

A single Trace Window is configured here to allow more detailed analysis of the proprietary protocol. In fixed mode, you can see the different cycle times of the two transferred messages in the Time column. If you switch to continuous output mode, you can see how the two messages are transferred in a single Ethernet packet.
