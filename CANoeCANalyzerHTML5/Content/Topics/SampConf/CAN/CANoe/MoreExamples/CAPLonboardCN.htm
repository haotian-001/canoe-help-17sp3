CAPL-on-Board

 

This page describes several sample configurations.

Click on the blue link to load the sample configuration in CANoe.

    Shows the improved realtime capabilities of the Vector CAPL-on-Board feature for all nodes.
  • CAPL-on-Board - Offline (CAN) â CAPL-on-Board_off.cfg
    Shows the improved realtime capabilities of the Vector CAPL-on-Board feature on the computer.

This pair of sample configurations shows the improved realtime capabilities of the Vector CAPL-on-Board feature in comparison to the normal execution mode:

  • more accurate timers / reduced jitter
  • shorter response times / lower latency

Requirements and Preparation

The configurations require a network interface which supports the Vector CAPL-on-Board feature with two CAN channels (e.g. VN1630, VN1640, â¦). Channel 1 has to be connected with channel 2 using a CANcable1.

Description

  • A CAPL node cyclically transmits a CAN message with ID 0x100 each 1 millisecond.
  • Another CAPL node receives the cyclic message and immediately transmits a response message with ID 0x102.
  • The cycle time is calculated as difference of the reception time stamps of two consecutive messages with ID 0x100.
  • The response time is calculated as difference between the transmission time stamp of a message with ID 0x100 and the start of frame time stamp of the corresponding message with ID 0x102.


The sample configuration also shows the improved accuracy of the timers available to CAPL nodes when running on the interface hardware.

Difference between the two Configurations

The two sample configurations are almost identical, except for the following difference:

CAPL-on-Board_on.CFG uses CAPL-on-Board for all nodes, whereas
CAPL-on-Board_off.CFG executes the CAPL code on the computer.

Graphics Windows

The Cycle Times Graphics Window shows the calculated current cycle time and the minimum and maximum over the whole measurement.

The Response Times Graphics Window shows the calculated current response time and the minimum and maximum over the whole measurement.

The yellow lines show the current cycle and reponse time for each CAN message, whereas the red lines show the minimum and maximum values for the cycle and reponse times, respectively.

In CAPL-on-Board_off.CFG, the influence of the Windows operating system is clearly visible (even though the CAPL code gets executed with elevated priority).

In comparison, in CAPL-on-Board_on.CFG, the reponse times are lots shorter, and there is almost no jitter visible at all.
