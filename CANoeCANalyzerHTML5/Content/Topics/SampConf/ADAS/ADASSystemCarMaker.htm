ADAS System CarMaker

 

Note

Click on the blue link to load the sample configuration in CANoe.

ADAS System CarMaker — ADASSystemCarMaker.cfg

Shows the use of all ADAS features based on the test of an ADAS algorithm. A CarMaker test run is used for stimulation.

Preconditions

At the start of the measurement, the configuration attempts to start a test run in IPG CarMaker. A CarMaker must be installed for this. The CANoe CarMaker Interface must also be installed. Further information on installation can be found here.

Before the CANoe measurement is started, the CarMaker must have started a test run or test series once. Otherwise the CANoe CarMaker Interface cannot establish a connection.

Introduction

At the start of the measurement, the system switches to the desktop Measurement. A CarMaker test run is loaded. This plays an emergency braking of a vehicle in front in a construction site. You can follow the scenario in the CarMaker windows, as well as the CANoe Scene Windows. The scenario is replayed several times with different parameters (speed of the first-person vehicle) and stimulates an ADAS algorithm integrated as an application model. A test unit tests the ADAS algorithm and generates a test report.

Desktops

This configuration contains several desktops.

Setup

The Measurement Setup, the Simulation Setup and the Communication Setup are displayed on this desktop.

A Trace Window and a Scene Window are integrated in the Measurement Setup, as well as an inactive Logging Block.

In the Communication Setup, the ADASNetwork vCDL is integrated as the database. In addition to the static object Radar defined here, which represents the object sensor of the CarMaker test run, an OutputSimpleAlgorithm object was defined. This contains the output of the ADAS algorithm to be tested.

In addition, several Application Models are included in the communication structure:

  • AlgorithmUnderTest: The ADAS algorithm to be tested as C# code. The example shows a very simplified emergency braking assistant, which warns the driver and brakes independently if the Time To Impact is less than 3s.

  • DashboardControl: Controls the display of the Dashboard panel with CAPL code.

  • Mapping: Creates the Detected Objects and the Sensor Objects using the ADAS Sensor API.

There is also a node in the Simulation Setup:

CarMakerControl: responds to the output of the ADAS algorithm in a short CAPL program. If the ADAS algorithm writes a warning on the output object, a brake signal is sent to the CarMaker test run. In addition, the connection to the CarMaker is established at the start of measurement.

Measurement

This desktop displays the Panel Dashboard, a Trace Window, the Scene Window, and the Test Configuration.

Panel Dashboard

Displays the current speed and RPM of the Ego vehicle at all times. If a warning is active from the ADAS algorithm for the Emergency Brake Assist, a warning is displayed centrally in the Information screen.

Trace Window

Displays the data received from the CarMaker test run. The radar.sensor_info object contains information such as the Sensor View (Sensor ID, Sensor Mounting Position) or the number of currently detected objects. The currently detected objects in turn are displayed as independent objects in the Trace Window (Detected_id_xxx_by_radar). These contain all known information about the objects such as dimension, position, speed or classification.

Additionally, the object OutputSimpleAlgorithm can be seen. In this the output of the AlgorithmUnderTest Application Model is stored.

Scene Window

Visualizes the data received from the CarMaker test run. Based on the information in the detected objects, the position and dimension of the objects are displayed.

Test Configuration

An included test unit checks the ADAS algorithm included in the Application Model AlgorithmUnderTest. Different tests check e.g. if a radar object was detected at all, if its measurement state is stable and valid, if the probability rate exceeds a certain threshold, if the radar object is in the same lane. If all this is the case, the outputs of the ADAS algorithm written to the OutputSimpleAlgorithm object are used to check whether (correctly) no warning is issued if no collision is to be feared or whether a warning is issued if a collision is to be feared. Afterwards a test report is generated. The tests are based on the ADAS Test Feature Set which is used in a C# test script and a CAPL test script and included in the test unit.
