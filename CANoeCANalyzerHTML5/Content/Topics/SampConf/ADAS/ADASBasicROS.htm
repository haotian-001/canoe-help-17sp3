ADAS Basic ROS

 

Note

Click on the blue link to load the sample configuration in CANoe.

ADAS Basic ROS - ADASBasicROS.cfg

Demonstrates the use of ADAS features to stimulate a ROS node with ADAS data from a created scenario. The results of the ADAS algorithm running in the ROS node are then evaluated using a test unit.

It is assumed that you already have experience using ROS 2 and know how to compile and start ROS packages. A ROS workspace containing the ROS node and the ROS messages used in the example is included with the sample configuration.

Introduction

At the start of the measurement, the system switches to the desktop Measurement. A scenario created with the Scenario Editor is loaded and started with the measurement. This scenario replays an emergency braking of a vehicle in front before an intersection. You can observe the scenario in the scene window and map window. The ADAS algorithm under test runs as a process in the ROS node and is stimulated with data from the scenario via an Application Model. A test unit tests the ADAS algorithm and generates a test report.

Schematic representation of the main components:

Desktops

This configuration contains several desktops.

Configuration

On this desktop, the Measurement Setup, the Communication Setup and the scenario manager are displayed.

Two Trace Windows, a Map Window and a Scene Window are integrated in the Measurement Setup, as well as an inactive Logging Block.

In the Communication Setup, static ADAS objects were created using the model editor. In addition to the objects that appear in the scenario (EgoVehicle, BrakingVehicle, EmergencyVehicle and FrontRadar), two other objects InputROS and OutputROS were also defined. InputROS is filled with the scenario data during the measurement and sent to the ROS node as a ROS message via the DDS binding. Similarly, the OutputROS object contains the results of the ADAS algorithm running in the ROS node and is evaluated within the test unit. In the example, the ADAS algorithm is a very simplified emergency braking assistant that warns the driver and brakes independently if the Time To Impact is less than 3s.

In the Scenario Manager, the EmergencyBrakeScenario previously created via the Scenario Editor was loaded. This scenario starts simultaneously with the measurement.

In addition, several application models are integrated in the communication structure:

  • ADASDataProvider.cs
    Fills the InputROS object with the scenario data required by the ADAS algorithm.

  • MapObjectsVisualizer.can
    Uses the Map API to display the objects of the scenario in the Map Window.

Measurement

On this desktop, two Trace Windows, the Scene Window, the Map Window, and the Test Configuration are displayed.

Trace Window ADAS

Displays the data received from the scenario. The FrontRadar.sensor_info object contains information such as the Sensor View (Sensor ID, Sensor Mounting Position) or the number of currently detected objects. The currently detected objects in turn are displayed as independent objects in the Trace Window (Detected_id_xxx_by_FrontRadar). These contain all known information about the objects such as dimension, position, speed or classification.

Trace Window ROS

With the help of an appropriate filter, displays separately only the data concerning the message exchange via DDS with ROS. Of particular interest here are the two objects InputROS and OutputROS, which encapsulate the input and output data to and from the ROS node respectively.

Scene Window

Visualizes the data received from the scenario. Based on the information in the detected objects, the position and dimension of the objects are displayed.

Map Window

Visualizes the scenario on a map. The Map API was used to display the objects.

Test Configuration

An included test unit tests the ADAS algorithm running in the process of the ROS node. For this purpose, the ROS node has to be started before the measurement starts using the corresponding ROS commands. Various tests check, for example, whether a radar object has been detected at all, whether the probability rate exceeds a certain threshold, whether the radar object is in the same lane. If all this is the case, the outputs of the ADAS algorithm written to the object OutputROS are used to check whether (correctly) no warning is issued if no collision is to be feared or whether a warning is issued if a collision is to be feared. Afterwards a test report is generated. The tests are based on the ADAS Test Feature Set that is used in a C# test script and included in the test unit.
